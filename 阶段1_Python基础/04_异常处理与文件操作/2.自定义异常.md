# 自定义异常

## 一、为什么需要自定义异常？

Python 内置异常（`ValueError`、`KeyError` 等）是通用的，但后端项目中的错误往往有**业务含义**：

```python
# ❌ 用内置异常，信息模糊
raise ValueError("找不到")    # 找不到什么？用户？商品？订单？

# ✅ 用自定义异常，语义清晰
raise UserNotFoundError(user_id=42)   # 一眼看出：用户 42 不存在
raise InsufficientBalanceError(balance=50, required=100)   # 余额不足
```

自定义异常的好处：
1. **语义明确**：异常名就能说明问题
2. **分层捕获**：可以按业务模块分别处理
3. **携带上下文**：附带错误码、相关 ID 等信息
4. **统一响应**：配合全局异常处理器，自动转成 HTTP 状态码

---

## 二、基本写法

### 最简单的自定义异常

```python
class AppError(Exception):
    """应用基础异常"""
    pass

# 使用
raise AppError("发生了一些错误")
```

### 带错误码和消息

```python
class AppError(Exception):
    """应用基础异常：所有业务异常的父类"""
    def __init__(self, message: str = "服务器内部错误", code: int = 500):
        self.message = message
        self.code = code
        super().__init__(self.message)

    def __str__(self):
        return f"[{self.code}] {self.message}"
```

## 三、构建异常体系

后端项目通常按 HTTP 状态码或业务模块组织异常层级：

### 按 HTTP 状态码分层

```python
class AppError(Exception):
    """基础异常"""
    def __init__(self, message: str = "服务器内部错误", code: int = 500):
        self.message = message
        self.code = code
        super().__init__(self.message)

# 4xx 客户端错误
class BadRequestError(AppError):
    """400 请求参数错误"""
    def __init__(self, message: str = "请求参数错误"):
        super().__init__(message, code=400)

class AuthenticationError(AppError):
    """401 未认证"""
    def __init__(self, message: str = "请先登录"):
        super().__init__(message, code=401)

class ForbiddenError(AppError):
    """403 无权限"""
    def __init__(self, message: str = "权限不足"):
        super().__init__(message, code=403)

class NotFoundError(AppError):
    """404 资源不存在"""
    def __init__(self, resource: str, resource_id = None):
        msg = f"{resource}不存在" if resource_id is None else f"{resource}(id={resource_id})不存在"
        super().__init__(msg, code=404)

class ConflictError(AppError):
    """409 资源冲突"""
    def __init__(self, message: str = "资源已存在"):
        super().__init__(message, code=409)

# 5xx 服务端错误
class DatabaseError(AppError):
    """数据库操作失败"""
    def __init__(self, message: str = "数据库操作失败"):
        super().__init__(message, code=500)

class ExternalServiceError(AppError):
    """外部服务调用失败"""
    def __init__(self, service: str, message: str = ""):
        super().__init__(f"调用{service}失败: {message}", code=502)
```

### 使用异常体系

```python
# 服务层
def get_user(user_id: int) -> dict:
    user = db.find(user_id)
    if not user:
        raise NotFoundError("用户", user_id)
    return user

def create_user(email: str) -> dict:
    if db.find_by_email(email):
        raise ConflictError(f"邮箱 {email} 已注册")
    return db.create({"email": email})

def delete_user(current_user: dict, user_id: int):
    if current_user["role"] != "admin":
        raise ForbiddenError("只有管理员可以删除用户")
    user = get_user(user_id)   # 可能抛 NotFoundError
    db.delete(user_id)

# 路由层：统一捕获
try:
    delete_user(current_user, 42)
except NotFoundError as e:
    return {"error": e.message}, 404
except ForbiddenError as e:
    return {"error": e.message}, 403
except AppError as e:
    return {"error": e.message}, e.code
```

## 四、FastAPI 中的异常处理（预告）

```python
from fastapi import FastAPI, HTTPException

app = FastAPI()

# 方式一：直接用 HTTPException
@app.get("/users/{user_id}")
def get_user(user_id: int):
    user = db.find(user_id)
    if not user:
        raise HTTPException(status_code=404, detail="用户不存在")
    return user

# 方式二：自定义异常 + 全局异常处理器（更推荐）
@app.exception_handler(AppError)
async def app_error_handler(request, exc: AppError):
    return JSONResponse(
        status_code=exc.code,
        content={"error": exc.message}
    )

@app.get("/users/{user_id}")
def get_user(user_id: int):
    user = db.find(user_id)
    if not user:
        raise NotFoundError("用户", user_id)   # 自动被全局处理器捕获
    return user
```

## 五、设计原则

| 原则 | 说明 |
|------|------|
| 继承 Exception，不要继承 BaseException | BaseException 包含 KeyboardInterrupt 等系统异常 |
| 一个项目一个基类（如 AppError） | 方便统一捕获所有业务异常 |
| 异常名以 Error 结尾 | 符合 Python 惯例 |
| 携带有用的上下文信息 | 错误码、资源名、ID 等 |
| 不要滥用自定义异常 | 只在需要语义区分的场景使用 |

## 六、常见坑

### 坑1：忘记调用 super().__init__()

```python
# ❌ 忘了 super()，str(e) 和 repr(e) 显示为空
class MyError(Exception):
    def __init__(self, message):
        self.message = message
        # 忘了 super().__init__(message)

e = MyError("出错了")
print(str(e))    # 空字符串！

# ✅ 调用 super()
class MyError(Exception):
    def __init__(self, message):
        self.message = message
        super().__init__(message)
```

### 坑2：异常层级设计过深

```python
# ❌ 过度设计
AppError → BusinessError → UserError → UserNotFoundError → UserNotFoundByIdError

# ✅ 扁平化，2-3 层足够
AppError → NotFoundError / AuthError / ForbiddenError / ...
```

### 坑3：在异常中存储敏感信息

```python
# ❌ 异常消息可能暴露给前端
raise AppError(f"数据库连接失败: host=192.168.1.100, password=secret123")

# ✅ 对外隐藏细节，对内记日志
logger.error(f"数据库连接失败: {connection_info}")
raise DatabaseError("数据库暂时不可用，请稍后重试")
```
