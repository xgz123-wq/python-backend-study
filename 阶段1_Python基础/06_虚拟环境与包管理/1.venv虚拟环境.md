# venv 虚拟环境

## 一、概念解释

虚拟环境是一个**独立的 Python 运行沙盒**，每个沙盒有自己的 Python 解释器和独立的第三方包目录，项目之间互不干扰。

Python 内置的 `venv` 模块可以一行命令创建虚拟环境，不需要额外安装任何工具。

---

## 二、为什么需要虚拟环境？

没有虚拟环境时，所有项目共用同一个全局 Python 环境：

```
项目A 依赖 requests==2.28
项目B 依赖 requests==2.31
```

两个版本只能安装一个 → 其中一个项目必然跑不起来。

**虚拟环境解决的核心问题：**
- 不同项目可以依赖同一个包的不同版本
- 避免"在我电脑上能跑"的经典问题
- 生产环境只安装必要的包，不带任何"历史垃圾"

**口诀**：不使用虚拟环境 = 项目必炸。

---

## 三、实际场景

- 本地开发时隔离各项目依赖
- CI/CD 流水线每次在干净环境里跑测试
- Docker 构建镜像时只打包必要依赖
- 团队协作时 `requirements.txt` 保证环境一致

---

## 四、核心用法

### 4.1 创建虚拟环境

```bash
# 在项目根目录创建 .venv 文件夹
python -m venv .venv
```

`.venv` 是约定俗成的命名（也有人用 `venv`、`env`，但 `.venv` 更常见且默认被 .gitignore 忽略）。

### 4.2 激活虚拟环境

```bash
# Windows（PowerShell）
.venv\Scripts\Activate.ps1

# Windows（CMD）
.venv\Scripts\activate.bat

# macOS / Linux
source .venv/bin/activate
```

激活后，命令行前缀会变成 `(.venv)` 提示你当前在虚拟环境里。

### 4.3 确认激活成功

```bash
# 查看当前 Python 路径（应指向 .venv 内部）
where python          # Windows
which python          # macOS/Linux

# 查看当前 pip 路径
where pip
```

### 4.4 退出虚拟环境

```bash
deactivate
```

### 4.5 虚拟环境内部结构

```
.venv/
├── Scripts/           # Windows：Python 可执行文件、pip、激活脚本
│   ├── python.exe
│   ├── pip.exe
│   └── Activate.ps1
├── Lib/
│   └── site-packages/ # 第三方包安装在这里
└── pyvenv.cfg         # 记录基础 Python 版本路径
```

## 五、常见坑

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| `Activate.ps1` 无法运行 | Windows 执行策略限制 | 运行 `Set-ExecutionPolicy -Scope CurrentUser RemoteSigned` |
| 激活后 `python` 还是系统版本 | 没有成功激活 | 检查命令行前缀是否有 `(.venv)` |
| 虚拟环境上传到了 Git | 没有加 `.gitignore` | 在 `.gitignore` 中添加 `.venv/` |
| 换电脑后环境丢失 | 虚拟环境不可移植 | 用 `requirements.txt` 重建，`pip install -r requirements.txt` |
| 不知道激活了哪个环境 | 多环境混乱 | `python -c "import sys; print(sys.prefix)"` |
