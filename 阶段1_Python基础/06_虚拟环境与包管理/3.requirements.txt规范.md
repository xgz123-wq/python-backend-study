# requirements.txt 规范

## 一、概念解释

`requirements.txt` 是 Python 项目的**依赖清单文件**，记录项目需要哪些包及其版本，是 Python 项目协作和部署的标准约定。

它本质上是一个纯文本文件，每行一个依赖，`pip install -r requirements.txt` 会逐行安装。

---

## 二、为什么需要规范的 requirements.txt？

混乱的 requirements.txt 会导致：
- 新人上手花几小时排查依赖问题
- 生产环境与开发环境版本不一致，线上出 bug
- 每次部署得担心包升级破坏功能

规范的写法能让任何人在任何机器上 `pip install -r requirements.txt` 后直接运行项目。

---

## 三、文件格式详解

### 3.1 基础格式

```
# 注释以 # 开头
包名==版本号          # 精确版本（推荐）
包名>=最低版本        # 最低版本约束
包名>=1.0,<2.0       # 版本范围
包名                  # 不限版本（不推荐用于项目）
```

### 3.2 完整示例（FastAPI 项目）

```
# Web 框架
fastapi==0.110.0
uvicorn[standard]==0.29.0

# 数据验证
pydantic==2.6.4
pydantic-settings==2.2.1

# 数据库
sqlalchemy==2.0.28
asyncpg==0.29.0
alembic==1.13.1

# 认证
python-jose[cryptography]==3.3.0
passlib[bcrypt]==1.7.4

# 工具
python-dotenv==1.0.1
httpx==0.27.0
```

### 3.3 分环境管理（进阶）

实际项目常分为多个文件：

```
requirements/
├── base.txt        # 所有环境共用
├── dev.txt         # 开发环境额外依赖（测试、调试工具）
└── prod.txt        # 生产环境额外依赖
```

`dev.txt` 内容示例：
```
-r base.txt         # 继承 base.txt

# 测试
pytest==8.1.1
pytest-asyncio==0.23.6
httpx==0.27.0

# 代码质量
ruff==0.3.7
mypy==1.9.0
```

## 四、最佳实践

### 4.1 只记录直接依赖

手写 requirements.txt 时，只写项目**直接 import** 的包，不写传递依赖：

```
# ✅ 正确：只写你直接使用的
fastapi==0.110.0
sqlalchemy==2.0.28

# ❌ 不必要：这些是 fastapi 的传递依赖
starlette==0.36.3
anyio==4.3.0
```

### 4.2 锁定精确版本用于部署

```
# ✅ 生产/团队协作：精确版本
fastapi==0.110.0

# ⚠️ 不稳定：可能每次装到不同版本
fastapi
```

### 4.3 配合 .gitignore

```gitignore
.venv/
__pycache__/
*.pyc
.env
```

`requirements.txt` 本身**必须提交到 Git**，`.venv/` 目录**绝不提交**。

## 五、常见坑

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| 装完后版本和 requirements.txt 不一致 | 没用精确版本号 | 改用 `==` 精确锁定 |
| 某台机器安装失败 | 依赖某个系统库（如 libpq） | 在注释里说明系统依赖 |
| 文件越来越乱 | 没有定期清理 | 定期对比实际使用，删除未用的包 |
| `pip freeze` 输出太多 | 混入了开发工具 | 分 base/dev/prod 管理 |
