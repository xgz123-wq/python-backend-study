TCP/IP 协议
后端开发的一切都建立在网络通信之上，TCP/IP 是这座大厦的地基。
目标 学习目标 ▼
  • 理解 TCP/IP 四层模型及各层职责
  • 能用自己的话解释三次握手与四次挥手的过程和原因
  • 掌握 TCP 与 UDP 的核心差异，根据场景正确选择
  • 了解 TCP 可靠性保证的五大机制（序列号、ACK、重传、滑动窗口、拥塞控制）
  • 能在面试中流畅回答 TIME_WAIT 的含义及其存在意义
💡 后端视角：你写的 FastAPI / Flask 代码工作在应用层，框架底层已处理传输层和网络层，但理解底层才能排查超时、连接等疑难问题。
概念 核心概念 ▼

TCP/IP 四层模型

┌─────────────────────────────────────────────┐ │ 应用层 │ HTTP · DNS · FTP · SMTP │ ← 你写代码的地方 ├─────────────────────────────────────────────┤ │ 传输层 │ TCP · UDP │ ← 决定数据怎么传 ├─────────────────────────────────────────────┤ │ 网络层 │ IP │ ← 决定数据发到哪 ├─────────────────────────────────────────────┤ │ 网络接口层 │ 以太网 · Wi-Fi │ ← 物理传输 └─────────────────────────────────────────────┘
展开查看：各层详细说明

应用层：你最常接触的一层。HTTP 请求、DNS 查询都发生在这层。你写的每一行 FastAPI 代码都在这里工作。

传输层：TCP 和 UDP 的家。负责把应用层的数据可靠（或快速）地送到目的地。端口号在这层发挥作用。

网络层：IP 协议的地盘。负责把数据包从源主机路由到目标主机。IP 地址在这层定义。

网络接口层：负责物理传输。以太网帧、Wi-Fi 信号等。后端开发者几乎不需要关心这层。

TCP 三次握手（建立连接）

客户端 服务器 │ │ │──── SYN (seq=x) ─────────────────>│ ① 客户端：我要连接 │ │ │<─── SYN+ACK (seq=y, ack=x+1) ────│ ② 服务器：好的，我也准备好了 │ │ │──── ACK (ack=y+1) ───────────────>│ ③ 客户端：确认，开始传数据 │ │ │═══════════ 连接建立 ═══════════════│
❓ 为什么是三次？两次无法确认客户端的接收能力。若有历史"过期连接请求"（网络延迟导致）到达服务器，两次握手就会误建连接，浪费服务器资源。三次握手确保双方都确认了对方的收发能力。

TCP 四次挥手（断开连接）

客户端 服务器 │ │ │──── FIN ─────────────────────────>│ ① 客户端：我发完了 │<─── ACK ──────────────────────────│ ② 服务器：知道了（可能还有数据要发） │ │ （服务器继续发送剩余数据） │<─── FIN ──────────────────────────│ ③ 服务器：我也发完了 │──── ACK ─────────────────────────>│ ④ 客户端：好，断开 │ │ │════════════ 连接关闭 ══════════════│
❓ 为什么是四次？ACK 和 FIN 不能合并发送。服务器收到 FIN 后可能还有数据未发完，必须先 ACK 确认收到，等数据发完后再发自己的 FIN。
展开查看：TIME_WAIT 详解

主动关闭连接的一方发出最后 ACK 后，不立即关闭，而是等待 2MSL（约 60 秒）。目的：

  • 确保对方收到了最后的 ACK。如果没收到，对方会重发 FIN
  • 让网络中残留的历史数据包有足够时间消失，避免干扰后续新连接
🔴 踩坑：高并发短连接场景下，大量 TIME_WAIT 会占满可用端口（最多 65535 个），导致新连接建立失败。解决方案：开启 SO_REUSEADDR、使用连接池复用连接。

TCP vs UDP 对比

特性TCPUDP
连接方式面向连接（三次握手）无连接（直接发）
可靠性✅ 可靠（确认重传）❌ 不可靠（发完不管）
有序性✅ 保证顺序❌ 不保证
速度较慢（有确认开销）更快（无额外开销）
头部大小20 字节8 字节
适用场景HTTP、数据库、文件传输DNS、视频直播、游戏

TCP 可靠性保证机制

机制作用
序列号（Sequence Number）给每个字节编号，保证接收方能按序重组
确认应答（ACK）接收方告知发送方"已收到"
超时重传规定时间内未收到 ACK，自动重发数据
滑动窗口控制发送速率，不让接收方缓冲区溢出
拥塞控制感知网络拥塞程度，动态调整发送速度
示例 代码示例（来自 demo/1_tcp_ip.py） ▼

1. TCP 服务端与客户端 — 体验三次握手

import socket, threading, time

# ── TCP 服务端 ──
def tcp_server():
    server = socket.socket(socket.AF_INET, socket.SOCK_STREAM)  # SOCK_STREAM = TCP
    server.setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)
    server.bind(("127.0.0.1", 9001))
    server.listen(1)
    conn, addr = server.accept()         # 阻塞，等三次握手完成
    data = conn.recv(1024)
    conn.send("TCP 服务端收到你的消息！".encode())
    conn.close(); server.close()

# ── TCP 客户端 ──
def tcp_client():
    time.sleep(0.3)
    client = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
    client.connect(("127.0.0.1", 9001))  # connect() 内部完成三次握手
    client.send("Hello TCP!".encode())
    response = client.recv(1024)
    client.close()                          # close() 内部完成四次挥手

关键行解读

代码行底层发生了什么
socket.SOCK_STREAM指定使用 TCP 协议（流式传输）
server.listen(1)开始监听，等待客户端连接
server.accept()阻塞直到三次握手完成，返回已建立的连接
client.connect()发起三次握手，建立 TCP 连接
client.close()发起四次挥手，断开 TCP 连接

2. UDP 通信 — 体验"无连接"

def udp_server():
    server = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)  # SOCK_DGRAM = UDP
    server.bind(("127.0.0.1", 9002))
    data, addr = server.recvfrom(1024)         # 等待数据，不需要建立连接
    server.sendto("UDP 服务端收到！".encode(), addr)

def udp_client():
    client = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
    client.sendto("Hello UDP!".encode(), ("127.0.0.1", 9002))  # 无 connect，直接发送
    response, _ = client.recvfrom(1024)
🔍 对比要点：UDP 客户端没有 connect()，直接 sendto()。服务端也没有 accept()，用 recvfrom() 同时接收数据和来源地址。

3. 浏览器输入 URL 后发生了什么？

1. DNS 解析：域名 → IP 地址 (应用层/UDP) 2. TCP 三次握手：建立与服务器的连接 (传输层/TCP) 3. TLS 握手：HTTPS 则协商加密 (传输层) 4. 发送 HTTP 请求 (应用层) 5. 服务器处理，返回 HTTP 响应 (应用层) 6. 浏览器渲染页面 (客户端) 7. TCP 四次挥手：断开连接 (传输层/TCP)
误区 常见误区 ▼
❌ 误区✅ 纠正
"TCP 比 UDP 好" 没有好坏之分，只有适合与否。视频直播用 TCP 反而体验更差（重传导致卡顿）。
"三次握手可以简化成两次" 不行。两次握手无法防止历史重复连接，会导致服务器为失效请求浪费资源。
"TCP 连接是一根物理线" TCP 连接是逻辑概念，靠双方内存中的状态信息维护，不是物理连接。
"UDP 不安全" UDP 不可靠 ≠ 不安全。安全性由上层协议（如 DTLS）或应用层加密保障。
"TIME_WAIT 越短越好" TIME_WAIT 是必要的安全机制。过短会导致新连接收到旧连接的残余数据包。

DNS 解析
你输入 google.com，但计算机只认 IP 地址——DNS 是互联网的"电话簿"。
目标 学习目标 ▼
  • 理解 DNS 解析完整流程（浏览器缓存 → OS 缓存 → 本地 DNS → 根 → 顶级 → 权威）
  • 掌握 A、CNAME、MX、TXT、NS 五种核心记录类型及其用途
  • 理解 TTL 对 DNS 传播的影响，知道上线前降低 TTL
  • 能使用 nslookup 和 Python socket 排查 DNS 故障
  • 了解 hosts 文件的优先级及后端开发中的应用场景
💡 DNS 修改后不会立即生效——记录 TTL 决定各级缓存过期时间。短则分钟，长则 48 小时。上线前降低 TTL，上线后再恢复！
概念 核心概念 ▼

DNS 记录类型

类型作用示例
A域名 → IPv4 地址api.example.com → 1.2.3.4
AAAA域名 → IPv6 地址api.example.com → 2001:db8::1
CNAME域名 → 另一个域名（别名）www.example.com → example.com
MX邮件服务器地址example.com → mail.example.com
TXT文本记录（域名验证等）"v=spf1 include:..."
NS指定域名服务器ns1.dnspod.net
🔑 后端最常用：A 记录（部署绑定服务器 IP）和 CNAME（CDN / 负载均衡配置）。

DNS 解析完整流程

浏览器输入 api.example.com │ ├─① 浏览器缓存 → 查过？直接用 │ ↓ 没有 ├─② 操作系统缓存/hosts → 系统缓存 & /etc/hosts │ ↓ 没有 ├─③ 本地 DNS 服务器 → 运营商/公司DNS（如 8.8.8.8） │ ↓ 没有（以下为迭代查询） ├─④ 根域名服务器 → "我不知道，去问 .com 服务器" │ ↓ ├─⑤ 顶级域名服务器 → ".com 说：去问 ns1.example.com" │ ↓ └─⑥ 权威域名服务器 → "IP 是 1.2.3.4！" ✓ ↓ 缓存结果，返回浏览器
展开查看：递归查询 vs 迭代查询

递归查询：客户端（浏览器）只问一次本地 DNS，本地 DNS 帮你跑完全程，最后把完整结果返回给你。

迭代查询：本地 DNS 依次询问根→顶级→权威服务器，每次只获得"下一步去找谁"的线索，直到最后拿到结果。

一句话：你（客户端）发出的是递归查询，本地 DNS 向上游发出的是迭代查询。

示例 代码示例（来自 demo/2_dns.py） ▼

1. 基础 DNS 查询 — 域名 → IP

import socket

domains = ["www.baidu.com", "github.com", "localhost"]
for domain in domains:
    try:
        ip = socket.gethostbyname(domain)      # 最简单的 DNS 查询
        print(f"  {domain} → {ip}")
    except socket.gaierror as e:
        print(f"  {domain} → 解析失败: {e}")

# localhost 总是解析为 127.0.0.1（在 hosts 文件中定义）

2. 测量 DNS 缓存效果

import time

domain = "www.baidu.com"
for i in range(3):
    start = time.perf_counter()
    socket.gethostbyname(domain)
    elapsed = (time.perf_counter() - start) * 1000
    print(f"  第{i+1}次查询: {elapsed:.2f}ms")

# 观察：第1次较慢（完整解析），后续明显更快（命中系统DNS缓存）
📊 预期输出：第一次 20-50ms，后续 0.01-1ms。这 10-50 倍的差距就是 DNS 缓存的效果。

3. 反向 DNS 查询 — IP → 域名

ips = ["8.8.8.8", "1.1.1.1", "127.0.0.1"]
for ip in ips:
    try:
        hostname, aliases, _ = socket.gethostbyaddr(ip)
        print(f"  {ip} → {hostname}")
    except socket.herror:
        print(f"  {ip} → 反向查询失败（未配置PTR记录）")

4. 部署后域名无法访问 — 排查清单

排查步骤（按顺序执行）： 1. nslookup api.example.com → DNS 是否解析到正确 IP？ 2. ping 1.2.3.4 → 服务器网络是否通？ 3. curl http://1.2.3.4:8000 → 直接访问 IP，服务是否正常？ 4. 检查 DNS 记录配置 → A 记录/CNAME 是否正确？ 5. 等待 DNS 传播 → 新记录可能需要几分钟到 48 小时生效
误区 常见误区 ▼
❌ 误区✅ 纠正
"DNS 修改后立即生效" DNS 有缓存和传播时间，TTL 短则几分钟，长则 48 小时。
"DNS 只做域名到 IP 的翻译" 还有 CNAME、MX、TXT 等多种记录类型，承担路由、邮件、验证等功能。
"DNS 用的是 TCP" DNS 默认使用 UDP（端口 53），快速且数据量小。超过 512 字节或区域传送时才用 TCP。
"hosts 文件没用了" 开发环境中非常有用，且优先级高于 DNS 查询，可用于模拟域名、内网别名、临时绕过 DNS。

常见端口与协议
端口 = 服务器上的"门牌号"。IP 定位到哪台机器，端口定位到哪个服务。
目标 学习目标 ▼
  • 熟记后端核心端口：HTTP(80)、HTTPS(443)、MySQL(3306)、PostgreSQL(5432)、Redis(6379)、SSH(22)
  • 理解端口号三大范围（0-1023 / 1024-49151 / 49152-65535）及权限限制
  • 能排查"端口被占用"错误，掌握 Windows/Linux 查端口命令
  • 理解 0.0.0.0 与 127.0.0.1 的核心区别及使用场景
  • 了解协议分层：你的一行 HTTP 请求在底层如何经历各协议层

0.0.0.0 vs 127.0.0.1

┌──────────────┬──────────────────────────────────────────┐ │ 127.0.0.1 │ 回环地址，只接受本机连接 │ │ (localhost) │ 其他机器无法访问 → 适合纯本地开发 │ ├──────────────┼──────────────────────────────────────────┤ │ 0.0.0.0 │ 监听所有网络接口 │ │ │ 局域网/Docker 内均可访问 → 适合服务暴露 │ └──────────────┴──────────────────────────────────────────┘
🔴 重要：使用 0.0.0.0 让服务暴露时，务必配置防火墙规则！数据库端口（3306、5432、6379）绝不应暴露在公网。

协议分层实例

# 你写的一行代码，底层经过了多个协议层：
response = requests.get("http://api.example.com/users")

# 实际发生：
# HTTP 协议（应用层）  → 构造 GET 请求报文
# TCP 协议（传输层）   → 建立连接、可靠传输
# IP  协议（网络层）   → 寻址路由到目标服务器
# 以太网（链路层）    → 物理传输
示例 代码示例（来自 demo/3_ports_protocols.py） ▼

1. 扫描本机常见端口

common_ports = {
    80: "HTTP", 443: "HTTPS", 3306: "MySQL",
    5432: "PostgreSQL", 6379: "Redis", 8000: "FastAPI"
}
for port, service in common_ports.items():
    sock = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
    sock.settimeout(0.5)
    result = sock.connect_ex(("127.0.0.1", port))   # 0 = 端口开放
    status = "✓ 开放" if result == 0 else "✗ 关闭"
    print(f"  {port:<8} {service:<20} {status}")
    sock.close()

2. 端口冲突演示 — "Address already in use"

server1 = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
server1.bind(("127.0.0.1", 9100))
server1.listen(1)
print("[服务1] 成功绑定端口 9100")

server2 = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
try:
    server2.bind(("127.0.0.1", 9100))   # 同一端口！
except OSError as e:
    print(f"[服务2] 绑定失败: {e}")  # Address already in use
🔧 排查命令：Windows → netstat -ano | findstr :端口号；Linux/Mac → lsof -i :端口号 或 ss -tlnp | grep 端口号

4. 端口连通性测试工具

def check_port(host, port, timeout=2):
    sock = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
    sock.settimeout(timeout)
    try:
        start = time.perf_counter()
        result = sock.connect_ex((host, port))
        elapsed = (time.perf_counter() - start) * 1000
        if result == 0:
            return True, f"{elapsed:.1f}ms"
        return False, "连接被拒绝"
    except socket.timeout:
        return False, "连接超时"
    finally:
        sock.close()
💡 实践技巧：后端连不上 MySQL？先用这个函数测试 check_port("db-host", 3306)，判断是网络问题还是认证问题。
误区 常见误区 ▼
❌ 误区✅ 纠正
"一个端口只能一个进程" 默认如此，但 SO_REUSEPORT 可让多个进程监听同一端口（负载均衡场景）。
"修改端口就安全了" 改端口只是"隐蔽"，真正的安全靠认证、加密和防火墙规则。
"0.0.0.0 是无效地址" 0.0.0.0 表示监听所有网络接口，是合法且常用的地址。Docker 容器内必须使用它。
"端口号可以随便选" 1024 以下需要管理员权限；生产服务要避免与已知服务冲突；团队需统一端口规范。
🔴 安全提醒：数据库端口（3306、5432、6379）绝不要暴露在公网！应使用防火墙/SG 规则限制，只允许应用服务器 IP 访问。
