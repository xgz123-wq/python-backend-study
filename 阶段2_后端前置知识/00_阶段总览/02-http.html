📌 本章 · 为什么要学 HTTP 协议？▼

第一节 · HTTP 请求与响应结构

HTTP 是浏览器、App、后端服务之间最常见的"对话格式"。理解请求和响应结构，是学习 API 的入口。
🎯 学习目标▼
  • 理解 HTTP 请求的 4 部分结构
  • 理解 HTTP 响应的 4 部分结构
  • 掌握 URL 各组成部分及含义
  • 区分 HTTP 与 TCP 的层次关系
  • 能通过浏览器或 Postman 观察请求头
与后端开发的关联：后端框架（FastAPI/Django）本质就是解析 HTTP 请求（路径、查询参数、Header、Body）并返回 HTTP 响应的程序。理解请求/响应结构是后续所有 API 开发的基础。
📖 核心概念▼

HTTP 是什么？

HTTP（HyperText Transfer Protocol）是应用层协议，规定客户端和服务器如何交换数据。底层可靠传输由 TCP 负责，HTTP 只管数据格式。

客户端（浏览器 / Postman / Python 程序） │ HTTP Request ▼ 服务器（Nginx / FastAPI / Django） │ HTTP Response ▼ 客户端接收结果

HTTP 请求的 4 部分

部分示例说明
请求行POST /api/users?page=1 HTTP/1.1方法 + 路径 + 协议版本
请求头Content-Type: application/json元数据：格式、认证、来源
空行空白行分隔 Header 和 Body
请求体{"name": "Alice"}提交给服务器的数据

HTTP 响应的 4 部分

部分示例说明
状态行HTTP/1.1 200 OK协议版本 + 状态码 + 状态文本
响应头Content-Type: application/json响应元数据
空行空白行分隔 Header 和 Body
响应体{"id": 1}返回给客户端的数据

URL 完整组成

https://api.example.com:443/users/1?verbose=true#profile │ │ │ │ │ │ 协议 域名 端口 路径 查询参数 片段

⚠️ #profile 片段标识符只在浏览器端使用，不会发送给服务器。

请求行与状态行详细拆解

请求行：POST /api/users?page=1 HTTP/1.1

  • POST — 请求方法，表示创建资源
  • /api/users?page=1 — 请求路径和查询参数
  • HTTP/1.1 — 协议版本

状态行：HTTP/1.1 200 OK

  • HTTP/1.1 — 协议版本
  • 200 — 状态码，表示成功
  • OK — 状态文本，给人看的说明

第二节 · HTTP 方法与状态码

HTTP 方法表达"想做什么"，状态码表达"结果怎么样"。这是 RESTful API 设计的基础。
🎯 学习目标▼
  • 掌握 5 种核心 HTTP 方法的语义
  • 理解幂等的概念和判断方法
  • 熟记常见状态码及分类（2xx/3xx/4xx/5xx）
  • 区分 401 vs 403 的关键差异
  • 能设计 RESTful 风格的方法与状态码映射
与后端开发的关联：方法语义和状态码是 RESTful API 设计的基石。后端必须根据操作类型选择正确方法、根据结果返回正确状态码，前后端才能高效联调。
📖 核心概念▼

5 种核心 HTTP 方法

方法语义常见用途是否应幂等
GET获取资源查询列表、查询详情✅ 是
POST创建资源 / 提交动作注册、登录、创建订单❌ 否
PUT全量更新资源替换用户资料✅ 是
PATCH局部更新资源修改昵称、修改状态✅ 通常是
DELETE删除资源删除文章、删除商品✅ 是

什么是幂等？

同一个请求执行一次和执行多次，最终结果相同。

GET /users/1 查询 1 次和 10 次，资源不变 → 幂等 DELETE /users/1 删除 1 次和 10 次，最终都是不存在 → 幂等 POST /orders 提交 1 次创 1 个订单，提交 10 次可能创 10 个 → 非幂等
完整状态码速查表（2xx/3xx/4xx/5xx）
类别状态码含义场景
2xx 成功200 OK请求成功查询成功、更新成功
201 Created创建成功新建用户、创建订单
204 No Content成功但无响应体删除成功
3xx 重定向301永久重定向域名永久迁移
302临时重定向临时跳转登录页
304资源未修改浏览器缓存命中
4xx 客户端400请求格式错误JSON 格式错误
401未认证没登录、Token 缺失
403无权限登录了但不能访问
404资源不存在用户 ID 不存在
422语义校验失败FastAPI 参数校验失败
5xx 服务端500服务端内部错误未捕获异常
502网关错误响应Nginx 后端服务异常
503服务不可用服务维护、过载
504网关超时后端处理太慢

class MethodStatusHandler(BaseHTTPRequestHandler):
    def do_GET(self):
        if self.path == "/users/1":
            self._send_json(200, self.users[1])    # 成功
        elif self.path == "/users/999":
            self._send_json(404, {"error": "用户不存在"}) # 资源不存在

    def do_POST(self):
        # 创建用户，返回 201
        self._send_json(201, {"message": "创建成功"})

    def do_DELETE(self):
        # 删除成功，返回 204 无响应体
        self.send_response(204)
        self.end_headers()

GET /users/1200 OK{"id": "1", "name": "Alice"}
GET /users/999404 Not Found{"error": "用户不存在"}
POST /users201 Created{"message": "创建成功", "user": ...}

⚠️ 常见误区▼
❌ 误区✅ 纠正
"所有成功都返回 200"创建资源更推荐 201，删除无内容更推荐 204
"未登录返回 403"未认证是 401，认证后无权限才是 403
"业务错误都返回 500"参数错误、资源不存在、权限问题属于 4xx 客户端错误
"POST 比 GET 安全"POST 参数不显示在 URL 中，但没有 HTTPS 仍然可能被窃听
🔒 安全警示：401（未认证）与 403（无权限）是面试高频考点。记住口诀："401 = 你是谁我不知道，403 = 我知道你是谁但你不能进"。

第三节 · Header 与 Content-Type

Header 是 HTTP 的"说明书"，Content-Type 告诉对方 Body 里装的是什么格式。
🎯 学习目标▼
  • 理解 Header 作为元数据的作用
  • 区分 Content-Type 与 Accept
  • 掌握常见 Content-Type 类型及场景
  • 知道 Authorization 的安全放置位置
  • 能根据请求体格式正确设置 Header
与后端开发的关联：Header 错误是前后端联调最常见的坑——前端发了 JSON 但 Content-Type 写错，后端就无法正确解析。
📖 核心概念▼

常见请求 Header

Header作用示例
Host目标主机api.example.com
Content-Type请求体格式application/json
Accept客户端期望的响应类型application/json
Authorization认证信息Bearer eyJ...
Cookie浏览器携带的 Cookiesession_id=abc
Origin跨域请求来源https://frontend.example.com

常见 Content-Type 类型

Content-Type含义常见场景
application/jsonJSON 数据RESTful API 最常用
application/x-www-form-urlencoded表单键值对传统 HTML 表单
multipart/form-data多部分表单文件上传
text/plain纯文本文本响应
text/htmlHTML 页面浏览器页面
application/octet-stream二进制流文件下载

Content-Type 与 Accept 的区别

请求方 响应方 ─────── ─────── Content-Type: application/json → "我发的 Body 是 JSON" Accept: application/json → "请给我 JSON 格式的响应"
Header方向含义
Content-Type发送方 → 接收方描述当前消息体的格式
Accept请求方 → 响应方表示期望得到的响应格式

⚠️ 常见误区▼
❌ 误区✅ 纠正
"Header 不重要"Header 决定 Body 怎么解析、是否认证、是否跨域、是否缓存
"Accept 和 Content-Type 一样"Accept 表示想接收什么；Content-Type 表示自己发送的 Body 是什么
"JSON 放 Body 就行"还必须设置 Content-Type: application/json
"Token 可以放 URL"不推荐，URL 容易进入日志；应放 Authorization Header
🔒 安全警示：永远不要把 Token 放在 URL 查询参数中！URL 会出现在浏览器历史、服务器日志、代理日志中，极容易泄露。

第四节 · Cookie 与 Session

HTTP 是无状态的，Cookie 和 Session 用来让服务器"记住你是谁"。
🎯 学习目标▼
  • 理解 HTTP 无状态的含义
  • 掌握 Cookie 的客户端存储机制
  • 掌握 Session 的服务端存储机制
  • 熟记 Cookie 的 3 个关键安全属性
  • 区分 Cookie/Session 与 JWT Token 方案
与后端开发的关联：登录态管理是后端最核心的功能之一。Cookie + Session 是传统方案，JWT Token 是前后端分离方案，生产环境需要理解两者的安全配置。
📖 核心概念▼

Cookie 工作流程

1. 用户登录（POST /login） 2. 服务器验证用户名密码 3. 服务器生成 session_id 4. 响应头返回 Set-Cookie: session_id=xxx 5. 浏览器保存 Cookie 6. 后续请求自动带上 Cookie: session_id=xxx 7. 服务器根据 session_id 查询用户状态

Cookie 常用安全属性

属性作用建议
HttpOnly禁止 JavaScript 读取 Cookie登录态 Cookie 必加
Secure只允许 HTTPS 发送 Cookie生产环境必加
SameSite限制跨站携带 Cookie推荐 Lax 或 Strict
Max-Age / Expires过期时间根据业务设置
PathCookie 生效路径通常为 /
DomainCookie 生效域名谨慎设置父域名

Session 存储方案对比

存储方式特点场景
内存快，但重启丢失，无法多实例共享本地学习 Demo
Redis快，可过期，多实例共享生产常用 ✅
数据库持久化强，但性能较低管理后台、低频场景
Cookie/Session vs JWT Token 方案对比
维度Cookie + SessionJWT Token
状态保存位置服务端保存状态客户端保存令牌，服务端校验签名
主动失效方便（删 Session 即可）较难（需黑名单机制）
扩展性需 Redis 集中存储无状态，天然可扩展
常见用途传统 Web 登录前后端分离、移动端 API

两者不是绝对替代关系：Cookie 是浏览器存储和自动携带机制，Session / JWT 是身份状态方案。

    SESSIONS[session_id] = {"user_id": "1", "username": "alice"}

        self._send_json(401, {"error": "未登录或 Session 已失效"})

⚠️ 常见误区▼
❌ 误区✅ 纠正
"Cookie 和 Session 是一回事"Cookie 在客户端，Session 在服务端
"Cookie 一定不安全"正确设置 HttpOnly、Secure、SameSite 后可用于安全登录态
"Session 不需要存储"Session 数据必须存储在内存、Redis 或数据库中
"JWT 一定比 Session 好"JWT 无法方便主动失效，Session 更适合强管控登录态
🔒 安全警示：生产环境的登录态 Cookie 必须同时设置 HttpOnly（防 XSS 窃取）+ Secure（仅 HTTPS 传输）+ SameSite=Lax（防 CSRF）。

第五节 · HTTPS 与 CORS

HTTPS 解决"传输是否安全"，CORS 解决"浏览器是否允许跨域访问"。
🎯 学习目标▼
  • 理解 HTTPS 解决的三个核心问题
  • 掌握同源策略的三要素判断
  • 理解 CORS 的工作原理和预检请求
  • 熟记核心 CORS 响应头
  • 了解 FastAPI 中 CORS 的安全配置
与后端开发的关联：生产环境 HTTPS 是强制要求，CORS 配置是前后端分离项目的必经之路。理解这两者才能正确处理跨域问题和安全部署。
📖 核心概念▼

HTTPS 解决的三个问题

目标说明
🔐 加密防止请求和响应内容被窃听
🔏 完整性防止数据在传输中被篡改
🛡️ 身份认证通过证书确认服务器身份，防止访问到假网站

HTTPS = HTTP + TLS 加密。生产环境中的登录、支付、用户隐私接口必须使用 HTTPS。

同源策略：三个完全相同才算同源

协议 + 域名 + 端口
URL 对比是否同源原因
https://a.com vs https://a.com✅ 是完全相同
http://a.com vs https://a.com❌ 否协议不同
https://a.com vs https://www.a.com❌ 否域名不同

CORS 核心响应头

Header作用示例
Access-Control-Allow-Origin允许哪个来源访问https://frontend.example.com
Access-Control-Allow-Methods允许哪些方法GET, POST, PUT, DELETE
Access-Control-Allow-Headers允许哪些请求头Content-Type, Authorization
Access-Control-Allow-Credentials是否允许携带 Cookietrue
Access-Control-Max-Age预检结果缓存时间（秒）600
简单请求 vs 预检请求（OPTIONS）

简单请求（直接发送）：GET/POST/HEAD 且 Header 简单、Content-Type 为表单类。

预检请求（先发 OPTIONS 询问）：复杂跨域请求前，浏览器自动发 OPTIONS 请求：

OPTIONS /api/users HTTP/1.1 Origin: https://frontend.example.com Access-Control-Request-Method: POST Access-Control-Request-Headers: Content-Type, Authorization

服务器返回允许的 CORS 头后，浏览器才发送真正的 POST 请求。

# 服务端：处理 CORS 预检 OPTIONS 请求
def do_OPTIONS(self):
    origin = self.headers.get("Origin", "")
    if origin == ALLOWED_ORIGIN:
        self.send_response(204)
        self.send_header("Access-Control-Allow-Origin", origin)
        self.send_header("Access-Control-Allow-Methods",
                          "GET, POST, PUT, PATCH, DELETE")
        self.send_header("Access-Control-Allow-Headers",
                          "Content-Type, Authorization")
    else:
        self.send_response(403)  # 拒绝不允许的来源
# FastAPI 中推荐的 CORS 配置方式
from fastapi.middleware.cors import CORSMiddleware

app.add_middleware(
    CORSMiddleware,
    allow_origins=["https://frontend.example.com"],  # 明确列出
    allow_credentials=True,
    allow_methods=["GET", "POST", "PUT", "PATCH", "DELETE"],
    allow_headers=["Content-Type", "Authorization"],
)

⚠️ 常见误区▼
❌ 误区✅ 纠正
"CORS 是后端拒绝请求"CORS 主要是浏览器拦截响应，服务端通常已经收到请求
"Postman 没跨域问题，说明浏览器也没问题"Postman 不受浏览器同源策略限制
"HTTPS 只用于加密密码"HTTPS 保护所有请求和响应，包括 Token、Cookie、个人数据
"allow_origins=['*'] 最方便"生产环境应只允许可信来源，尤其是带 Cookie 的请求
🔒 安全警示：浏览器不允许 Access-Control-Allow-Origin: * 与 Access-Control-Allow-Credentials: true 同时用于凭证请求。生产环境必须明确列出可信前端域名。
