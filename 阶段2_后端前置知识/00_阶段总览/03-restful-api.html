RESTful 设计原则
REST 不是规范，是一组约束——用好它，接口自带文档属性。
目标 学习目标 ▼
  • 理解 REST 六大约束及其实际含义，知道为什么"无状态"对横向扩展重要
  • 能将"动作导向"的旧接口改造为"资源导向"的 RESTful 接口
  • 掌握 HTTP 方法（GET/POST/PUT/PATCH/DELETE）与 CRUD 的精确映射
  • 理解 PUT 和 PATCH 的语义区别，知道何时用哪个
  • 能设计嵌套资源 URL，并知道嵌套不超过 2 层的原因
💡 后端视角：RESTful 不是强制规范，而是业界最广泛认可的约定。遵循它能让你的 API 被前端、移动端、第三方开发者一眼看懂，减少沟通成本。
概念 核心概念 ▼

REST 六大约束

约束说明实际意义
统一接口资源通过固定接口暴露，HTTP 方法 + URL 唯一标识操作降低学习成本，方便工具集成
无状态服务端不存储客户端状态，每次请求携带完整信息易横向扩展，请求可独立重试
缓存响应标注是否可缓存，客户端/代理可缓存减少重复请求，提升性能
客户端-服务端分离前端只关心 UI，后端只关心数据，接口解耦前后端独立演化
分层系统客户端不感知中间层（CDN、网关、负载均衡）灵活扩展基础设施
按需代码（可选）服务端可以下发可执行代码给客户端少用，了解即可

动作导向 vs 资源导向

动作导向（反模式）： 资源导向（RESTful）： POST /getUserInfo GET /users/{id} POST /createNewUser POST /users POST /deleteUser DELETE /users/{id} POST /updateUserName PATCH /users/{id} GET /getAllOrders GET /orders POST /doPayOrder POST /orders/{id}/pay
🔑 转换思路：① 找出操作的核心名词（资源）；② 用 HTTP 方法表达动作；③ 特殊操作用子资源或动名词兜底（如 /orders/{id}/pay）

HTTP 方法与 CRUD 映射

HTTP 方法CRUD含义幂等？示例
GETRead读取资源✅ 是GET /users/1
POSTCreate创建资源❌ 否POST /users
PUTUpdate全量更新（替换整个资源）✅ 是PUT /users/1
PATCHUpdate部分更新（只改传入字段）⚠️ 通常是PATCH /users/1
DELETEDelete删除资源✅ 是DELETE /users/1
❓ PUT vs PATCH：PUT 传入完整对象，未传字段会被清空（覆盖语义）；PATCH 只传需要修改的字段，其余保持不变（部分更新语义）。实战中 PATCH 更常用。

嵌套资源（层级关系）

# 正确：最多嵌套 2 层
GET  /users/{userId}/orders           # 用户的所有订单
POST /users/{userId}/orders           # 在该用户下创建订单
GET  /users/{userId}/orders/{orderId} # 获取该用户的某个订单

# 错误：超过 2 层，冗长难维护
GET /users/{uid}/orders/{oid}/items/{iid}/reviews/{rid}  # ❌ 太深
GET /order-items/{itemId}/reviews                         # ✅ 扁平化更好

FastAPI 完整 RESTful 路由示例

from fastapi import FastAPI

app = FastAPI()

# ── 用户集合 ──
@app.get("/users")
def list_users(): ...           # 列出所有用户（支持分页/过滤）

@app.post("/users")
def create_user(): ...          # 创建用户，Body 含用户信息

# ── 单个用户 ──
@app.get("/users/{user_id}")
def get_user(user_id: int): ...  # 获取指定用户

@app.patch("/users/{user_id}")
def update_user(user_id: int): ... # 部分更新（只改传入字段）

@app.delete("/users/{user_id}")
def delete_user(user_id: int): ... # 删除用户

# ── 嵌套资源：用户的订单 ──
@app.get("/users/{user_id}/orders")
def list_user_orders(user_id: int): ...

# ── 特殊操作：支付订单 ──
@app.post("/orders/{order_id}/pay")
def pay_order(order_id: int): ... # 特殊操作用子路径

@app.post("/orders/{order_id}/cancel")
def cancel_order(order_id: int): ...

改造前改造后说明

POST /createOrderPOST /orders动词去掉，用方法表达

误区 常见误区 ▼
❌ 误区✅ 纠正
URL 里加动词：POST /createUser 用 HTTP 方法表达动词，URL 只留名词：POST /users
一律用 POST（"POST 万能"） 丢失 GET 缓存特性、接口无幂等保证，可读性差；正确映射语义方法。
嵌套层级超过 2 层 超 2 层改用扁平化 + 查询参数：GET /projects?teamId=x&memberId=y
REST 必须 100% 纯粹，否则不算 REST 批量删除、复杂搜索等无法纯 CRUD 映射时，实用优先：POST /users/batch-delete 也是可接受的方案。
用 PUT 做部分更新 PUT 是全量替换，未传字段会被清空；部分更新应用 PATCH。

URL 命名规范
URL 是接口的"门牌号"，好的命名让接口一看就懂，差的命名是团队协作的噩梦。
目标 学习目标 ▼
  • 掌握 URL 命名七条核心规范：名词复数、小写连字符、层级关系、查询参数用途等
  • 理解路径参数和查询参数的本质区别，知道各自适用场景
  • 能看懂并分析一个完整 URL 的每个部分（协议、主机、前缀、路径、参数）
  • 能将不规范的旧接口改造为符合规范的 URL
💡 规范的 URL 是自文档化的：光看 GET /users/123/orders?status=pending&page=1 就知道"获取用户 123 的待处理订单第 1 页"。
概念 核心概念 ▼

七条核心命名规范

#规范正确示例错误示例
1用名词，不用动词GET /usersGET /getUsers
2用复数名词表示集合/users, /orders/user, /userList
3小写 + 连字符（kebab-case）/product-categories/productCategories, /product_categories
4路径层级表资源关系（最多 2 层）/users/{id}/orders/getUserOrders?userId=x
5查询参数做过滤/分页/排序/users?status=active&page=2/users/active（状态作路径）
6特殊操作用子路径动名词兜底POST /orders/{id}/payPOST /payOrder
7路径加 /api 前缀/api/v1/users/users（与前端路由冲突）

完整 URL 结构解析

https://api.example.com/api/v1/users/123/orders?status=pending&page=1&size=10 ├── https:// 协议（scheme） ├── api.example.com 主机名（host） ├── /api/v1 路径前缀（前缀 + 版本） ├── /users/123/orders 资源路径（集合/ID/子集合） └── ?status=pending&... 查询参数（过滤 + 分页）

路径参数查询参数
位置/users/{id}/users?id=123
语义标识一个具体资源过滤/修饰资源集合
必填性通常必填通常可选
典型用法单个资源 CRUD搜索、分页、排序
示例GET /users/42GET /users?role=admin

FastAPI 路径参数与查询参数示例

from fastapi import FastAPI
from typing import Optional

app = FastAPI()

# 路径参数：标识具体资源
@app.get("/users/{user_id}")
def get_user(user_id: int):
    return {"user_id": user_id}

# 查询参数：过滤、分页
@app.get("/users")
def list_users(
    status: Optional[str] = None,     # 过滤
    page: int = 1,                    # 分页
    size: int = 20,
):
    return {"status": status, "page": page, "size": size}

# 嵌套资源：用户的订单
@app.get("/users/{user_id}/orders")
def list_user_orders(user_id: int, status: Optional[str] = None):
    return {"user_id": user_id, "status": status}
误区 常见误区 ▼
❌ 误区✅ 纠正
ID 用查询参数传：GET /users?id=123 ID 是资源唯一标识，属于路径参数：GET /users/123
混用 camelCase 和 kebab-case 统一用小写连字符：/product-categories，团队内风格一致最重要
路径结尾带斜杠：/users/ 去掉尾部斜杠：/users（某些框架会视为不同路由）
文件扩展名出现在 URL：/users.json 数据格式通过 Accept: application/json 请求头协商，不放 URL
把过滤条件放到路径里：/users/active 过滤是修饰集合，用查询参数：/users?status=active

统一响应格式
让前端能写一个拦截器搞定所有接口的错误处理——统一格式的价值所在。
目标 学习目标 ▼
  • 理解为什么需要统一响应格式，能解释它对前端和联调的具体好处
  • 掌握推荐的响应结构：code + message + data 三字段设计
  • 理解 HTTP 状态码 vs 业务 code 的双层设计逻辑，能用表格区分各场景
  • 能设计合理的业务错误码编码规则（如 40001、40101、50001）
  • 掌握分页响应的完整格式：items + pagination 元数据
💡 统一格式的最大价值：前端可以写一个全局拦截器处理 Loading 状态、错误弹窗、Token 过期，而不是在每个接口里单独处理。
概念 核心概念 ▼

推荐响应结构

// 成功响应 { "code": 0, "message": "success", "data": { "id": 1, "name": "Alice" } } // 失败响应 { "code": 40401, "message": "用户不存在", "data": null }
字段类型说明
codeint业务状态码：0 代表成功，非 0 代表具体错误
messagestr人类可读描述，成功时为 "success"，失败时为错误原因（面向用户）
dataany业务数据，失败时明确返回 null

HTTP 状态码 vs 业务 code 双层设计（推荐）

场景HTTP 状态码业务 codemessage
获取用户成功2000success
创建用户成功2010success
用户名已存在40040002用户名已被注册
Token 过期40140102登录已过期，请重新登录
无权限40340301无访问权限
用户不存在40440401用户不存在
服务器异常50050001服务器内部错误
🔑 双层的价值：HTTP 状态码让网关/监控工具识别失败；业务 code 让前端知道具体原因（如弹不同的错误文案）。

业务错误码编码规则

XXYYYY 格式： XX → HTTP 状态码前两位（40 = 4xx，50 = 5xx） YYYY → 具体业务错误序号 40001 → 请求参数缺失或格式错误 40002 → 用户名已存在 40101 → Token 无效 40102 → Token 已过期 40401 → 用户不存在 40402 → 订单不存在 50001 → 数据库操作失败

分页响应格式

{ "code": 0, "message": "success", "data": { "items": [ {...}, {...} ], "pagination": { "page": 1, "size": 20, "total": 100, ← 满足过滤条件的总记录数 "totalPages": 5, "hasNext": true, "hasPrev": false } } }
示例 代码示例：FastAPI 统一响应封装 ▼

通用响应模型（Pydantic）

from typing import TypeVar, Generic, Optional
from pydantic import BaseModel

T = TypeVar("T")

class Response(BaseModel, Generic[T]):
    code: int = 0
    message: str = "success"
    data: Optional[T] = None

    @classmethod
    def ok(cls, data: T = None, message: str = "success"):
        return cls(code=0, message=message, data=data)

    @classmethod
    def error(cls, code: int, message: str):
        return cls(code=code, message=message, data=None)

分页响应封装

from pydantic import BaseModel
from typing import List

class Pagination(BaseModel):
    page: int
    size: int
    total: int
    total_pages: int
    has_next: bool
    has_prev: bool

class PagedData(BaseModel, Generic[T]):
    items: List[T]
    pagination: Pagination

@app.get("/users")
def list_users(page: int = 1, size: int = 20):
    total = db.query(User).count()        # 先查总数
    items = db.query(User).offset((page-1)*size).limit(size).all()
    paged = PagedData(
        items=items,
        pagination=Pagination(
            page=page, size=size, total=total,
            total_pages=(total + size - 1) // size,
            has_next=page * size < total,
            has_prev=page > 1
        )
    )
    return Response.ok(data=paged)
误区 常见误区 ▼
❌ 误区✅ 纠正
HTTP 200 但 code 是 404：{"code": 404, "msg": "..."} HTTP 状态码和业务 code 各司其职：HTTP 404 + 业务 code 40401。混用会让监控误判为成功。
data 字段直接返回数组：[{...}, {...}] 用对象包裹：{"items": [...], "pagination": {...}}。一旦需要加元数据，不用改所有调用方。
错误时省略 data 字段 明确返回 "data": null，让前端知道字段存在但为空，而不是接口设计遗漏了该字段。
message 暴露技术细节："NullPointerException at line 42" message 应面向用户："服务器内部错误"。技术细节写日志，不暴露给客户端。

分页、过滤与排序
列表接口的三大核心能力——不做分页的接口，数据量大了必挂。
目标 学习目标 ▼
  • 掌握 Offset 分页和 Cursor 分页的原理、SQL 实现、适用场景及各自缺点
  • 理解过滤参数的命名规范：等值过滤、范围过滤（_gt/_gte/_lt/_lte）、模糊搜索
  • 掌握排序参数设计，并理解为什么 sort 参数必须做白名单校验
  • 能看懂并写出带分页+过滤+排序的完整列表接口 URL
🔴 安全提醒：sort 参数若不做白名单校验，直接拼接进 SQL，会有 SQL 注入风险！
概念 核心概念 ▼

两种分页方式对比

特性页码分页（Offset）游标分页（Cursor）
可跳页✅ 可以❌ 不可以
大数据量性能差（高 OFFSET 慢）好（按 ID/时间戳索引）
数据一致性差（实时变更会漏/重）好
实现复杂度简单较复杂
适用场景搜索结果、数据管理后台动态 Feed 流、消息列表

Offset 分页 SQL 实现

请求：GET /users?page=2&size=20 SQL： SELECT * FROM users ORDER BY id LIMIT 20 OFFSET 20; -- offset = (page-1) * size = (2-1)*20 = 20

Cursor 分页 SQL 实现

请求：GET /posts?cursor=eyJpZCI6MTAwfQ&size=20 （cursor 是上一页最后一条记录 ID=100 的 Base64 编码） SQL： SELECT * FROM posts WHERE id > 100 -- cursor 解码后的 id ORDER BY id ASC LIMIT 20;

写法含义示例

示例 代码示例：带分页过滤排序的 FastAPI 接口 ▼

Offset 分页实现

from fastapi import FastAPI, Query
from typing import Optional

app = FastAPI()

ALLOWED_SORT_FIELDS = {"id", "name", "created_at", "updated_at"}

@app.get("/api/v1/users")
def list_users(
    status: Optional[str] = None,
    age_gt: Optional[int] = None,
    q: Optional[str] = None,
    sort: str = "id",
    order: str = "asc",
    page: int = Query(1, ge=1),
    size: int = Query(20, ge=1, le=100),
):
    # ⚠️ sort 白名单校验，防止 SQL 注入
    if sort not in ALLOWED_SORT_FIELDS:
        raise ValueError(f"不支持按 {sort} 排序")

    query = db.query(User)
    if status:
        query = query.filter(User.status == status)
    if age_gt:
        query = query.filter(User.age > age_gt)
    if q:
        query = query.filter(User.name.contains(q))

    total = query.count()                                # 先查总数
    items = query.offset((page-1)*size).limit(size).all()
    return {"items": items, "total": total, "page": page}

Cursor 分页实现

import base64, json

def encode_cursor(last_id: int) -> str:
    return base64.b64encode(json.dumps({"id": last_id}).encode()).decode()

def decode_cursor(cursor: str) -> int:
    return json.loads(base64.b64decode(cursor)).get("id", 0)

@app.get("/api/v1/posts")
def list_posts(cursor: Optional[str] = None, size: int = 20):
    last_id = decode_cursor(cursor) if cursor else 0
    items = db.query(Post).filter(Post.id > last_id).limit(size + 1).all()
    has_next = len(items) > size
    items = items[:size]
    next_cursor = encode_cursor(items[-1].id) if has_next else None
    return {"items": items, "nextCursor": next_cursor, "hasNext": has_next}
误区 常见误区 ▼
❌ 误区✅ 纠正
不做分页直接返回全表：db.query(User).all() 无论数据量多少，列表接口都应有分页上限（不传参数时默认最多返回 100 条）
过滤条件放路径里：/users/active/admin 过滤条件是修饰集合的，用查询参数：/users?status=active&role=admin
total 用当前页数量代替："total": len(items) total 是全量 COUNT 查询（不含 LIMIT），需单独执行 query.count()
sort 参数不校验，直接拼接进 SQL 必须做白名单校验（ALLOWED_SORT_FIELDS），防止 SQL 注入

API 版本管理
破坏性变更时让新旧客户端同时工作——避免"改了接口、所有调用方都炸"的灾难。
目标 学习目标 ▼
  • 理解什么是"破坏性变更"，能判断哪些改动需要升版本，哪些不需要
  • 掌握四种版本管理方式（URL路径/Header/查询参数/子域名）及各自优劣
  • 知道推荐方案是 URL 路径版本（/api/v1/），能说出原因
  • 了解 FastAPI 中用 APIRouter prefix 实现多版本路由的方式
  • 了解旧版本下线前的注意事项（通知策略、弃用头、迁移窗口）
💡 版本管理不是"API 改了就换版本"，而是"破坏性变更时才换"。非破坏性变更（加字段、加接口）直接改，不需要新版本。
概念 核心概念 ▼

破坏性变更 vs 非破坏性变更

需要新版本（破坏性变更）不需要新版本（非破坏性变更）
删除字段：响应中去掉 email新增字段：响应中增加 avatar_url
修改字段名：user_name → username新增接口：增加 /users/{id}/followers
修改字段类型：age 从 int 改为 string松化参数限制：必填改为可选
删除接口：/users/search 下线Bug 修复：不改结构只修正数据
修改认证方式：Cookie → Bearer Token性能优化：响应内容不变

四种版本管理方式对比

方案示例直观性实现难度推荐度
URL 路径（首选✅）/api/v1/users⭐⭐⭐ 高简单⭐⭐⭐
请求头API-Version: 2⭐ 低中⭐⭐ 次选
查询参数/api/users?version=2⭐⭐ 中简单⭐ 不推荐
子域名v2.api.example.com⭐⭐⭐ 高复杂⭐⭐ 特殊场景
🔑 推荐 URL 路径版本的原因：直观（一眼知道版本）、易调试（浏览器直接访问）、对缓存友好、路由层面隔离实现简单。

旧版本下线策略

1. 发布 v2 时，v1 继续维护（至少 6-12 个月） 2. 在 v1 响应头中加弃用警告： Deprecation: true Sunset: Sat, 01 Jan 2025 00:00:00 GMT Link: <https://api.example.com/api/v2/users>; rel="successor-version" 3. 监控 v1 流量，降至 0 时下线 4. 下线前发邮件/公告通知老客户端开发者
示例 代码示例：FastAPI 多版本路由 ▼

方式一：路由前缀（推荐）

from fastapi import FastAPI, APIRouter

app = FastAPI()

v1_router = APIRouter(prefix="/api/v1")
v2_router = APIRouter(prefix="/api/v2")

@v1_router.get("/users/{user_id}")
def get_user_v1(user_id: int):
    # v1 旧格式
    return {"user_name": "alice", "user_email": "alice@x.com"}

@v2_router.get("/users/{user_id}")
def get_user_v2(user_id: int):
    # v2 新格式（字段名变更 = 破坏性变更）
    return {
        "username": "alice",
        "email": "alice@x.com",
        "created_at": "2024-01-01T00:00:00Z",
        "avatar_url": None
    }

app.include_router(v1_router)
app.include_router(v2_router)

方式二：独立应用挂载

app_v1 = FastAPI(title="API v1")
app_v2 = FastAPI(title="API v2")

# 各自注册路由...

main_app = FastAPI()
main_app.mount("/api/v1", app_v1)
main_app.mount("/api/v2", app_v2)

字段v1（旧）v2（新）变更类型
用户名user_nameusername❌ 破坏性（字段重命名）
邮箱user_emailemail❌ 破坏性（字段重命名）
创建时间create_time（无时区）created_at（ISO 8601）❌ 破坏性（格式改变）
头像—（无此字段）avatar_url: null✅ 非破坏性（新增）
误区 常见误区 ▼
❌ 误区✅ 纠正
所有改动都发新版本 只有破坏性变更才需要新版本；非破坏性变更（加字段、加接口）直接改，不增加维护负担
v2 上线立即下线 v1 旧版本必须有充足迁移窗口（移动端 App 发版周期可能数月），强行下线直接影响用户
版本号用语义版本：/api/v1.2.3/ API 对外版本只需主版本号（v1、v2）；次版本和补丁变更不改 URL
文档中忘记标注弃用状态 旧版本弃用时，必须在 API 文档、Swagger/OpenAPI 中明确标注 deprecated: true
