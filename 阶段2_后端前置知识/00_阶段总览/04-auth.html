第一节 · 认证与授权概念

认证（Authentication）= 你是谁？授权（Authorization）= 你能干什么？
🎯 学习目标 ▼
  • 区分认证（AuthN）与授权（AuthZ）的核心含义
  • 理解 HTTP 无状态性及 Session/Token 两种解决方案
  • 掌握 RBAC（基于角色的权限控制）模型
  • 正确区分 401（未认证）与 403（未授权）状态码
  • 了解密码安全存储的基本要求
与后端开发的关联：每个 API 接口都需要先认证用户身份、再校验操作权限。FastAPI 中通过 Depends(get_current_user) 实现认证依赖注入，再通过角色检查实现授权——这是后端安全的两层防线。
📖 核心概念 ▼

认证（AuthN）验证用户身份（你是谁），授权（AuthZ）验证操作权限（你能做什么）。认证是授权的前提。

维度认证（AuthN）授权（AuthZ）
核心问题你是谁？你能做什么？
发生时机登录时每次请求时
常见技术密码、JWT、OAuth2RBAC、ACL、策略
HTTP 状态码401 Unauthorized403 Forbidden
类比查身份证检查通行证

Session vs Token 对比

维度Session（传统）Token/JWT（现代）
存储位置服务端（内存/Redis）客户端（浏览器/App）
服务器状态有状态无状态
横向扩展难（需共享 Session）易（独立验证）
注销方式删除服务端 Session黑名单机制
移动端支持差（Cookie 不友好）好（Header 传递）
Session 方案流程（展开查看完整步骤）
登录：用户提交账号密码 → 服务器验证 → 创建 Session 存 Redis → 返回 Session ID（Set-Cookie）
后续请求：浏览器自动携带 Cookie → 服务器用 Session ID 查 Redis → 确认身份 → 处理请求
登出：服务器删除 Redis 中 Session → Session ID 立即失效
痛点：多服务器部署时必须共享 Redis，否则用户被路由到另一台服务器时会"掉线"
Token/JWT 方案流程（展开查看完整步骤）
登录：用户提交账号密码 → 服务器验证 → 生成 JWT（含用户信息+签名） → 返回 Token
后续请求：客户端在 Authorization Header 携带 Token → 服务器解密验证签名和有效期 → 读取用户信息 → 处理请求
优势：每台服务器只需知道 secret_key 就能独立验证，无需共享存储，天然支持水平扩展
RBAC 权限模型原理（展开查看）
用户 → 角色 → 权限：不直接给用户分配权限，而是通过角色间接关联。
示例：用户 Alice 被赋予"普通用户"角色，"普通用户"角色拥有 [read_own_profile, read_public_posts, create_comment] 权限。
当需要修改 Alice 的权限时，只需调整她的角色，或修改角色的权限集合——改动一处即可影响所有同角色用户。
💻 代码示例 ▼

Part 1：认证方法 — authenticate()

# 认证（Authentication）：验证用户身份，失败返回 None → HTTP 401
def authenticate(self, username: str, password: str) -> dict | None:
    user = self.users.get(username)
    if not user:
        return None  # 用户不存在
    password_hash = hashlib.sha256(password.encode()).hexdigest()
    if password_hash != user["password_hash"]:
        return None  # 密码错误
    return {"username": username, "role": user["role"]}

Part 2：授权方法 — authorize()

# 授权（Authorization）：验证用户是否有权执行某操作，失败 → HTTP 403
def authorize(self, user: dict, action: str) -> bool:
    allowed_actions = self.permissions.get(user["role"], [])
    return action in allowed_actions

Part 3：Session 机制模拟

class SessionStore:
    def create_session(self, user_data: dict, ttl_seconds: int = 3600) -> str:
        session_id = secrets.token_urlsafe(32)  # 随机安全 ID
        self._sessions[session_id] = {
            "user": user_data,
            "expires_at": time.time() + ttl_seconds,
        }
        return session_id  # 存入 Cookie 返回给客户端

    def get_session(self, session_id: str) -> dict | None:
        session = self._sessions.get(session_id)
        if not session or time.time() > session["expires_at"]:
            return None
        return session["user"]

Part 4：401 vs 403 演示 — API 请求处理

def simulate_api_request(session_id, action, user_session_store):
    # Step 1: 认证（你是谁？）
    if not session_id:
        return 401, "未提供认证凭证（请先登录）"
    current_user = user_session_store.get_session(session_id)
    if not current_user:
        return 401, "Session 无效或已过期"

    # Step 2: 授权（你能做什么？）
    if not auth.authorize(current_user, action):
        return 403, f"权限不足，{current_user['role']} 角色无法执行 [{action}]"

    return 200, f"操作成功！{current_user['username']} 执行了 [{action}]"
⚠️ 常见误区 ▼
❌ 误区✅ 纠正
"401 和 403 都是权限问题"401 = 未认证（没登录/Token 过期），403 = 未授权（登录了但没权限）。两者在 FastAPI 中处理方式不同
"Token 比 Session 一定更安全"各有利弊。Token 短时效 + HTTPS 才安全；Session 在服务端可控，注销更直接
"密码用 MD5 加密就安全了"MD5 是哈希不是加密，且可被彩虹表攻破。应使用 bcrypt/argon2 加盐哈希
"JWT 可以直接存敏感信息"Payload 只是 Base64 编码，任何人可解码读取，切勿存密码、身份证号等敏感信息
"登出就是前端删除 Token"Token 在有效期内仍然有效，完整登出需要服务端黑名单机制
🔴 安全警告：生产环境中，绝不能明文存储密码，即使"只有内部人员能看到数据库"也不行——数据库泄露（SQL 注入、备份泄露、离职员工）是真实发生的安全事件。

第二节 · JWT 详解

JWT 是现代后端认证的主流方案：服务器不存状态，把用户信息加密签名后交给客户端保管。
🎯 学习目标 ▼
  • 掌握 JWT 的三部分结构（Header / Payload / Signature）
  • 理解签名防篡改原理（HMAC 对称签名）
  • 能手动 Base64URL 解码 JWT Payload
  • 掌握 Access Token + Refresh Token 双 Token 机制
  • 记住 JWT 的安全底线（Payload 不是加密）
与后端开发的关联：JWT 是前后端分离项目的认证标配。FastAPI 中配合 OAuth2PasswordBearer 和 Depends 实现声明式认证：保护任何接口只需一行 current_user = Depends(get_current_user)。
📖 核心概念 ▼

JWT（JSON Web Token）是一个开放标准（RFC 7519），由三部分用 . 分隔组成：

┌──────────────────────────────────────────────────────────────────────┐
│                        JWT 完整结构                                   │
├──────────────┬─────────────────────────────────┬────────────────────┤
│   Header     │          Payload                │     Signature      │
│   (头部)     │          (载荷)                 │      (签名)        │
│              │                                 │                    │
│ {            │ {                               │ HMACSHA256(        │
│  "alg":"HS256",│  "sub":"1234567890",            │  base64(header)+  │
│  "typ":"JWT" │  "username":"alice",            │  "."+             │
│ }            │  "role":"admin",                │  base64(payload),  │
│              │  "exp":1700000000,              │  secret_key        │
│  → Base64URL │  "iat":1699996400               │ )                  │
│    编码      │ }                               │                    │
│              │  → Base64URL 编码（非加密！）    │  → Base64URL 编码   │
└──────────────┴─────────────────────────────────┴────────────────────┘

标准 Payload 字段（Registered Claims）

字段全称说明
subSubject用户唯一标识（通常是用户 ID）
expExpiration Time过期时间（Unix 时间戳）— 必须检查！
iatIssued At签发时间
issIssuer签发者
nbfNot Before生效时间（此时间前不接收）
jtiJWT ID唯一标识，可用于防重放攻击
双 Token 机制详解（Access + Refresh）
问题：Token 有效期太长不安全（泄露后长时间可用），太短则用户频繁重新登录体验差。
解决：
· Access Token：有效期短（15分钟~2小时），用于访问 API
· Refresh Token：有效期长（7~30天），仅用于在 Access Token 过期后刷新新的 Access Token
流程：Access Token 过期 → 客户端自动用 Refresh Token 请求 /auth/refresh → 服务器验证 Refresh Token → 返回新的 Access Token → 用户无感知
JWT 安全注意事项（展开查看完整清单）
1. Payload 不是加密的——只是 Base64URL 编码，任何人都能解码，绝不可存敏感信息
2. 必须 HTTPS 传输——防止中间人窃取完整 Token
3. 短有效期——Access Token 推荐 15分钟~2小时，降低泄露风险窗口
4. 强密钥——secret_key 至少 32 位随机字符串，从环境变量读取，不写死在代码中
5. 必须验证签名——不要只解码 Payload 就用，签名验证是防篡改的唯一屏障
💻 代码示例 ▼

Part 1：手动 Base64URL 解码 Payload（证明 Payload 可被任何人读取）

def b64_decode(data: str) -> dict:
    padding = 4 - len(data) % 4
    if padding != 4:
        data += "=" * padding
    decoded_bytes = base64.urlsafe_b64decode(data)
    return json.loads(decoded_bytes)

# ⚠️ 任何人都能解码 Payload！绝不可放密码等敏感信息
payload = b64_decode("eyJzdWIiOiIxMjM0NTYiLCJ1c2VybmFtZSI6ImFsaWNlIn0")
# → {"sub": "123456", "username": "alice"}  一目了然！

Part 2：手动实现 HS256 签名（理解防篡改原理）

def create_jwt_manually(payload, secret):
    header_enc = base64url_encode(json.dumps({"alg":"HS256","typ":"JWT"}).encode())
    payload_enc = base64url_encode(json.dumps(payload).encode())

    # 签名 = HMACSHA256(header.payload, secret_key)
    signing_input = f"{header_enc}.{payload_enc}"
    signature = hmac.new(secret.encode(), signing_input.encode(), hashlib.sha256).digest()
    signature_enc = base64url_encode(signature)

    return f"{header_enc}.{payload_enc}.{signature_enc}"

def verify_jwt_manually(token, secret):
    parts = token.split(".")
    signing_input = f"{parts[0]}.{parts[1]}"
    expected_sig = base64url_encode(
        hmac.new(secret.encode(), signing_input.encode(), hashlib.sha256).digest()
    )
    # 防时序攻击的比较方式
    if not hmac.compare_digest(expected_sig, parts[2]):
        return None  # 签名不匹配，Token 被篡改！
    payload = b64_decode(parts[1])
    if "exp" in payload and time.time() > payload["exp"]:
        return None  # Token 已过期
    return payload

Part 3：PyJWT 生产环境用法 — 双 Token 机制

import jwt

def create_access_token(user_id, username, role):
    payload = {
        "sub": str(user_id), "username": username,
        "role": role, "type": "access",
        "exp": datetime.now(timezone.utc) + timedelta(hours=2),
    }
    return jwt.encode(payload, SECRET, algorithm="HS256")

def verify_token(token, expected_type="access"):
    payload = jwt.decode(token, SECRET, algorithms=["HS256"])
    if payload.get("type") != expected_type:
        raise ValueError("Token 类型错误")
    return payload
⚠️ 常见误区 ▼
❌ 误区✅ 纠正
"JWT 是加密的，Payload 内容安全"Payload 只是 Base64URL 编码，不是加密！任何人都能解码读取。只有签名保证完整性（防篡改）
"JWT 登出就是客户端删除 Token"在有效期内 Token 仍然可用，真正登出需服务端黑名单（Redis 存已失效 Token，设置 TTL 与原 Token 过期时间一致）
"JWT 越大越安全，Payload 多放信息"Payload 越大每次请求携带数据越多，影响性能。只放必要信息（sub/role/exp），业务详情从数据库查
"用对称加密（HS256）就够了"多服务场景需共享密钥有泄露风险，可考虑 RS256（非对称：私钥签发、公钥验证）
🔴 安全警告：如果 JWT 的 secret_key 设置为 "secret" 或 "123456" 等弱密钥，攻击者可以自己签发任意用户 Token 绕过认证。生产环境务必使用 32+ 位随机字符串。

第三节 · 密码哈希：bcrypt 与 argon2

永远不要明文存储密码。密码哈希是用户数据安全的最后防线。
🎯 学习目标 ▼
  • 理解为什么 MD5/SHA1 不能用于密码存储
  • 掌握 bcrypt 加盐哈希的工作原理
  • 了解 argon2 相比 bcrypt 的优势
  • 能使用 passlib 统一接口处理密码
  • 理解"哈希慢是优点"的安全逻辑
与后端开发的关联：用户注册/登录是后端最常见的需求。FastAPI + passlib + bcrypt 是最佳组合：注册时 pwd_context.hash(password)，登录时 pwd_context.verify(password, hash)——两行代码搞定密码安全。
📖 核心概念 ▼

什么是密码哈希？哈希是单向函数——输入任意数据，输出固定长度摘要，无法反推。密码哈希额外要求：慢 + 加盐。

彩虹表攻击原理

攻击者预先计算常见密码的 MD5：
  MD5("123456")    → "e10adc3949ba59abbe56e057f20f883e"
  MD5("password")  → "5f4dcc3b5aa765d61d8327deb882cf99"
  MD5("qwerty")    → "d8578edf8458ce06fbc5bb76a58c5ca4"
  ...（预计算数百亿条）

数据库泄露 → 攻击者拿到 MD5 哈希 → 查彩虹表 → 秒出原始密码！

防御方法：每次哈希加入随机盐（Salt），即使相同密码每次结果也不同，
彩虹表彻底失效——因为要针对每个可能的盐都预计算一次。

bcrypt vs argon2 对比

维度bcryptargon2
设计年代1999 年2015 年（密码哈希竞赛冠军）
内存占用固定（很小）可配置（推荐高内存，如 64MB）
抗 GPU 攻击一般强（高内存使 GPU 并行无效）
参数灵活性只有 cost（轮次）time、memory、parallelism 三维控制
推荐场景大多数 Web 应用高安全要求、新项目
bcrypt cost factor 详解（展开查看）
cost 控制迭代次数 = 2^cost 次：
cost=10 → 2^10 = 1,024 轮 → ~100ms（最低推荐）
cost=12 → 2^12 = 4,096 轮 → ~300ms（推荐值，安全与性能平衡）
cost=14 → 2^14 = 16,384 轮 → ~1s（高安全要求）
随着 CPU 性能提升，可以逐步调高 cost 值，passlib 的 needs_update() 可自动检测并重新哈希。

Part 2：✅ bcrypt 正确使用

import bcrypt

def hash_password(plain_password: str, rounds: int = 12) -> str:
    salt = bcrypt.gensalt(rounds=rounds)  # 生成随机盐
    hashed = bcrypt.hashpw(plain_password.encode("utf-8"), salt)
    return hashed.decode("utf-8")

def verify_password(plain_password: str, hashed_password: str) -> bool:
    return bcrypt.checkpw(
        plain_password.encode("utf-8"),
        hashed_password.encode("utf-8")
    )

# 演示：相同密码，每次哈希结果不同（因为盐是随机的！）
hash1 = hash_password("my_secret")  # $2b$12$R4xC...（盐=R4xC）
hash2 = hash_password("my_secret")  # $2b$12$Z9qA...（盐=Z9qA）
# hash1 ≠ hash2，彩虹表彻底失效！

Part 3：passlib 统一接口（生产环境推荐）

from passlib.context import CryptContext

pwd_context = CryptContext(schemes=["bcrypt"], deprecated="auto")

# 注册：哈希密码存入数据库
hashed = pwd_context.hash("user_password_123")

# 登录：验证密码（自动从哈希中提取盐）
is_valid = pwd_context.verify("user_password_123", hashed)  # True

# 升级：检查是否需要重新哈希（cost 参数升级后自动迁移）
if pwd_context.needs_update(hashed):
    new_hash = pwd_context.hash(plain_password)  # 用新参数重新哈希

Part 4：argon2 使用（新项目推荐）

from argon2 import PasswordHasher
from argon2.exceptions import VerifyMismatchError

ph = PasswordHasher(
    time_cost=2,         # 迭代次数（时间成本）
    memory_cost=65536,   # 64MB 内存（抗 GPU 关键！）
    parallelism=2,       # 并行度
)

hashed = ph.hash("my_password")
# → "$argon2id$v=19$m=65536,t=2,p=2$..."

try:
    ph.verify(hashed, "my_password")  # 正确 → 不抛异常
except VerifyMismatchError:
    print("密码错误")
⚠️ 常见误区 ▼
❌ 误区✅ 纠正
"MD5 加个固定盐就安全了"手动加固定盐的 MD5 依然很快，GPU 可暴力破解。必须用专门的慢哈希算法（bcrypt/argon2）
"SHA256 比 MD5 更安全，所以可以存密码"对密码存储而言，SHA256 和 MD5 面临同样问题——都太快了。安全在于"慢"而非"长"
"bcrypt cost=12 表示 12 次迭代"错！cost 是指数：cost=12 实际是 2^12 = 4,096 次迭代
"每次登录都重新哈希太慢了"设计如此：每次验证几百毫秒用户无感，但攻击者暴力破解代价极高——这正是安全目标
"可以把原始密码打印到日志调试"绝对不行！日志可能被多人查看、存储、备份。禁止在任何日志中记录明文密码

📖 核心概念 ▼

OAuth2.0 是授权框架，解决"如何让第三方应用访问你的数据，而不需要把密码给它"。

四个角色

授权码模式完整流程

 用户 → [1.点击登录] → 客户端 → [2.重定向到 GitHub]
                                    ↓
 用户 → [3.跳转 GitHub 登录并同意授权] → 授权服务器
                                    ↓
 授权服务器 → [4.重定向回客户端 + authorization_code]
                                    ↓
 客户端 → [5.用 code + client_secret 换 access_token] → 授权服务器
                                    ↓
 客户端 → [6.用 access_token 获取用户信息] → 资源服务器
                                    ↓
 客户端 ← [7.拿到用户信息，创建本地用户，颁发自己的 JWT]

关键设计：code 通过浏览器 URL 传递（可被记录），是一次性且短期的；
access_token 的换取需要 client_secret（只有服务端知道），多加一道安全验证。

四种授权模式

模式适用场景安全性
授权码模式有服务端的 Web 应用（最常用）⭐⭐⭐⭐⭐
简化模式（Implicit）纯前端应用（已不推荐）⭐⭐

为什么授权码要通过浏览器 URL 传递，而 access_token 不这样？（展开）
授权码（authorization_code）通过浏览器 URL 传递，可能被浏览器历史记录、服务器日志、Referer 头等泄露。但它是一次性的（用完即失效）且有效期极短（通常 10 分钟），即使泄露也无法重复使用。

Access Token是长期有效的凭证，如果通过 URL 传递风险太大。所以设计为两步：先用 URL 传低风险的一次性 code，再用 code + client_secret（只有服务端有）在服务端之间交换高风险的 access_token——用户和浏览器全程看不到 access_token 的换取过程。
💻 代码示例 ▼

Part 1：构建授权 URL + state 防 CSRF

def step1_build_authorization_url(self, scope, auth_server_url):
    state = secrets.token_urlsafe(32)  # 随机 state，防 CSRF
    self._state_store[state] = {"created_at": time.time(), "scope": scope}

    params = {
        "client_id": self.client_id,
        "redirect_uri": self.redirect_uri,
        "scope": " ".join(scope),
        "state": state,           # 携带到授权服务器
        "response_type": "code",
    }
    return f"{auth_server_url}/oauth/authorize?{urlencode(params)}"

Part 2：回调处理 — 验证 state + 用 code 换 Token

def step3_handle_callback(self, callback_url, auth_server):
    params = parse_qs(urlparse(callback_url).query)
    code = params.get("code", [None])[0]
    state = params.get("state", [None])[0]

    # 关键：验证 state 防 CSRF 攻击！
    if state not in self._state_store:
        raise ValueError("❌ state 不匹配，可能是 CSRF 攻击！拒绝处理！")
    del self._state_store[state]  # 使用后删除，防重放

    # 用 code + client_secret 换取 access_token（服务端操作）
    token_data = auth_server.exchange_code_for_token(
        code=code, client_id=self.client_id,
        client_secret=self.client_secret,  # 只在服务端！绝不暴露！
        redirect_uri=self.redirect_uri,
    )

    # 用 access_token 获取用户信息
    user_info = auth_server.get_user_info(token_data["access_token"], "read:user")
    return {"token_data": token_data, "user_info": user_info}

Part 3：Scope 权限控制 — 最小权限原则

# ❌ 坏做法：申请过多权限（用户不信任，拒绝授权）
scope = ["admin:org", "repo", "delete_repo"]

# ✅ 好做法：只申请需要的权限（最小权限原则）
scope = ["read:user", "user:email"]  # 用 GitHub 登录只需这些

# 服务端验证 scope
def get_user_info(access_token, required_scope):
    token_data = self._access_tokens.get(access_token)
    if required_scope not in token_data["scope"]:
        raise PermissionError(f"token 没有 {required_scope} 权限")
⚠️ 常见误区 ▼
❌ 误区✅ 纠正
"OAuth2 是认证协议"OAuth2 是授权框架，不是认证协议。认证是 OpenID Connect（基于 OAuth2 之上构建的身份层）
"client_secret 可以放在前端代码"绝对不行！前端代码所有人可见。client_secret 必须只在服务端使用
"authorization_code 可以重复使用"授权码是一次性的，使用后立即失效，且有效期很短（通常 10 分钟）
"scope 多多益善"最小权限原则：只申请实际需要的权限。过多 scope 增加安全风险，用户也更不愿意授权
🔴 安全警告：如果 state 参数不验证或设为固定值，攻击者可以构造恶意链接诱导用户点击，将自己的第三方账号绑定到受害者的本地账号上——这叫 CSRF 登录劫持。每次 OAuth 授权都必须生成随机 state 并验证！
