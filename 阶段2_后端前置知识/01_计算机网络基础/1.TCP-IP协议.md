# TCP/IP 协议

> 后端开发的一切都建立在网络通信之上，TCP/IP 是这座大厦的地基。

---

## 一、概念解释

### 什么是 TCP/IP？

TCP/IP 不是一个协议，而是一组协议的统称（协议族），定义了计算机之间如何通信。名字来源于其中最核心的两个协议：

- **TCP**（Transmission Control Protocol）：传输控制协议，负责可靠传输
- **IP**（Internet Protocol）：网际协议，负责寻址和路由

### 四层模型

TCP/IP 把网络通信分成四层，每层各司其职：

```
┌─────────────────┐
│   应用层          │ ← HTTP、DNS、FTP、SMTP（你写的代码在这层）
├─────────────────┤
│   传输层          │ ← TCP、UDP（决定数据怎么传）
├─────────────────┤
│   网络层          │ ← IP（决定数据发到哪）
├─────────────────┤
│   网络接口层      │ ← 以太网、Wi-Fi（物理传输）
└─────────────────┘
```

**后端视角**：你写的 FastAPI / Flask 代码工作在应用层，框架底层帮你处理了传输层（TCP）和网络层（IP），你只需要关注"发什么数据、收什么数据"。

## 三、核心知识点

### 1. TCP 三次握手（建立连接）

TCP 是面向连接的协议，通信前必须先建立连接。三次握手的过程：

```
客户端                          服务器
  │                               │
  │ ──── SYN (seq=x) ──────────> │  第1次：客户端说"我要连接"
  │                               │
  │ <──── SYN+ACK (seq=y,ack=x+1) │  第2次：服务器说"好的，我也准备好了"
  │                               │
  │ ──── ACK (ack=y+1) ────────> │  第3次：客户端说"确认，开始传数据"
  │                               │
  │ ======= 连接建立，开始传输 ====== │
```

**为什么是三次，不是两次？**
- 两次握手无法确认客户端的接收能力。如果客户端发了一个"过期的连接请求"（网络延迟导致），服务器两次握手就会误建连接，浪费资源。
- 三次握手确保双方都确认了对方的发送和接收能力。

**面试口诀**：SYN → SYN+ACK → ACK

### 2. TCP 四次挥手（断开连接）

断开连接比建立连接多一步，因为 TCP 是全双工的（双方可以同时发送数据），需要分别关闭两个方向：

```
客户端                          服务器
  │                               │
  │ ──── FIN ──────────────────> │  第1次：客户端说"我发完了"
  │                               │
  │ <──── ACK ────────────────── │  第2次：服务器说"知道了，我可能还有数据要发"
  │                               │
  │    （服务器继续发送剩余数据）      │
  │                               │
  │ <──── FIN ────────────────── │  第3次：服务器说"我也发完了"
  │                               │
  │ ──── ACK ──────────────────> │  第4次：客户端说"好的，断开"
  │                               │
  │ ======= 连接关闭 ============= │
```

**为什么是四次，不是三次？**
- 收到 FIN 后，服务器可能还有数据没发完，所以 ACK 和 FIN 分开发送。
- 如果服务器刚好也没数据了，实际上可以合并为三次（FIN+ACK 一起发）。

**关键概念：TIME_WAIT**
- 客户端发出最后一个 ACK 后，不会立即关闭，而是等待 2MSL（约 60 秒）
- 目的：确保服务器收到了最后的 ACK，如果没收到，服务器会重发 FIN
- 后端踩坑：高并发短连接场景下，大量 TIME_WAIT 会占满端口

### 3. TCP vs UDP

| 特性 | TCP | UDP |
|------|-----|-----|
| 连接方式 | 面向连接（三次握手） | 无连接（直接发） |
| 可靠性 | 可靠（确认重传机制） | 不可靠（发了就不管） |
| 有序性 | 保证顺序 | 不保证顺序 |
| 速度 | 较慢（有确认开销） | 更快（无额外开销） |
| 适用场景 | HTTP、数据库连接、文件传输 | DNS 查询、视频直播、游戏 |
| 头部大小 | 20 字节 | 8 字节 |

**后端开发中的选择**：
- **99% 的场景用 TCP**：HTTP API、数据库连接、消息队列，都需要可靠传输
- **UDP 的典型场景**：DNS 查询（快速、数据量小）、视频/音频流（丢几帧无所谓）、游戏（延迟比丢包更致命）

### 4. TCP 的可靠性保证机制

TCP 不只是"能传数据"，它保证数据**完整、有序、不重复**地到达：

| 机制 | 作用 |
|------|------|
| 序列号（Sequence Number） | 给每个字节编号，保证顺序 |
| 确认应答（ACK） | 接收方告诉发送方"收到了" |
| 超时重传 | 发出去没收到 ACK，就重新发 |
| 滑动窗口 | 控制发送速度，不让接收方被淹没 |
| 拥塞控制 | 感知网络状况，避免网络堵塞 |

## 四、实际场景

### 场景 1：浏览器输入 URL 后发生了什么？

```
1. DNS 解析：域名 → IP 地址                    （应用层/UDP）
2. TCP 三次握手：建立与服务器的连接              （传输层/TCP）
3. TLS 握手：如果是 HTTPS，协商加密              （传输层）
4. 发送 HTTP 请求                               （应用层）
5. 服务器处理请求，返回 HTTP 响应                （应用层）
6. 浏览器渲染页面                               （客户端）
7. TCP 四次挥手：断开连接                        （传输层）
```

### 场景 2：数据库连接

```python
# 当你写 SQLAlchemy 连接数据库时，底层发生了什么：
# 1. TCP 三次握手：Python 进程与 MySQL 服务器建立 TCP 连接
# 2. MySQL 握手协议：在 TCP 连接之上，交换版本、认证信息
# 3. 执行 SQL：在已建立的连接上发送 SQL 命令、接收结果
# 4. 连接池：复用 TCP 连接，避免频繁握手的开销
```

## 五、常见误区

| 误区 | 纠正 |
|------|------|
| "TCP 比 UDP 好" | 没有好坏之分，只有适合不适合。视频直播用 TCP 反而体验更差 |
| "三次握手可以简化成两次" | 不行，两次无法防止历史重复连接，会导致资源浪费 |
| "TCP 连接是物理连接" | TCP 连接是逻辑概念，靠双方维护状态实现，不是一根"线" |
| "UDP 不安全" | UDP 不可靠 ≠ 不安全，安全性由上层协议（如 DTLS）保障 |

## 六、面试高频题

**Q：TCP 三次握手的过程？**
A：客户端发 SYN → 服务器回 SYN+ACK → 客户端发 ACK。三次确保双方的收发能力都正常。

**Q：为什么 TCP 握手是三次，挥手是四次？**
A：握手时服务器的 SYN+ACK 可以合并发送；挥手时服务器收到 FIN 后可能还有数据未发完，ACK 和 FIN 必须分开。

**Q：TCP 和 UDP 的区别？什么时候用 UDP？**
A：TCP 面向连接、可靠有序；UDP 无连接、速度快但不可靠。DNS 查询、视频直播、游戏用 UDP。

**Q：什么是 TIME_WAIT？为什么存在？**
A：主动关闭连接的一方在发出最后一个 ACK 后等待 2MSL。目的：确保对方收到了最后的 ACK，防止历史数据包干扰新连接。
