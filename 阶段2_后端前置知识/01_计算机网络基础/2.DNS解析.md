# DNS 解析

> 你输入 `google.com`，但计算机只认 IP 地址。DNS 就是互联网的"电话簿"。

---

## 一、概念解释

### 什么是 DNS？

**DNS**（Domain Name System，域名系统）是一个分布式的"域名 → IP 地址"翻译系统。

```
人类友好：  www.google.com
计算机友好：142.250.80.4

DNS 的工作 = 把前者翻译成后者
```

### 为什么不直接用 IP？

- IP 地址是数字，人记不住（`142.250.80.4` vs `google.com`）
- 域名可以不变，背后的 IP 可以随时切换（服务器迁移、负载均衡）
- 一个域名可以对应多个 IP（CDN、多机房部署）

## 二、为什么需要了解 DNS？

| 场景 | 为什么需要 |
|------|-----------|
| 部署上线 | 需要配置域名解析（A 记录、CNAME） |

| CDN 配置 | CDN 加速依赖 CNAME 域名解析 |
| 微服务 | 服务发现本质上就是 DNS 的变种 |

## 三、核心知识点

### 1. DNS 记录类型

| 记录类型 | 作用 | 示例 |
|----------|------|------|
| **A** | 域名 → IPv4 地址 | `api.example.com → 1.2.3.4` |
| **AAAA** | 域名 → IPv6 地址 | `api.example.com → 2001:db8::1` |
| **CNAME** | 域名 → 另一个域名（别名） | `www.example.com → example.com` |
| **MX** | 邮件服务器地址 | `example.com → mail.example.com` |
| **TXT** | 文本记录（域名验证等） | `example.com → "v=spf1 ..."` |
| **NS** | 指定域名服务器 | `example.com → ns1.dnspod.net` |

**后端最常用**：A 记录（部署时绑定服务器 IP）和 CNAME（CDN / 负载均衡配置）。

### 2. DNS 解析流程

当你在浏览器输入 `api.example.com` 时，解析经历以下步骤：

```
1. 浏览器缓存        → 看看之前查过没？有就直接用
       ↓ 没有
2. 操作系统缓存      → 系统 DNS 缓存 / hosts 文件
       ↓ 没有
3. 本地 DNS 服务器   → 运营商或公司的 DNS（如 8.8.8.8）
       ↓ 没有
4. 根域名服务器      → "我不知道，但 .com 的域名去问 .com 服务器"
       ↓
5. 顶级域名服务器    → ".com 服务器说：example.com 去问 ns1.example.com"
       ↓
6. 权威域名服务器    → "api.example.com 的 IP 是 1.2.3.4"
       ↓
7. 本地 DNS 缓存结果，返回给浏览器
```

**关键特征**：
- **递归查询**：客户端只问一次本地 DNS，本地 DNS 帮你跑完全程
- **迭代查询**：本地 DNS 依次问根→顶级→权威服务器，每次获得下一步线索
- **缓存机制**：每级都有缓存，TTL（Time To Live）决定缓存时间

### 3. hosts 文件

操作系统本地的"手动 DNS"，优先级高于远程 DNS 查询：

```
# Windows: C:\Windows\System32\drivers\etc\hosts
# Linux/Mac: /etc/hosts

127.0.0.1    localhost
127.0.0.1    myapp.local        # 开发环境自定义域名
192.168.1.100  db.internal      # 内网数据库别名
```

**后端用途**：
- 开发环境模拟域名（不用真的买域名）
- 内网服务起别名
- 临时绕过 DNS 解析（排查问题时指向特定 IP）

### 4. 常用公共 DNS

| DNS 服务器 | IP 地址 | 特点 |

## 四、实际场景

### 场景 1：部署后域名无法访问

```
排查步骤：
1. nslookup api.example.com     → DNS 是否解析到正确 IP？
2. ping 1.2.3.4                 → 服务器网络是否通？
3. curl http://1.2.3.4:8000     → 直接访问 IP，服务是否正常？
4. 检查 DNS 记录配置            → A 记录/CNAME 是否正确？
5. 等待 DNS 传播                → 新记录可能需要几分钟到 48 小时生效
```

### 场景 2：开发环境配置

```python
# 本地开发时，API 地址通常用环境变量：
# 开发环境：API_HOST=localhost
# 测试环境：API_HOST=api-test.example.com
# 生产环境：API_HOST=api.example.com
```

## 五、常见误区

| 误区 | 纠正 |
|------|------|
| "DNS 修改后立即生效" | DNS 有缓存和传播时间，TTL 短则几分钟，长则 48 小时 |
| "DNS 只做域名到 IP 的翻译" | 还有 CNAME、MX、TXT 等多种记录类型 |
| "DNS 用的是 TCP" | DNS 默认使用 UDP（快速、数据量小），超过 512 字节或区域传送才用 TCP |
| "hosts 文件没用了" | 开发环境中非常有用，且优先级高于 DNS 查询 |

## 六、面试高频题

**Q：DNS 解析的完整流程？**
A：浏览器缓存 → 系统缓存/hosts → 本地 DNS 服务器 → 根域名服务器 → 顶级域名服务器 → 权威域名服务器。本地 DNS 做递归查询，向上游做迭代查询。

**Q：DNS 用的是 TCP 还是 UDP？**
A：默认 UDP（端口 53），因为 DNS 查询数据小、追求速度。响应超过 512 字节或 DNS 区域传送时使用 TCP。

**Q：如何加速 DNS 解析？**
A：使用更快的公共 DNS、设置合理的 TTL、利用 DNS 预解析（`<link rel="dns-prefetch">`）、CDN 就近解析。
