# HTTP 方法与状态码

> HTTP 方法表达“想做什么”，状态码表达“结果怎么样”。这是 RESTful API 设计的基础。

---

## 一、概念解释

### HTTP 方法是什么？

HTTP 方法（Method）表示客户端对资源想执行的动作。

```http
GET /users/1 HTTP/1.1
POST /users HTTP/1.1
DELETE /users/1 HTTP/1.1
```

同一个 URL，使用不同方法可以表达不同操作。

### HTTP 状态码是什么？

状态码是服务器对请求处理结果的标准化说明。

```http
HTTP/1.1 404 Not Found
```

`404` 告诉客户端：你请求的资源不存在。

---

## 二、为什么需要方法和状态码？

| 场景 | 作用 |
|------|------|
| API 设计 | 用标准方法表达 CRUD，接口语义清晰 |
| 前后端联调 | 前端根据状态码判断成功、失败、未登录、无权限 |
| 日志排查 | 通过大量 `4xx` / `5xx` 判断问题来源 |
| 网关监控 | 统计接口成功率、错误率、延迟 |

---

## 三、常见 HTTP 方法

| 方法 | 语义 | 常见用途 | 是否应幂等 |
|------|------|----------|------------|
| `GET` | 获取资源 | 查询列表、查询详情 | 是 |
| `POST` | 创建资源 / 提交动作 | 注册、登录、创建订单 | 否 |
| `PUT` | 全量更新资源 | 替换用户资料 | 是 |
| `PATCH` | 局部更新资源 | 修改昵称、修改状态 | 通常是 |
| `DELETE` | 删除资源 | 删除文章、删除商品 | 是 |

### 幂等是什么意思？

**幂等**：同一个请求执行一次和执行多次，最终结果相同。

```text
GET /users/1        查询 1 次和查询 10 次，资源不变 → 幂等
DELETE /users/1     删除 1 次和删除 10 次，最终都是不存在 → 幂等
POST /orders        提交 1 次创建 1 个订单，提交 10 次可能创建 10 个订单 → 非幂等
```

## 四、常见状态码

### 2xx：成功

| 状态码 | 含义 | 场景 |
|--------|------|------|
| `200 OK` | 请求成功 | 查询成功、更新成功 |
| `201 Created` | 创建成功 | 新建用户、创建订单 |
| `204 No Content` | 成功但无响应体 | 删除成功 |

### 3xx：重定向

| 状态码 | 含义 | 场景 |
|--------|------|------|
| `301 Moved Permanently` | 永久重定向 | 域名永久迁移 |
| `302 Found` | 临时重定向 | 临时跳转登录页 |
| `304 Not Modified` | 资源未修改 | 浏览器缓存命中 |

### 4xx：客户端错误

| 状态码 | 含义 | 场景 |
|--------|------|------|
| `400 Bad Request` | 请求格式错误 | JSON 格式错误、参数非法 |
| `401 Unauthorized` | 未认证 | 没登录、Token 缺失 |
| `403 Forbidden` | 无权限 | 登录了但不能访问 |
| `404 Not Found` | 资源不存在 | 用户 ID 不存在 |
| `409 Conflict` | 资源冲突 | 用户名已存在、重复提交 |
| `422 Unprocessable Entity` | 语义校验失败 | FastAPI 参数校验失败 |
| `429 Too Many Requests` | 请求过多 | 触发限流 |

### 5xx：服务器错误

| 状态码 | 含义 | 场景 |
|--------|------|------|
| `500 Internal Server Error` | 服务端内部错误 | 未捕获异常 |
| `502 Bad Gateway` | 网关收到错误响应 | Nginx 后端服务异常 |
| `503 Service Unavailable` | 服务不可用 | 服务维护、过载 |
| `504 Gateway Timeout` | 网关超时 | 后端处理太慢 |

## 五、实际场景

### 用户资源的 RESTful 映射

| 操作 | 方法 + URL | 推荐状态码 |
|------|------------|------------|
| 查询用户列表 | `GET /users` | `200` |
| 查询单个用户 | `GET /users/1` | `200` / `404` |
| 创建用户 | `POST /users` | `201` |
| 全量更新用户 | `PUT /users/1` | `200` |
| 局部更新用户 | `PATCH /users/1` | `200` |
| 删除用户 | `DELETE /users/1` | `204` |

---

## 六、常见误区

| 误区 | 纠正 |
|------|------|
| “所有成功都返回 200” | 创建资源更推荐 `201`，删除无内容更推荐 `204` |
| “未登录返回 403” | 未认证是 `401`，认证后无权限才是 `403` |
| “业务错误都返回 500” | 参数错误、资源不存在、权限问题属于 `4xx` |
| “POST 比 GET 安全” | POST 参数不显示在 URL 中，但没有 HTTPS 仍然可能被窃听 |

## 七、面试高频题

**Q：GET 和 POST 的区别？**  
A：GET 用于获取资源，参数通常放 URL 中，应幂等；POST 用于创建资源或提交动作，请求体通常携带数据，通常不幂等。

**Q：401 和 403 的区别？**  
A：401 表示未认证；403 表示已认证但无权限。

**Q：什么时候用 PUT，什么时候用 PATCH？**  
A：PUT 表示全量替换资源；PATCH 表示局部更新资源。
