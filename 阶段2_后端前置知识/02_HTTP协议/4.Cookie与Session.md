# Cookie 与 Session

> HTTP 是无状态的，Cookie 和 Session 用来让服务器“记住你是谁”。

---

## 一、概念解释

### HTTP 无状态是什么意思？

HTTP 协议本身不会记住上一次请求是谁发的。

```text
第 1 次请求：登录成功
第 2 次请求：访问个人中心
```

如果没有额外机制，服务器无法天然知道第 2 次请求来自刚才登录过的用户。

### Cookie 是什么？

Cookie 是服务器让浏览器保存的一小段数据。之后浏览器访问同一站点时，会自动带上 Cookie。

```http
Set-Cookie: session_id=abc123; HttpOnly; Path=/
```

浏览器后续请求：

```http
Cookie: session_id=abc123
```

### Session 是什么？

Session 是服务器端保存的用户状态。浏览器只保存一个 `session_id`，真正的数据放在服务器。

```text
浏览器 Cookie: session_id=abc123
服务器 Session 存储:
abc123 -> {"user_id": 1, "username": "alice"}
```

---

## 二、为什么需要 Cookie / Session？

| 场景 | 作用 |
|------|------|
| 登录状态 | 记住用户已经登录 |
| 购物车 | 未登录或已登录购物车状态保存 |
| CSRF 防护 | Cookie + Token 校验请求来源 |
| 个性化设置 | 语言、主题等用户偏好 |

---

## 三、Cookie 工作流程

```text
1. 用户登录
2. 服务器验证用户名密码
3. 服务器生成 session_id
4. 响应头返回 Set-Cookie: session_id=xxx
5. 浏览器保存 Cookie
6. 后续请求自动带上 Cookie: session_id=xxx
7. 服务器根据 session_id 查询用户状态
```

---

## 四、Cookie 常用属性

| 属性 | 作用 | 建议 |
|------|------|------|
| `HttpOnly` | 禁止 JavaScript 读取 Cookie | 登录态 Cookie 必加 |
| `Secure` | 只允许 HTTPS 发送 Cookie | 生产环境必加 |
| `SameSite` | 限制跨站携带 Cookie | 推荐 `Lax` 或 `Strict` |
| `Max-Age` / `Expires` | 过期时间 | 根据业务设置 |
| `Path` | Cookie 生效路径 | 通常为 `/` |
| `Domain` | Cookie 生效域名 | 谨慎设置父域名 |

### 安全 Cookie 示例

```http
Set-Cookie: session_id=abc123; HttpOnly; Secure; SameSite=Lax; Path=/
```

---

## 五、Session 的存储位置

| 存储方式 | 特点 | 场景 |
|----------|------|------|
| 内存 | 快，但服务重启丢失，无法多实例共享 | 本地学习 Demo |
| Redis | 快，可过期，多实例共享 | 生产常用 |
| 数据库 | 持久化强，但性能较低 | 管理后台、低频场景 |

生产环境中，Session 常放 Redis，因为它支持过期时间和多服务实例共享。

---

## 六、Cookie / Session vs Token

| 方案 | 状态保存位置 | 常见用途 |
|------|--------------|----------|
| Cookie + Session | 服务端保存状态 | 传统 Web 登录 |
| JWT Token | 客户端保存令牌，服务端校验签名 | 前后端分离、移动端 API |

两者不是绝对替代关系：Cookie 是浏览器存储和自动携带机制，Session / JWT 是身份状态方案。

---

## 七、常见误区

| 误区 | 纠正 |
|------|------|
| “Cookie 和 Session 是一回事” | Cookie 在客户端，Session 在服务端 |
| “Cookie 一定不安全” | 正确设置 `HttpOnly`、`Secure`、`SameSite` 后可用于安全登录态 |
| “Session 不需要存储” | Session 数据必须存储在内存、Redis 或数据库中 |
| “JWT 一定比 Session 好” | JWT 无法方便主动失效，Session 更适合强管控登录态 |

---

## 八、面试高频题

**Q：Cookie 和 Session 的区别？**  
A：Cookie 保存在客户端，Session 保存在服务端；客户端通常通过 Cookie 携带 `session_id`，服务端用它查询 Session 数据。

**Q：为什么 Cookie 要设置 HttpOnly？**  
A：防止 JavaScript 读取 Cookie，降低 XSS 窃取登录态的风险。

**Q：Session 在分布式部署中有什么问题？**  
A：如果 Session 只存在单机内存，多实例之间无法共享；通常用 Redis 集中存储。
