# HTTPS 与 CORS

> HTTPS 解决“传输是否安全”，CORS 解决“浏览器是否允许跨域访问”。

---

## 一、HTTPS 概念解释

### HTTPS 是什么？

HTTPS = HTTP + TLS 加密。

```text
HTTP：明文传输
HTTPS：HTTP 数据先经过 TLS 加密，再通过网络传输
```

HTTPS 主要解决三个问题：

| 目标 | 说明 |
|------|------|
| 加密 | 防止请求和响应内容被窃听 |
| 完整性 | 防止数据在传输中被篡改 |
| 身份认证 | 通过证书确认服务器身份，防止访问到假网站 |

---

## 二、HTTPS 为什么重要？

| 场景 | 风险 |
|------|------|
| 登录 | HTTP 明文传输密码，可能被窃听 |
| Token | HTTP 明文传输 Token，可能被盗用 |
| 支付 | 请求金额、订单可能被篡改 |
| Cookie | 没有 HTTPS 时，`Secure` Cookie 不会发送 |

生产环境中的登录、支付、用户隐私接口必须使用 HTTPS。

---

## 三、HTTPS 简化流程

```text
1. 客户端访问 https://example.com
2. 服务器返回 TLS 证书
3. 客户端验证证书是否可信、域名是否匹配、是否过期
4. 双方协商会话密钥
5. 后续 HTTP 数据使用会话密钥加密传输
```

真实 TLS 握手比这个更复杂，但后端学习阶段先掌握核心目的即可。

## 四、CORS 概念解释

### 什么是同源？

同源要求三个部分完全相同：

```text
协议 + 域名 + 端口
```

| URL | 是否同源 | 原因 |
|-----|----------|------|
| `https://api.example.com` vs `https://api.example.com` | 是 | 完全相同 |
| `http://api.example.com` vs `https://api.example.com` | 否 | 协议不同 |
| `https://api.example.com` vs `https://www.example.com` | 否 | 域名不同 |
| `https://api.example.com:443` vs `https://api.example.com:8443` | 否 | 端口不同 |

### 什么是 CORS？

**CORS**（Cross-Origin Resource Sharing，跨源资源共享）是浏览器的安全机制，用来限制网页脚本跨域请求接口。

```text
前端页面：https://frontend.example.com
后端 API：https://api.example.com

协议相同，但域名不同 → 跨域 → 浏览器检查 CORS 响应头
```

注意：CORS 是浏览器限制，不是服务器之间的限制。后端服务用 Python 请求另一个服务时，不受浏览器 CORS 限制。

---

## 五、CORS 核心响应头

| Header | 作用 | 示例 |
|--------|------|------|
| `Access-Control-Allow-Origin` | 允许哪个来源访问 | `https://frontend.example.com` |
| `Access-Control-Allow-Methods` | 允许哪些方法 | `GET, POST, PUT, DELETE` |
| `Access-Control-Allow-Headers` | 允许哪些请求头 | `Content-Type, Authorization` |
| `Access-Control-Allow-Credentials` | 是否允许携带 Cookie | `true` |
| `Access-Control-Max-Age` | 预检结果缓存时间 | `600` |

## 六、简单请求与预检请求

### 简单请求

满足特定条件的请求会直接发送，例如：

- `GET` / `POST` / `HEAD`
- Header 很简单
- `Content-Type` 是表单类类型

### 预检请求

复杂跨域请求前，浏览器会先发一个 `OPTIONS` 请求询问服务器是否允许。

```http
OPTIONS /api/users HTTP/1.1
Origin: https://frontend.example.com
Access-Control-Request-Method: POST
Access-Control-Request-Headers: Content-Type, Authorization
```

服务器允许后返回：

```http
HTTP/1.1 204 No Content
Access-Control-Allow-Origin: https://frontend.example.com
Access-Control-Allow-Methods: GET, POST
Access-Control-Allow-Headers: Content-Type, Authorization
```

然后浏览器才会发送真正的 `POST` 请求。

## 七、实际场景

### FastAPI 常见 CORS 配置

```python
from fastapi import FastAPI
from fastapi.middleware.cors import CORSMiddleware

app = FastAPI()

app.add_middleware(
    CORSMiddleware,
    allow_origins=["https://frontend.example.com"],
    allow_credentials=True,
    allow_methods=["GET", "POST", "PUT", "PATCH", "DELETE"],
    allow_headers=["Content-Type", "Authorization"],
)
```

### 重要安全点

如果需要携带 Cookie，不能这样配置：

```python
allow_origins=["*"]
allow_credentials=True
```

原因：浏览器不允许 `Access-Control-Allow-Origin: *` 与 `Access-Control-Allow-Credentials: true` 同时用于凭证请求。生产环境应明确列出可信前端域名。

## 八、常见误区

| 误区 | 纠正 |
|------|------|
| “CORS 是后端拒绝请求” | CORS 主要是浏览器拦截响应，服务端通常已经收到请求 |
| “Postman 没跨域问题，说明浏览器也没问题” | Postman 不受浏览器同源策略限制 |
| “HTTPS 只用于加密密码” | HTTPS 保护所有请求和响应，包括 Token、Cookie、个人数据 |
| “允许 `*` 最方便” | 生产环境应只允许可信来源，尤其是带 Cookie 的请求 |

## 九、面试高频题

**Q：HTTPS 解决了什么问题？**  
A：加密传输、防篡改、服务器身份认证。

**Q：什么是同源策略？**  
A：浏览器要求协议、域名、端口都相同才是同源，不同源的脚本请求会受到限制。

**Q：CORS 预检请求是什么？**  
A：浏览器在复杂跨域请求前先发送 `OPTIONS` 请求，询问服务器是否允许该跨域请求。
