# RESTful 设计原则

## 概念解释

REST（Representational State Transfer，表现层状态转移）是一种 **Web 架构风格**，由 Roy Fielding 在 2000 年博士论文中提出。REST 本身不是协议、不是规范，而是一组**设计约束**。

符合 REST 约束的 API，就叫做 **RESTful API**。

> 核心思想：**一切皆资源（Resource），通过 HTTP 方法操作资源**

---

## 为什么需要 RESTful 设计

### 没有规范时的 API 长什么样？

```
POST /getUserInfo
POST /createNewUser
POST /deleteUser
POST /updateUserName
GET  /getAllOrders
POST /doPayOrder
```

**问题**：
- URL 里充满动词（`get`, `create`, `delete`, `do`），命名不统一
- 全部用 POST，HTTP 方法的语义被浪费
- 接口数量随业务增长爆炸式增加
- 新人接手一脸懵

### RESTful 之后：

```
GET    /users/{id}        # 获取用户
POST   /users             # 创建用户
PATCH  /users/{id}        # 更新用户
DELETE /users/{id}        # 删除用户
GET    /orders            # 获取所有订单
POST   /orders/{id}/pay   # 支付订单（特殊操作）
```

**好处**：
- URL 语义清晰，一眼知道操作的是什么资源
- HTTP 方法承担动作职责（GET=查、POST=增、PUT/PATCH=改、DELETE=删）
- 接口数量有限，自带文档属性

---

## REST 六大约束

| 约束 | 说明 | 实际意义 |
|------|------|----------|
| **统一接口** | 资源通过固定接口暴露，HTTP 方法 + URL 唯一标识操作 | 降低学习成本，方便工具集成 |
| **无状态** | 服务端不存储客户端状态，每次请求携带完整信息 | 易横向扩展，请求可独立重试 |
| **缓存** | 响应标注是否可缓存，客户端/代理可缓存 | 减少重复请求，提升性能 |
| **客户端-服务端分离** | 前端只关心 UI，后端只关心数据，接口解耦 | 前后端独立演化 |
| **分层系统** | 客户端不感知中间层（CDN、网关、负载均衡） | 灵活扩展基础设施 |
| **按需代码**（可选） | 服务端可以下发可执行代码（如 JS）给客户端 | 少用，了解即可 |

## 资源导向 vs 动作导向

### 动作导向（反模式）

```
POST /api/sendEmail         # 动词 URL
POST /api/resetPassword     # 动词 URL
GET  /api/getBannerList     # 动词 URL
POST /api/doCheckout        # 动词 URL
```

**识别特征**：URL 里有动词（get/create/delete/do/handle/process）

### 资源导向（RESTful）

```
POST   /api/emails                    # 发送邮件 = 创建一条邮件记录
POST   /api/users/{id}/password-reset # 重置密码 = 创建一条重置请求
GET    /api/banners                   # 获取 Banner 列表
POST   /api/orders/{id}/checkout      # 结算 = 在订单上执行结算操作
```

**转换思路**：
1. 找出操作的**核心名词**（资源）
2. 用 **HTTP 方法**表达动作
3. 特殊操作用**子资源**或**动名词**兜底（`/actions/reset`）

## HTTP 方法与 CRUD 映射

| HTTP 方法 | CRUD | 含义 | 幂等？ | 示例 |
|-----------|------|------|--------|------|
| GET | Read | 读取资源 | ✅ 是 | `GET /users/1` |
| POST | Create | 创建资源 | ❌ 否 | `POST /users` |
| PUT | Update | **全量**更新（替换整个资源） | ✅ 是 | `PUT /users/1` |
| PATCH | Update | **部分**更新（只改传入字段） | ⚠️ 通常是 | `PATCH /users/1` |
| DELETE | Delete | 删除资源 | ✅ 是 | `DELETE /users/1` |

> **PUT vs PATCH 区别**：
> - PUT：传入完整对象，未传字段会被清空（覆盖语义）
> - PATCH：只传需要修改的字段，其余保持不变（部分更新语义）
> - 实战中 **PATCH 更常用**（更新接口很少需要全量传输）

## 集合与单个资源

```
GET    /users          # 资源集合：列出所有用户
POST   /users          # 资源集合：创建新用户

GET    /users/{id}     # 单个资源：获取指定用户
PUT    /users/{id}     # 单个资源：全量更新用户
PATCH  /users/{id}     # 单个资源：部分更新用户
DELETE /users/{id}     # 单个资源：删除用户
```

**规律**：同一个 URL `/users` 和 `/users/{id}` 通过 HTTP 方法分别表达"集合操作"和"单体操作"。

## 嵌套资源（层级关系）

当资源之间有明确的所属关系时，可以用嵌套 URL 表达：

```
GET  /users/{userId}/orders         # 获取用户的所有订单
POST /users/{userId}/orders         # 在该用户下创建订单
GET  /users/{userId}/orders/{orderId}  # 获取该用户的某个订单
```

> **注意**：嵌套不要超过 **2 层**，否则 URL 冗长难维护：
> ```
> ❌ /users/{uid}/orders/{oid}/items/{iid}/reviews/{rid}   # 太深
> ✅ /order-items/{itemId}/reviews                          # 扁平化
> ```

## 实际场景：用户+订单系统 URL 设计

| 操作 | 方法 | URL | 说明 |
|------|------|-----|------|
| 获取所有用户 | GET | `/users` | 支持分页/过滤 |
| 创建用户 | POST | `/users` | Body 含用户信息 |
| 获取指定用户 | GET | `/users/{id}` | |
| 更新用户昵称 | PATCH | `/users/{id}` | 只传修改字段 |
| 删除用户 | DELETE | `/users/{id}` | |
| 获取用户订单 | GET | `/users/{id}/orders` | 嵌套资源 |
| 创建订单 | POST | `/orders` | 也可不嵌套 |
| 支付订单 | POST | `/orders/{id}/pay` | 特殊操作用子路径 |
| 取消订单 | POST | `/orders/{id}/cancel` | 特殊操作用子路径 |

## 常见误区

### ❌ 误区1：URL 里加动词

```
POST /api/createUser     ❌
GET  /api/getUserById    ❌
POST /api/deleteOrder    ❌
```

**正确**：用 HTTP 方法表达动词，URL 只留名词。

### ❌ 误区2：一律用 POST

很多团队因为"POST 万能"省事，全用 POST。代价是：
- 无法利用 GET 请求的缓存特性
- 接口无幂等保证，重试会产生副作用
- 失去 REST 规范约定，文档可读性差

### ❌ 误区3：嵌套层级过深

```
GET /orgs/{orgId}/teams/{teamId}/members/{memberId}/projects/{projectId}  ❌
```

超过 2 层嵌套后，用扁平化 + 查询参数更好：
```
GET /projects?teamId={teamId}&memberId={memberId}  ✅
```

### ❌ 误区4：严格 REST vs 实用主义

并非所有操作都能用 CRUD 完美映射（如：批量操作、搜索、导出）。遇到这些情况：
- 批量删除：`DELETE /users` + Body 传 ID 列表，或 `POST /users/batch-delete`
- 搜索：`GET /users?q=keyword`（简单搜索）或 `POST /search/users`（复杂搜索）
- 导出：`GET /users/export?format=csv`

**原则：优先 REST，实用优先于教条。**

## 面试高频题

**Q1：REST 是什么？RESTful API 和普通 API 的区别？**
> REST 是一种架构风格，RESTful API 遵循"资源导向 + HTTP 方法语义 + 无状态"三大核心约束。普通 API 通常动词 URL + 全 POST，没有统一规范。

**Q2：GET 和 POST 有什么本质区别？**
> GET 是幂等的只读操作，结果可缓存，参数在 URL 中；POST 是非幂等操作，用于创建资源，参数在 Body 中。区别不在于"安全性高低"，而在于**语义和幂等性**。

**Q3：PUT 和 PATCH 有什么区别？**
> PUT 是全量替换（未传字段置空），PATCH 是部分更新（只修改传入字段）。实战中 PATCH 更常用。

**Q4：如何表达"启用/禁用用户"这类状态变更？**
> 两种方式：① `PATCH /users/{id}` + Body `{"status": "disabled"}`（通用）；② `PUT /users/{id}/status` + Body `{"status": "disabled"}`（语义明确）。均可，以团队约定为准。
