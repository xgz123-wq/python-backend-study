# URL 命名规范

## 概念解释

URL（Uniform Resource Locator，统一资源定位符）是 RESTful API 的"门牌号"。  
好的 URL 命名让接口**一看就懂**，差的 URL 命名是团队协作的噩梦。

---

## 为什么需要命名规范

同一个"获取用户列表"的接口，不同人可能写出：

```
/GetAllUsers
/get_user_list
/userList
/users/list
/api/Users
/users          ← 正确
```

没有规范时每个人风格不同，接口多了就是"大型迷宫"。

---

## 七条核心命名规范

### 规范1：用名词，不用动词

URL 代表资源，动作由 HTTP 方法表达。

```
❌ GET /getUsers
❌ POST /createUser
❌ DELETE /deleteUser

✅ GET    /users
✅ POST   /users
✅ DELETE /users/{id}
```

> **一句话**：URL = 名词，HTTP 方法 = 动词

---

### 规范2：用复数名词表示集合

资源通常以集合形式存在，统一用复数避免混乱。

```
✅ /users           # 用户集合
✅ /orders          # 订单集合
✅ /products        # 商品集合
✅ /articles        # 文章集合

❌ /user            # 单数易混淆：是集合还是单个？
❌ /userList        # 多余的 "List"
```

> **例外**：确实只有一个实例的资源用单数，如 `/profile`（当前用户的个人资料）

---

### 规范3：用小写字母 + 连字符（kebab-case）

URL 大小写不敏感（标准约定小写），单词分隔用 `-`，不用 `_` 或驼峰。

```
✅ /product-categories
✅ /order-items
✅ /user-addresses

❌ /productCategories    # 驼峰：URL 不友好
❌ /product_categories   # 下划线：易被下划线链接遮挡
❌ /ProductCategories    # 大写：非标准
```

> **原因**：`_` 在某些字体中会被超链接下划线遮住；驼峰在 URL 中视觉混乱；`-` 对搜索引擎 SEO 更友好。

---

### 规范4：用层级路径表达资源关系

当资源有明确所属关系时，用路径嵌套表达（最多 2 层）。

```
✅ /users/{userId}/orders           # 用户的订单列表
✅ /orders/{orderId}/items          # 订单的商品列表
✅ /posts/{postId}/comments         # 文章的评论列表

❌ /getUserOrders?userId=123        # 用查询参数替代层级关系（不推荐）
❌ /users/{uid}/orders/{oid}/items/{iid}/reviews/{rid}  # 超过2层，过深
```

---

### 规范5：查询参数用于过滤、分页、排序

不要把过滤条件放到路径里：

```
✅ GET /users?status=active&age_gt=18        # 过滤：活跃的18岁以上用户
✅ GET /users?page=2&size=20                 # 分页
✅ GET /users?sort=created_at&order=desc     # 排序

❌ GET /users/active                         # 把状态当路径（不规范）
❌ GET /users/age/18                         # 查询条件混入路径
```

---

### 规范6：特殊操作用子路径动名词兜底

某些操作无法完美映射为 CRUD（如：支付、发布、归档），用子路径表达：

```
✅ POST /orders/{id}/pay            # 支付订单
✅ POST /orders/{id}/cancel         # 取消订单
✅ POST /articles/{id}/publish      # 发布文章
✅ POST /users/{id}/avatar          # 上传头像（POST 到子资源）

❌ POST /payOrder                   # 动词 URL
❌ GET  /cancelOrderById?id=123     # GET 不应有副作用
```

---

### 规范7：API 路径加前缀 `/api`

后端接口统一加 `/api` 前缀，与前端静态页面路由隔离。

```
✅ /api/users
✅ /api/orders
✅ /api/v1/users      # 加版本号

❌ /users             # 与前端路由可能冲突
```

---

## 完整 URL 结构解析

```
https://api.example.com/api/v1/users/123/orders?status=pending&page=1&size=10

├── https://             协议（scheme）
├── api.example.com      主机名（host）
├── /api/v1              路径前缀（前缀 + 版本）
├── /users/123/orders    资源路径（集合/ID/子集合）
└── ?status=pending&...  查询参数（过滤+分页）
```

---

## 路径参数 vs 查询参数

| | 路径参数 | 查询参数 |
|--|---------|---------|
| 位置 | `/users/{id}` | `/users?id=123` |
| 语义 | **标识**一个具体资源 | **过滤/修饰**资源集合 |
| 必填性 | 通常必填 | 通常可选 |
| 典型用法 | 单个资源 CRUD | 搜索、分页、排序 |
| 示例 | `GET /users/42` | `GET /users?role=admin` |

## 实战对比：改造前 vs 改造后

### 电商系统 API

| 改造前 | 改造后 | 说明 |
|--------|--------|------|
| `POST /getProduct` | `GET /products/{id}` | 动词→方法，单数→单ID路径 |
| `POST /createOrder` | `POST /orders` | 动词→方法 |
| `POST /getAllUserOrders` | `GET /users/{id}/orders` | 嵌套资源 |
| `GET /searchUsers?name=x` | `GET /users?name=x` | 已正确，search 动词去掉 |
| `POST /cancelOrder?id=1` | `POST /orders/1/cancel` | 特殊操作→子路径 |
| `GET /userDetail?userId=1` | `GET /users/1` | detail 多余，ID 进路径 |

## 常见误区

### ❌ 误区1：把 ID 用查询参数传

```
GET /users?id=123    ❌
GET /users/123       ✅
```

ID 是资源的**唯一标识**，属于路径参数，不是过滤条件。

### ❌ 误区2：混用 camelCase 和 kebab-case

```
/productCategory    ❌  (驼峰)
/product_category   ❌  (下划线)
/product-category   ✅  (连字符)
```

统一风格，团队内一致最重要。

### ❌ 误区3：路径结尾带斜杠

```
/users/    ❌  (尾部斜杠)
/users     ✅
```

某些框架会将 `/users/` 和 `/users` 视为不同路由。

### ❌ 误区4：文件扩展名出现在 URL

```
/users.json    ❌
/users.xml     ❌
/users         ✅  (通过 Accept header 协商格式)
```

数据格式通过 `Accept: application/json` 请求头协商，不放 URL。

## 面试高频题

**Q1：RESTful API 的 URL 命名有哪些规范？**
> 名词复数、小写连字符、路径层级表资源关系（不超过2层）、查询参数做过滤/分页/排序、特殊操作用子路径动名词兜底。

**Q2：路径参数和查询参数有什么区别？什么时候用哪个？**
> 路径参数用于标识具体资源（`/users/{id}`）；查询参数用于过滤、搜索、分页、排序（`/users?status=active`）。区别在于**是否是资源的唯一标识**。

**Q3：如何设计一个"搜索"接口？**
> 简单搜索：`GET /users?q=keyword`（语义是"过滤用户集合"）；复杂搜索（多字段组合）：`POST /users/search`（Body 传复杂条件）。以实用为主。
