# 统一响应格式

## 概念解释

统一响应格式（Unified Response Format）是指：**后端所有接口都按照同一个 JSON 结构返回数据**，无论成功还是失败。

---

## 为什么需要统一响应格式

### 没有规范时的混乱

```json
// 接口 A 成功
{"id": 1, "name": "Alice"}

// 接口 B 成功
{"success": true, "result": {...}, "errMsg": ""}

// 接口 C 失败
{"error": "User not found"}

// 接口 D 失败
{"code": 404, "message": "Not Found", "data": null}
```

**问题**：
- 前端需要为每个接口单独写解析逻辑
- 拦截器（统一处理 Loading、错误弹框）无法写
- 联调效率低，出错概率高

### 统一格式后

```json
// 所有接口成功
{"code": 0, "message": "success", "data": {...}}

// 所有接口失败
{"code": 40001, "message": "用户不存在", "data": null}
```

前端拦截器一次搞定所有接口的错误处理。

---

## 常见统一响应格式设计

### 方案一：业务 code + message + data（推荐）

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "id": 1,
    "name": "Alice",
    "email": "alice@example.com"
  }
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| `code` | int | 业务状态码，`0` 或 `200` 代表成功，其他代表具体错误 |
| `message` | str | 人类可读的状态描述，成功时为 `"success"`，失败时为错误原因 |
| `data` | any | 业务数据，失败时为 `null` |

---

### 方案二：带 timestamp + requestId 的增强版（生产环境推荐）

```json
{
  "code": 0,
  "message": "success",
  "data": {...},
  "timestamp": 1716288000000,
  "requestId": "550e8400-e29b-41d4-a716"
}
```

新增字段：
- `timestamp`：服务端响应时间戳（毫秒），便于排查时间相关问题
- `requestId`：每次请求唯一 ID，日志追踪用（配合链路追踪）

---

## HTTP 状态码 vs 业务 code 双层设计

这是最容易争议的设计点，两种流派：

### 流派一：HTTP 状态码 + 业务 code 双层（推荐）

```
HTTP 状态码 → 表达"请求有没有正常处理"
业务 code   → 表达"业务逻辑是否成功"
```

| 场景 | HTTP 状态码 | 业务 code | message |
|------|------------|-----------|---------|
| 获取用户成功 | 200 | 0 | success |
| 创建用户成功 | 201 | 0 | success |
| 用户名已存在 | 400 | 40001 | 用户名已被注册 |
| Token 过期 | 401 | 40100 | 登录已过期，请重新登录 |
| 无权限 | 403 | 40300 | 无访问权限 |
| 用户不存在 | 404 | 40400 | 用户不存在 |
| 服务器异常 | 500 | 50000 | 服务器内部错误 |

**优点**：HTTP 层面可以被网关、监控、日志工具识别；业务层面可以细化错误原因。

### 流派二：全部 200 + 业务 code（部分团队）

```
HTTP 始终返回 200，业务是否成功完全看 code 字段
```

| 场景 | HTTP 状态码 | 业务 code |
|------|------------|-----------|
| 任何响应 | 200 | 0（成功）/ 非0（失败） |

**优点**：某些代理/防火墙对非200有特殊处理，全200可规避。  
**缺点**：监控和日志工具无法通过 HTTP 状态码区分成功失败。

> **本课程推荐**：双层设计，HTTP 状态码语义正确 + 业务 code 细化原因。

## 业务错误码设计规范

### 编码规则（示例）

```
XXYYYY
XX   → HTTP 状态码前两位（40 = 4xx，50 = 5xx）
YYYY → 具体业务错误序号
```

| 范围 | 含义 |
|------|------|
| `0` | 成功 |
| `40001 - 40099` | 参数/请求错误（400） |
| `40101 - 40199` | 认证失败（401） |
| `40301 - 40399` | 无权限（403） |
| `40401 - 40499` | 资源不存在（404） |
| `50001 - 50099` | 服务器内部错误（500） |

### 示例错误码表

| 错误码 | 含义 |
|--------|------|
| `0` | 成功 |
| `40001` | 请求参数缺失或格式错误 |
| `40002` | 用户名已存在 |
| `40003` | 邮箱格式不正确 |
| `40101` | Token 无效 |
| `40102` | Token 已过期 |
| `40301` | 无访问该资源的权限 |
| `40401` | 用户不存在 |
| `40402` | 订单不存在 |
| `50001` | 数据库操作失败 |

## 分页响应格式

列表接口需要额外的分页元数据：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "items": [
      {"id": 1, "name": "Alice"},
      {"id": 2, "name": "Bob"}
    ],
    "pagination": {
      "page": 1,
      "size": 20,
      "total": 100,
      "totalPages": 5,
      "hasNext": true,
      "hasPrev": false
    }
  }
}
```

---

## Python 实现示例（FastAPI 风格）

```python
from typing import TypeVar, Generic, Optional
from pydantic import BaseModel

T = TypeVar("T")

class Response(BaseModel, Generic[T]):
    code: int = 0
    message: str = "success"
    data: Optional[T] = None

    @classmethod
    def ok(cls, data: T = None, message: str = "success"):
        return cls(code=0, message=message, data=data)

    @classmethod
    def error(cls, code: int, message: str):
        return cls(code=code, message=message, data=None)


# 使用示例
response = Response.ok(data={"id": 1, "name": "Alice"})
error = Response.error(code=40401, message="用户不存在")
```

## 常见误区

### ❌ 误区1：把 HTTP 状态码和业务 code 混用

```json
// ❌ HTTP 200 但 code 是 404
HTTP/1.1 200 OK
{"code": 404, "message": "用户不存在"}
```

这样做会让监控系统误判为成功，日志中找不到真正的错误。

### ❌ 误区2：data 字段直接返回数组

```json
// ❌ 数组直接作为响应体
[{"id": 1}, {"id": 2}]
```

一旦需要加分页元数据或其他字段，要改所有调用方。从一开始就用对象包裹。

### ❌ 误区3：错误时不填 data（留字段不存在）

```json
// ❌ 缺少 data 字段
{"code": 40401, "message": "用户不存在"}
```

应该明确返回 `"data": null`，让前端知道字段存在但为空。

### ❌ 误区4：message 写给机器看

```json
// ❌ 技术信息暴露给用户
{"code": 50001, "message": "NullPointerException at line 42"}
```

`message` 应面向用户，技术细节写日志，不暴露给客户端。

## 面试高频题

**Q1：为什么需要统一响应格式？**
> 前端可以写统一的拦截器处理错误弹框/Loading；联调效率提高；监控和日志工具可以统一解析。

**Q2：HTTP 状态码和业务 code 有什么关系？应该用哪个？**
> 两层含义不同：HTTP 状态码表示"请求是否被正常处理"（协议层），业务 code 表示"业务操作是否成功"（应用层）。推荐双层设计：HTTP 状态码语义正确，业务 code 细化具体原因。

**Q3：分页响应应该返回哪些元数据？**
> 必需：当前页码（page）、每页大小（size）、总记录数（total）。可选：总页数（totalPages）、是否有下一页（hasNext）。
