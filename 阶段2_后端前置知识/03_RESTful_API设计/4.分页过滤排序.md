# 分页、过滤与排序

## 概念解释

当资源集合数据量大时（几百、几千、几百万条），不能一次返回全部数据。  
**分页**（Pagination）、**过滤**（Filtering）、**排序**（Sorting）是列表接口的三大核心能力。

---

## 分页

### 为什么必须分页

- 数据库查询全表 + 网络传输 → 接口超时
- 前端渲染大量 DOM → 页面卡死
- 用户也不可能一次看完 100 万条数据

### 方案一：页码分页（Offset Pagination）

最常见，适合"跳页"场景（如电商列表、搜索结果）。

**请求参数**：

```
GET /users?page=2&size=20
```

| 参数 | 含义 | 默认值 |
|------|------|--------|
| `page` | 页码（从 1 开始） | 1 |
| `size` | 每页条数 | 20 |

**响应数据**：

```json
{
  "code": 0,
  "data": {
    "items": [...],
    "pagination": {
      "page": 2,
      "size": 20,
      "total": 100,
      "totalPages": 5,
      "hasNext": true,
      "hasPrev": true
    }
  }
}
```

**SQL 实现**：
```sql
SELECT * FROM users
ORDER BY id
LIMIT 20 OFFSET 20;   -- page=2, size=20 → offset = (page-1) * size
```

**缺点**：数据量超大时，高偏移量（`OFFSET 100000`）性能差；数据实时变化时翻页会出现"重复/漏条目"问题。

---

### 方案二：游标分页（Cursor Pagination）

适合**无限滚动**（如微博 Feed 流、评论区）和**大数据量**场景。

**请求参数**：

```
GET /posts?cursor=eyJpZCI6MTAwfQ&size=20
```

| 参数 | 含义 |
|------|------|
| `cursor` | 游标（通常是 Base64 编码的上一页最后一条记录的 ID 或时间戳） |
| `size` | 每页条数 |

**响应数据**：

```json
{
  "code": 0,
  "data": {
    "items": [...],
    "pagination": {
      "nextCursor": "eyJpZCI6MTIwfQ",
      "hasNext": true
    }
  }
}
```

**SQL 实现**：
```sql
SELECT * FROM posts
WHERE id > 100          -- cursor 解码后的 id
ORDER BY id ASC
LIMIT 20;
```

**优点**：性能稳定（无大 OFFSET），数据一致性好（不受插入/删除影响）。  
**缺点**：无法跳页，只能前进/后退一页。

---

### 两种分页方式对比

| 特性 | 页码分页 | 游标分页 |
|------|---------|---------|
| 可跳页 | ✅ 可以 | ❌ 不可以 |
| 大数据量性能 | 差（高 OFFSET） | 好（按 ID/时间戳索引） |
| 数据一致性 | 差（实时变更会漏/重） | 好 |
| 实现复杂度 | 简单 | 较复杂 |
| 适用场景 | 搜索结果、数据管理后台 | 动态 Feed 流、消息列表 |

## 过滤

过滤让客户端精确查询满足条件的资源子集，通过查询参数传递条件。

### 简单等值过滤

```
GET /users?status=active
GET /users?role=admin&status=active
GET /orders?user_id=123
```

**规则**：参数名 = 字段名，参数值 = 期望值。

---

### 范围过滤（数字/日期）

常见约定：

| 写法 | 含义 |
|------|------|
| `age_gt=18` | age > 18 |
| `age_gte=18` | age >= 18 |
| `age_lt=60` | age < 60 |
| `age_lte=60` | age <= 60 |
| `created_at_from=2024-01-01` | created_at >= 2024-01-01 |
| `created_at_to=2024-12-31` | created_at <= 2024-12-31 |

**示例**：

```
GET /orders?amount_gte=100&amount_lte=1000
GET /users?created_at_from=2024-01-01&created_at_to=2024-03-31
```

---

### 模糊搜索

```
GET /users?name=alice           # 精确匹配
GET /users?name__contains=ali   # 包含（模糊）
GET /users?q=alice              # 通用搜索参数（全文搜索）
```

---

### 多值过滤（IN 查询）

```
GET /users?ids=1,2,3            # id IN (1,2,3)
GET /products?category=book,game  # category IN ('book', 'game')
GET /users?status[]=active&status[]=pending  # 数组写法
```

---

### 过滤参数规范建议

1. **参数名小写 + 下划线**：`user_id`、`created_at`
2. **范围用 `_gt/_gte/_lt/_lte` 后缀**，全文搜索用 `q`
3. **布尔值用 `true/false` 字符串**：`?is_vip=true`
4. **日期格式统一 ISO 8601**：`2024-01-01` 或 `2024-01-01T00:00:00Z`

## 排序

### 基本排序

```
GET /users?sort=created_at&order=desc
GET /users?sort=name&order=asc
```

| 参数 | 含义 | 默认值 |
|------|------|--------|
| `sort` | 排序字段 | 通常是 `id` 或 `created_at` |
| `order` | 排序方向：`asc`（升序）/ `desc`（降序） | `asc` |

---

### 多字段排序

有两种常见写法：

**写法一：`-` 前缀表示降序（Django REST Framework 风格）**

```
GET /users?ordering=-created_at,name
# → ORDER BY created_at DESC, name ASC
```

**写法二：分隔符写法**

```
GET /users?sort=created_at:desc,name:asc
```

---

### 排序安全注意事项

⚠️ **后端必须对 `sort` 参数做白名单校验**：

```python
ALLOWED_SORT_FIELDS = {"id", "name", "created_at", "updated_at"}

def validate_sort(sort: str) -> str:
    if sort not in ALLOWED_SORT_FIELDS:
        raise ValueError(f"不支持按 {sort} 排序")
    return sort
```

**原因**：直接拼接 `sort` 参数进 SQL 会有 SQL 注入风险（即使 ORM 也要谨慎）。

## 综合示例：完整列表接口

```
GET /api/v1/orders?
    status=pending&             ← 过滤：待处理订单
    amount_gte=100&             ← 过滤：金额 >= 100
    created_at_from=2024-01-01& ← 过滤：日期范围
    sort=created_at&            ← 排序：按创建时间
    order=desc&                 ← 排序：降序
    page=1&                     ← 分页：第1页
    size=20                     ← 分页：每页20条
```

响应：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "items": [
      {"id": 201, "amount": 299, "status": "pending", "created_at": "2024-03-15"},
      {"id": 198, "amount": 150, "status": "pending", "created_at": "2024-03-14"}
    ],
    "pagination": {
      "page": 1,
      "size": 20,
      "total": 47,
      "totalPages": 3,
      "hasNext": true,
      "hasPrev": false
    }
  }
}
```

## 常见误区

### ❌ 误区1：不做分页直接返回全表

```python
# ❌
users = db.query(User).all()   # 百万数据直接全量返回
```

无论数据量多少，列表接口都应该有分页上限（即使不传 page 参数，默认也限制最多返回 100 条）。

### ❌ 误区2：把过滤条件放路径里

```
GET /users/active/admin    ❌  （状态和角色作为路径段）
GET /users?status=active&role=admin   ✅
```

### ❌ 误区3：total 不准确

分页查询用了 LIMIT，但 total 字段却写成 `len(current_page_data)` 而不是全量 count：

```python
# ❌ 错误：total 只是当前页的数量
items = db.query(User).offset(offset).limit(size).all()
return {"items": items, "total": len(items)}  # len() 最多返回 size

# ✅ 正确：total 是过滤后的总数（不含分页限制）
total = db.query(User).count()
items = db.query(User).offset(offset).limit(size).all()
return {"items": items, "total": total}
```

### ❌ 误区4：sort 参数不校验，SQL 注入风险

如上"排序安全"小节，`sort` 必须白名单校验。

## 面试高频题

**Q1：页码分页和游标分页的区别？各自适用什么场景？**
> 页码分页（offset）实现简单、支持跳页，但大数据量时 OFFSET 性能差；游标分页性能稳定、不受实时数据变化影响，适合 Feed 流无限滚动，但不支持跳页。

**Q2：列表接口如何防止 sort 参数 SQL 注入？**
> 对 sort 参数做白名单校验，只允许固定的字段名列表，不直接将参数值拼接进 SQL。

**Q3：分页的 total 字段应该怎么计算？**
> total 是**满足过滤条件的总记录数**，需要单独执行 COUNT 查询（不带 LIMIT/OFFSET），不能用当前页返回的数量代替。
