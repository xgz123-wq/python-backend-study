# API 版本管理

## 概念解释

API 版本管理（API Versioning）是指：当 API 需要做**破坏性变更**（Breaking Change）时，通过版本号让新旧客户端能同时工作，避免"改了接口、所有调用方都炸"的灾难。

---

## 为什么需要版本管理

### 什么是破坏性变更？

以下改动会破坏已有客户端：

| 变更类型 | 示例 |
|----------|------|
| 删除字段 | 响应中去掉 `email` 字段 |
| 修改字段名 | `user_name` 改为 `username` |
| 修改字段类型 | `age` 从 int 改为 string |
| 删除接口 | `/users/search` 接口下线 |
| 修改 URL 结构 | `/user/{id}` 改为 `/users/{id}` |
| 修改认证方式 | 从 Cookie 改为 Bearer Token |

### 非破坏性变更（不需要新版本）

| 变更类型 | 示例 |
|----------|------|
| 新增字段 | 响应中增加 `avatar_url` 字段 |
| 新增接口 | 增加 `/users/{id}/followers` |
| 松化参数限制 | 原来必填的参数改为可选 |
| Bug 修复 | 修复返回数据错误（不改结构） |

## 四种版本管理方式

### 方案一：URL 路径版本（最常用 ✅）

版本号直接放在 URL 路径中。

```
GET /api/v1/users
GET /api/v2/users
POST /api/v1/orders
```

**优点**：
- 直观、易理解，一眼知道用哪个版本
- 易于调试（浏览器直接访问）
- 路由层面隔离，实现简单

**缺点**：
- URL "不纯粹"（版本信息混入了资源标识）
- 新版本需要复制大量路由代码

**推荐场景**：绝大多数公司 API、对外开放 API（Open API）

---

### 方案二：请求头版本（Header Versioning）

版本信息放在自定义请求头中。

```
GET /api/users
Headers:
  API-Version: 2
  Accept: application/vnd.myapp.v2+json
```

**优点**：URL 保持整洁，符合 REST "URL 只标识资源"的纯粹原则

**缺点**：
- 不直观，调试麻烦（无法直接用浏览器测试）
- 某些代理/网关无法根据 Header 路由

**推荐场景**：纯粹主义者、内部服务 API

---

### 方案三：查询参数版本

版本号作为查询参数传递。

```
GET /api/users?version=2
GET /api/users?api-version=v2
```

**优点**：可选参数，不传时回退到默认版本

**缺点**：语义混乱（version 不是过滤条件）；容易被遗忘

**推荐场景**：不推荐，仅作兜底向后兼容

---

### 方案四：子域名版本

```
https://v1.api.example.com/users
https://v2.api.example.com/users
```

**优点**：完全隔离，可部署到不同服务器

**缺点**：运维成本高，证书、DNS 配置麻烦

**推荐场景**：大型平台、需要完全物理隔离的场景

---

## 四种方案对比

| 方案 | 直观性 | 实现难度 | 缓存友好 | 推荐度 |
|------|--------|---------|---------|--------|
| URL 路径（`/v1/`） | ⭐⭐⭐ 高 | 简单 | ✅ 好 | ⭐⭐⭐ 首选 |
| 请求头 | ⭐ 低 | 中 | ❌ 差 | ⭐⭐ 次选 |
| 查询参数（`?v=1`） | ⭐⭐ 中 | 简单 | ✅ 好 | ⭐ 不推荐 |
| 子域名 | ⭐⭐⭐ 高 | 复杂 | ✅ 好 | ⭐⭐ 特殊场景 |

## URL 路径版本的实践规范

### 版本号命名

```
✅ /api/v1/users    (推荐：简洁)
✅ /api/v2/users
❌ /api/1.0/users   (小数点不友好)
❌ /api/version1/users  (太长)
```

### 版本何时升级？

| 场景 | 是否需要新版本 |
|------|---------------|
| 字段重命名 | ✅ 需要（破坏性变更） |
| 删除字段 | ✅ 需要 |
| 修改认证方式 | ✅ 需要 |
| 新增字段 | ❌ 不需要 |
| 新增接口 | ❌ 不需要 |
| Bug 修复 | ❌ 不需要 |

### 版本共存与下线策略

1. **发布 v2 时，v1 继续维护**（至少 6-12 个月）
2. **提前通知弃用**：在 v1 响应头中加 `Deprecation: true` + 弃用日期
3. **监控 v1 流量**，降至 0 时下线
4. **下线前发邮件/公告通知**老客户端开发者

```
# v1 返回的弃用警告头
Deprecation: true
Sunset: Sat, 01 Jan 2025 00:00:00 GMT
Link: <https://api.example.com/api/v2/users>; rel="successor-version"
```

## FastAPI 中的版本管理实现

### 方式一：路由前缀

```python
from fastapi import FastAPI, APIRouter

app = FastAPI()

v1_router = APIRouter(prefix="/api/v1")
v2_router = APIRouter(prefix="/api/v2")

@v1_router.get("/users")
def get_users_v1():
    return {"version": "v1", "users": [...]}

@v2_router.get("/users")
def get_users_v2():
    # v2 新增 pagination 字段
    return {"version": "v2", "data": {"items": [...], "pagination": {...}}}

app.include_router(v1_router)
app.include_router(v2_router)
```

### 方式二：独立应用挂载

```python
app_v1 = FastAPI(title="API v1")
app_v2 = FastAPI(title="API v2")

# 各自注册路由...

main_app = FastAPI()
main_app.mount("/api/v1", app_v1)
main_app.mount("/api/v2", app_v2)
```

## 实际场景：v1 → v2 迁移示例

### v1 响应（旧格式）

```json
GET /api/v1/users/1

{
  "user_name": "alice",       ← 字段名
  "user_email": "alice@x.com",
  "create_time": "2024-01-01"
}
```

### v2 响应（新格式）

```json
GET /api/v2/users/1

{
  "username": "alice",         ← 字段名改了（破坏性变更）
  "email": "alice@x.com",
  "created_at": "2024-01-01T00:00:00Z",  ← 格式改了（ISO 8601）
  "avatar_url": null           ← 新增字段（非破坏性）
}
```

这种情况下**必须**引入 v2，同时保留 v1 供老客户端使用。

## 常见误区

### ❌ 误区1：所有改动都发新版本

非破坏性变更（加字段、加接口）不需要发新版本，滥用版本号会增加维护负担。

### ❌ 误区2：新版本上线立即下线旧版本

旧版本必须有充足的迁移窗口（移动端 App 发版周期可能需要数月），强行下线会直接影响用户。

### ❌ 误区3：版本号用语义版本（如 v1.2.3）

API 对外版本只需要**主版本号**（v1、v2），次版本和补丁版本变更不需要改 URL。

### ❌ 误区4：忘记在文档中标注弃用状态

旧版本弃用时，必须在 API 文档、Swagger/OpenAPI 中明确标注 `deprecated: true`。

## 面试高频题

**Q1：API 版本管理有哪几种方式？你推荐哪种？**
> 四种：URL 路径、请求头、查询参数、子域名。推荐 URL 路径（`/api/v1/`），最直观、最易调试、对缓存友好。

**Q2：什么情况下需要升级 API 版本？**
> 破坏性变更时必须升版本，包括：删除字段、重命名字段、修改字段类型、删除接口、修改认证方式。新增字段/接口属于非破坏性变更，不需要新版本。

**Q3：旧版本下线前需要做哪些准备？**
> 提前通知（文档标注 deprecated + Sunset 日期），监控旧版本流量降至 0，给客户端充足迁移窗口（移动端至少 3-6 个月）。
