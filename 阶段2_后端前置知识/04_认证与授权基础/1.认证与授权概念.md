# 认证与授权概念

> 认证（Authentication）= 你是谁？授权（Authorization）= 你能干什么？

---

## 一、概念解释

### 认证（Authentication，简称 AuthN）

验证用户身份，确认"你是你所说的那个人"。

```
用户：我是 Alice，密码是 123456
系统：验证密码正确 → 确认你是 Alice ✅
```

常见认证方式：
- **账号密码**：最传统，用户提交凭证，服务器验证
- **手机验证码**：基于手机号所有权验证
- **第三方登录**：通过微信/Google 等已信任平台认证（OAuth2）
- **生物识别**：指纹、人脸（移动端）

### 授权（Authorization，简称 AuthZ）

确认用户有权限做某件事。认证之后才谈授权。

```
Alice 已登录（已认证）
Alice 想删除订单 → 检查权限 → Alice 是普通用户，无权删除其他人订单 ❌
Alice 想查看自己订单 → 检查权限 → 有权限 ✅
```

常见授权方式：
- **基于角色（RBAC）**：用户 → 角色 → 权限（最常用）
- **基于权限**：直接给用户分配具体权限
- **基于资源**：对特定资源设置访问策略

### 两者的关键区别

| 维度 | 认证（AuthN） | 授权（AuthZ） |
|------|--------------|--------------|
| 核心问题 | 你是谁？ | 你能做什么？ |
| 发生时机 | 登录时 | 每次请求时 |
| 常见技术 | 密码、JWT、OAuth2 | RBAC、ACL、策略 |
| HTTP 状态码 | 401 Unauthorized | 403 Forbidden |
| 类比 | 查身份证 | 检查通行证 |

## 二、为什么需要认证与授权？

| 场景 | 说明 |
|------|------|
| 保护用户数据 | 只有本人才能查看自己的订单、收货地址 |
| 区分用户权限 | 普通用户/VIP/管理员看到不同内容 |
| 防止越权操作 | 用户 A 不能删除用户 B 的数据 |
| 计费与限流 | 识别用户身份后按套餐限制 API 调用次数 |
| 审计日志 | 记录"谁、在什么时候、做了什么操作" |

## 三、核心知识点

### 1. HTTP 的无状态性与会话管理

HTTP 是无状态协议——每次请求都是独立的，服务器不记得你之前的请求。

```
用户登录请求   → 服务器：你是谁？验证密码... OK
用户下一个请求 → 服务器：你是谁？（又不认识了！）
```

**问题**：如何在无状态 HTTP 上实现"登录后持续认证"？

**解决方案**：
1. **Session + Cookie**：服务器记住登录状态，发给客户端一个标识符
2. **Token（JWT）**：服务器生成加密凭证，客户端每次携带，服务器解密验证

### 2. Session vs Token 对比

| 维度 | Session（传统方案） | Token/JWT（现代方案） |
|------|-------------------|---------------------|
| 存储位置 | 服务端（内存/Redis） | 客户端（浏览器/App） |
| 服务器状态 | 有状态（需要存储） | 无状态（不需要存储） |
| 横向扩展 | 难（多服务器需共享 Session） | 易（每台服务器独立验证） |
| 注销方式 | 直接删除服务端 Session | 需要黑名单机制 |
| 移动端支持 | 差（Cookie 不友好） | 好（Header 传递） |
| 安全性 | Session ID 泄露可被利用 | Token 有过期时间，有效期短则更安全 |
| 适用场景 | 传统 Web 应用 | 前后端分离、移动端、微服务 |

### 3. 认证流程全貌

```
Session 方案流程：
1. 用户提交账号密码
2. 服务器验证密码，创建 Session（存在 Redis）
3. 返回 Session ID（存在 Cookie）
4. 后续请求自动携带 Cookie
5. 服务器用 Session ID 查 Redis，确认用户身份

Token/JWT 方案流程：
1. 用户提交账号密码
2. 服务器验证密码，生成 JWT（含用户信息，签名加密）
3. 返回 JWT 给客户端（存在 localStorage 或内存）
4. 后续请求在 Authorization Header 携带 JWT
5. 服务器解密 JWT，验证签名和有效期，读取用户信息
```

### 4. 密码安全存储

**绝对不能**明文存储密码！一旦数据库泄露，所有用户密码暴露。

```python
# ❌ 错误：明文存储
user.password = "123456"

# ❌ 错误：MD5/SHA1（可被彩虹表破解）
user.password = hashlib.md5(password.encode()).hexdigest()

# ✅ 正确：使用加盐哈希算法
# bcrypt/argon2 会自动加盐，每次哈希结果都不同
import bcrypt
hashed = bcrypt.hashpw(password.encode(), bcrypt.gensalt())
```

**为什么普通哈希不够？**
- MD5/SHA1 是**确定性函数**：相同输入必然相同输出
- 攻击者可以预先计算所有常见密码的哈希值（彩虹表）
- bcrypt/argon2 使用**随机盐值**（Salt），每次哈希结果都不同，彩虹表失效

## 四、实际场景

### 场景 1：用户登录完整流程

```
前端：POST /api/auth/login  {"username": "alice", "password": "xxx"}
          ↓
后端：查数据库找到 alice，验证密码哈希
          ↓
后端：密码正确，生成 JWT（有效期 2 小时）
          ↓
后端：返回 {"access_token": "eyJ...", "token_type": "bearer"}
          ↓
前端：保存 token，后续请求加 Header: Authorization: Bearer eyJ...
          ↓
后端：验证 token 签名和有效期，从 token 读取用户 ID
          ↓
后端：处理业务逻辑，返回数据
```

### 场景 2：权限检查（RBAC）

```python
# 用户角色：user / admin / super_admin

@router.delete("/users/{user_id}")
async def delete_user(user_id: int, current_user: User = Depends(get_current_user)):
    # 普通用户只能删除自己
    if current_user.role == "user" and current_user.id != user_id:
        raise HTTPException(status_code=403, detail="无权操作其他用户")
    # 管理员可以删除任何用户
    if current_user.role not in ["admin", "super_admin"]:
        raise HTTPException(status_code=403, detail="权限不足")
    # 执行删除...
```

## 五、常见误区

| 误区 | 纠正 |
|------|------|
| "401 和 403 都是权限问题" | 401 = 未认证（没登录），403 = 未授权（登录了但没权限） |
| "Token 比 Session 一定更安全" | 各有利弊，关键在于实现方式。Token 短时效 + HTTPS 才安全 |
| "密码用 MD5 加密就安全了" | MD5 是哈希不是加密，且可被彩虹表攻破，应使用 bcrypt/argon2 |
| "JWT 可以直接存敏感信息" | JWT Payload 只是 Base64 编码，任何人都能解码读取，切勿存密码等敏感信息 |
| "登出就是前端删除 token" | Token 在有效期内仍然有效，完整登出需要服务端黑名单机制 |

## 六、面试高频题

**Q：认证和授权的区别？**
A：认证（Authentication）验证用户身份（你是谁），授权（Authorization）验证操作权限（你能做什么）。认证失败返回 401，授权失败返回 403。

**Q：Session 和 JWT 的区别？适用场景？**
A：Session 有状态（服务端存储），适合传统 Web 应用，横向扩展需 Redis 共享；JWT 无状态（客户端存储），适合前后端分离、移动端和微服务，但注销需要额外机制。

**Q：为什么不能用 MD5 存储密码？**
A：MD5 是确定性哈希，相同输入输出相同，攻击者可以用彩虹表（预计算的哈希字典）直接反查密码。应使用 bcrypt/argon2，它们自动加随机盐，使彩虹表攻击失效。
