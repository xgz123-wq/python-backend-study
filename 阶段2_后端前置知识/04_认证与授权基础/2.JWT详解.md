# JWT（JSON Web Token）

> JWT 是现代后端认证的主流方案：服务器不存状态，把用户信息加密签名后交给客户端保管。

---

## 一、概念解释

### 什么是 JWT？

**JWT**（JSON Web Token）是一个开放标准（RFC 7519），定义了一种紧凑、自包含的方式，用于在各方之间安全传递 JSON 数据，通常用于认证和信息交换。

**自包含**是 JWT 的核心特点：Token 本身就包含了用户信息，服务器无需查数据库就能知道你是谁。

### JWT 的结构

JWT 由三部分组成，用 `.` 分隔：

```
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IkFsaWNlIiwiZXhwIjoxNzAwMDAwMDAwfQ.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c
│                Header                │ │                  Payload                      │ │           Signature           │
```

**1. Header（头部）** — 描述算法

```json
{
  "alg": "HS256",   // 签名算法：HMAC SHA256
  "typ": "JWT"      // Token 类型
}
```

**2. Payload（载荷）** — 存放声明（Claims）

```json
{
  "sub": "1234567890",      // Subject：用户 ID（标准字段）
  "username": "alice",       // 自定义字段
  "role": "admin",           // 自定义字段
  "exp": 1700000000,         // Expiration：过期时间（标准字段）
  "iat": 1699996400          // Issued At：签发时间（标准字段）
}
```

**3. Signature（签名）** — 防篡改

```
HMACSHA256(
  base64UrlEncode(header) + "." + base64UrlEncode(payload),
  secret_key    ← 只有服务器知道的密钥
)
```

## 二、为什么需要 JWT？

### 传统 Session 的问题

```
           用户1         用户2         用户3
             ↓             ↓             ↓
        [服务器1]     [服务器2]     [服务器3]
             ↑             ↑             ↑
           用户1的         ???           ???
           Session        
           在这里！

问题：负载均衡下，用户1的第二次请求可能被路由到服务器2，
     服务器2没有 Session，用户被迫重新登录！
```

### JWT 如何解决这个问题

```
           用户（携带 JWT Token）
             ↓         ↓        ↓
        [服务器1]  [服务器2]  [服务器3]

每台服务器只需要知道 secret_key，就能独立验证 JWT！
不需要共享存储，天然支持水平扩展。
```

## 三、核心知识点

### 1. JWT 完整工作流程

```
登录阶段：
1. 用户：POST /login  {username, password}
2. 服务器：验证密码 → 正确 → 用 secret_key 生成 JWT
3. 服务器：返回 {access_token: "eyJ...", expires_in: 7200}
4. 客户端：保存 Token（localStorage / 内存 / HTTP-only Cookie）

访问受保护资源：
5. 客户端：GET /api/profile
           Header: Authorization: Bearer eyJ...
6. 服务器：取出 Token → 验证签名 → 检查过期时间 → 读取 Payload
7. 服务器：从 Payload 获取用户 ID，处理业务逻辑
8. 服务器：返回数据
```

### 2. 标准 Payload 字段（Claims）

| 字段 | 全称 | 说明 |
|------|------|------|
| `sub` | Subject | 用户唯一标识（通常是用户 ID） |
| `iss` | Issuer | 签发者（如 `api.example.com`） |
| `aud` | Audience | 受众（接收方） |
| `exp` | Expiration Time | 过期时间（Unix 时间戳） |
| `iat` | Issued At | 签发时间 |
| `nbf` | Not Before | 生效时间（此时间前不接受） |
| `jti` | JWT ID | 唯一标识，防重放攻击 |

### 3. Access Token + Refresh Token 双 Token 机制

**问题**：Token 有效期太长，安全风险大；有效期太短，用户频繁重新登录，体验差。

**解决方案**：

```
Access Token  → 有效期短（2 小时），用于访问 API
Refresh Token → 有效期长（30 天），用于刷新 Access Token

正常流程：
用户登录 → 获得 Access Token（2h）+ Refresh Token（30d）
         → 用 Access Token 访问接口
         
Access Token 过期时：
客户端自动用 Refresh Token 请求新的 Access Token
         → 用户无感刷新，无需重新登录
         
Refresh Token 过期时：
用户需要重新登录
```

```python
# 返回双 Token
{
    "access_token": "eyJ...",      # 短期，用于 API 访问
    "refresh_token": "eyJ...",     # 长期，用于刷新
    "token_type": "bearer",
    "expires_in": 7200             # access_token 2 小时后过期
}
```

### 4. JWT 的安全注意事项

**⚠️ Payload 是可被任何人读取的！**

```python
import base64, json

# JWT 的 Payload 部分只是 Base64 编码，不是加密！
payload = "eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IkFsaWNlIn0"
decoded = base64.b64decode(payload + "==")  # 加 == 补齐
print(json.loads(decoded))  # 任何人都能看到内容！
```

**安全规范**：
1. **不存敏感信息**：Payload 不要放密码、身份证号、银行卡号
2. **HTTPS 传输**：Token 在传输中必须加密
3. **短有效期**：Access Token 推荐 15 分钟～2 小时
4. **强密钥**：`secret_key` 使用随机生成的长字符串（至少 32 位），不要用 `"secret"` 这类弱密钥
5. **验证签名**：不要只解码 Payload，必须验证签名！

## 四、实际场景

### 场景 1：FastAPI 中实现 JWT

```python
from datetime import datetime, timedelta
from jose import JWTError, jwt
from passlib.context import CryptContext

SECRET_KEY = "your-very-long-random-secret-key-here"
ALGORITHM = "HS256"
ACCESS_TOKEN_EXPIRE_MINUTES = 30

def create_access_token(data: dict) -> str:
    """生成 Access Token"""
    to_encode = data.copy()
    expire = datetime.utcnow() + timedelta(minutes=ACCESS_TOKEN_EXPIRE_MINUTES)
    to_encode.update({"exp": expire})
    return jwt.encode(to_encode, SECRET_KEY, algorithm=ALGORITHM)

def verify_token(token: str) -> dict:
    """验证 Token，返回 Payload"""
    try:
        payload = jwt.decode(token, SECRET_KEY, algorithms=[ALGORITHM])
        return payload
    except JWTError:
        raise HTTPException(status_code=401, detail="Token 无效或已过期")
```

### 场景 2：前端如何携带 Token

```javascript
// 登录后保存
localStorage.setItem("access_token", response.data.access_token);

// 每次请求自动携带
axios.interceptors.request.use(config => {
    const token = localStorage.getItem("access_token");
    if (token) {
        config.headers.Authorization = `Bearer ${token}`;
    }
    return config;
});
```

## 五、常见误区

| 误区 | 纠正 |
|------|------|
| "JWT 是加密的" | Payload 只是 Base64 编码，不是加密！只有签名保证完整性 |
| "JWT 登出就是客户端删除 token" | Token 在有效期内仍然有效，真正安全的登出需要服务端黑名单 |
| "JWT 越大越安全，放更多信息" | Payload 越大，每次请求携带的数据越多，影响性能。只放必要信息 |
| "用对称加密就够了" | 微服务中如果多个服务都需要验证 token，对称密钥需要在多服务间共享，有泄露风险。可考虑 RS256（非对称加密） |

## 六、面试高频题

**Q：JWT 的组成结构？**
A：三部分用 `.` 分隔：Header（算法类型）、Payload（用户数据和过期时间）、Signature（签名，防篡改）。Header 和 Payload 是 Base64URL 编码，不是加密。

**Q：JWT 的 Payload 安全吗，可以存密码吗？**
A：不安全，Payload 只是 Base64 编码，任何人都可以解码读取。绝对不能存密码、身份证等敏感信息。通过 HTTPS 传输防止中间人窃取整个 Token。

**Q：Access Token 和 Refresh Token 的区别？**
A：Access Token 有效期短（分钟/小时级），用于访问 API；Refresh Token 有效期长（天/月级），用于在 Access Token 过期后无感刷新，避免用户频繁重新登录。

**Q：如何实现 JWT 登出？**
A：纯 JWT 本身无法立即失效。常见方案：1）维护 Token 黑名单（存 Redis，查询 O(1)）；2）使用短有效期（Token 很快自然失效）；3）在 Token 中加入版本号，改密码时递增版本。
