# OAuth2.0 授权

> OAuth2.0 解决的问题：如何让第三方应用访问你的数据，而不需要把密码给它。

---

## 一、概念解释

### 什么是 OAuth2.0？

**OAuth2.0**（Open Authorization 2.0）是一个授权框架，允许第三方应用在用户授权的前提下，访问用户在某平台上的资源，**全程无需用户提供密码给第三方**。

**生活类比**：
```
传统方式（危险）：
你告诉快递员你家门锁密码，让他帮你取包裹。
→ 快递员知道了密码，他可以随时进入你家

OAuth2.0 方式（安全）：
你去物业办了一张临时通行证给快递员，通行证只有权限
进门厅，不能进卧室，且 24 小时后自动失效。
→ 快递员只能在授权范围内操作，你的"密码"（真钥匙）从未泄露
```

### OAuth2.0 的四个角色

| 角色 | 中文名 | 说明 | 举例 |
|------|--------|------|------|
| **Resource Owner** | 资源所有者 | 拥有数据的用户 | 你（GitHub 用户） |
| **Client** | 客户端/第三方 | 请求访问数据的应用 | 某 IDE 插件 |
| **Authorization Server** | 授权服务器 | 负责认证用户、颁发 Token | GitHub 的认证系统 |
| **Resource Server** | 资源服务器 | 存放受保护资源的服务器 | GitHub API |

## 二、为什么需要 OAuth2.0？

**真实场景**：你想用某个应用"用 GitHub 登录"

**没有 OAuth2.0 的做法（危险）**：
```
应用：请把你的 GitHub 账号密码给我
用户：好，密码是 xxx123
（此后应用可以以你的身份做任何事：删仓库、改个人信息、查私有仓库...）
```

**有 OAuth2.0 的做法（安全）**：
```
应用：请跳转到 GitHub 登录
GitHub：这个应用申请读取你的用户名和邮箱，是否允许？
用户：允许
GitHub：颁发 Token（只有读取用户名和邮箱的权限，1小时后失效）
应用：用 Token 访问 GitHub API，只能读取被授权的信息
```

## 三、核心知识点

### 1. OAuth2.0 的四种授权模式

| 模式 | 适用场景 | 安全性 |
|------|---------|--------|
| **授权码模式**（Authorization Code） | 有服务端的 Web 应用（最常用） | ⭐⭐⭐⭐⭐ |
| 简化模式（Implicit） | 纯前端应用（已不推荐，被 PKCE 替代） | ⭐⭐ |
| 密码模式（Resource Owner Password） | 高度信任的第一方应用 | ⭐⭐⭐ |
| 客户端凭证模式（Client Credentials） | 机器对机器，无用户参与 | ⭐⭐⭐⭐ |

### 2. 授权码模式（最重要）完整流程

```
┌─────────┐                    ┌──────────────┐              ┌──────────────────┐
│  用 户   │                    │  客户端应用   │              │ GitHub 授权服务器 │
└────┬────┘                    └──────┬───────┘              └────────┬─────────┘
     │                                │                               │
     │  1. 点击"用 GitHub 登录"        │                               │
     │───────────────────────────────>│                               │
     │                                │ 2. 重定向到 GitHub 授权页面    │
     │<───────────────────────────────│ （携带 client_id、scope、      │
     │ 3. 浏览器跳转到 GitHub 登录页   │    redirect_uri、state）       │
     │────────────────────────────────────────────────────────────────>
     │ 4. 用户在 GitHub 上登录并同意授权                                │
     │<────────────────────────────────────────────────────────────────
     │ 5. GitHub 重定向回应用（携带 authorization_code）                │
     │───────────────────────────────>│                               │
     │                                │ 6. 用 code + client_secret   │
     │                                │    换取 access_token          │
     │                                │────────────────────────────-->│
     │                                │ 7. 返回 access_token          │
     │                                │<──────────────────────────────│
     │                                │ 8. 用 access_token 访问资源   │
     │                                │────────────────────────────-->│
     │                                │ 9. 返回用户信息                │
     │  10. 登录成功，显示用户界面      │<──────────────────────────────│
     │<───────────────────────────────│                               │
```

**关键参数说明**：
- `client_id`：应用在 GitHub 注册时获得的唯一标识
- `client_secret`：应用的秘密凭证（**只在服务端使用，绝不暴露给前端**）
- `scope`：申请的权限范围（如 `read:user email`）
- `redirect_uri`：授权完成后跳转回的地址
- `state`：随机字符串，防止 CSRF 攻击
- `authorization_code`：临时授权码，**一次性、短期有效**（防止泄露后被重复使用）

### 3. state 参数的安全作用

```python
import secrets

# 发起授权请求时，生成随机 state 并存 session
state = secrets.token_urlsafe(32)
session["oauth_state"] = state

# 构造授权 URL
auth_url = (
    f"https://github.com/login/oauth/authorize"
    f"?client_id={CLIENT_ID}"
    f"&redirect_uri={REDIRECT_URI}"
    f"&scope=read:user+user:email"
    f"&state={state}"    # 携带 state
)

# 回调时验证 state
@app.get("/callback")
def callback(code: str, state: str):
    if state != session.get("oauth_state"):
        raise HTTPException(400, "state 不匹配，可能是 CSRF 攻击！")
    # state 验证通过，继续换取 token...
```

### 4. Scope（权限范围）

Scope 限制了 Token 能访问的资源范围，最小权限原则：

```python
# GitHub 常见 Scope
scopes = {
    "read:user":  "读取用户基本信息（头像、用户名）",
    "user:email": "读取用户邮箱",
    "repo":       "访问仓库（读写权限）",
    "delete_repo": "删除仓库",
    "admin:org":  "管理组织",
}

# 最小权限原则：只申请你实际需要的权限
# 用 GitHub 登录只需要：read:user user:email
# 不要申请 repo admin:org 等不必要的权限
```

### 5. FastAPI 中的 OAuth2（简化版）

FastAPI 内置了 OAuth2 支持，常用于 API 认证：

```python
from fastapi import FastAPI, Depends, HTTPException
from fastapi.security import OAuth2PasswordBearer, OAuth2PasswordRequestForm

app = FastAPI()

# OAuth2PasswordBearer 告诉 FastAPI 从哪里获取 token
oauth2_scheme = OAuth2PasswordBearer(tokenUrl="/token")

@app.post("/token")
async def login(form_data: OAuth2PasswordRequestForm = Depends()):
    """OAuth2 密码模式：用账号密码换取 token"""
    user = authenticate_user(form_data.username, form_data.password)
    if not user:
        raise HTTPException(status_code=401, detail="账号或密码错误")
    
    access_token = create_access_token(data={"sub": user.username})
    return {"access_token": access_token, "token_type": "bearer"}

async def get_current_user(token: str = Depends(oauth2_scheme)):
    """从 token 中提取当前用户（作为依赖注入）"""
    payload = verify_token(token)
    return payload

@app.get("/me")
async def read_me(current_user = Depends(get_current_user)):
    """受保护的接口，需要携带 token"""
    return {"user": current_user}
```

### 6. OAuth2 vs JWT 的关系

```
OAuth2    → 授权框架，定义了"如何获取 token"的流程
JWT       → Token 格式，定义了"token 长什么样"

关系：OAuth2 常用 JWT 作为 Access Token 的格式，
     但 OAuth2 也可以用不透明 Token（Opaque Token）

类比：
OAuth2 = 货币系统（如何流通货币）
JWT    = 纸币的印刷格式（货币长什么样）
```

## 四、实际场景

### 场景 1：实现第三方 GitHub 登录

```python
import httpx
import secrets
from fastapi import FastAPI, Request
from fastapi.responses import RedirectResponse

app = FastAPI()

GITHUB_CLIENT_ID = "your_client_id"
GITHUB_CLIENT_SECRET = "your_client_secret"  # 只在服务端使用！
REDIRECT_URI = "http://localhost:8000/auth/github/callback"

@app.get("/auth/github/login")
async def github_login(request: Request):
    """步骤1：重定向到 GitHub 授权页"""
    state = secrets.token_urlsafe(32)
    request.session["oauth_state"] = state
    
    github_auth_url = (
        f"https://github.com/login/oauth/authorize"
        f"?client_id={GITHUB_CLIENT_ID}"
        f"&redirect_uri={REDIRECT_URI}"
        f"&scope=read:user+user:email"
        f"&state={state}"
    )
    return RedirectResponse(github_auth_url)

@app.get("/auth/github/callback")
async def github_callback(code: str, state: str, request: Request):
    """步骤2：GitHub 回调，用 code 换取 access_token"""
    # 验证 state 防 CSRF
    if state != request.session.get("oauth_state"):
        return {"error": "CSRF 验证失败"}
    
    # 用 code 换 access_token
    async with httpx.AsyncClient() as client:
        token_response = await client.post(
            "https://github.com/login/oauth/access_token",
            headers={"Accept": "application/json"},
            data={
                "client_id": GITHUB_CLIENT_ID,
                "client_secret": GITHUB_CLIENT_SECRET,  # 服务端才有！
                "code": code,
                "redirect_uri": REDIRECT_URI,
            }
        )
        token_data = token_response.json()
        access_token = token_data["access_token"]
    
    # 用 access_token 获取用户信息
    async with httpx.AsyncClient() as client:
        user_response = await client.get(
            "https://api.github.com/user",
            headers={"Authorization": f"Bearer {access_token}"}
        )
        github_user = user_response.json()
    
    # 创建或更新本地用户，返回我们自己的 JWT
    local_token = create_access_token({"sub": str(github_user["id"])})
    return {"access_token": local_token, "github_username": github_user["login"]}
```

## 五、常见误区

| 误区 | 纠正 |
|------|------|
| "OAuth2 是认证协议" | OAuth2 是**授权**框架，不是认证协议。认证是 OpenID Connect（基于 OAuth2 构建的） |
| "client_secret 可以放在前端代码" | 绝对不行！client_secret 必须只在服务端保存，前端代码会被所有人看到 |
| "authorization_code 可以重复使用" | 授权码是一次性的，使用后立即失效，且有效期很短（通常 10 分钟） |
| "scope 随便申请，多多益善" | 最小权限原则：只申请需要的权限，过宽的 scope 增加安全风险，用户也更不愿意授权 |

## 六、面试高频题

**Q：OAuth2.0 授权码模式的完整流程？**
A：1) 客户端重定向用户到授权服务器；2) 用户登录并同意授权；3) 授权服务器重定向回客户端（携带 code）；4) 客户端用 code + client_secret 向授权服务器换取 access_token；5) 客户端用 access_token 访问资源服务器。

**Q：为什么要有 authorization_code，不直接返回 access_token？**
A：授权码通过浏览器 URL 传递（可能被日志/历史记录看到），设计为一次性、短期有效，降低泄露风险。access_token 的换取需要 client_secret（只有服务端有），多了一道安全验证。

**Q：OAuth2 和 OpenID Connect 的区别？**
A：OAuth2 是授权框架（解决"你能做什么"），OpenID Connect（OIDC）是在 OAuth2 基础上构建的身份认证层（解决"你是谁"），它在 OAuth2 的 Token 中增加了 `id_token`（一个 JWT，包含用户身份信息）。
