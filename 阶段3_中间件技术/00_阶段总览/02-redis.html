结构特点底层实现典型场景

🔍 展开：Hash vs String 存对象的选择策略

存一个用户对象有两种方式：

方式 A：String + JSON 序列化 — SET user:1 '{"name":"zs","age":28}'，整体读写，适合对象小且整体读取的场景。

方式 B：Hash 存字段 — HSET user:1 name zs age 28，可单独更新某字段而无需重写整个对象，适合频繁更新个别属性的场景。

选择原则：频繁更新对象的某一字段 → Hash（只更新那个 field）；对象整体读取/很少修改 → String + JSON。

🔍 展开：List 底层结构演变与性能陷阱

Redis 3.2 前用 linkedlist + ziplist，3.2+ 统一为 quicklist（ziplist 组成的双向链表）。

⚠️ LRANGE 是 O(N)，不要用于超大列表全量获取；LINDEX 也是 O(N)，按索引取中间元素性能差——需要按位置取元素时应用 ZSet 而非 List。

1. String — 缓存与计数器

# 基础读写
SET user:1:name "张三"
GET user:1:name            # → "张三"

# 带过期时间的写入（原子操作，防止死锁）
SETEX token:abc123 7200 "session_data"

# 计数器（原子自增，并发安全）
INCR article:1:views         # → 1 → 2 → 3 ...
INCRBY article:1:views 10   # 一次加 10

# 分布式锁关键命令
SETNX lock:order:123 "owner_id"  # 仅当不存在时设置

2. Hash — 对象字段操作

# 存用户信息（可单独更新某字段）
HSET user:1 name "张三" age 28 email "zs@qq.com"
HGET user:1 name              # → "张三"
HMGET user:1 name age         # → ["张三", "28"]
HINCRBY user:1 age 1        # age 变为 29（原子操作）
HKEYS user:1                  # → ["name","age","email"]
HLEN user:1                   # → 3

3. List — 消息队列

# 生产者：从右端入队
RPUSH queue:tasks "task1" "task2" "task3"

# 消费者：从左端阻塞弹出（队列空时等待）
BLPOP queue:tasks 5          # 阻塞最多 5 秒

# 获取最新 20 条（最新的在左边）
LRANGE messages:room:1 0 19

4. Set — 社交关系

# 用户关注列表
SADD user:1:follows 101 205 333
SADD user:2:follows 205 333 404

# 共同关注（交集）→ {205, 333}
SINTER user:1:follows user:2:follows

# 我关注了但对方没关注的（差集）→ {101}
SDIFF user:1:follows user:2:follows

# 随机抽 3 个幸运用户
SRANDMEMBER online:users 3

5. ZSet — 排行榜

# 添加/更新分数
ZADD leaderboard 9800 "player:101"
ZINCRBY leaderboard 200 "player:101"  # → score=10000

# Top 10（倒序，分数高在前）
ZREVRANGE leaderboard 0 9 WITHSCORES

# 查某玩家排名（从 0 开始）
ZREVRANK leaderboard "player:101"  # → 0（第一名）

# 按分数段查询（8000~10000 分的玩家）
ZRANGEBYSCORE leaderboard 8000 10000

⚠️ 常见误区

避坑指南

▼
❌ 误区✅ 纠正
Key 随便起名，如 name、token用冒号分隔层级：user:1:token，避免键名冲突
用 KEYS * 查看所有 Key生产环境禁止！全量扫描阻塞 Redis。用 SCAN 0 游标迭代
单个 List/Hash 存几万元素无所谓大 Key 问题：超过 1 万元素要警惕，序列化慢、内存大、迁移困难
Redis 是单线程所以可以随便用 O(N) 命令恰恰相反！单线程意味着耗时命令（SORT 大集合）会阻塞所有请求
把 Redis 当成 MySQL 用，存了不设过期Redis 是缓存，不设过期时间 → 内存泄漏 → OOM Kill
🚨 安全警告：KEYS * 在生产环境是灾难级操作——它遍历整个键空间，执行期间 Redis 无法处理其他请求。曾有案例因运维误执行 KEYS 导致服务雪崩、P99 延迟从 2ms 飙升到 2000ms。

📖 核心概念

四大场景全景

▼

场景一：缓存（Cache-Aside 模式）

┌─────────────────────────────────────────────────────────┐
│              Cache-Aside（旁路缓存）模式                    │
├─────────────────────┬───────────────────────────────────┤
│      读流程          │          写流程                    │
│                     │                                   │
│  请求 → 查 Redis    │  请求 → 写数据库                   │
│    ↓命中     ↓未命中 │           ↓                       │
│  直接返回   查数据库  │      删除 Redis 缓存              │
│             ↓        │      （不更新，下次读时重建）        │
│          写入 Redis  │                                   │
│             ↓        │                                   │
│           返回数据    │                                   │
└─────────────────────┴───────────────────────────────────┘

写时为什么删缓存而不是更新缓存？并发写场景下，线程 A 和 B 同时更新数据库和缓存 → 谁最后更新缓存谁的值就"胜出" → 但数据库里可能是另一个值 → 数据不一致。删除缓存更安全：下次读时自动从库拉最新。

场景二：分布式 Session

传统服务器端 Session 存在单机内存中 → 多台部署时，负载均衡导致 Session 丢失。Redis 是独立外部存储，所有节点共享。

⚠️ 关键设计：滑动过期——用户每次活动都应刷新 TTL，避免正在操作时突然登出。

场景三：排行榜（ZSet）

ZSet 天然支持按 score 排序，ZINCRBY 原子更新分数，ZREVRANGE 获取 Top N。多维度（日/周/月榜）用不同 Key 区分。

场景四：分布式锁

设计要点原因
SET NX EX 而非 SETNX+EXPIRESETNX 和 EXPIRE 是两步，中间崩溃 → 死锁
锁 value 用唯一 UUID防止 A 超时后 B 获取锁，A 再来误删 B 的锁
Lua 脚本释放锁GET + DEL 两步必须原子执行
合理过期时间业务超时后自动释放，防止死锁

1. Cache-Aside 缓存模式

import json
import redis

r = redis.Redis(host='localhost', port=6379, db=0)

def get_user(user_id: int) -> dict:
    cache_key = f"user:{user_id}"
    cached = r.get(cache_key)          # 1. 先查缓存
    if cached:
        return json.loads(cached)        # 命中 → 直接返回

    user = db.query_user(user_id)          # 2. 未命中 → 查库
    r.setex(cache_key, 3600, json.dumps(user)) # 3. 写缓存 + 设过期
    return user

def update_user(user_id: int, data: dict):
    db.update_user(user_id, data)         # 1. 先写库
    r.delete(f"user:{user_id}")          # 2. 删缓存（下次读时重建）

2. 分布式 Session（滑动过期）

import uuid, json
SESSION_EXPIRE = 1800  # 30分钟

def create_session(user_id: int, role: str) -> str:
    session_id = str(uuid.uuid4())
    data = {"user_id": user_id, "role": role}
    r.setex(f"session:{session_id}", SESSION_EXPIRE, json.dumps(data))
    return session_id

def get_session(session_id: str) -> dict | None:
    data = r.get(f"session:{session_id}")
    if not data:
        return None
    r.expire(f"session:{session_id}", SESSION_EXPIRE)  # 滑动过期
    return json.loads(data)

3. 排行榜（ZSet）

def add_score(player_id: int, points: int):
    r.zincrby("leaderboard:game:weekly", points, f"player:{player_id}")

def get_top_n(n: int = 10) -> list:
    results = r.zrevrange("leaderboard:game:weekly", 0, n-1, withscores=True)
    return [{"player": m.decode(), "score": int(s)} for m, s in results]

def get_rank(player_id: int) -> int | None:
    rank = r.zrevrank("leaderboard:game:weekly", f"player:{player_id}")
    return rank + 1 if rank is not None else None

4. 分布式锁（SET NX EX + Lua 释放）

def acquire_lock(lock_key: str, expire: int = 10) -> str | None:
    lock_value = str(uuid.uuid4())  # 唯一标识，防止误删
    result = r.set(lock_key, lock_value, nx=True, ex=expire)
    return lock_value if result else None

def release_lock(lock_key: str, lock_value: str) -> bool:
    lua = """
    if redis.call("GET", KEYS[1]) == ARGV[1] then
        return redis.call("DEL", KEYS[1])
    else
        return 0
    end
    """
    return bool(r.eval(lua, 1, lock_key, lock_value))

▼
❌ 误区✅ 纠正
写操作后更新缓存而非删除缓存并发写时更新缓存导致数据不一致，删缓存更安全

📖 核心概念

三问题对比与原理图解

▼

一、缓存穿透 — 查不存在的数据

请求: GET /product/-1（数据库中不存在的 ID）
  ↓
查 Redis → Miss（从未缓存过）
  ↓
查 MySQL → 不存在（NULL）
  ↓
什么都没缓存 ← 每次相同请求都走完整链路
  ↓
恶意攻击：构造 -1、0、随机字符串狂发请求 → 数据库被打爆

解决方案：

① 缓存空值：查到 NULL 也写入 Redis，设短过期（5 分钟）。缺点：攻击者用无穷不同 ID 会填满 Redis。

② 布隆过滤器 ⭐ 推荐：在 Redis 前加一层，存储所有合法 ID。请求先进布隆 → 不在则直接返回 404。

布隆过滤器原理：
  存数据: ID=1001 → hash1→bit[3]=1, hash2→bit[7]=1, hash3→bit[15]=1
  查数据: ID=1001 → 查 bit[3]、bit[7]、bit[15] 是否都为 1
          ✓ 都为 1 → "可能存在"（有误判率 0.1%~1%）
          ✗ 任一位为 0 → "一定不存在"（100% 准确）
特点：占用极少内存，不支持删除，误判率可控

二、缓存击穿 — 热点 Key 突然过期

T 时刻：热点 Key "product:1001" 过期
  ↓
1000 个并发请求同时查缓存 → 全部 Miss
  ↓
1000 个请求同时查数据库 ← 击穿！
  ↓
数据库可能崩溃（瞬时并发太高）

解决方案：① 互斥锁：缓存 Miss 时只允许一个请求去查库，其他等待。② 逻辑过期 ⭐：缓存永不过期，value 存逻辑过期时间，过期后异步更新，当前请求返回旧数据。

三、缓存雪崩 — 大量 Key 同时过期

场景 A：整点批量刷新，所有 Key 设相同 TTL=1h
       → 1小时后全部同时过期 → 雪崩
场景 B：Redis 服务器宕机
       → 所有请求没有缓存层 → 全打数据库 → 系统崩溃

解决方案：① 过期时间加随机抖动（±20%）。② 热点数据永不过期 + 后台定时刷新。③ Redis 高可用部署（主从+哨兵/Cluster）。

问题触发条件影响范围核心解决方案
穿透查询不存在的数据每次特定 Key布隆过滤器 / 缓存空值
击穿热点 Key 突然过期某个热点 Key 并发互斥锁 / 逻辑过期
雪崩大量 Key 同时过期或宕机整个系统随机 TTL / 永不过期 / 高可用
🧠 记忆口诀：
穿透 → 根本没有 → 用布隆过滤器拦在门口
击穿 → 热点消失 → 用锁保护，一个人去取数据
雪崩 → 集体失效 → 时间错开，不要集体退场

def get_product(product_id: int) -> dict | None:
    cache_key = f"product:{product_id}"
    cached = r.get(cache_key)
    if cached is not None:
        if cached == b"NULL":           # 空值标记
            return None
        return json.loads(cached)

    product = db.get_product(product_id)
    if product is None:
        r.setex(cache_key, 300, "NULL")  # 缓存空值 5 分钟
        return None
    r.setex(cache_key, 3600, json.dumps(product))
    return product

2. 缓存击穿 — 互斥锁方案

def get_hot_product(product_id: int) -> dict | None:
    cache_key = f"product:{product_id}"
    lock_key  = f"lock:product:{product_id}"

    cached = r.get(cache_key)
    if cached:
        return json.loads(cached)

    # 缓存 Miss，尝试获取互斥锁
    lock_val = str(uuid.uuid4())
    if r.set(lock_key, lock_val, nx=True, ex=5):
        try:
            product = db.get_product(product_id)  # 只有一人查库
            if product:
                r.setex(cache_key, 3600, json.dumps(product))
            return product
        finally:
            release_lock(lock_key, lock_val)       # Lua 原子释放
    else:
        time.sleep(0.05)                        # 等待后重试
        return get_hot_product(product_id)

3. 缓存击穿 — 逻辑过期方案 ⭐

def get_hot_product_v2(product_id: int) -> dict | None:
    cache_key = f"product:{product_id}"
    cached = r.get(cache_key)
    if not cached: return None

    data = json.loads(cached)
    if time.time() > data["__expire_at__"]:    # 逻辑过期
        lock_key = f"lock:product:{product_id}"
        if r.set(lock_key, "1", nx=True, ex=5):
            threading.Thread(target=rebuild, args=(product_id,)).start()

    # 返回旧数据（允许短暂不一致，但避免击穿）
    return {k:v for k,v in data.items() if k != "__expire_at__"}

过期、淘汰与持久化全景

▼

一、Key 过期策略（两种组合）

┌──────────────────────────────────────────────────────┐
│              Redis Key 过期策略（双机制协作）           │
├─────────────────────┬────────────────────────────────┤
│   惰性删除           │        定期删除                 │
│   (Lazy Expiration) │   (Periodic Expiration)        │
├─────────────────────┼────────────────────────────────┤
│ 访问 Key 时才检查     │ 每 100ms 随机抽取一批           │
│ 过期 → 删除 + 返null │ 检查并删除已过期的 Key           │
│                     │                                │
│ 优点: 零 CPU 开销     │ 优点: 主动回收内存              │
│ 缺点: 不访问不删 →   │ 缺点: 随机抽取 → 不保证         │
│      内存泄漏风险     │      全部过期 Key 被清理        │
└─────────────────────┴────────────────────────────────┘
协作：惰性删除兜底（访问即检查），定期删除扫墓（定时主动清理）

二、内存淘汰策略（8 种）

策略说明适用场景
noeviction不淘汰，内存满时报错数据不允许丢失
allkeys-lru ⭐淘汰最近最少使用的通用缓存场景，最常用
volatile-lru仅淘汰有过期时间中最少使用的混合存储（持久+缓存）
allkeys-lfu淘汰访问频率最低的热点数据分布不均
volatile-lfu有过期时间中淘汰频率最低的-
allkeys-random随机淘汰任意 Key数据均匀访问
volatile-random随机淘汰有过期时间的 Key-
volatile-ttl淘汰剩余存活时间最短的希望快过期的先淘汰
🔍 LRU vs LFU 深度对比
维度LRULFU
含义Least Recently Used（最久未访问）Least Frequently Used（最少被访问）
淘汰逻辑上次访问距今最久的淘汰总访问次数最少的淘汰
缺点热点数据可能被一次批量操作冲掉新数据访问少容易被误淘汰
Redis 版本所有版本Redis 4.0+
推荐通用缓存首选热点数据分布严重不均时

三、持久化：RDB vs AOF

┌────────────────────────────────────────────────────────┐
│                    RDB（快照持久化）                       │
│  主进程 fork() → 子进程遍历内存 → 写入 dump.rdb           │
│  技术: Copy-On-Write，子进程看到的是 fork 时刻的快照        │
│                                                         │
│  触发: save 3600 1  # 3600s内1次修改                    │
│        save 300 100  # 300s内100次修改                   │
│        save 60 10000 # 60s内10000次修改                  │
│        手动: BGSAVE（异步）/ SAVE（同步，禁用）            │
├────────────────────────────────────────────────────────┤
│                    AOF（追加日志持久化）                    │
│  每条写命令追加写入 appendonly.aof → 恢复时重放所有命令     │
│                                                         │
│  fsync 策略:                                            │
│    always   — 每条都刷盘，最安全，最慢                     │
│    everysec — 每秒刷盘，最多丢1秒 ⭐ 默认推荐              │
│    no       — OS 决定，性能最好，可能丢较多                 │
│                                                         │
│  AOF 重写: BGREWRITEAOF 压缩日志（防止文件无限增长）        │
└────────────────────────────────────────────────────────┘
对比维度RDBAOF
持久化方式快照（全量）日志（增量）
数据丢失快照间隔内的数据会丢最多丢失 1 秒
文件体积小，紧凑大，文本格式
恢复速度快慢（重放命令）
适用场景定期备份、允许少量丢失数据安全性要求高
⭐ 生产推荐：RDB + AOF 混合持久化（Redis 4.0+，aof-use-rdb-preamble yes）
AOF 文件头部存储 RDB 快照 + 后续追加增量 AOF 日志 → 恢复速度接近 RDB，数据完整性接近 AOF。

避坑指南

▼
❌ 误区✅ 纠正
生产环境未配置 maxmemory内存用尽 → OOM Kill → 服务中断。必须设 maxmemory + 淘汰策略
RDB save 全部注释掉 + AOF 也没开重启直接丢所有数据！至少开一个持久化方案
AOF 文件大到几十 GB 不管定期触发 BGREWRITEAOF 或合理配置自动重写阈值
大内存（>10GB）fork() 阻塞不关心fork() 会短暂阻塞 Redis 主线程，内存越大阻塞越久，需监控慢日志
持久化目录权限不对，Redis 静默失败启动后检查 INFO persistence 确认 rdb_last_bgsave_status 为 ok
🚨 真实事故：某团队 RDB 配置被注释、AOF 未开启，运维重启 Redis 后发现所有缓存数据丢失。由于缓存预热脚本也有 bug，数据库瞬间被请求打爆，P99 延迟从 5ms 飙升到 30 秒，服务不可用长达 40 分钟。
