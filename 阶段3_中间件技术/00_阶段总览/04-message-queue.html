第一节 · 消息队列概念与场景

消息队列用一条队列换来异步、解耦、削峰三件套，是后端架构的减压阀。
🎯 学习目标▼
  • 理解 Producer、Consumer、Broker、Queue。
  • 区分异步、解耦、削峰三类价值。
  • 知道消息可靠性和幂等的关系。
📖 核心概念▼
Producer → Broker/Queue → Consumer
下单主流程 → 发消息 → 后台发短信/积分/推送
作用含义典型场景
异步不等慢操作结果邮件、报表、通知
解耦服务不直接调用订单事件广播
削峰先排队再消费秒杀下单
💻 代码示例▼

来源：demo/demo_1_send_tasks.py

# Web 请求中只提交任务，不等待执行完成
result = send_email.delay("user@example.com", "欢迎注册", "感谢您注册")
return {"task_id": result.id, "status": "queued"}
⚠️ 常见误区▼
误区纠正
用了队列就不会丢消息还需要 ACK、重试、持久化、死信策略。
所有操作都异步支付扣款、强一致库存不应随意异步。

第二节 · Celery 异步任务

Celery 把耗时任务丢给 Worker 异步执行，让 Web 请求秒级返回。
🎯 学习目标▼
  • 掌握 Celery broker/backend/worker 架构。
  • 会定义任务、提交任务、查询状态。
  • 理解重试、ACK、预取与定时任务。
📖 核心概念▼
Web 代码 → delay/apply_async → Broker
Worker ← 消费任务 ← Broker
Backend ← 存储任务结果
💻 代码示例▼

来源：demo/demo_1_celery_worker.py

app = Celery("demo", broker="redis://localhost:6379/0", backend="redis://localhost:6379/1")

@app.task(name="tasks.add", bind=True)
def add(self, x: int, y: int) -> int:
    return x + y
⚠️ 常见误区▼
误区纠正
在 Web 请求中 result.get()会阻塞请求，应该返回 task_id 轮询。
Worker 预取过多长任务会堆在单个 Worker，可设 prefetch=1。

第三节 · Broker 选型：Redis vs RabbitMQ

Broker 选 Redis 还是 RabbitMQ，本质是在简单部署和强路由能力之间做权衡。
🎯 学习目标▼
  • 理解 Redis 与 RabbitMQ 的能力边界。
  • 知道 Exchange、Queue、Routing Key。
  • 能按可靠性、路由、运维成本选型。
📖 核心概念▼
维度RedisRabbitMQ
部署简单，项目常已有需额外运维
路由简单队列名Exchange/Binding/Routing Key
可靠性依赖持久化配置原生能力更完整
适合中小项目复杂路由/高可靠项目
💻 代码示例▼

来源：demo/demo_2_advanced.py

app.conf.task_queues = (
    Queue("default", routing_key="default"),
    Queue("high", routing_key="high"),
)
app.conf.task_routes = {"tasks.urgent_task": {"queue": "high"}}
⚠️ 常见误区▼
误区纠正
Redis Broker 等于专业 MQRedis 简单快速，但路由/死信能力弱。
RabbitMQ 一定更好小项目可能运维复杂度大于收益。
