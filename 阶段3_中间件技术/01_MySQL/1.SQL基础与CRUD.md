# 1. SQL 基础与 CRUD

> 掌握 SQL 最核心的四种操作：增（INSERT）、查（SELECT）、改（UPDATE）、删（DELETE）。

---

## 一、概念解释

**CRUD** 是数据库操作的四个基本动作，对应 SQL 的四条核心语句：

| 操作 | SQL 语句 | HTTP 方法对比 |
|------|---------|-------------|
| 增（Create）| `INSERT` | POST |
| 查（Read）  | `SELECT` | GET  |
| 改（Update）| `UPDATE` | PUT/PATCH |
| 删（Delete）| `DELETE` | DELETE |

---

## 二、为什么需要

任何应用的数据持久化都离不开这四种操作。电商系统中：
- 用户注册 → INSERT users
- 查看商品列表 → SELECT products
- 修改订单状态 → UPDATE orders
- 取消订单 → UPDATE（逻辑删除）或 DELETE

## 三、建表语法

### 3.1 CREATE TABLE 完整语法

```sql
CREATE TABLE 表名 (
    字段名  数据类型  [约束条件]  [COMMENT '注释'],
    ...
    [CONSTRAINT 约束名 约束类型 (字段)]
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

### 3.2 常用数据类型

| 分类 | 类型 | 说明 | 示例场景 |
|------|------|------|---------|
| 整数 | `INT` / `INT UNSIGNED` | 4字节，有/无符号 | 普通ID、数量 |
| 整数 | `BIGINT` | 8字节 | 雪花ID、大流量系统 |
| 小数 | `DECIMAL(10,2)` | 精确小数 | **金额必用**，不用 FLOAT |
| 字符 | `VARCHAR(n)` | 可变长字符串 | 用户名、标题 |
| 字符 | `CHAR(n)` | 固定长字符串 | 手机号、状态码 |
| 文本 | `TEXT` | 长文本（最大 65535） | 商品描述、日志 |
| 日期 | `DATETIME` | 日期+时间，不含时区 | 创建时间 |
| 日期 | `TIMESTAMP` | 时间戳（含时区，范围到2038年） | 慎用 |
| 枚举 | `ENUM('v1','v2')` | 枚举值 | 状态字段 |

> **金额为什么用 DECIMAL 而不是 FLOAT？**  
> FLOAT 是近似值：`0.1 + 0.2 = 0.30000000000000004`。DECIMAL 是精确计算，金融场景必须用 DECIMAL。

### 3.3 常用约束

```sql
PRIMARY KEY      -- 主键（唯一 + 非空）
NOT NULL         -- 非空
UNIQUE           -- 唯一
DEFAULT 值       -- 默认值
AUTO_INCREMENT   -- 自增（仅 INT 类型）
FOREIGN KEY      -- 外键
```

## 四、INSERT — 插入数据

### 4.1 基本语法

```sql
-- 插入单行（推荐：指定字段名，字段顺序与表定义无关）
INSERT INTO users (username, email, phone, password_hash, balance, status)
VALUES ('小明', 'xiaoming@example.com', '13912345678', '$2b$12$xxx', 100.00, 'active');

-- 插入多行（一次性提交，比多条 INSERT 性能更好）
INSERT INTO users (username, email, phone, password_hash) VALUES
('小红', 'xiaohong@example.com', '13987654321', '$2b$12$yyy'),
('小蓝', 'xiaolan@example.com',  '13800000001', '$2b$12$zzz');
```

### 4.2 坑：字段值顺序必须与字段列表一一对应

```sql
-- ❌ 危险：不写字段名，依赖表定义顺序（表结构变更后必出错）
INSERT INTO users VALUES (NULL, '小明', 'xxx@example.com', ...);

-- ✅ 推荐：明确写出字段名
INSERT INTO users (username, email, ...) VALUES ('小明', 'xxx@example.com', ...);
```

## 五、SELECT — 查询数据

### 5.1 基础查询

```sql
-- 查所有字段（不推荐在生产代码中使用 *，会多传输不需要的字段）
SELECT * FROM users;

-- 查指定字段
SELECT id, username, email, status FROM users;

-- 给字段取别名
SELECT id, username AS 用户名, balance AS 账户余额 FROM users;
```

### 5.2 WHERE 条件过滤

```sql
-- 等值查询
SELECT * FROM users WHERE status = 'active';

-- 比较运算符：=  !=  >  >=  <  <=
SELECT * FROM products WHERE price > 1000;

-- 逻辑运算：AND  OR  NOT
SELECT * FROM products WHERE price > 100 AND stock > 0;

-- 范围：BETWEEN ... AND ...（含两端）
SELECT * FROM products WHERE price BETWEEN 100 AND 1000;

-- 集合：IN (...)
SELECT * FROM users WHERE status IN ('active', 'inactive');

-- 模糊查询：LIKE（% 匹配任意多字符，_ 匹配单个字符）
SELECT * FROM products WHERE title LIKE '%手机%';
SELECT * FROM users WHERE phone LIKE '138%';

-- 空值判断（必须用 IS NULL / IS NOT NULL，不能用 = NULL）
SELECT * FROM products WHERE description IS NULL;
```

### 5.3 ORDER BY 排序

```sql
-- 升序（默认 ASC）
SELECT id, title, price FROM products ORDER BY price;

-- 降序
SELECT id, title, price FROM products ORDER BY price DESC;

-- 多字段排序：先按 status 升序，再按 price 降序
SELECT id, title, price, status
FROM products
ORDER BY status ASC, price DESC;
```

### 5.4 LIMIT 分页

```sql
-- 取前 5 条
SELECT id, title, price FROM products LIMIT 5;

-- 分页：LIMIT offset, count
-- 第 1 页（跳过 0 条，取 10 条）
SELECT id, title, price FROM products LIMIT 0, 10;

-- 第 2 页（跳过 10 条，取 10 条）
SELECT id, title, price FROM products LIMIT 10, 10;

-- 或者用 LIMIT count OFFSET offset 语法（更清晰）
SELECT id, title, price FROM products LIMIT 10 OFFSET 10;
```

## 六、UPDATE — 修改数据

```sql
-- 修改单个字段
UPDATE users SET balance = 200.00 WHERE id = 1;

-- 修改多个字段
UPDATE orders
SET status = 'paid', updated_at = NOW()
WHERE order_no = 'ORD20260101001';

-- 基于原值计算（给用户余额增加 50 元）
UPDATE users SET balance = balance + 50.00 WHERE id = 1;
```

> **安全警告**：UPDATE 不加 WHERE 条件会更新全表所有行！生产环境一定先用 SELECT 确认范围再 UPDATE。

---

## 七、DELETE — 删除数据

```sql
-- 删除单条
DELETE FROM users WHERE id = 10;

-- 删除多条
DELETE FROM orders WHERE status = 'cancelled' AND created_at < '2026-01-01';

-- ⚠️ 不加 WHERE 会清空整张表！
-- DELETE FROM users;  -- 危险！
```

### 逻辑删除 vs 物理删除

| 方式 | 实现 | 优点 | 缺点 |
|------|------|------|------|
| 物理删除 | `DELETE` | 彻底清除，不占空间 | 无法恢复，影响关联数据 |
| 逻辑删除 | 增加 `is_deleted` 字段，`UPDATE is_deleted=1` | 可恢复，保留历史 | 查询时需加 `WHERE is_deleted=0` |

> 电商系统普遍使用**逻辑删除**，用户信息/订单数据一般不物理删除。

## 八、NULL 处理

NULL 是数据库中的"未知值"，与空字符串 `''` 不同：

```sql
-- NULL 的比较必须用 IS NULL / IS NOT NULL
SELECT * FROM products WHERE description IS NULL;     -- ✅
SELECT * FROM products WHERE description = NULL;      -- ❌ 永远返回空结果！

-- NULL 参与运算结果仍为 NULL
SELECT NULL + 1;     -- 结果：NULL
SELECT NULL = NULL;  -- 结果：NULL（不是 1！）

-- COALESCE：返回第一个非 NULL 的值（常用于设置默认值）
SELECT COALESCE(description, '暂无描述') AS 描述 FROM products;

-- IFNULL：MySQL 专有，等同于 COALESCE 的两参数版
SELECT IFNULL(description, '暂无描述') AS 描述 FROM products;
```

## 九、防 SQL 注入（参数化查询）

**SQL 注入**是最常见的安全漏洞，通过构造恶意输入来篡改 SQL 语句：

```python
# ❌ 危险：字符串拼接 SQL（生产代码绝对禁止）
username = "admin' OR '1'='1"
sql = f"SELECT * FROM users WHERE username = '{username}'"
# 实际执行：SELECT * FROM users WHERE username = 'admin' OR '1'='1'
# 结果：返回所有用户！

# ✅ 安全：使用参数占位符（具体写法见 03_Python_ORM 章节）
# Python + PyMySQL 示例：
cursor.execute("SELECT * FROM users WHERE username = %s", (username,))
```

> SQL 文件中的练习均为直接 SQL，生产代码中通过 ORM 或参数化查询传参，永远不要手动拼接 SQL 字符串。

## 十、常见坑

| 坑 | 说明 | 解决方案 |
|----|------|---------|
| 忘写 WHERE | UPDATE/DELETE 影响全表 | 养成习惯：先 SELECT 确认范围 |
| `= NULL` | 查不到任何结果 | 改为 `IS NULL` |
| FLOAT 精度 | 金额计算出现小数误差 | 金额字段用 `DECIMAL` |
| LIKE 全模糊 | `LIKE '%xxx%'` 无法使用索引 | 尽量用前缀匹配 `LIKE 'xxx%'` |
| 大小写 | VARCHAR 字段默认大小写不敏感 | 明文密码比对用 `BINARY` 关键字 |
