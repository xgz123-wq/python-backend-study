# 2. 多表联查与子查询

> 现实业务数据都分散在多张表中，JOIN 是连接它们的桥梁。

---

## 一、概念解释

**多表联查（JOIN）**：将多张表按某个关联字段合并成一张临时结果表再查询。  
**子查询（Subquery）**：将一个 SELECT 语句嵌套在另一个 SQL 语句中，先执行内层查询，再将结果传给外层。

---

## 二、为什么需要

电商系统中，订单表只存了 `user_id` 和 `product_id`，若要显示"张三购买了 iPhone 16 Pro，花了 9999 元"，就必须把三张表联起来查。单表查询无法满足这个需求。

---

## 三、JOIN 联表查询

### 3.1 JOIN 类型对比

| 类型 | 关键字 | 结果 |
|------|--------|------|
| 内连接 | `INNER JOIN` / `JOIN` | 只保留两表都有匹配的行 |
| 左连接 | `LEFT JOIN` | 保留左表所有行，右表无匹配显示 NULL |
| 右连接 | `RIGHT JOIN` | 保留右表所有行，左表无匹配显示 NULL |
| 全连接 | 无（MySQL 不支持 `FULL JOIN`）| 可用 `LEFT JOIN UNION RIGHT JOIN` 模拟 |

```
INNER JOIN 示意：
 users           orders
┌──┬──────┐     ┌──┬─────────┬─────────┐
│1 │ 张三 │────▶│1 │ user_id=1│ 9999.00│  ← 有匹配，保留
│2 │ 李四 │────▶│2 │ user_id=2│  899.00│  ← 有匹配，保留
│3 │ 王五 │  ✗  │  │         │        │  ← 无订单，不保留
└──┴──────┘     └──┴─────────┴─────────┘

LEFT JOIN 示意（以 users 为左表）：
 users           orders
┌──┬──────┐     ┌──┬─────────┬─────────┐
│1 │ 张三 │────▶│1 │ user_id=1│ 9999.00│  ← 保留
│2 │ 李四 │────▶│2 │ user_id=2│  899.00│  ← 保留
│3 │ 王五 │──▶ NULL   NULL      NULL   ← 无订单，用 NULL 填充
└──┴──────┘
```

### 3.2 INNER JOIN（内连接）

```sql
-- 查询订单信息，同时显示用户名（只显示有订单的用户）
SELECT
    o.order_no,
    u.username,
    o.total_amount,
    o.status
FROM orders o
INNER JOIN users u ON o.user_id = u.id
ORDER BY o.created_at DESC;

-- 三表联查：订单 + 用户名 + 商品名
SELECT
    o.order_no,
    u.username  AS 买家,
    p.title     AS 商品,
    o.quantity  AS 数量,
    o.total_amount AS 总金额,
    o.status    AS 订单状态
FROM orders o
JOIN users   u ON o.user_id    = u.id
JOIN products p ON o.product_id = p.id
ORDER BY o.created_at DESC;
```

### 3.3 LEFT JOIN（左连接）

```sql
-- 查询所有用户及其订单数（包含没有下过单的用户）
SELECT
    u.id,
    u.username,
    COUNT(o.id) AS 订单数
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
GROUP BY u.id, u.username
ORDER BY 订单数 DESC;

-- 找出从未下过单的用户（RIGHT NULL 技巧）
SELECT u.id, u.username, u.email
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
WHERE o.id IS NULL;  -- 右表无匹配的 → 从未下单
```

## 四、聚合函数

| 函数 | 说明 |
|------|------|
| `COUNT(*)` | 统计行数（含 NULL） |
| `COUNT(字段)` | 统计非 NULL 的行数 |
| `SUM(字段)` | 求和 |
| `AVG(字段)` | 求平均值 |
| `MAX(字段)` | 最大值 |
| `MIN(字段)` | 最小值 |

```sql
-- 统计已付款订单的总金额和平均金额
SELECT
    COUNT(*)            AS 订单总数,
    SUM(total_amount)   AS 总收入,
    AVG(total_amount)   AS 平均客单价,
    MAX(total_amount)   AS 最大订单,
    MIN(total_amount)   AS 最小订单
FROM orders
WHERE status = 'paid';
```

## 五、GROUP BY + HAVING

**GROUP BY**：按某个字段分组，每组产生一行聚合结果。  
**HAVING**：对分组后的结果再过滤（WHERE 在分组前过滤，HAVING 在分组后过滤）。

```sql
-- 统计每个用户的订单数和消费总额
SELECT
    u.username,
    COUNT(o.id)         AS 订单数,
    SUM(o.total_amount) AS 消费总额
FROM users u
JOIN orders o ON u.id = o.user_id
WHERE o.status = 'paid'          -- WHERE：分组前过滤（只统计已付款订单）
GROUP BY u.id, u.username
HAVING SUM(o.total_amount) > 500  -- HAVING：分组后过滤（消费超 500 的用户）
ORDER BY 消费总额 DESC;

-- 统计每种商品分类的在售商品数和平均价格
SELECT
    c.name       AS 分类,
    COUNT(p.id)  AS 商品数,
    AVG(p.price) AS 平均售价,
    MIN(p.price) AS 最低价,
    MAX(p.price) AS 最高价
FROM categories c
LEFT JOIN products p ON c.id = p.category_id AND p.status = 'on_sale'
GROUP BY c.id, c.name
ORDER BY 商品数 DESC;
```

> **WHERE vs HAVING 区别**：
> - `WHERE` 在 `GROUP BY` 之前执行，不能用聚合函数
> - `HAVING` 在 `GROUP BY` 之后执行，可以用聚合函数

## 六、子查询

### 6.1 WHERE 子查询

```sql
-- 查询购买了最贵商品的订单（先找最高价格，再查对应订单）
SELECT order_no, user_id, total_amount
FROM orders
WHERE product_id = (
    SELECT id FROM products ORDER BY price DESC LIMIT 1
);

-- 查询消费总额超过平均消费的用户
SELECT username, (
    SELECT SUM(total_amount)
    FROM orders
    WHERE user_id = u.id AND status = 'paid'
) AS 消费总额
FROM users u
WHERE (
    SELECT COALESCE(SUM(total_amount), 0)
    FROM orders
    WHERE user_id = u.id AND status = 'paid'
) > (
    SELECT AVG(user_total)
    FROM (
        SELECT user_id, SUM(total_amount) AS user_total
        FROM orders WHERE status = 'paid'
        GROUP BY user_id
    ) AS sub
);
```

### 6.2 IN 子查询

```sql
-- 查询有未支付订单的用户信息
SELECT id, username, email, balance
FROM users
WHERE id IN (
    SELECT DISTINCT user_id
    FROM orders
    WHERE status = 'unpaid'
);

-- 查询从未被购买过的商品（库存虽然有但没订单）
SELECT id, title, price, stock
FROM products
WHERE id NOT IN (
    SELECT DISTINCT product_id FROM orders
);
```

### 6.3 FROM 子查询（派生表）

```sql
-- 先按用户汇总消费，再找出消费最多的前 3 名
SELECT username, 消费总额
FROM (
    SELECT
        u.username,
        SUM(o.total_amount) AS 消费总额
    FROM users u
    JOIN orders o ON u.id = o.user_id
    WHERE o.status = 'paid'
    GROUP BY u.id, u.username
) AS user_stats          -- 派生表必须有别名
ORDER BY 消费总额 DESC
LIMIT 3;
```

## 七、SQL 执行顺序

理解 SQL 执行顺序是写出正确 SQL 的关键：

```
FROM        → 1. 确定数据来源（包括 JOIN）
WHERE       → 2. 行级过滤（分组前）
GROUP BY    → 3. 分组
HAVING      → 4. 组级过滤（分组后）
SELECT      → 5. 选取字段（计算别名）
ORDER BY    → 6. 排序（此时才能用 SELECT 中定义的别名）
LIMIT       → 7. 限制返回行数
```

> 常见坑：在 WHERE 中使用 SELECT 定义的别名会报错，因为 WHERE 比 SELECT 先执行。

```sql
-- ❌ 报错：WHERE 先执行，total_price 别名还不存在
SELECT price * quantity AS total_price
FROM orders
WHERE total_price > 1000;

-- ✅ 正确：用 HAVING 或重复写表达式
SELECT price * quantity AS total_price
FROM orders
HAVING total_price > 1000;
-- 或
SELECT total_amount FROM orders WHERE total_amount > 1000;
```

## 八、常见坑

| 坑 | 说明 | 解决方案 |
|----|------|---------|
| GROUP BY 字段不完整 | SELECT 了非聚合字段但没放入 GROUP BY | 把所有非聚合字段都放入 GROUP BY |
| LEFT JOIN 后 WHERE 变 INNER JOIN | `WHERE right_table.col = xxx` 会过滤掉 NULL 行 | 把右表条件移到 `ON` 子句中 |
| 子查询返回多行 | `= (子查询)` 但子查询返回多行 | 改用 `IN (子查询)` |
| JOIN 顺序影响性能 | 小表驱动大表更快 | 把小结果集放在前面（优化器通常会自动处理） |
