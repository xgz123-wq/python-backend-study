# 3. 索引原理与优化

> 索引是数据库性能的核心，理解它才能写出真正"快"的 SQL。

---

## 一、概念解释

**索引**是数据库为加速查询而建立的数据结构，类比于书的目录：没有目录要翻遍全书找内容，有了目录直接定位到页码。

没有索引 → **全表扫描**：遍历每一行，数据量大时极慢。  
有了索引 → **索引扫描**：按索引数据结构快速定位，O(log n) 时间复杂度。

---

## 二、为什么需要

电商系统典型场景：
- 用户登录：`WHERE email = 'xxx'`，无索引需扫描全部用户
- 商品搜索：`WHERE category_id = 1 AND status = 'on_sale'`，百万商品中快速筛选
- 订单查询：`WHERE user_id = 123 ORDER BY created_at DESC`，用户查自己的历史订单

---

## 三、B+ 树索引原理

MySQL InnoDB 的索引底层使用 **B+ 树**（Balanced+ Tree）。

```
B+ 树特性：
- 所有数据存储在叶节点（Leaf Node）
- 叶节点之间有双向链表连接（支持范围查询）
- 非叶节点只存储键值和指针（可以存更多键，树高更低）
- 通常只需 3~4 层即可存储千万级数据
- 树高固定，每次查询磁盘 IO 次数恒定（通常 2~4 次）

树高 3 能存多少数据？
每个节点 16KB，存储 INT 型键值（4字节）+ 指针（6字节）= 10字节/条
→ 每个非叶节点可存 16384 / 10 ≈ 1638 个键
→ 树高 3：1638 × 1638 × 叶节点数据 ≈ 2000万 行
```

---

## 四、聚簇索引 vs 二级索引

### 4.1 聚簇索引（Clustered Index）

- 即**主键索引**，叶节点直接存储**完整行数据**
- InnoDB 每张表有且仅有一个聚簇索引
- 按主键查询：叶节点直接返回数据，一次 B+ 树遍历搞定

```
聚簇索引叶节点：
┌──────────────────────────────────┐
│ id=1 │ username='张三' │ email=... │ balance=1580 │ ...完整数据 │
└──────────────────────────────────┘
```

### 4.2 二级索引（Secondary Index）

- 也叫辅助索引，叶节点只存储**索引字段值 + 主键值**
- 通过二级索引查到主键后，再用主键回聚簇索引取完整数据 → **回表**
- 如果 SELECT 的字段都在索引中，可以不回表 → **覆盖索引**

```
email 字段上的二级索引叶节点：
┌──────────────────────────────┐
│ email='zhangsan@...' │ id=1 │  ← 只存 email + 主键 id
└──────────────────────────────┘
查询 WHERE email='...' 时：
1. 遍历 email 索引，找到 id=1
2. 拿 id=1 回聚簇索引取完整数据（回表）
```

## 五、索引类型与创建

```sql
-- 1. 主键索引（建表时自动创建）
PRIMARY KEY (id)

-- 2. 唯一索引（值唯一，可以有 NULL）
CREATE UNIQUE INDEX idx_users_email ON users(email);

-- 3. 普通索引（最常用）
CREATE INDEX idx_orders_user_id ON orders(user_id);

-- 4. 联合索引（多字段组合）
CREATE INDEX idx_orders_user_status ON orders(user_id, status);

-- 5. 查看表的所有索引
SHOW INDEX FROM orders;

-- 6. 删除索引
DROP INDEX idx_orders_user_id ON orders;
```

## 六、联合索引与最左前缀原则

联合索引 `(user_id, status, created_at)` 在 B+ 树中先按 `user_id` 排序，`user_id` 相同时按 `status` 排序，以此类推。

```sql
CREATE INDEX idx_orders_composite ON orders(user_id, status, created_at);

-- ✅ 能用到索引（满足最左前缀）
WHERE user_id = 1
WHERE user_id = 1 AND status = 'paid'
WHERE user_id = 1 AND status = 'paid' AND created_at > '2026-01-01'

-- ⚠️ 只能用到 user_id 部分（中间字段跳过了 status）
WHERE user_id = 1 AND created_at > '2026-01-01'

-- ❌ 完全不能用索引（没有从最左字段开始）
WHERE status = 'paid'
WHERE created_at > '2026-01-01'
```

**记忆方法**：像电话簿按姓 → 名排序，查"李明"可以；查名字叫"明"的人就得全翻。

## 七、EXPLAIN 分析执行计划

```sql
EXPLAIN SELECT * FROM orders WHERE user_id = 1;
```

关键字段解读：

| 字段 | 含义 | 关注点 |
|------|------|--------|
| `type` | 访问类型 | 从好到差：`system > const > eq_ref > ref > range > index > ALL` |
| `key` | 实际使用的索引 | NULL 表示没用索引（全表扫描） |
| `rows` | 预估扫描行数 | 越小越好 |
| `Extra` | 额外信息 | `Using index`（覆盖索引）最优；`Using filesort`（需要排序，可能慢） |

```
type 解读：
ALL    → 全表扫描（最差，百万数据下非常慢）
index  → 全索引扫描（比 ALL 稍好，遍历索引树）
range  → 索引范围扫描（WHERE id > 100 这类）
ref    → 非唯一索引等值查询（常见于 WHERE user_id = 1）
const  → 唯一索引等值查询（WHERE id = 1，至多返回一行，最优之一）
```

## 八、索引失效的 6 大场景

```sql
-- 场景 1：对索引字段使用函数
-- ❌ 失效：对 created_at 使用了 YEAR() 函数
WHERE YEAR(created_at) = 2026
-- ✅ 改写：用范围查询
WHERE created_at BETWEEN '2026-01-01' AND '2026-12-31 23:59:59'

-- 场景 2：隐式类型转换
-- ❌ 失效：phone 是 VARCHAR，但传入数字（MySQL 会隐式转换）
WHERE phone = 13800001001
-- ✅ 改写：保持类型一致
WHERE phone = '13800001001'

-- 场景 3：LIKE 前缀通配符
-- ❌ 失效：% 在前面，无法用 B+ 树
WHERE title LIKE '%手机%'
-- ✅ 能用索引：前缀匹配
WHERE title LIKE '手机%'

-- 场景 4：OR 条件中有字段没有索引
-- ❌ 失效：若 balance 没有索引，整个 OR 条件退化为全表扫描
WHERE email = 'a@b.com' OR balance > 1000
-- ✅ 改写：保证 OR 两边的字段都有索引，或改用 UNION
SELECT * FROM users WHERE email = 'a@b.com'
UNION
SELECT * FROM users WHERE balance > 1000;

-- 场景 5：NOT IN / NOT EXISTS（通常无法用索引）
-- ❌ 通常失效
WHERE status NOT IN ('active', 'disabled')
-- ✅ 考虑改写为范围查询

-- 场景 6：联合索引违反最左前缀
-- ❌ 失效（见上节示例）
WHERE status = 'paid'  -- 联合索引 (user_id, status)，跳过了 user_id
```

## 九、常见坑

| 坑 | 说明 | 解决方案 |
|----|------|---------|
| 索引不是越多越好 | 每个索引都占磁盘空间，且 INSERT/UPDATE/DELETE 时需维护 | 只给高频查询的 WHERE/JOIN/ORDER BY 字段加索引 |
| 小表不需要索引 | 数据量 < 1000 行时，全表扫描比索引快（索引有额外开销） | 表数据量超过 1万 行再考虑加索引 |
| 覆盖索引的价值 | 若 SELECT 的字段全在索引中，可避免回表 | 高频只读查询考虑联合索引覆盖 SELECT 字段 |
| 区分度低的字段不适合单独建索引 | 如 status 只有 4 个值，选择性极低，单独索引效果差 | 与高区分度字段组成联合索引 |
