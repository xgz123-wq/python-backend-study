# 4. 事务与隔离级别

> 事务是数据库的"原子操作"保障，隔离级别决定并发时数据的可见性。

---

## 一、概念解释

**事务（Transaction）**：一组 SQL 操作，要么全部成功，要么全部回滚，不存在"执行了一半"的中间态。

**隔离级别**：多个事务并发执行时，一个事务能看到其他事务的哪些修改，由隔离级别控制。

---

## 二、为什么需要

电商下单场景：
1. 扣减库存：`UPDATE products SET stock = stock - 1 WHERE id = 1`
2. 创建订单：`INSERT INTO orders (...)`
3. 扣减余额：`UPDATE users SET balance = balance - 9999 WHERE id = 1`

如果第 2 步成功、第 3 步失败（余额不足），没有事务则会出现：**有订单但没扣钱**的数据不一致问题。事务确保三步操作要么全成功，要么全回滚。

---

## 三、ACID 四大特性

| 特性 | 全称 | 说明 | 实现机制 |
|------|------|------|---------|
| **A** 原子性 | Atomicity | 事务内操作不可分割，全成功或全回滚 | Undo Log |
| **C** 一致性 | Consistency | 事务前后数据库从一个合法状态到另一个合法状态 | 其他三个特性共同保证 |
| **I** 隔离性 | Isolation | 并发事务互不干扰，各自看到的数据一致 | MVCC + 锁 |
| **D** 持久性 | Durability | 事务提交后，数据永久保存，不会因宕机丢失 | Redo Log |

---

## 四、事务控制语句

```sql
-- 开启事务
BEGIN;
-- 或
START TRANSACTION;

-- 提交事务（所有操作生效）
COMMIT;

-- 回滚事务（所有操作撤销）
ROLLBACK;

-- 查看自动提交状态（默认 ON，即每条 SQL 自动提交）
SHOW VARIABLES LIKE 'autocommit';

-- 关闭自动提交（当前会话有效）
SET autocommit = 0;
-- 开启自动提交
SET autocommit = 1;
```

## 五、事务隔离级别

### 5.1 四种隔离级别

| 隔离级别 | 脏读 | 不可重复读 | 幻读 | 说明 |
|---------|------|-----------|------|------|
| `READ UNCOMMITTED` | ✅ 有 | ✅ 有 | ✅ 有 | 可读取未提交数据（几乎不用） |
| `READ COMMITTED` | ❌ 无 | ✅ 有 | ✅ 有 | 每次读最新已提交（Oracle 默认） |
| `REPEATABLE READ` | ❌ 无 | ❌ 无 | ⚠️ 有* | 同一事务内重复读结果一致（MySQL 默认） |
| `SERIALIZABLE` | ❌ 无 | ❌ 无 | ❌ 无 | 串行执行，性能极差（极少使用） |

> *InnoDB 的 REPEATABLE READ 通过 MVCC 解决了大多数幻读问题

### 5.2 三种并发问题

**脏读（Dirty Read）**：事务 A 读到了事务 B **未提交**的数据，如果 B 回滚，A 读到的数据就是"脏数据"。

```
时间线：
T1: BEGIN; UPDATE users SET balance=0 WHERE id=1;  -- 未 COMMIT
T2: SELECT balance FROM users WHERE id=1;           -- 读到 0（脏读！）
T1: ROLLBACK;  -- 实际 balance 没变，但 T2 已经读到了错误值
```

**不可重复读（Non-Repeatable Read）**：同一事务内，两次读取同一行，结果不同（因为其他事务在中间提交了修改）。

```
时间线：
T1: BEGIN; SELECT balance FROM users WHERE id=1;  -- 读到 1580
T2: UPDATE users SET balance=100 WHERE id=1; COMMIT;
T1: SELECT balance FROM users WHERE id=1;          -- 读到 100（不可重复读！）
T1: COMMIT;
```

**幻读（Phantom Read）**：同一事务内，两次按条件查询，第二次出现了新的行（因为其他事务插入了数据）。

```
时间线：
T1: BEGIN; SELECT COUNT(*) FROM orders WHERE user_id=1;  -- 返回 3
T2: INSERT INTO orders (...) user_id=1; COMMIT;
T1: SELECT COUNT(*) FROM orders WHERE user_id=1;          -- 返回 4（幻读！）
T1: COMMIT;
```

### 5.3 查看和设置隔离级别

```sql
-- 查看当前隔离级别（MySQL 8.0）
SELECT @@transaction_isolation;

-- 查看（MySQL 5.7）
SELECT @@tx_isolation;

-- 设置当前会话的隔离级别（不影响其他连接）
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 设置全局隔离级别（影响所有新连接）
SET GLOBAL TRANSACTION ISOLATION LEVEL REPEATABLE READ;
```

## 六、MVCC 简介

**MVCC（Multi-Version Concurrency Control，多版本并发控制）**是 InnoDB 实现高并发读写的核心机制。

核心思想：**读不加锁，写时创建新版本**，读写不互相阻塞。

```
每行数据隐含两个字段：
- trx_id：最后修改本行的事务 ID
- roll_pointer：指向 Undo Log 中的旧版本数据

Undo Log 链（版本链）：
当前版本(trx_id=5) → 旧版本(trx_id=3) → 更旧版本(trx_id=1)

READ VIEW（读视图）：
- 事务开始时生成一个快照，记录当前活跃事务列表
- 读取数据时，沿版本链找到对本事务可见的最新版本
- REPEATABLE READ：整个事务期间使用同一个 READ VIEW
- READ COMMITTED：每次读都生成新的 READ VIEW
```

## 七、死锁

**死锁**：两个事务互相持有对方需要的锁，双方都在等待，永久阻塞。

```
场景：
T1: UPDATE orders SET status='paid' WHERE id=1;  -- 持有 id=1 的锁
T2: UPDATE orders SET status='paid' WHERE id=2;  -- 持有 id=2 的锁
T1: UPDATE orders SET status='cancelled' WHERE id=2;  -- 等待 T2 释放
T2: UPDATE orders SET status='cancelled' WHERE id=1;  -- 等待 T1 释放
→ 死锁！MySQL 会自动检测并回滚代价小的事务
```

**避免死锁的原则**：
1. 按固定顺序访问表和行（如都按 id 升序加锁）
2. 事务尽量短，减少持锁时间
3. 避免在事务中等待用户输入

## 八、长事务的危害

```sql
-- 查看当前运行超过 60 秒的事务（生产环境监控必备）
SELECT
    trx_id,
    trx_started,
    TIMESTAMPDIFF(SECOND, trx_started, NOW()) AS 运行秒数,
    trx_query AS 当前SQL
FROM information_schema.INNODB_TRX
WHERE TIMESTAMPDIFF(SECOND, trx_started, NOW()) > 60;
```

**长事务的危害**：
1. 大量 Undo Log 无法回收，导致磁盘暴涨
2. 持锁时间长，其他事务阻塞等待
3. 主从复制延迟

## 九、常见坑

| 坑 | 说明 | 解决方案 |
|----|------|---------|
| 忘记 COMMIT | 修改数据后没提交，关闭连接时自动回滚 | 显式 BEGIN + COMMIT，或用框架的事务上下文管理器 |
| 事务包含非事务操作 | DDL（CREATE/ALTER/DROP）在 InnoDB 中会隐式提交 | DDL 不要放在事务内 |
| SELECT FOR UPDATE | 悲观锁，会阻塞其他写操作 | 只在必须强一致性的场景（如扣库存）使用 |
| 事务太大 | 一个事务操作了大量行，持锁时间长 | 拆分成小事务，批量处理用分批提交 |
