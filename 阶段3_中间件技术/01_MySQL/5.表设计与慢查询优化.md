# 5. 表设计与慢查询优化

> 好的表设计是性能的基础，慢查询优化是后端工程师必备技能。

---

## 一、概念解释

**表设计**：在建表阶段就做出正确的字段类型选择、索引规划和范式取舍，避免后期大规模重构。  
**慢查询优化**：找出执行时间超过阈值的 SQL，通过 EXPLAIN、索引调整、SQL 改写等手段提速。

---

## 二、三大范式

数据库设计的规范化理论，帮助消除冗余数据：

| 范式 | 要求 | 违反后果 |
|------|------|---------|
| **1NF** 第一范式 | 每列不可再分（原子性） | 字段存多值（如 "iPhone,iPad"）难以查询 |
| **2NF** 第二范式 | 非主键字段完全依赖于主键（消除部分依赖） | 数据冗余，更新异常 |
| **3NF** 第三范式 | 非主键字段不依赖于其他非主键字段（消除传递依赖） | 冗余更严重，增删改需同步多处 |

### 实例：违反三范式 vs 遵循三范式

```
❌ 违反设计（orders 表中冗余用户名和商品名）：
orders(id, order_no, user_id, username, product_id, product_title, price, ...)
→ username 和 product_title 直接写在 orders 表中
→ 用户改名时，需要更新所有订单记录！

✅ 遵循设计（只存外键，通过 JOIN 获取详情）：
orders(id, order_no, user_id, product_id, ...)
users(id, username, ...)
products(id, title, price, ...)
```

> **实际工程中的反范式化**：为了性能，有时故意冗余某些字段（如订单表冗余商品名称快照），但需要明确记录设计理由和同步策略。

## 三、列类型选择

### 3.1 整数类型

| 类型 | 字节 | 范围（有符号） | 使用场景 |
|------|------|-------------|---------|
| `TINYINT` | 1 | -128~127 | 枚举、标志位（`is_deleted`） |
| `SMALLINT` | 2 | -32768~32767 | 小范围计数 |
| `INT` | 4 | -21亿~21亿 | 常规 ID（`UNSIGNED` 可到 42亿） |
| `BIGINT` | 8 | -922亿亿~922亿亿 | 雪花 ID、高并发系统 |

```sql
-- 推荐：用 INT UNSIGNED 作为自增主键（够用且节省空间）
id INT UNSIGNED AUTO_INCREMENT PRIMARY KEY

-- 大流量系统用 BIGINT（雪花算法生成的 ID 超过 INT 范围）
id BIGINT UNSIGNED NOT NULL COMMENT '雪花ID'
```

### 3.2 金额字段——必须用 DECIMAL

```sql
-- ✅ 正确：精确小数，(10,2) 表示最多10位，其中2位小数
price       DECIMAL(10, 2) NOT NULL COMMENT '单价（元）'
balance     DECIMAL(10, 2) NOT NULL DEFAULT 0.00

-- ❌ 错误：FLOAT/DOUBLE 是近似值，金额计算有误差
price FLOAT   -- 严禁！
```

### 3.3 字符串类型

| 类型 | 特点 | 适用场景 |
|------|------|---------|
| `CHAR(n)` | 固定长度，不足补空格，查询快 | 固定格式：手机号 `CHAR(11)`、国家代码 `CHAR(2)` |
| `VARCHAR(n)` | 可变长，按实际存储，需额外1-2字节存长度 | 用户名、邮箱、标题 |
| `TEXT` | 最大 65535 字节，不能设默认值 | 商品描述、评论内容 |
| `MEDIUMTEXT` | 最大 16MB | 文章内容 |

```sql
-- VARCHAR 的 n 是字符数（不是字节数），utf8mb4 下每个汉字最多 4 字节
username    VARCHAR(50)  -- 最多 50 个字符（中英文均可）
email       VARCHAR(100)
phone       CHAR(11)     -- 手机号固定 11 位，用 CHAR
```

### 3.4 时间类型

| 类型 | 范围 | 时区 | 推荐 |
|------|------|------|------|
| `DATETIME` | 1000-9999年 | 不存时区，存入什么显示什么 | ✅ 推荐 |
| `TIMESTAMP` | 1970-2038年 | 存 UTC，显示时转换为服务器时区 | ⚠️ 2038年问题 |
| `DATE` | 只有日期 | - | 生日、有效期 |

```sql
-- 推荐写法：DATETIME + DEFAULT CURRENT_TIMESTAMP
created_at  DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
updated_at  DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
```

## 四、逻辑删除设计

```sql
-- 方案 1：is_deleted 标志位（TINYINT，最简单）
ALTER TABLE products ADD COLUMN is_deleted TINYINT(1) NOT NULL DEFAULT 0;
-- 删除操作
UPDATE products SET is_deleted = 1 WHERE id = 18;
-- 查询时过滤已删除
SELECT * FROM products WHERE is_deleted = 0;

-- 方案 2：deleted_at 时间戳（可以知道删除时间）
ALTER TABLE products ADD COLUMN deleted_at DATETIME DEFAULT NULL;
UPDATE products SET deleted_at = NOW() WHERE id = 18;
SELECT * FROM products WHERE deleted_at IS NULL;
```

> **逻辑删除注意事项**：唯一索引字段（如 email、order_no）的逻辑删除需要特殊处理，否则软删除后无法重新注册相同邮箱。解决方案：删除时将 email 改为 `email_deleted_时间戳` 或使用组合唯一索引 `(email, is_deleted)`。

## 五、慢查询优化

### 5.1 开启慢查询日志

```sql
-- 查看慢查询配置
SHOW VARIABLES LIKE 'slow_query%';
SHOW VARIABLES LIKE 'long_query_time';  -- 默认 10 秒，一般设为 1 秒

-- 开启慢查询日志（当前会话）
SET GLOBAL slow_query_log = ON;
SET GLOBAL long_query_time = 1;  -- 超过 1 秒才记录

-- 查看慢查询日志位置
SHOW VARIABLES LIKE 'slow_query_log_file';
```

### 5.2 深分页优化

电商商品列表翻到第 10000 页（LIMIT 99990, 10）性能极差：

```sql
-- ❌ 深分页：扫描并丢弃 99990 行，越翻越慢
SELECT id, title, price
FROM products
ORDER BY id
LIMIT 99990, 10;

-- ✅ 方案 1：游标分页（记住上次最后一个 id）
-- 前端记录上次最后的 id（如 last_id = 99990）
SELECT id, title, price
FROM products
WHERE id > 99990   -- 直接从上次停的地方继续，走主键索引
ORDER BY id
LIMIT 10;

-- ✅ 方案 2：延迟关联（先用覆盖索引取 id，再 JOIN 取完整数据）
SELECT p.id, p.title, p.price
FROM products p
JOIN (
    SELECT id FROM products ORDER BY id LIMIT 99990, 10
) AS ids ON p.id = ids.id;
```

### 5.3 ORDER BY 优化

```sql
-- ❌ 排序字段没有索引，产生 Using filesort（内存/磁盘排序）
SELECT * FROM orders ORDER BY total_amount DESC LIMIT 10;

-- ✅ 给排序字段加索引，走索引排序（Using index）
CREATE INDEX idx_orders_amount ON orders(total_amount);

-- ✅ 联合索引同时覆盖 WHERE + ORDER BY
-- 索引 (user_id, created_at) 可以优化：
SELECT * FROM orders WHERE user_id = 1 ORDER BY created_at DESC;
```

## 六、字段命名与设计规范

| 规范 | 示例 | 说明 |
|------|------|------|
| 小写+下划线 | `order_no`、`created_at` | 大小写兼容，Linux 下不易出错 |
| 不用缩写 | `description` 而非 `desc`（`DESC` 是 SQL 关键字！） | 避免和关键字冲突 |
| 时间字段后缀 | `created_at`、`updated_at`、`deleted_at` | 统一命名，ORM 框架可自动处理 |
| 状态字段 | `status ENUM(...)` 或 `TINYINT` | 枚举值语义清晰，TINYINT 更灵活 |
| 金额字段 | `price DECIMAL(10,2)` | 字段名包含"price/amount/balance"，类型统一用 DECIMAL |
| 主键 | `id INT UNSIGNED AUTO_INCREMENT` | 统一命名为 `id` |
| 外键 | `user_id`、`product_id` | `被引用表名_id` 格式 |

## 七、常见坑

| 坑 | 说明 | 解决方案 |
|----|------|---------|
| TEXT 字段不能设默认值 | MySQL 中 TEXT/BLOB 不支持 DEFAULT | 在应用层处理默认值，或改用 VARCHAR |
| VARCHAR 过长建索引 | MySQL 索引键最大 767 字节，VARCHAR(255) utf8mb4 超过限制 | 索引前缀：`INDEX(title(50))` |
| NULL vs 空字符串 | NULL 表示"未知"，`''` 表示"已知为空" | 设计时明确语义，金额字段必须 NOT NULL |
| 自增 ID 回绕 | INT UNSIGNED 最大约 42亿，高并发系统可能用尽 | 预估数据量，必要时用 BIGINT |
| 时间存 VARCHAR | `created_at VARCHAR(20)` 存 "2026-01-01 10:00:00" | 必须用 DATETIME，支持范围查询和索引 |
