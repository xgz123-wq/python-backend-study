第 0 节 · MySQL 环境与基础操作

入门预备课：在学 SQL 语法之前，先把 Linux 下操控 MySQL 的基本功打牢。
🎯 目标 学习目标 ▼

与后端开发的关联：排查线上数据问题、本地初始化开发环境，命令行操作 MySQL 都是每天必备的技能。
📖 概念 核心概念 ▼

MySQL 是最流行的开源关系型数据库，广泛用于 Web 后端。使用 Linux 终端操作 MySQL 是后端工程师的基本功。

字符集对比

字符集最大字节/字符能否存 Emoji
utf8（MySQL专有）3 字节❌ 不支持（Emoji 需要 4 字节）

结论：新项目一律使用 utf8mb4 + utf8mb4_unicode_ci（大小写不敏感）。

第 1 节 · SQL 基础与 CRUD

掌握 SQL 最核心的四种操作：增（INSERT）、查（SELECT）、改（UPDATE）、删（DELETE）。
🎯 目标 学习目标 ▼
  • 掌握创建表的基本语法及常用数据类型选择（特别是 DECIMAL）
  • 熟练编写增删改查（CRUD）操作语句
  • 掌握 WHERE 条件过滤、ORDER BY 排序与 LIMIT 分页
  • 理解并正确处理数据库中的 NULL 值
与后端开发的关联：任何应用的数据持久化核心都是 CRUD。电商系统中，商品列表展示对应 SELECT 分页，购买结算对应 INSERT 和 UPDATE。
📖 概念 核心概念 ▼

CRUD 是数据库操作的四个基本动作，对应 SQL 的四条核心语句：

操作SQL 语句HTTP 方法对比
增（Create）INSERTPOST
查（Read）SELECTGET
改（Update）UPDATEPUT / PATCH
删（Delete）DELETEDELETE

常用数据类型避坑指南

  • 整数：通常使用 INT；对于高并发系统生成的雪花 ID 需要使用 BIGINT（8 字节）。
  • 金额：必须使用 DECIMAL(10,2)，严禁使用 FLOAT，否则会导致进度丢失（0.1 + 0.2 = 0.30000000000000004）。
  • 空值（NULL）：不能用 = NULL 进行判断，必须使用 IS NULL 或 IS NOT NULL。
💻 示例 代码示例 ▼

1. 建表与数据类型

CREATE TABLE users (
    id INT AUTO_INCREMENT PRIMARY KEY,
    username VARCHAR(50) NOT NULL,
    
    balance DECIMAL(10,2) DEFAULT 0.00,  -- 金额必用 DECIMAL

    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

2. SELECT 基础查询、排序与分页

-- 查询分页商品列表（第 2 页，每页 5 条）

SELECT id, title, price, stock, status
FROM products
WHERE status = 'on_sale' AND price BETWEEN 100 AND 1000
ORDER BY price DESC
LIMIT 5 OFFSET 5;  -- 跳过前 5 条，取 5 条


-- 模糊查询与 NULL 判断

SELECT id, title, price
FROM products
WHERE title LIKE '%手机%' AND description IS NOT NULL;

3. INSERT / UPDATE / DELETE 数据更新

-- 插入多行并明确指定列名（推荐）

INSERT INTO users (username, balance) VALUES
('小红', 100.00), ('小蓝', 50.00);


-- 基于原值计算修改（给用户充值）

UPDATE users SET balance = balance + 50.00 WHERE id = 1;


-- 删除指定数据

DELETE FROM orders WHERE status = 'cancelled';
⚠️ 误区 常见误区 ▼
❌ 误区
使用 FLOAT 类型存储用户账户余额。

判断 description 为空时使用：WHERE description = NULL。

INSERT INTO users VALUES (NULL, 'name', ...);
✅ 纠正
使用 DECIMAL(10,2) 保证精度。

判断 NULL 必须使用：WHERE description IS NULL。

推荐明确写出字段名：INSERT INTO users (username, ...) VALUES ('name', ...);
安全警告： 执行 UPDATE 和 DELETE 操作如果不加 WHERE 条件，会导致全表数据被修改或清空！生产环境务必谨慎，建议先 SELECT 确认影响范围。

第 2 节 · 多表联查与子查询

现实业务数据都分散在多张表中，JOIN 是连接它们的桥梁。
🎯 目标 学习目标 ▼
  • 理解并熟练使用 INNER JOIN、LEFT JOIN 解决多表数据关联
  • 掌握使用 GROUP BY 与聚合函数进行数据统计
  • 区分 WHERE 与 HAVING 的执行时机和适用场景
  • 理解子查询（WHERE / IN / FROM）的用法
  • 牢记 SQL 的真实执行顺序
与后端开发的关联：现实业务数据往往分布在多个表中，报表统计和复杂列表查询必须依赖多表联查与聚合分析。
📖 概念 核心概念 ▼

多表联查（JOIN）：将多张表按某个关联字段合并成一张临时结果表再查询。

子查询（Subquery）：将一个 SELECT 语句嵌套在另一个 SQL 语句中，先执行内层查询，再将结果传给外层。

JOIN 类型对比

类型关键字结果
内连接INNER JOIN只保留两表都有匹配的行
左连接LEFT JOIN保留左表所有行，右表无匹配显示 NULL
右连接RIGHT JOIN保留右表所有行，左表无匹配显示 NULL

SQL 执行顺序

FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT
为什么 SELECT 别名不能在 WHERE 中使用？

根据 SQL 执行顺序，WHERE 在第 2 步执行（行级过滤），而 SELECT 及其别名定义在第 5 步才执行。因此在 WHERE 中使用未定义的别名会直接报错。如果需要对计算结果过滤，应使用 HAVING，因为它在 SELECT 和 GROUP BY 之后执行。

💻 示例 代码示例 ▼

1. 多表 JOIN

-- 查询所有用户及其订单数（包含没有下过单的用户用 LEFT JOIN）

SELECT
    u.id, u.username, COUNT(o.id) AS 订单数
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
GROUP BY u.id, u.username;


-- 找出从未下过单的用户（右表匹配为 NULL）

SELECT u.id, u.username
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
WHERE o.id IS NULL;

2. GROUP BY 与 HAVING

-- 统计消费超 500 的用户

SELECT
    u.username,
    SUM(o.total_amount) AS 消费总额
FROM users u
JOIN orders o ON u.id = o.user_id
WHERE o.status = 'paid'          -- 分组前过滤
GROUP BY u.id, u.username
HAVING SUM(o.total_amount) > 500 -- 分组后过滤
ORDER BY 消费总额 DESC;

3. 子查询

-- 查询有未支付订单的用户信息

SELECT id, username, balance
FROM users
WHERE id IN (
    SELECT DISTINCT user_id FROM orders WHERE status = 'unpaid'
);
⚠️ 误区 常见误区 ▼
❌ 误区
把 SELECT 中定义的别名直接放在 WHERE 中过滤。

在 LEFT JOIN 中，把右表的过滤条件放在 WHERE 里（如 WHERE o.status='paid'）。
✅ 纠正
WHERE 比 SELECT 先执行，必须把别名过滤换成 HAVING 或者重复表达式。

WHERE 中的过滤会剔除没有右表的行，导致 LEFT JOIN 变成 INNER JOIN，应当将右表条件放入 ON 中。

第 3 节 · 索引原理与优化

索引是数据库性能的核心，理解它才能写出真正"快"的 SQL。
🎯 目标 学习目标 ▼
  • 理解索引的作用及其底层 B+ 树数据结构
  • 区分聚簇索引和二级索引（非聚簇索引）的区别
  • 掌握回表查询与覆盖索引的概念及优化方案
  • 理解联合索引的“最左前缀原则”
与后端开发的关联：索引是数据库性能优化的核心。理解索引才能写出不拖垮数据库的高效 SQL。
📖 概念 核心概念 ▼

索引是数据库为加速查询而建立的数据结构，类比于书的目录。有了索引，查询不再是全表扫描，而是 O(log n) 级别的快速定位。

B+ 树特性

  • 所有数据存储在叶节点，且叶节点之间有双向链表连接，支持高效的范围查询。
  • 非叶节点只存储键值和指针。树高固定，通常 3~4 层即可存储千万级数据。

聚簇索引 vs 二级索引

类型特点叶子节点存储内容
聚簇索引InnoDB 每张表有且仅有一个（通常是主键）完整的行数据
二级索引用户自定义创建的普通索引或唯一索引索引字段值 + 对应的主键值
什么是“回表”与“覆盖索引”？

通过二级索引查询时，先在二级索引 B+ 树上查到主键 ID，然后再去聚簇索引的 B+ 树上查完整的行数据，这个过程称为**回表**。
如果查询的列恰好包含在二级索引中（比如通过 SELECT id, name FROM users WHERE name='张三'，而 name 有索引），不需要回表直接返回结果，这就是**覆盖索引**，性能极高。

💻 示例 代码示例 ▼

1. 索引的创建与查看

-- 为单列创建普通索引

CREATE INDEX idx_email ON users(email);


-- 为多列创建联合索引

CREATE INDEX idx_status_price ON products(status, price);


-- 查看表上的所有索引

SHOW INDEX FROM products;

2. 联合索引与最左前缀原则

-- 假设有联合索引 (category_id, status, price)



-- ✅ 索引生效（匹配最左前缀）

SELECT * FROM products WHERE category_id = 1;
SELECT * FROM products WHERE category_id = 1 AND status = 'on_sale';


-- ❌ 索引失效（没有匹配最左侧字段）

SELECT * FROM products WHERE status = 'on_sale';
SELECT * FROM products WHERE price > 100;
⚠️ 误区 常见误区 ▼
❌ 误区
表里的索引建得越多越好。

在索引字段上使用函数，如：WHERE YEAR(created_at) = 2026。

使用 LIKE '%关键词%' 进行模糊查询。
✅ 纠正
索引会占用磁盘空间，并且在 INSERT/UPDATE/DELETE 时需维护索引树，导致写入变慢。应按需创建。

在字段上使用函数会导致索引失效。应改为：created_at >= '2026-01-01'。

左侧带 % 会导致索引失效，应尽量使用前缀匹配 LIKE '关键词%' 或借助 ElasticSearch。

第 4 节 · 事务与隔离级别

事务是数据库的"原子操作"保障，隔离级别决定并发时数据的可见性。
🎯 目标 学习目标 ▼
  • 理解事务的作用与 ACID 四大特性
  • 掌握事务的基本控制语句（START TRANSACTION、COMMIT、ROLLBACK）
  • 理解并发事务带来的问题（脏读、不可重复读、幻读）
  • 掌握 MySQL 的四种隔离级别及其对应的解决方案
与后端开发的关联：只要涉及到资金扣减、库存变化等多步组合操作，必须使用事务，否则极易出现系统数据不一致。
📖 概念 核心概念 ▼

事务（Transaction）：一组 SQL 操作，要么全部成功，要么全部回滚，不存在执行了一半的中间态。

ACID 四大特性

特性全称说明
A 原子性Atomicity事务内操作不可分割，全成功或全回滚（依赖 Undo Log）
C 一致性Consistency事务前后数据库数据从一个合法状态到另一个合法状态
I 隔离性Isolation并发事务互不干扰，各自看到的数据一致（依赖 MVCC + 锁）
D 持久性Durability提交后数据永久保存，宕机不丢失（依赖 Redo Log）

并发问题与隔离级别

  • 脏读：读到了别的事务未提交的数据。
  • 不可重复读：同一个事务内两次读取同一行数据，结果不一样（别人 UPDATE 了）。
  • 幻读：同一个事务内两次相同的查询条件，查出的行数不一样（别人 INSERT/DELETE 了）。
隔离级别脏读不可重复读幻读
读未提交 (RU)❌ 会❌ 会❌ 会
读已提交 (RC)✅ 免❌ 会❌ 会
可重复读 (RR) - MySQL默认✅ 免✅ 免⚠️ 很大程度避免
串行化 (Serializable)✅ 免✅ 免✅ 免
💻 示例 代码示例 ▼

1. 事务控制基本操作

-- 开启事务

START TRANSACTION;
-- 也可以用 BEGIN;


-- 业务 SQL：扣减库存与扣除余额

UPDATE products SET stock = stock - 1 WHERE id = 1;
UPDATE users SET balance = balance - 100 WHERE id = 1;


-- 提交事务（永久生效）

COMMIT;


-- 或回滚事务（撤销上述所有修改）

ROLLBACK;
⚠️ 误区 常见误区 ▼
❌ 误区
只要将多条 SQL 放在一起执行就具有事务保护。

隔离级别设为越高越好，直接用 Serializable 保证绝对安全。
✅ 纠正
必须显式执行 START TRANSACTION 和 COMMIT/ROLLBACK 才能确保原子性。

串行化隔离级别会让所有请求排队，极大降低并发性能。通常保持 MySQL 默认的 RR 级别即可满足绝大多数业务需求。

第 5 节 · 表设计与慢查询优化

好的表设计是性能的基础，慢查询优化是后端工程师必备技能。
🎯 目标 学习目标 ▼
  • 理解并掌握数据库表设计的三大范式及反范式化的应用场景
  • 学会利用 EXPLAIN 关键字分析 SQL 慢查询并找到性能瓶颈
  • 掌握常见慢查询的优化思路及改进方案
与后端开发的关联：良好的表设计能预防未来数据灾难；当你接手一个老项目或者服务响应慢时，排查并优化慢 SQL 是体现高级工程师价值的核心工作。
📖 概念 核心概念 ▼

三大范式：数据库设计的规范化理论，帮助消除数据冗余和更新异常：

  • 第一范式（1NF）：确保每列的原子性，不可再分（如不能在一个字段存“手机,电脑”）。
  • 第二范式（2NF）：消除部分依赖，非主键列必须完全依赖于主键。
  • 第三范式（3NF）：消除传递依赖，非主键列不依赖于其他非主键列。

反范式化：为了减少 JOIN 的性能开销，在业务允许范围内故意冗余少量字段（如在订单表冗余“商品名称”快照）。

EXPLAIN 执行计划（慢查询排查利器）

在 SQL 语句前加上 EXPLAIN，MySQL 会模拟优化器执行该 SQL 并返回执行计划。

  • type：决定了查找方式。从好到坏：system > const > eq_ref > ref > range > index > ALL。（ALL 表示全表扫描，是必须优化的重点）
  • key：实际使用的索引。如果为 NULL 表示没用到索引。
  • rows：预计要扫描的行数。
  • Extra：额外信息。如 Using index 表示覆盖索引（极好）；Using filesort 或 Using temporary 通常意味着性能较差。
💻 示例 代码示例 ▼

1. 使用 EXPLAIN 分析查询

-- 在原本的 SQL 前加 EXPLAIN

EXPLAIN SELECT * FROM users WHERE email = 'test@example.com';

-- 结果输出大致如下：
-- id | select_type | table | type | possible_keys | key | rows | Extra
--  1 | SIMPLE      | users | ALL  | NULL          | NULL| 1000 | Using where

-- type 为 ALL 且 key 为 NULL 表示这引发了全表扫描，必须建立索引优化！

2. 优化 LIMIT 大偏移量分页

-- 慢查询：偏移量极大时，即使有索引，数据库也要丢弃 100 万条数据，极慢。
SELECT * FROM orders ORDER BY id LIMIT 1000000, 10;


-- 优化方案（延迟关联 / 子查询）：先利用覆盖索引获取主键，再联表查详情

SELECT o.* FROM orders o
INNER JOIN (
    SELECT id FROM orders ORDER BY id LIMIT 1000000, 10
) AS tmp ON o.id = tmp.id;
⚠️ 误区 常见误区 ▼
❌ 误区
表设计必须严格遵循第三范式，哪怕牺牲大量性能做复杂多表 JOIN。

只要加了索引，SQL 就一定会变快。
✅ 纠正
工程实践中，为了避免高并发下严重的性能开销，常常会适当“反范式化”来用空间换时间。

索引设计错误（未遵循最左前缀）、使用函数计算、或者隐式类型转换，都会导致优化器放弃使用索引（type=ALL）。
