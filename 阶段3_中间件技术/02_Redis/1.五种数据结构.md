# Redis 五种核心数据结构

> 五种数据结构：String/Hash/List/Set/ZSet 各有擅长，选错就让性能砸盘。

## 概念解释

Redis 是一个开源的**内存数据库**，所有数据存在内存中，因此读写速度极快（10万+ QPS）。

Redis 的数据以 **Key-Value** 形式存储，其中 Key 始终是字符串，Value 支持 5 种不同的数据结构，这 5 种结构是 Redis 能覆盖多种应用场景的核心原因。

---

## 为什么需要这 5 种结构？

不同业务场景对数据的操作模式不同：
- 存一个用户名 → 字符串就够了
- 存用户的多个属性（姓名、年龄、邮箱）→ 哈希更合适
- 存消息队列、最新评论 → 列表
- 统计在线用户（去重）→ 集合
- 积分排行榜 → 有序集合

用对数据结构，代码简单、性能好。

---

## 五种数据结构详解

### 1. String（字符串）

**最基础的类型，Key 对应一个字符串值。**

```
key: "user:1:name"
value: "张三"
```

| 命令 | 说明 | 示例 |
|------|------|------|
| `SET key value` | 设置值 | `SET name "张三"` |
| `GET key` | 获取值 | `GET name` |
| `DEL key` | 删除 | `DEL name` |
| `EXPIRE key seconds` | 设置过期时间 | `EXPIRE name 3600` |
| `TTL key` | 查看剩余时间 | `TTL name` |
| `INCR key` | 整数自增 1 | `INCR counter` |
| `INCRBY key n` | 整数自增 n | `INCRBY counter 10` |
| `SETNX key value` | 仅当不存在时设置 | `SETNX lock "1"` |
| `SETEX key seconds value` | 设置值同时设过期 | `SETEX token 7200 "abc123"` |

**典型场景**：
- 缓存 HTML 页面、接口返回值
- 存 Session Token（带过期时间）
- 计数器（文章浏览数、点赞数）
- 分布式锁（SETNX）

**底层**：Short string（≤44字节）用 embstr 编码，长字符串用 raw 编码，纯数字用 int 编码。

---

### 2. Hash（哈希）

**一个 Key 对应一个字典（field → value），适合存对象属性。**

```
key: "user:1"
fields: {name: "张三", age: "28", email: "zs@qq.com"}
```

| 命令 | 说明 |
|------|------|
| `HSET key field value` | 设置字段 |
| `HGET key field` | 获取字段 |
| `HMSET key f1 v1 f2 v2` | 批量设置 |
| `HMGET key f1 f2` | 批量获取 |
| `HGETALL key` | 获取所有字段 |
| `HDEL key field` | 删除字段 |
| `HKEYS key` | 所有字段名 |
| `HLEN key` | 字段数量 |
| `HINCRBY key field n` | 字段整数自增 |

**典型场景**：
- 缓存用户信息（可单独更新某字段，不必重写整个对象）
- 购物车（`cart:{userId}` → `{商品ID: 数量}`）

**Hash vs String 的选择**：
- 频繁更新对象的某一字段 → 用 Hash（只更新那个 field）
- 对象整体读取/很少修改 → 用 String + JSON 序列化

---

### 3. List（列表）

**有序、可重复的列表，底层是双向链表，两端操作 O(1)。**

```
key: "messages:room:1"
value: ["消息3", "消息2", "消息1"]   ← 最新的在左边
```

| 命令 | 说明 |
|------|------|
| `LPUSH key v1 v2` | 从左端插入 |
| `RPUSH key v1 v2` | 从右端插入 |
| `LPOP key` | 从左端弹出 |
| `RPOP key` | 从右端弹出 |
| `LRANGE key start stop` | 获取范围（0 -1 获取全部）|
| `LLEN key` | 获取长度 |
| `LINDEX key index` | 按下标获取 |
| `LTRIM key start stop` | 裁剪列表 |
| `BLPOP key timeout` | 阻塞弹出（实现消息队列）|

**典型场景**：
- 最新消息列表（`LPUSH` 插入 + `LRANGE 0 19` 取最新20条）
- 消息队列（生产者 `RPUSH`，消费者 `BLPOP`）
- 操作历史记录

**踩坑**：
- `LRANGE` 是 O(N)，不要用于超大列表的全量获取
- `LINDEX` 是 O(N)，按索引取值性能差，中间元素要用有序集合

---

### 4. Set（集合）

**无序、不重复的元素集合，支持集合运算（交集/并集/差集）。**

```
key: "online:users"
value: {101, 205, 333, 404}
```

| 命令 | 说明 |
|------|------|
| `SADD key v1 v2` | 添加元素 |
| `SREM key v1` | 删除元素 |
| `SMEMBERS key` | 获取所有元素 |
| `SCARD key` | 元素数量 |
| `SISMEMBER key v` | 是否包含某元素 |
| `SRANDMEMBER key n` | 随机获取 n 个 |
| `SPOP key` | 随机弹出一个 |
| `SINTER k1 k2` | 交集 |
| `SUNION k1 k2` | 并集 |
| `SDIFF k1 k2` | 差集 |

**典型场景**：
- 在线用户列表（去重）
- 用户关注/粉丝（`SINTER` 求共同关注）
- 随机抽奖（`SRANDMEMBER` 或 `SPOP`）
- 标签系统

---

### 5. ZSet（有序集合）

**有序、不重复的集合，每个元素关联一个 score（分数），按 score 排序。**

```
key: "leaderboard"
value: {张三:9800, 李四:9500, 王五:8800}  ← 按分数排序
```

| 命令 | 说明 |
|------|------|
| `ZADD key score member` | 添加元素 |
| `ZSCORE key member` | 获取分数 |
| `ZINCRBY key n member` | 分数自增 |
| `ZRANK key member` | 获取正序排名（从0开始）|
| `ZREVRANK key member` | 获取倒序排名 |
| `ZRANGE key start stop` | 正序范围查询 |
| `ZREVRANGE key start stop` | 倒序范围查询 |
| `ZRANGEBYSCORE key min max` | 按分数范围查询 |
| `ZCARD key` | 元素数量 |
| `ZREM key member` | 删除元素 |

**典型场景**：
- 排行榜（积分榜、游戏排名）
- 延时队列（score 存时间戳）
- 带权重的任务调度

**ZSet vs Set**：
- 不需要排序 → Set（内存省）
- 需要按分数排序 → ZSet

---

## 命令速查总结

| 类型 | 最常用命令 | 核心场景 |
|------|----------|---------|
| String | SET/GET/INCR/SETEX/SETNX | 缓存、计数器、分布式锁 |
| Hash | HSET/HGET/HGETALL/HINCRBY | 对象缓存、购物车 |
| List | LPUSH/LRANGE/BLPOP | 消息队列、最新列表 |
| Set | SADD/SISMEMBER/SINTER | 去重、关系运算 |
| ZSet | ZADD/ZREVRANGE/ZINCRBY | 排行榜、延时队列 |

---

## 常见坑

1. **Key 命名规范**：使用冒号分隔的层级结构，如 `user:1:token`，避免键名冲突
2. **KEYS * 禁止在生产环境用**：全量扫描会阻塞 Redis（用 SCAN 代替）
3. **大 Key 问题**：单个 List/Hash/Set 元素超过 1 万要警惕，序列化慢、内存大
4. **Value 序列化**：存复杂对象时要 JSON 序列化，取出后要反序列化
5. **Redis 是单线程处理命令**：耗时操作（如 SORT 大集合）会阻塞所有请求
