# Redis 典型使用场景

> 典型使用场景：缓存、Session、排行榜、分布式锁，覆盖后端 80% Redis 需求。

## 概念解释

Redis 凭借其极快的内存读写速度和丰富的数据结构，在后端开发中承担了多种重要角色。
本章从**缓存、Session、排行榜、分布式锁**四大高频场景切入，讲解如何选型和落地实现。

---

## 场景一：缓存（Cache）

### 为什么需要缓存？

数据库查询（磁盘 IO）通常需要几毫秒到几十毫秒，而 Redis 读取在**微秒级**。
对于热点数据（反复读、很少改），直接走缓存可以减少数据库压力、提升响应速度。

### 缓存的基本模式：Cache-Aside（旁路缓存）

```
读流程：
  1. 先查 Redis
  2. 命中 → 直接返回（缓存命中）
  3. 未命中 → 查数据库 → 写入 Redis → 返回（缓存未命中）

写流程：
  1. 先写数据库
  2. 再删除（或更新）Redis 缓存
```

**为什么写时删缓存而不是更新缓存？**

更新缓存在并发场景下容易产生数据不一致（线程A写库，线程B也写库，谁后更新缓存谁覆盖谁是随机的）。删除缓存更安全：下次读时自动从库中拉最新数据。

### 代码示意（Python + redis-py）

```python
import json
import redis

r = redis.Redis(host='localhost', port=6379, db=0)

def get_user(user_id: int) -> dict:
    cache_key = f"user:{user_id}"

    # 1. 先查缓存
    cached = r.get(cache_key)
    if cached:
        return json.loads(cached)

    # 2. 缓存未命中，查数据库
    user = db.query_user(user_id)  # 假设的数据库查询

    # 3. 写入缓存，设置过期时间（防止缓存永久占用）
    r.setex(cache_key, 3600, json.dumps(user))
    return user

def update_user(user_id: int, data: dict):
    # 1. 更新数据库
    db.update_user(user_id, data)
    # 2. 删除缓存（下次读时会重建）
    r.delete(f"user:{user_id}")
```

### 缓存过期时间设置建议

| 数据类型 | 推荐过期时间 |
|---------|------------|
| 用户基本信息 | 30分钟 ~ 1小时 |
| 商品详情 | 10分钟 ~ 30分钟 |
| 热点列表（首页推荐）| 1分钟 ~ 5分钟 |
| 验证码 | 5分钟 |
| Token | 与业务 Token 有效期一致 |

**原则**：写频率越高，过期时间越短；越是静态数据，可以越长甚至不过期。

## 场景二：Session 存储

### 为什么用 Redis 存 Session？

传统服务器端 Session 存在内存中，多台服务器部署时，用户的 Session 只在某一台机器上，导致**负载均衡时 Session 丢失**问题。

Redis 是独立的外部存储，所有服务器节点都能访问，天然支持分布式 Session。

### Session 存储结构

```
key:  "session:{session_id}"          # 如 "session:abc123xyz"
value: {"user_id": 1, "role": "admin", "login_time": "2024-01-01"}
expire: 1800s（30分钟不活动则过期）
```

### 代码示意

```python
import uuid
import json
import redis

r = redis.Redis(host='localhost', port=6379, db=1)
SESSION_EXPIRE = 1800  # 30分钟

def create_session(user_id: int, role: str) -> str:
    session_id = str(uuid.uuid4())
    session_data = {"user_id": user_id, "role": role}
    r.setex(f"session:{session_id}", SESSION_EXPIRE, json.dumps(session_data))
    return session_id

def get_session(session_id: str) -> dict | None:
    data = r.get(f"session:{session_id}")
    if not data:
        return None
    # 每次访问刷新过期时间（滑动过期）
    r.expire(f"session:{session_id}", SESSION_EXPIRE)
    return json.loads(data)

def delete_session(session_id: str):
    r.delete(f"session:{session_id}")
```

**注意**：现代后端更多用 JWT 替代 Session，但 Redis Session 在需要强制下线、实时撤销权限的场景仍有优势（JWT 一旦颁发无法撤销，除非引入黑名单机制，而黑名单本身也要存 Redis）。

## 场景三：排行榜

### 为什么用 ZSet？

ZSet（有序集合）天然支持按 score 排序，`ZREVRANGE` 可以直接获取 Top N，`ZINCRBY` 可以原子地更新分数，完美契合排行榜需求。

### 实现方案

```
key:   "leaderboard:game:weekly"
value: {
    "player:101": 9800,   # score = 积分
    "player:205": 9500,
    "player:333": 8800,
}
```

```python
def add_score(player_id: int, points: int):
    """增加玩家积分"""
    r.zincrby("leaderboard:game:weekly", points, f"player:{player_id}")

def get_top_n(n: int = 10) -> list:
    """获取 Top N 排行榜（倒序，分数高的在前）"""
    # withscores=True 同时返回分数
    results = r.zrevrange("leaderboard:game:weekly", 0, n - 1, withscores=True)
    return [{"player": m.decode(), "score": int(s)} for m, s in results]

def get_player_rank(player_id: int) -> int:
    """获取玩家当前名次（从1开始）"""
    rank = r.zrevrank("leaderboard:game:weekly", f"player:{player_id}")
    return rank + 1 if rank is not None else None
```

### 多维度排行榜

如果需要同时维护"日榜/周榜/月榜"，只需用不同的 Key：
```
leaderboard:game:daily
leaderboard:game:weekly
leaderboard:game:monthly
```
每天/每周用定时任务清空或重置对应 Key。

## 场景四：分布式锁

### 为什么需要分布式锁？

多台服务器同时处理同一资源时（如秒杀商品库存），如果没有互斥机制，会出现**超卖**问题。

单机锁（threading.Lock）只在单进程内有效，分布式环境需要一个**所有节点都能访问的外部锁**。

### 基本实现：SETNX + EXPIRE

```python
import time
import uuid

def acquire_lock(lock_key: str, expire: int = 10) -> str | None:
    """
    尝试获取锁，返回锁的唯一标识（用于释放时验证）
    lock_key: 锁名，如 "lock:order:sku_123"
    expire: 锁的最大持有时间（秒），防止崩溃后死锁
    """
    lock_value = str(uuid.uuid4())  # 唯一值，防止误删他人的锁
    # SET key value NX EX seconds ← 原子操作（设值+过期一步完成）
    result = r.set(lock_key, lock_value, nx=True, ex=expire)
    return lock_value if result else None

def release_lock(lock_key: str, lock_value: str) -> bool:
    """
    释放锁，必须验证是自己的锁才能删除（防止误删）
    使用 Lua 脚本保证「查询+删除」的原子性
    """
    lua_script = """
    if redis.call("GET", KEYS[1]) == ARGV[1] then
        return redis.call("DEL", KEYS[1])
    else
        return 0
    end
    """
    result = r.eval(lua_script, 1, lock_key, lock_value)
    return bool(result)

# 使用示例
def process_order(sku_id: int, user_id: int):
    lock_key = f"lock:sku:{sku_id}"
    lock_value = acquire_lock(lock_key, expire=5)

    if not lock_value:
        raise Exception("系统繁忙，请稍后重试")  # 获取锁失败

    try:
        # 临界区：扣库存等操作
        stock = get_stock(sku_id)
        if stock <= 0:
            raise Exception("库存不足")
        decr_stock(sku_id)
        create_order(user_id, sku_id)
    finally:
        release_lock(lock_key, lock_value)  # 必须释放
```

### 关键设计要点

| 要点 | 原因 |
|------|------|
| 用 `SET NX EX` 而不是 SETNX + EXPIRE | SETNX 和 EXPIRE 是两步，中间崩溃会死锁 |
| 锁的 value 用唯一 UUID | 防止持有者 A 超时后 B 获取了锁，A 再来误删 B 的锁 |
| 用 Lua 脚本释放锁 | GET + DEL 两步操作必须原子执行 |
| 设置合理的过期时间 | 业务超时后锁自动释放，防死锁 |

**进阶**：生产环境建议用 `Redlock` 算法（多节点 Redis）或直接用 `redis-py` 的 `Lock` 类，它已内置上述逻辑。

## 场景对比总结

| 场景 | 使用结构 | 核心命令 | 关键设计 |
|------|---------|---------|---------|
| 数据缓存 | String | SETEX / GET / DEL | Cache-Aside 模式，写时删缓存 |
| Session 存储 | String（JSON）| SETEX / GET / EXPIRE | 滑动过期，分布式共享 |
| 排行榜 | ZSet | ZINCRBY / ZREVRANGE | Key 按时间维度区分日/周/月 |
| 分布式锁 | String | SET NX EX / Lua DEL | 唯一 value + Lua 原子释放 |

## 常见坑

1. **缓存与数据库不一致**：写操作后必须删缓存，顺序是「先写库，再删缓存」
2. **忘记设过期时间**：缓存永不过期会导致内存耗尽
3. **锁过期时间太短**：业务还没执行完锁就到期，其他请求进入临界区
4. **锁没有 finally 释放**：业务异常时锁没有释放，等待超时才能自动解锁
5. **Session 未滑动过期**：用户每次活动都应刷新过期时间，避免正在操作时突然登出
