# Redis 缓存三大问题：穿透、击穿、雪崩

> 穿透/击穿/雪崩：三种缓存崩塌的脸，对应三种防护手段，面试与生产都绕不开。

## 概念解释

Redis 作为缓存层，正常情况下可以拦截大量请求，减轻数据库压力。但在特定情形下，缓存失去防护作用，大量请求直接冲击数据库，甚至导致系统崩溃。

这三种经典问题是**Redis 面试必考题**，也是真实系统中最常见的稳定性隐患。

---

## 一、缓存穿透（Cache Penetration）

### 现象

查询一个**数据库中根本不存在**的数据（如不存在的商品 ID），缓存中也没有（因为从未写入过），每次请求都会穿过缓存，直接打到数据库。

**恶意攻击场景**：攻击者构造大量不存在的 ID（如 `-1`、`0`、随机字符串），用脚本狂发请求 → 缓存全部未命中 → 数据库被打爆。

```
请求: GET /product/-1
  ↓ 查 Redis → Miss（不存在）
  ↓ 查 MySQL → 不存在（NULL）
  ↓ 什么都没缓存
  ↓ 下次同样请求，重复上面步骤 ← 无限循环
```

### 解决方案

#### 方案一：缓存空值（Cache Null）

查到空结果时，把 `null` 也写入 Redis，设置较短过期时间（如 5 分钟）。

```python
def get_product(product_id: int) -> dict | None:
    cache_key = f"product:{product_id}"
    cached = r.get(cache_key)

    if cached is not None:
        if cached == b"NULL":  # 空值标记
            return None
        return json.loads(cached)

    # 查数据库
    product = db.get_product(product_id)

    if product is None:
        # 缓存空值，防止穿透，5分钟后过期
        r.setex(cache_key, 300, "NULL")
        return None

    r.setex(cache_key, 3600, json.dumps(product))
    return product
```

**缺点**：如果攻击者用无数不同的 ID，Redis 会被大量空值填满。

#### 方案二：布隆过滤器（Bloom Filter）⭐ 推荐

在 Redis 前再加一层**布隆过滤器**，存储所有合法 ID。请求进来先检查是否在布隆过滤器中，不在则直接返回，完全不查 Redis 和数据库。

```
请求 → 布隆过滤器（一定不存在 → 直接返回404）
              ↓ 可能存在
           查 Redis
              ↓ Miss
           查数据库
```

```python
# 使用 redis-py-bloom 或 RedisBloom 模块
from redis.commands.bf import BFCommands

# 初始化时加载所有合法商品 ID
def init_bloom_filter():
    bf = r.bf()
    bf.create("product:bloom", 0.001, 1000000)  # 错误率0.1%，预计100万数据
    all_ids = db.get_all_product_ids()
    for id in all_ids:
        bf.add("product:bloom", id)

def get_product_safe(product_id: int) -> dict | None:
    # 先过布隆过滤器
    if not r.bf().exists("product:bloom", product_id):
        return None  # 一定不存在，直接返回

    # 通过了再走正常缓存逻辑
    return get_product(product_id)
```

**布隆过滤器原理**：
- 存数据时，用多个哈希函数映射到位数组的多个位置，置为 1
- 查询时，检查所有对应位是否都为 1
- 若有任一位为 0 → 一定不存在（100% 准确）
- 若所有位都为 1 → 可能存在（有误判率，默认 0.1%~1%）
- 特点：占用极少内存，不支持删除

## 二、缓存击穿（Cache Breakdown）

### 现象

某个**热点 Key**（如热销商品、热门话题）在缓存中**恰好过期**的瞬间，大量并发请求同时涌入，全部穿透缓存查数据库，数据库瞬间压力激增。

```
T时刻：热点 Key "product:1001" 过期
  ↓ 1000 个并发请求同时查缓存 → Miss
  ↓ 1000 个请求同时查数据库 ← 击穿！
  ↓ 数据库可能崩溃
```

### 解决方案

#### 方案一：互斥锁（Mutex）

缓存 Miss 时，只允许一个请求去查数据库，其他请求等待。

```python
import time

def get_hot_product(product_id: int) -> dict | None:
    cache_key = f"product:{product_id}"
    lock_key = f"lock:product:{product_id}"

    # 查缓存
    cached = r.get(cache_key)
    if cached:
        return json.loads(cached)

    # 缓存 Miss，尝试获取锁
    lock_value = str(uuid.uuid4())
    acquired = r.set(lock_key, lock_value, nx=True, ex=5)

    if acquired:
        try:
            # 获得锁，查数据库并重建缓存
            product = db.get_product(product_id)
            if product:
                r.setex(cache_key, 3600, json.dumps(product))
            return product
        finally:
            # 释放锁（Lua 原子操作）
            lua = "if redis.call('GET',KEYS[1])==ARGV[1] then return redis.call('DEL',KEYS[1]) else return 0 end"
            r.eval(lua, 1, lock_key, lock_value)
    else:
        # 未获得锁，等待后重试
        time.sleep(0.05)
        return get_hot_product(product_id)  # 递归重试
```

#### 方案二：逻辑过期（Logical Expiration）⭐ 推荐

缓存**永不过期**，但在 value 里存一个逻辑过期时间。读取时检查是否已逻辑过期，过期则开启异步线程更新，当前请求返回旧数据（允许短暂不一致）。

```python
import threading

def get_hot_product_v2(product_id: int) -> dict | None:
    cache_key = f"product:{product_id}"
    cached = r.get(cache_key)

    if not cached:
        return None  # 第一次预热时需要主动加载

    data = json.loads(cached)
    expire_time = data.get("__expire_at__")

    if expire_time and time.time() > expire_time:
        # 逻辑过期，异步更新（不阻塞当前请求）
        lock_key = f"lock:product:{product_id}"
        if r.set(lock_key, "1", nx=True, ex=5):
            # 启动后台线程更新缓存
            threading.Thread(target=_rebuild_cache, args=(product_id,)).start()

    # 返回旧数据（可能已过期，但避免击穿）
    return {k: v for k, v in data.items() if k != "__expire_at__"}

def _rebuild_cache(product_id: int):
    product = db.get_product(product_id)
    if product:
        product["__expire_at__"] = time.time() + 3600
        r.set(f"product:{product_id}", json.dumps(product))
    r.delete(f"lock:product:{product_id}")
```

## 三、缓存雪崩（Cache Avalanche）

### 现象

**大量 Key 在同一时间集中过期**（或 Redis 宕机），导致大批请求同时穿透缓存打到数据库，数据库承受不住，系统整体崩溃。

```
场景一：整点批量刷新缓存，大量 Key 同时设了相同的 TTL=1小时
  → 1小时后全部同时过期 → 雪崩

场景二：Redis 服务器宕机
  → 所有请求没有缓存层 → 全打数据库 → 雪崩
```

### 解决方案

#### 方案一：过期时间加随机抖动

不要给所有 Key 设置相同的过期时间，加上随机偏移量，让过期时间分散。

```python
import random

def set_cache_with_jitter(key: str, value, base_ttl: int):
    """
    base_ttl: 基础过期时间（秒）
    随机抖动：±20% 范围
    """
    jitter = random.randint(-base_ttl // 5, base_ttl // 5)
    actual_ttl = base_ttl + jitter
    r.setex(key, actual_ttl, json.dumps(value))

# 原来：3600秒，所有商品同时过期
# 现在：2880~4320秒之间随机，过期时间分散
set_cache_with_jitter("product:1001", product_data, 3600)
```

#### 方案二：热点数据永不过期

对访问量极高的 Key（首页数据、热销商品），缓存设为永不过期，改用**后台定时刷新**。

```python
import schedule
import threading

def refresh_hot_cache():
    """后台定时刷新热点缓存（无过期时间）"""
    hot_products = db.get_hot_products(limit=100)
    for product in hot_products:
        r.set(f"product:{product['id']}", json.dumps(product))
    print(f"热点缓存刷新完成：{len(hot_products)} 条")

# 每5分钟刷新一次
schedule.every(5).minutes.do(refresh_hot_cache)
```

#### 方案三：Redis 高可用部署

- **主从复制 + 哨兵模式**：主节点宕机后，哨兵自动选出新主节点（切换时间通常 < 30秒）
- **Redis Cluster**：数据分片到多个节点，单节点故障不影响整体服务
- 应用层实现**熔断降级**：Redis 不可用时，降级返回默认数据或错误提示，而不是继续打数据库

## 三大问题对比总结

| 问题 | 触发条件 | 影响范围 | 核心解决方案 |
|------|---------|---------|------------|
| **缓存穿透** | 查询不存在的数据 | 每次特定 Key 的请求 | 布隆过滤器 / 缓存空值 |
| **缓存击穿** | 热点 Key 突然过期 | 某个热点 Key 的并发请求 | 互斥锁 / 逻辑过期 |
| **缓存雪崩** | 大量 Key 同时过期 / Redis 宕机 | 整个系统 | 随机 TTL / 永不过期 / 高可用部署 |

---

## 记忆口诀

- **穿透**：根本没有 → 用布隆过滤器拦在门口
- **击穿**：热点消失 → 用锁保护，一个人去取数据
- **雪崩**：集体失效 → 时间错开，不要集体退场

---

## 常见坑

1. **缓存空值忘设过期时间**：防穿透的空值如果不过期，真正有数据后永远返回 null
2. **互斥锁等待时间设太短**：数据库查询还没完成，其他线程就超时放弃了
3. **部署时统一初始化缓存**：系统启动时所有缓存同一时间写入、相同 TTL → 雪崩定时炸弹
4. **布隆过滤器不支持删除**：商品下架后布隆过滤器无法删除，需要周期性重建
