# Redis 过期策略、淘汰策略与持久化

> 过期、淘汰、持久化：决定 Redis 内存会不会爆、重启会不会丢数据。

## 概念解释

Redis 所有数据在内存中，内存是有限资源。当数据不再需要时（过期）或内存不足时（淘汰），Redis 需要清理数据。而为了防止重启数据丢失，Redis 提供了两种持久化机制。

---

## 一、Key 过期策略

### 为什么需要过期策略？

- 缓存数据不能永久占用内存
- 验证码、Token、Session 有生命周期
- 数据库不知道 Key 什么时候过期，需要 Redis 自己管理

### Redis 如何处理过期 Key？

Redis 同时采用**两种策略**组合使用：

#### 1. 惰性删除（Lazy Expiration）

Key 过期后**不立刻删除**，等到下次访问该 Key 时，才检查是否过期，若已过期则删除并返回 null。

```
优点：不消耗额外 CPU（只在访问时检查）
缺点：长期不访问的过期 Key 会一直占用内存（内存泄漏风险）
```

#### 2. 定期删除（Periodic Expiration）

Redis 默认**每隔 100ms** 随机抽取一批设置了过期时间的 Key，检查并删除已过期的。

```
优点：主动回收内存，不完全依赖访问触发
缺点：是随机抽取，不能保证所有过期 Key 都被及时清理
```

**实际效果**：两者结合，惰性删除兜底（你来访问我检查），定期删除扫墓（我定期主动清理）。

## 二、内存淘汰策略（Eviction Policy）

### 什么时候触发淘汰？

当 Redis 内存使用量达到 `maxmemory` 配置的上限时，新写入操作会触发淘汰策略，Redis 会按照设定的策略删除部分 Key 来释放空间。

### 8 种淘汰策略

| 策略 | 说明 | 适用场景 |
|------|------|---------|
| `noeviction` | 不淘汰，内存满时新写入报错 | 数据不允许丢失的场景 |
| `allkeys-lru` | 从所有 Key 中淘汰最近最少使用的 | **通用缓存场景，最常用** ⭐ |
| `volatile-lru` | 仅从设置了过期时间的 Key 中淘汰最近最少使用的 | 混合存储（持久+缓存）|
| `allkeys-lfu` | 从所有 Key 中淘汰访问频率最低的 | 热点数据分布不均场景 |
| `volatile-lfu` | 仅从有过期时间的 Key 中淘汰频率最低的 | - |
| `allkeys-random` | 随机淘汰任意 Key | 数据均匀访问的缓存 |
| `volatile-random` | 随机淘汰有过期时间的 Key | - |
| `volatile-ttl` | 淘汰剩余存活时间最短的 Key | 希望快过期的先被淘汰 |

### LRU vs LFU

- **LRU（Least Recently Used）**：最长时间没访问 → 淘汰
  - 缺点：热点数据可能被一次批量操作冲掉
- **LFU（Least Frequently Used）**：总访问次数最少 → 淘汰
  - 优点：真正的热点数据不会被淘汰
  - Redis 4.0+ 支持

**生产推荐**：
- 纯缓存用途 → `allkeys-lru` 或 `allkeys-lfu`
- 既存缓存又存持久数据 → `volatile-lru`

### 配置方式

```bash
# redis.conf 中配置
maxmemory 2gb               # 最大内存限制
maxmemory-policy allkeys-lru # 淘汰策略
```

```python
# 通过客户端动态修改（临时，重启失效）
r.config_set("maxmemory-policy", "allkeys-lru")
```

## 三、持久化：RDB（快照）

### 什么是 RDB？

RDB（Redis Database）是 Redis 的**快照持久化**方式，将某一时刻的内存数据完整保存为一个二进制文件（`dump.rdb`）。

### RDB 的工作原理

```
Redis 主进程 fork() 出子进程
  ↓
子进程遍历内存数据，写入 RDB 文件
  ↓
写完后替换旧的 RDB 文件
  ↓
主进程继续处理请求（不阻塞）
```

使用 `fork()` + 写时复制（Copy-On-Write）技术，子进程看到的是创建时的内存快照，即使主进程继续修改数据也不影响子进程的写入。

### 触发 RDB 的条件

```bash
# redis.conf 中的 save 配置（满足任一条件则触发）
save 3600 1     # 3600秒内有1次修改
save 300 100    # 300秒内有100次修改
save 60 10000   # 60秒内有10000次修改
```

也可以手动触发：
```bash
BGSAVE   # 异步保存（推荐，不阻塞）
SAVE     # 同步保存（阻塞，生产环境禁用）
```

### RDB 优缺点

| 优点 | 缺点 |
|------|------|
| 文件紧凑，体积小，适合备份和全量复制 | 快照时间点之间的数据可能丢失 |
| 恢复速度快（直接加载二进制文件）| fork() 时如果内存大，短暂阻塞（毫秒级）|
| 对性能影响小 | 不适合要求数据零丢失的场景 |

## 四、持久化：AOF（追加日志）

### 什么是 AOF？

AOF（Append Only File）是 Redis 的**日志持久化**方式，将每一条**写命令**以文本形式追加写入文件（`appendonly.aof`），恢复时重放所有命令。

### AOF 写入策略（三种 fsync 模式）

```bash
# redis.conf
appendonly yes
appendfsync always      # 每条命令都 fsync，最安全，性能最差
appendfsync everysec    # 每秒 fsync 一次，最多丢失1秒数据 ⭐ 默认推荐
appendfsync no          # 交给 OS 决定，性能最好，可能丢较多数据
```

### AOF 重写（防止文件无限增长）

AOF 文件记录的是命令，随着时间增长文件会越来越大。Redis 提供**AOF 重写**功能，将内存当前状态转换为最精简的命令集，替换旧 AOF 文件。

```bash
BGREWRITEAOF   # 手动触发 AOF 重写

# 自动触发配置
auto-aof-rewrite-percentage 100    # AOF 文件大小比上次翻倍时触发
auto-aof-rewrite-min-size 64mb     # 且文件大于 64MB
```

### AOF 优缺点

| 优点 | 缺点 |
|------|------|
| 数据更安全，最多丢失1秒（everysec模式）| 文件体积比 RDB 大很多 |
| 可读性强（文本命令，可人工修复）| 恢复速度比 RDB 慢（重放命令）|
| 支持增量写入 | 高写入量时性能略低于 RDB |

---

## 五、RDB vs AOF 对比与选型

| 对比维度 | RDB | AOF |
|---------|-----|-----|
| 持久化方式 | 快照（全量）| 日志（增量）|
| 数据丢失 | 快照间隔内的数据会丢 | 最多丢失 1 秒 |
| 文件体积 | 小，紧凑 | 大，文本格式 |
| 恢复速度 | 快 | 慢（重放命令）|
| 适用场景 | 定期备份、数据允许少量丢失 | 对数据安全要求高 |

### 生产推荐：RDB + AOF 混合持久化

Redis 4.0+ 支持混合持久化，AOF 文件头部存储 RDB 快照，后续追加增量 AOF 日志：

```bash
# redis.conf
aof-use-rdb-preamble yes   # 开启混合持久化
```

效果：恢复速度接近 RDB（加载快照），数据完整性接近 AOF（最多丢失1秒）。

## 六、完整配置参考

```bash
# redis.conf 生产推荐配置片段

# 内存限制
maxmemory 4gb
maxmemory-policy allkeys-lru

# RDB（定期快照备份）
save 3600 1
save 300 100
dbfilename dump.rdb
dir /data/redis/

# AOF（日志持久化）
appendonly yes
appendfilename appendonly.aof
appendfsync everysec
auto-aof-rewrite-percentage 100
auto-aof-rewrite-min-size 64mb
aof-use-rdb-preamble yes   # 混合持久化
```

## 总结速记

| 概念 | 一句话 |
|------|-------|
| 惰性删除 | 用到才检查，不访问不删除 |
| 定期删除 | 定时随机抽查过期 Key |
| `allkeys-lru` | 最久没用的先删，纯缓存首选 |
| RDB | 定时全量快照，恢复快，会丢数据 |
| AOF | 每条写命令记日志，最多丢1秒 |
| 混合持久化 | 两者优势兼得，生产推荐 |

---

## 常见坑

1. **生产环境未配置 maxmemory**：内存用尽后 Redis 进程被 OOM Kill，服务中断
2. **RDB save 配置全部注释掉**：关闭了 RDB，但忘了开启 AOF，重启直接丢所有数据
3. **AOF 文件过大影响重启速度**：定期触发 BGREWRITEAOF 或合理配置自动重写
4. **fork() 阻塞**：内存很大（>10GB）时，fork() 可能阻塞几百毫秒，监控慢日志排查
5. **持久化目录权限问题**：Redis 进程没有写入权限，持久化静默失败
