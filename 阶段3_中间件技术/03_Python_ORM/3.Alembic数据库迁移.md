# Alembic 数据库迁移

> Alembic 用迁移文件管理表结构版本，让上线和回滚都可追溯。

## 概念解释

**数据库迁移**是指以受控、可追溯的方式管理数据库表结构的变更（建表、加字段、改索引、删列等）。

**为什么不直接修改数据库？**

- 手动改表结构无法追踪版本，多人协作时互相覆盖
- 生产环境无法重现数据库状态
- 回滚到之前版本时不知道要撤销哪些改动

**Alembic** 是 SQLAlchemy 官方推荐的迁移工具，将每次结构变更记录为一个迁移文件，文件之间形成版本链，支持升级（upgrade）和回滚（downgrade）。

---

## 为什么需要迁移工具？

```
没有迁移工具：
  开发环境: users 表有 avatar 字段
  生产环境: users 表没有 avatar 字段  ← 部署后程序报错！
  同事A：我加了 email 索引
  同事B：我也改了 email 列  ← 冲突！

有了 Alembic：
  每次结构变更 → 生成迁移文件（版本号 + 变更内容）
  部署时 alembic upgrade head → 自动同步到最新结构
  出问题 alembic downgrade -1 → 回滚上一步
```

## 一、初始化 Alembic 项目

```bash
# 在项目根目录执行（会生成 alembic/ 目录和 alembic.ini 配置文件）
alembic init alembic
```

生成的目录结构：
```
项目根目录/
├── alembic/
│   ├── env.py          # 迁移环境配置（最重要，需要修改）
│   ├── script.py.mako  # 迁移文件模板
│   └── versions/       # 存放所有迁移文件
└── alembic.ini         # 数据库连接配置
```

---

## 二、配置连接（alembic.ini）

打开 `alembic.ini`，修改数据库连接字符串：

```ini
# alembic.ini
sqlalchemy.url = mysql+pymysql://root:123456@localhost:3306/mydb
```

**更好的做法**：从环境变量读取，不硬编码到配置文件：

```python
# alembic/env.py 中修改
import os
from sqlalchemy import engine_from_config

config.set_main_option(
    "sqlalchemy.url",
    os.getenv("DATABASE_URL", "mysql+pymysql://root:123456@localhost:3306/mydb")
)
```

## 三、关联 Model（env.py）

这是最关键的一步：让 Alembic 知道你的 Model 定义在哪里，这样它才能**自动检测 Model 变更**并生成迁移文件。

打开 `alembic/env.py`，找到 `target_metadata = None` 这行，修改为：

```python
# alembic/env.py

# 导入你的 Base（包含所有 Model 定义）
import sys
import os
sys.path.insert(0, os.path.dirname(os.path.dirname(__file__)))  # 将项目根目录加入 PATH

from app.models import Base   # 根据你的实际路径修改
target_metadata = Base.metadata  # ← 关键：把 None 改为 Base.metadata
```

## 四、核心工作流

### 第一次建表（初始迁移）

```bash
# 1. 根据当前 Model 自动生成迁移文件
alembic revision --autogenerate -m "initial tables"
# 会在 alembic/versions/ 下生成类似 abc123_initial_tables.py 的文件

# 2. 执行迁移（建表）
alembic upgrade head
# head 表示升级到最新版本
```

### 修改 Model 后更新数据库

```python
# 假设你在 User Model 中新加了 avatar 字段
class User(Base):
    # ... 原有字段 ...
    avatar: Mapped[str] = mapped_column(String(255), nullable=True)  # 新增
```

```bash
# 1. 生成迁移文件（自动检测 Model 变更）
alembic revision --autogenerate -m "add avatar to users"

# 2. 查看生成的迁移文件，确认内容正确
# alembic/versions/xyz789_add_avatar_to_users.py

# 3. 执行迁移
alembic upgrade head
```

### 回滚操作

```bash
# 回滚最近一次迁移
alembic downgrade -1

# 回滚到指定版本（版本号在文件名中）
alembic downgrade abc123

# 回滚到最初状态（删除所有表）
alembic downgrade base
```

## 五、迁移文件解析

```python
# alembic/versions/abc123_add_avatar_to_users.py

"""add avatar to users

Revision ID: abc123
Revises: def456          # 上一个版本的 ID（形成版本链）
Create Date: 2024-01-15 10:30:00.000000
"""

from alembic import op
import sqlalchemy as sa

# 版本标识
revision = 'abc123'
down_revision = 'def456'
branch_labels = None
depends_on = None


def upgrade() -> None:
    """升级操作：新增 avatar 列"""
    op.add_column(
        'users',
        sa.Column('avatar', sa.String(255), nullable=True)
    )


def downgrade() -> None:
    """降级操作：删除 avatar 列（撤销升级）"""
    op.drop_column('users', 'avatar')
```

每个迁移文件都有 `upgrade()` 和 `downgrade()` 两个函数，保证可以双向操作。

## 六、手动编写迁移文件

自动生成不能检测到所有变更（如初始数据、复杂索引、存储过程），有时需要手动写：

```bash
# 生成空白迁移文件
alembic revision -m "seed initial data"
```

```python
def upgrade() -> None:
    # 手动操作：插入初始数据
    op.execute("""
        INSERT INTO roles (name, description) VALUES
        ('admin', '超级管理员'),
        ('user', '普通用户')
    """)
    
    # 手动操作：创建复合索引
    op.create_index(
        "ix_posts_author_created",
        "posts",
        ["author_id", "created_at"],
    )

def downgrade() -> None:
    op.drop_index("ix_posts_author_created", "posts")
    op.execute("DELETE FROM roles")
```

## 七、Aerich（Tortoise ORM 的迁移工具）

如果使用 Tortoise ORM，应使用配套的 **Aerich** 而非 Alembic：

```bash
pip install aerich
```

```bash
# 初始化（需要 tortoise 配置字典，通常在 main.py 或 config.py 中定义）
aerich init -t app.config.TORTOISE_ORM

# 初始化数据库（第一次建表）
aerich init-db

# Model 变更后生成迁移
aerich migrate --name "add_avatar_to_users"

# 执行迁移
aerich upgrade

# 回滚
aerich downgrade -v 1   # 回滚到版本1
```

## 八、常用命令速查

```bash
# Alembic 常用命令
alembic current                    # 查看当前数据库版本
alembic history                    # 查看迁移历史（版本链）
alembic heads                      # 查看最新版本

alembic revision --autogenerate -m "描述"  # 自动生成迁移文件
alembic revision -m "描述"                 # 生成空白迁移文件

alembic upgrade head               # 升级到最新
alembic upgrade +1                 # 升级一步
alembic upgrade <revision_id>      # 升级到指定版本

alembic downgrade -1               # 回滚一步
alembic downgrade base             # 回滚到最初（删除所有表）
alembic downgrade <revision_id>    # 回滚到指定版本

alembic show <revision_id>         # 查看某个版本的迁移内容
```

## 九、生产部署最佳实践

1. **迁移文件纳入版本控制**：`alembic/versions/` 目录要 commit 到 git
2. **部署脚本中执行迁移**：`alembic upgrade head` 在应用启动前执行
3. **迁移文件必须可回滚**：每个 `upgrade()` 必须有对应的 `downgrade()`
4. **先迁移再部署**：避免新代码访问旧数据库结构报错
5. **大表加列要谨慎**：百万行以上的表 `ADD COLUMN` 会锁表，需要用在线 DDL 工具（如 `pt-online-schema-change`）

---

## 常见坑

1. **`--autogenerate` 检测不到所有变更**：表名大小写变更、CHECK 约束等需要手动写
2. **忘记关联 `target_metadata`**：`env.py` 中没有导入 `Base.metadata`，自动生成的迁移是空的
3. **多人同时生成迁移**：两个迁移文件指向同一个 `down_revision` 会产生分支，需要手动合并
4. **生产环境直接 `drop_all` 重建**：数据直接丢失，必须用迁移
5. **迁移和代码版本不同步**：只升级了代码没有执行 `alembic upgrade head`，程序查新字段报错
