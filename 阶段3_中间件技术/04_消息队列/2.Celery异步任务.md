# Celery 异步任务

> Celery 把耗时任务丢给 Worker 异步执行，让 Web 请求秒级返回。

## 概念解释

**Celery** 是 Python 生态中最流行的**分布式任务队列框架**，用于在后台异步执行耗时操作。

Celery 本身不存储消息，它需要一个**消息代理（Broker）** 来传递任务，以及一个**后端（Backend）** 来存储任务执行结果。

```
[你的 Web 代码]
  ↓ send_task()
[Broker：Redis / RabbitMQ]   ← 任务排队等待
  ↓ 分发任务
[Celery Worker 进程]          ← 真正执行任务的地方
  ↓ 存储结果
[Backend：Redis / 数据库]     ← 可选，用于查询任务结果
```

## 一、基础配置

```python
# celery_app.py（Celery 实例通常放在单独文件）
from celery import Celery

# 创建 Celery 实例
# 第一个参数：模块名（通常是项目名或当前文件名）
# broker：任务队列（任务发往哪里）
# backend：结果存储（任务结果存到哪里）
app = Celery(
    "myproject",
    broker="redis://localhost:6379/1",   # db=1 存任务
    backend="redis://localhost:6379/2",  # db=2 存结果
)

# 配置项（也可以用字典方式）
app.conf.update(
    task_serializer="json",        # 任务序列化格式
    result_serializer="json",      # 结果序列化格式
    accept_content=["json"],       # 接受的内容类型
    timezone="Asia/Shanghai",      # 时区
    enable_utc=True,
    task_track_started=True,       # 任务开始时记录状态
    task_acks_late=True,           # 任务执行完成后才 ACK（提高可靠性）
    worker_prefetch_multiplier=1,  # Worker 每次只预取1个任务（防止任务堆积在单个 Worker）
)
```

## 二、定义任务

```python
# tasks.py
from celery_app import app
import time

# @app.task 装饰器将函数注册为 Celery 任务
@app.task
def send_email(to: str, subject: str, body: str) -> dict:
    """发送邮件（模拟耗时操作）"""
    time.sleep(2)  # 模拟发送耗时
    print(f"邮件已发送至 {to}: {subject}")
    return {"status": "sent", "to": to}


@app.task
def generate_report(user_id: int) -> str:
    """生成用户报告（模拟耗时操作）"""
    time.sleep(5)
    report_url = f"https://example.com/reports/{user_id}.pdf"
    return report_url


# bind=True：将任务实例本身传入（可以访问 self.request.id 等信息）
@app.task(bind=True)
def long_task(self, data: list) -> dict:
    """可报告进度的长任务"""
    total = len(data)
    for i, item in enumerate(data):
        # 更新进度（存入 Backend）
        self.update_state(
            state="PROGRESS",
            meta={"current": i + 1, "total": total, "percent": int((i + 1) / total * 100)},
        )
        time.sleep(0.1)
    return {"processed": total}
```

## 三、调用任务（发送到队列）

```python
# 在 Web 代码或脚本中调用任务
from tasks import send_email, generate_report, long_task

# ── 方式1：delay()（最简单，等价于 apply_async）──
result = send_email.delay("user@example.com", "欢迎注册", "感谢您注册！")
print(f"任务已提交，ID = {result.id}")
# 此时函数立即返回，邮件在后台发送

# ── 方式2：apply_async()（完整参数控制）──
result = send_email.apply_async(
    args=["user@example.com", "欢迎注册", "感谢您注册！"],
    countdown=60,          # 60秒后执行（延迟任务）
    expires=300,           # 300秒内没被消费则丢弃
    retry=True,            # 失败时重试
    retry_policy={
        "max_retries": 3,
        "interval_start": 2,   # 第一次重试等待2秒
        "interval_step": 2,    # 每次重试多等2秒
    },
)

# 指定在哪个队列执行（多队列场景）
generate_report.apply_async(args=[101], queue="reports")
```

## 四、获取任务结果

```python
from celery.result import AsyncResult
from celery_app import app

# 通过任务 ID 查询结果（适合轮询场景）
def check_task_status(task_id: str) -> dict:
    result = AsyncResult(task_id, app=app)

    return {
        "task_id": task_id,
        "status": result.status,
        # 状态值：PENDING / STARTED / PROGRESS / SUCCESS / FAILURE / RETRY / REVOKED
        "result": result.result if result.ready() else None,
    }

# 等待结果（阻塞，不推荐在 Web 请求中用）
result = send_email.delay("user@example.com", "test", "test")
output = result.get(timeout=10)  # 最多等10秒
print(output)  # {"status": "sent", "to": "user@example.com"}
```

## 五、启动 Worker

```bash
# 基础启动（在项目根目录执行）
celery -A celery_app worker --loglevel=info

# 参数说明：
# -A celery_app：Celery 实例所在的模块（celery_app.py）
# worker：启动 Worker 模式
# --loglevel=info：日志级别（debug/info/warning/error）

# 指定并发数（默认=CPU核心数）
celery -A celery_app worker --concurrency=4 --loglevel=info

# 指定消费的队列（多队列场景）
celery -A celery_app worker -Q emails,reports --loglevel=info

# 后台运行（守护进程）
celery -A celery_app worker --detach --logfile=celery.log --pidfile=celery.pid
```

## 六、定时任务（Celery Beat）

除了异步任务，Celery 还支持**定时/周期任务**（类似 cron）：

```python
# celery_app.py 中添加定时任务配置
from celery.schedules import crontab

app.conf.beat_schedule = {
    # 每天凌晨2点清理过期 Token
    "clean-expired-tokens": {
        "task": "tasks.clean_expired_tokens",
        "schedule": crontab(hour=2, minute=0),
    },
    # 每10分钟刷新热点缓存
    "refresh-hot-cache": {
        "task": "tasks.refresh_hot_cache",
        "schedule": 600,  # 秒数
    },
    # 每周一早9点发汇总报告
    "weekly-report": {
        "task": "tasks.send_weekly_report",
        "schedule": crontab(hour=9, minute=0, day_of_week="monday"),
    },
}
```

```bash
# 启动定时任务调度器（和 Worker 分开启动）
celery -A celery_app beat --loglevel=info

# 生产环境：Worker 和 Beat 同时启动
celery -A celery_app worker --beat --loglevel=info
```

## 七、任务重试

```python
@app.task(
    bind=True,
    max_retries=3,          # 最多重试3次
    default_retry_delay=5,  # 默认重试间隔5秒
)
def call_third_party_api(self, user_id: int) -> dict:
    """调用第三方 API（网络可能失败，需要重试）"""
    try:
        response = requests.get(f"https://api.example.com/users/{user_id}", timeout=5)
        response.raise_for_status()
        return response.json()
    except requests.RequestException as exc:
        # 出错时触发重试
        raise self.retry(
            exc=exc,
            countdown=2 ** self.request.retries,  # 指数退避：2s, 4s, 8s
        )
```

## 八、多队列（任务优先级）

实际项目中，不同任务的优先级不同，应分队列处理：

```python
# celery_app.py
from kombu import Queue

app.conf.task_queues = (
    Queue("high_priority"),   # 高优先级：用户通知、支付回调
    Queue("default"),         # 默认：普通业务任务
    Queue("low_priority"),    # 低优先级：报表、统计
)
app.conf.task_default_queue = "default"

# 任务发到指定队列
send_email.apply_async(args=[...], queue="high_priority")
generate_report.apply_async(args=[...], queue="low_priority")
```

```bash
# 不同 Worker 消费不同队列
celery -A celery_app worker -Q high_priority --concurrency=8    # 多并发处理高优先级
celery -A celery_app worker -Q default --concurrency=4
celery -A celery_app worker -Q low_priority --concurrency=2     # 少并发处理低优先级
```

## 常见坑

1. **任务函数不能有不可序列化的参数**：参数必须能 JSON 序列化（避免传 ORM 对象，改传 ID）
2. **任务幂等性**：`task_acks_late=True` + 网络重传可能导致任务重复执行，业务逻辑必须幂等
3. **result.get() 阻塞**：在 Web 请求中用 `get()` 等结果会阻塞请求，失去异步意义
4. **Worker 没有启动**：任务一直 PENDING 不处理，检查 Worker 进程是否在运行
5. **Broker 连接失败**：任务发不出去，检查 Redis/RabbitMQ 是否运行、连接字符串是否正确
6. **定时任务重复执行**：多个 Beat 实例同时运行会重复触发，生产环境只能启动一个 Beat
