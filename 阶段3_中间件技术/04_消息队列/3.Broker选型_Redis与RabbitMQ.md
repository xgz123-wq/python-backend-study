# Broker 选型：Redis vs RabbitMQ

> Broker 选 Redis 还是 RabbitMQ，本质是在简单部署和强路由能力之间做权衡。

## 概念解释

**Broker（消息代理）** 是 Celery 任务队列的核心基础设施，负责接收生产者发来的任务、暂存在队列中，并将任务分发给 Worker 消费。

Celery 支持多种 Broker，最常用的是 **Redis** 和 **RabbitMQ**，两者在功能和适用场景上有明显差异。

---

## 一、Redis 作为 Broker

### 为什么用 Redis 做 Broker？

Redis 本来是缓存数据库，但其 **List 数据结构（LPUSH/BRPOP）** 天然具备消息队列的特性：
- `LPUSH`：生产者把任务推入队列左端
- `BRPOP`：Worker 阻塞等待，取出队列右端的任务

Redis 的优点是**已经是项目基础设施**，无需额外部署新服务。

### Celery + Redis Broker 配置

```python
from celery import Celery

app = Celery(
    "myproject",
    # Broker：Redis db=0（也可以指定 db 号分开）
    broker="redis://:password@localhost:6379/0",
    # Backend：用 Redis 存储任务结果
    backend="redis://:password@localhost:6379/1",
)

# 更完整的配置方式（推荐，方便集中管理）
app.config_from_object({
    "broker_url": "redis://:password@localhost:6379/0",
    "result_backend": "redis://:password@localhost:6379/1",
    "broker_transport_options": {
        "visibility_timeout": 3600,  # Worker 处理任务的最长时间（秒），超时则重新入队
        "retry_policy": {
            "timeout": 5.0,          # 连接 Broker 超时时间
        },
    },
    "result_expires": 3600,          # 结果保留时间（秒），防止 Redis 内存被撑满
})
```

### Redis Broker 的局限性

| 局限 | 说明 |
|------|------|
| 消息持久化弱 | Redis 默认是内存，宕机可能丢消息（需配置 AOF 持久化）|
| 不支持消息路由 | 没有 Exchange/Binding 等路由规则，只能用简单队列名区分 |
| 不支持死信队列（原生）| 需要自己实现失败消息的处理逻辑 |
| 单线程瓶颈 | 高并发写入时 Redis 单线程可能成为瓶颈 |

**适用场景**：中小项目、任务量不大、已部署 Redis、不需要复杂路由。

## 二、RabbitMQ 作为 Broker

### RabbitMQ 核心概念

RabbitMQ 是专业的消息队列服务，基于 **AMQP（Advanced Message Queuing Protocol）** 协议，提供了更完整的消息队列语义：

```
生产者
  ↓ publish
Exchange（交换机）  ← 决定消息路由规则
  ↓ 按路由键绑定
Queue（队列）       ← 消息存储
  ↓ consume
消费者（Worker）
```

| 组件 | 说明 |
|------|------|
| **Exchange（交换机）** | 接收消息并按规则分发到队列，不存储消息 |
| **Queue（队列）** | 真正存储消息，消费者从队列取消息 |
| **Binding（绑定）** | Exchange 和 Queue 之间的路由规则 |
| **Routing Key（路由键）** | 消息的标签，Exchange 据此决定发到哪个 Queue |

### Exchange 类型

| 类型 | 路由规则 | 使用场景 |
|------|---------|---------|
| **direct** | 精确匹配路由键 | 任务按类型分队列（发邮件→email队列，发短信→sms队列）|
| **fanout** | 广播，发到所有绑定的队列 | 系统事件通知（多个服务都需要处理）|
| **topic** | 路由键通配符（`*`单词，`#`多词）| 按模式分类（`order.created`、`order.paid`）|
| **headers** | 按消息头匹配 | 较少使用 |

### Celery + RabbitMQ 配置

```python
from celery import Celery
from kombu import Exchange, Queue

app = Celery(
    "myproject",
    broker="amqp://user:password@localhost:5672//",  # AMQP 协议
    backend="redis://localhost:6379/1",               # 结果仍可用 Redis 存储
)

# 定义 Exchange 和 Queue（实现消息路由）
default_exchange = Exchange("default", type="direct")
priority_exchange = Exchange("priority", type="direct")

app.conf.task_queues = (
    Queue("default",        default_exchange,  routing_key="default"),
    Queue("email",          default_exchange,  routing_key="email"),
    Queue("high_priority",  priority_exchange, routing_key="high"),
)
app.conf.task_default_queue = "default"
app.conf.task_default_exchange = "default"
app.conf.task_default_routing_key = "default"

# 任务路由规则（任务名 → 队列）
app.conf.task_routes = {
    "tasks.send_email": {"queue": "email", "routing_key": "email"},
    "tasks.urgent_task": {"queue": "high_priority", "routing_key": "high"},
}
```

### RabbitMQ 独有功能

#### 死信队列（Dead Letter Queue）

任务失败/超时后，自动路由到死信队列，方便排查问题：

```python
from kombu import Queue

# 定义普通队列 + 死信队列
app.conf.task_queues = (
    Queue(
        "default",
        arguments={
            "x-dead-letter-exchange": "dlx",    # 死信交换机
            "x-dead-letter-routing-key": "dead", # 死信路由键
            "x-message-ttl": 300000,             # 消息最长存活时间（毫秒）
        },
    ),
    Queue("dead_letters"),  # 死信存放队列
)
```

#### 延迟队列（Delayed Messages）

通过 RabbitMQ 插件或 TTL + 死信队列实现延迟执行：

```bash
# 安装延迟消息插件
rabbitmq-plugins enable rabbitmq_delayed_message_exchange
```

## 三、Redis vs RabbitMQ 选型指南

| 对比维度 | Redis Broker | RabbitMQ Broker |
|---------|-------------|-----------------|
| 部署复杂度 | 低（通常已有）| 高（需额外部署）|
| 消息可靠性 | 中（依赖 AOF）| 高（磁盘持久化）|
| 消息路由 | 简单（队列名）| 复杂（Exchange/Binding）|
| 死信队列 | 需手动实现 | 原生支持 |
| 消息延迟 | 通过 ETA 实现 | 插件支持 |
| 吞吐量 | 高 | 中 |
| 监控界面 | 需 Flower | 内置 Web 管理界面（15672端口）|
| 学习成本 | 低 | 中 |
| 适用规模 | 中小项目 | 中大项目 |

### 选型结论

```
项目已用 Redis + 任务量 < 每秒数千 + 不需要复杂路由
  → Redis Broker ✓

需要高可靠性 + 复杂路由规则 + 死信队列 + 大规模任务
  → RabbitMQ Broker ✓

超大规模（每秒百万消息）+ 消息回溯 + 流处理
  → Kafka（Celery 支持但不推荐，Kafka 更适合自定义消费者）
```

## 四、生产环境最佳实践

### Redis Broker 生产配置

```python
app.conf.update(
    # Broker 连接池（提高并发性能）
    broker_pool_limit=10,

    # 连接失败重试
    broker_connection_retry_on_startup=True,
    broker_connection_max_retries=10,

    # 任务可见性超时（Worker 崩溃后，任务多久重新入队）
    broker_transport_options={"visibility_timeout": 43200},  # 12小时

    # 结果过期（防止 Redis 被结果数据撑满）
    result_expires=86400,  # 24小时
)
```

### RabbitMQ 连接字符串格式

```python
# 基础格式
broker = "amqp://username:password@hostname:port/vhost"

# 示例
broker = "amqp://admin:secret@localhost:5672//"  # // 表示使用默认 vhost

# SSL 连接
broker = "amqps://admin:secret@rabbitmq.prod.com:5671//"
```

### 连接字符串安全建议

```python
import os

# 从环境变量读取，不硬编码密码
BROKER_URL = os.getenv("CELERY_BROKER_URL", "redis://localhost:6379/0")
BACKEND_URL = os.getenv("CELERY_RESULT_BACKEND", "redis://localhost:6379/1")

app = Celery("myproject", broker=BROKER_URL, backend=BACKEND_URL)
```

## 常见坑

1. **Redis Broker 宕机丢消息**：需要配置 Redis AOF 持久化，并设置 `visibility_timeout`
2. **`visibility_timeout` 设置不当**：比任务实际执行时间短 → 任务被重复执行
3. **RabbitMQ 连接数耗尽**：每个 Worker 子进程都建连接，需合理设置连接池和并发数
4. **Backend 不配置就用 result.get()**：没有 Backend 时 `get()` 会永久阻塞
5. **多环境 Broker 地址不分离**：开发和生产使用同一个 Redis/RabbitMQ，导致任务互相干扰
