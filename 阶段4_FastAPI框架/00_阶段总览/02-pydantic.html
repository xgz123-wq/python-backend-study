第二节 · Pydantic 数据验证

用 Python 类型注解自动做数据验证和序列化，FastAPI 的「请求体」和「响应体」都靠它。

用类型注解写一份数据契约，前端发错格式自动返回 422。
⏱️ 预估阅读 7 分钟
🎯 学习目标 本节你要带走什么 ▶
  • 理解 Pydantic = Python 类型注解驱动的验证库
  • 会用 BaseModel + Field(...) 写带约束的字段
  • 区分 Create / Update / Response 三种 Schema 用途
  • 理解 response_model 的字段过滤作用（防泄露）
  • 知道 from_attributes=True 是 ORM 转换开关
与后端开发的关联：Schema 是「接口契约」的代码化——前端传错字段类型后端立即 422，无需写 if/else 校验。响应 Schema 决定「暴露给前端什么字段」，是把密码、token 等敏感字段挡在 API 外的第一道闸。
📖 核心概念 验证流程 + Schema 三件套 ▶
核心定义 Pydantic 用 Python 的类型注解（str、int、Optional[...]）声明字段，FastAPI 在收到请求时自动实例化这个类做验证；实例化失败 → 422 + 详细错误信息。

1. 验证流程

客户端发 JSON
     ↓
FastAPI 看到参数类型是 Pydantic BaseModel
     ↓
Pydantic 把 JSON 转成 Python 对象（按类型 + Field 约束校验）
     ↓
✓ 成功 → 调用 handler，对象是合法 Python 实例
✗ 失败 → 自动返回 422 + 错误详情（不用写 try/except）

2. 三种 Schema 拆分（不要混用同一个类）

Schema作用字段特点
XxxCreate接收「创建」请求无 id，必填字段用 Field(...)
XxxUpdate接收「更新」请求所有字段 Optional，只更新传入的
XxxResponse返回给客户端有 id，可排除密码等敏感字段

3. Field(...) 常用约束

name:        str = Field(..., min_length=1, max_length=100)
price:       float = Field(..., gt=0)               # gt: 严格大于；ge: 大于等于
age:         int  = Field(..., ge=0, le=150)
email:       str  = Field(..., regex=r"^.+@.+$")   # Pydantic v1 用 regex，v2 改 pattern
description: Optional[str] = Field(None, max_length=500)
tags:        list[str] = Field(default_factory=list)

4. response_model 字段过滤原理

路由函数的返回值如果比 response_model 声明的字段多，Pydantic 只取 Schema 里有的字段——多余的字段被丢弃。这就是「敏感字段防泄露」的工作机制。

ORM 对象 {id, name, password_hash, created_at}
                ↓
response_model = UserResponse(id, name)  # 只声明 id, name
                ↓
返回给前端 {id, name}   ← password_hash 和 created_at 不会泄露

① 最小 Schema 演示（来自 2.Pydantic数据验证.md）

from pydantic import BaseModel, Field
from typing import Optional

# 请求 Schema（接收客户端数据）
class ItemCreate(BaseModel):
    name: str = Field(..., min_length=1, max_length=100)
    price: float = Field(..., gt=0)          # gt=0：必须大于0
    description: Optional[str] = None  # 可选字段

# 响应 Schema（返回给客户端的数据，可以不同于存库的字段）
class ItemResponse(BaseModel):
    id: int
    name: str
    price: float

    model_config = {"from_attributes": True}  # 允许从 ORM 对象转换

# 路由中使用
@app.post("/items/", response_model=ItemResponse)
async def create_item(item: ItemCreate):
    # item 已经过验证，name 一定有值，price 一定 > 0
    ...

② 完整三件套（来自 schemas.py 真实项目）

from pydantic import BaseModel, Field
from typing import Optional


class ItemBase(BaseModel):
    name: str = Field(..., min_length=1, max_length=100, description="商品名称")
    price: float = Field(..., gt=0, description="价格，必须大于0")
    description: Optional[str] = Field(None, max_length=500)


class ItemCreate(ItemBase):
    """创建商品的请求体"""
    pass


class ItemUpdate(BaseModel):
    """更新商品的请求体（全部可选，只改传入的字段）"""
    name: Optional[str] = Field(None, min_length=1, max_length=100)
    price: Optional[float] = Field(None, gt=0)
    description: Optional[str] = None


class ItemResponse(ItemBase):
    """返回给客户端的格式"""
    id: int

    # from_attributes=True：允许从 SQLAlchemy ORM 对象直接转换
    model_config = {"from_attributes": True}

⚠️ 常见误区 最常踩的坑 ▶
❌ 误区✅ 正确做法
用同一个 Schema 既做请求体又做响应体（class Item(BaseModel): id, name, price 一把抓） 拆成 Create / Update / Response 三件套。Create 没有 id，Response 包含 id，Update 全 Optional
把 password_hash 等敏感字段放进 UserResponse，忘记过滤 返回 Schema 只声明需要暴露的字段；多余的字段 Pydantic 自动丢弃
默认字段写成可变对象：tags: list = []，所有实例共享同一个列表 用 Field(default_factory=list)，每个实例独立初始化
忘加 model_config = {"from_attributes": True}，ORM 对象无法转响应体 SQLAlchemy ORM 对象需要这个开关才能被 Pydantic 序列化
安全：响应 Schema 漏字段 = 密码/Token 直接泄露。生产代码审查时把 XxxResponse 列表当作「对外公开接口」严格走查。
