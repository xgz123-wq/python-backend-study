第四节 · async 异步

让程序在等待 IO 时不阻塞，去处理其他请求——FastAPI 路由的并发能力全部来自这里。

等数据库时不阻塞别的请求，用 async/await 撑住高并发。
⏱️ 预估阅读 6 分钟
🎯 学习目标 本节你要带走什么 ▶
  • 理解「异步 = IO 等待期间不阻塞，去处理其他请求」
  • 会用 async def + await 写异步函数
  • 区分「必须 await 的 IO 场景」与「纯 CPU 计算不需 async」
  • 识别「异步路由 + 同步驱动」这个最致命组合
  • 理解事件循环与协程的关系
与后端开发的关联：异步是 FastAPI 高并发的核心机制。但 async 是个契约——一旦路由是 async def，它调用的所有 IO 库都必须是异步的（asyncpg / aioredis / httpx），否则整个事件循环被卡死，等于单线程。
📖 核心概念 同步 vs 异步 · 协程 · 事件循环 ▶
核心定义 async def 声明的函数是「协程」（coroutine），调用时返回协程对象，await 才真正执行。事件循环（event loop）负责在 IO 等待时切换去处理其他协程——这是「并发」不是「并行」。

1. 同步 vs 异步的执行流

同步（blocking）：                    异步（non-blocking）：

请求1: |==查 DB 100ms==|响应         请求1: |==查 DB........|
请求2:          |==查 DB 100ms==|响应 请求2:        |==查 DB........|响应
请求3:                   |==查 DB...| 请求3:               |==查 DB....|响应
                                      事件循环在 DB 等待时切去处理下一个请求

总耗时: 300ms  (串行)                  总耗时: ≈100ms (并发)

2. 协程三要素

1. async def fetch_user(...):  ← 声明「这是一个协程函数」
2. result = fetch_user(123)    ← 调用时返回协程对象（不执行）
3. user = await result         ← await 触发执行并等待结果

3. 何时用 / 何时不用 async

场景要求
数据库查询（SQLAlchemy async）必须 await，驱动必须是异步的（aiosqlite/asyncpg/aiomysql）
Redis 操作必须 await（用 redis.asyncio）
HTTP 请求必须 await（用 httpx.AsyncClient，不要用 requests）
文件 IO（大文件）用 aiofiles + await
纯 CPU 计算（循环/排序/JSON 序列化）不需要 async，直接普通函数

4. FastAPI 的「async 容忍」机制

FastAPI 路由 async def 可以 await 异步函数；如果你写的是普通 def，FastAPI 会自动放到线程池里跑，不会阻塞事件循环。但你必须清楚自己在哪种模式下——错了就有性能问题。

💻 代码示例 从 demo/app/database.py 异步引擎 + routers/items.py 摘录 ▶

① 最小异步演示（来自 4.async异步.md）

import asyncio

# async def 声明异步函数
async def fetch_user(user_id: int):
    await asyncio.sleep(0.1)   # await = 等待这个 IO 操作，期间可处理其他事
    return {"id": user_id}

# FastAPI 路由：async def 路由 + await 调用异步函数
@app.get("/users/{user_id}")
async def get_user(user_id: int):
    user = await fetch_user(user_id)   # await 异步调用
    return user

③ 路由里 await 异步查询（来自 routers/items.py）

from fastapi import APIRouter, Depends, HTTPException
from sqlalchemy.ext.asyncio import AsyncSession
from sqlalchemy import select

from app.deps import get_db
from app.models.item import Item
from app.schemas import ItemResponse

router = APIRouter(prefix="/items", tags=["商品"])

# 查询列表：async 路由 + await 异步查询
@router.get("/", response_model=list[ItemResponse])
async def list_items(
    skip: int = 0,
    limit: int = 10,
    db: AsyncSession = Depends(get_db),
):
    # await 让出控制权，期间事件循环可处理其他请求
    result = await db.execute(select(Item).offset(skip).limit(limit))
    return result.scalars().all()

# 创建：await commit + refresh
@router.post("/", response_model=ItemResponse, status_code=201)
async def create_item(
    data: ItemCreate,
    db: AsyncSession = Depends(get_db),
):
    db_item = Item(**data.model_dump())
    db.add(db_item)
    await db.commit()           # 异步提交
    await db.refresh(db_item)    # 异步拿自增 id
    return db_item
⚠️ 常见误区 最常踩的坑 ▶
❌ 误区✅ 正确做法
异步路由里调用同步数据库驱动 pymysql.connect() / psycopg2.connect()，请求来了直接卡死整个事件循环 异步路由必须用异步驱动：asyncpg / aiomysql / aiosqlite，配合 SQLAlchemy AsyncSession
以为 async def 自动加速——结果没用 await，等于普通函数套了层壳 async 函数里必须有 await 才会真正让出控制权；纯计算不需要 async
用 requests 库（同步）发 HTTP 请求：requests.get(...) 用 httpx.AsyncClient + await client.get(...)
async def 里调 time.sleep(1)——同步 sleep 也会阻塞事件循环 1 秒 用 await asyncio.sleep(1)
致命组合：async def 路由 + 同步 IO 驱动 = 「假异步」。FastAPI 看起来在跑，但所有请求实际上串行排队，整个系统并发能力被一个 requests.get() 拉回单线程水平。压测才能发现，但生产已经爆炸。
