第六节 · 中间件与异常处理

中间件拦截每个请求（日志/响应头/CORS）；异常处理把程序错误转成统一 JSON 格式——告别 500 堆栈露给前端。

给每个请求装一层拦截器，统一打日志、统一返错，不再泄露 500 堆栈。
⏱️ 预估阅读 7 分钟
🎯 学习目标 本节你要带走什么 ▶
  • 理解中间件 = 每个请求都过一遍的拦截层（洋葱模型）
  • 会用 @app.middleware("http") 自定义中间件（打日志/加响应头）
  • 理解 CORSMiddleware 的作用和顺序坑
  • 会用 @app.exception_handler 把业务异常转成统一 JSON 响应
  • 理解 HTTPException 和自定义业务异常的分工
与后端开发的关联：中间件是「横切关注点」的标准位置（日志、鉴权、限流、压缩、TraceID），路由不需要写这些。异常处理是「不把内部错误露给前端」的第一道安全屏障——线上 500 堆栈泄露是严重事故。
📖 核心概念 洋葱模型 + 中间件顺序 + 异常处理 ▶
核心定义 中间件是一个 async def(request, call_next) → response 函数，注册后 每个请求 都会穿过它；call_next 调用下一个中间件或最终 handler。多个中间件按「后进先出」顺序执行（洋葱模型）。

1. 洋葱模型执行流

请求进入
   ↓
中间件 A 入口 (打时间戳)
   ↓
中间件 B 入口 (记 trace_id)
   ↓
中间件 C 入口 (CORS 检查)
   ↓
路由 handler (业务)
   ↓
中间件 C 出口
   ↓
中间件 B 出口
   ↓
中间件 A 出口 (算耗时写响应头)
   ↓
响应返回

2. add_middleware 是后进先出

代码顺序执行顺序（请求方向）
app.add_middleware(A) 先B 先执行入口
app.add_middleware(B) 后A 后执行入口
→ 后 add 的反而最外层，所以 CORS 写最后 add（最外层）

3. 异常处理两种风格

风格 A：直接用 FastAPI 自带的 HTTPException
   raise HTTPException(status_code=404, detail="not found")
   → 自动返回 404 + {"detail": "not found"}

风格 B：自定义业务异常 + @app.exception_handler 统一拦截
   class ItemNotFound(Exception): ...
   raise ItemNotFound(item_id=42)
   @app.exception_handler(ItemNotFound)
   async def handler(request, exc):
       return JSONResponse(status_code=404, content={...})

   → 自定义错误格式（统一 code/message 结构）

4. 常见中间件场景

中间件作用
CORSMiddleware处理跨域（前后端分离必须）
GZipMiddleware响应自动 gzip 压缩
TrustedHostMiddleware限制允许访问的 Host（防 Host 头攻击）
自定义 @app.middleware("http")打日志、加响应头、记录耗时、注入 trace_id

① 最小演示（来自 6.中间件与异常处理.md）

import time
from fastapi import Request

# 自定义中间件：记录每个请求耗时
@app.middleware("http")
async def add_process_time(request: Request, call_next):
    start = time.time()
    response = await call_next(request)   # 调用下一个处理层
    process_time = time.time() - start
    response.headers["X-Process-Time"] = str(process_time)
    return response

# CORS 中间件（前后端分离必须加）
from fastapi.middleware.cors import CORSMiddleware

app.add_middleware(
    CORSMiddleware,
    allow_origins=["http://localhost:3000"],  # 生产改为实际前端域名
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)

② 真实项目 main.py（中间件 + 异常处理 + lifespan）

import time
from contextlib import asynccontextmanager

from fastapi import FastAPI, Request
from fastapi.middleware.cors import CORSMiddleware
from fastapi.responses import JSONResponse

from app.database import init_db
from app.routers.items import router as items_router


# 生命周期管理（替代已废弃的 on_event）
@asynccontextmanager
async def lifespan(app: FastAPI):
    await init_db()
    yield


app = FastAPI(
    title="FastAPI 学习 Demo",
    version="1.0.0",
    lifespan=lifespan,
)

# ---- CORS 中间件（必须最先 add）----
app.add_middleware(
    CORSMiddleware,
    allow_origins=["*"],   # 开发用 *，生产改为实际前端域名
    allow_credentials=True,
    allow_methods=["*"],
    allow_headers=["*"],
)

# ---- 自定义中间件（记录请求耗时）----
@app.middleware("http")
async def log_process_time(request: Request, call_next):
    start = time.time()
    response = await call_next(request)
    duration = round((time.time() - start) * 1000, 2)
    response.headers["X-Process-Time"] = f"{duration}ms"
    print(f"{request.method} {request.url.path} → {response.status_code} ({duration}ms)")
    return response

# ---- 全局异常处理（把未捕获异常转为统一 JSON）----
@app.exception_handler(Exception)
async def global_exception_handler(request: Request, exc: Exception):
    return JSONResponse(
        status_code=500,
        content={"code": 500, "message": "Internal Server Error", "detail": str(exc)},
    )

# ---- 注册路由 ----
app.include_router(items_router)


@app.get("/", tags=["health"])
async def health_check():
    return {"status": "ok", "docs": "/docs"}

③ 业务异常 + 自定义 Handler

from fastapi import HTTPException, Request
from fastapi.responses import JSONResponse

# 自定义业务异常
class ItemNotFound(Exception):
    def __init__(self, item_id: int):
        self.item_id = item_id

# 注册异常处理器
@app.exception_handler(ItemNotFound)
async def item_not_found_handler(request, exc):
    return JSONResponse(
        status_code=404,
        content={"code": 404, "message": f"Item {exc.item_id} not found"}
    )

# 路由里直接 raise，不用写 if/return
@app.get("/items/{item_id}")
async def get_item(item_id: int, db: AsyncSession = Depends(get_db)):
    item = await db.get(Item, item_id)
    if not item:
        raise ItemNotFound(item_id)   # 自动走 handler，返回 404 JSON
    return item

⚠️ 常见误区 最常踩的坑 ▶
❌ 误区✅ 正确做法
CORS 写在其他中间件 下面（最后 add），预检请求被其他中间件拦截，前端报 CORS 错误 CORS 写在最上面（第一个 add）——后进先出，最先 add 反而最内层。CORSMiddleware 必须在最外层处理 OPTIONS 预检
中间件里漏写 await call_next(request)，请求卡住不返回 call_next 一定要 await，它返回的是协程
把 detail=str(exc) 原样返回给客户端（500 时泄露堆栈） 生产环境只返回友好 message，str(exc) 写到日志/监控系统（不要回给前端）
业务代码到处 try/except 然后 return JSONResponse(...) 凑响应 统一 raise 业务异常 + @app.exception_handler 拦截，路由保持干净
CORS 顺序：这是 FastAPI 部署时最常见的前端联调问题。代码里 add_middleware 第一个写的就是执行最外层；CORSMiddleware 必须 第一个 add，否则浏览器 OPTIONS 预检被其他中间件截掉。
