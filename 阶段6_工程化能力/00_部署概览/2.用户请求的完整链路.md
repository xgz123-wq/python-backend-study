# 2. 用户请求的完整链路

> 一次请求从浏览器到数据库走了哪些节点，这是排查线上问题的地图。

## 概念解释

当用户在浏览器输入 `https://api.example.com/users` 并按下回车，到看到返回结果，中间经历了一系列复杂的环节。作为后端开发者，理解这条完整链路是排查问题、优化性能的基础。

```
用户输入域名
    ↓
① DNS 解析（域名 → IP 地址）
    ↓
② CDN 缓存检查（是否有缓存的响应）
    ↓
③ TCP 连接建立（三次握手）
    ↓
④ TLS 加密握手（HTTPS）
    ↓
⑤ 发送 HTTP 请求到服务器
    ↓
⑥ Web 服务器（Nginx）接收请求
    ↓
⑦ 应用服务器（Python）处理业务逻辑
    ↓
⑧ 数据库查询（MySQL/PostgreSQL）
    ↓
⑨ 原路返回响应给用户
```

## 为什么需要理解完整链路

| 场景 | 不理解链路时 | 理解链路后 |
|------|-------------|-----------|
| 用户反馈"网站打不开" | 不知道问题在哪 | 能逐步排查：DNS？服务器？数据库？ |
| 网站加载很慢 | 只能猜测原因 | 知道是 DNS 慢、TTFB 高还是传输慢 |
| 需要优化性能 | 无从下手 | 知道在哪个环节可以做缓存、压缩 |
| 配置 HTTPS | 照着教程盲目操作 | 理解 TLS 握手过程和证书链 |

## 环节 1：DNS 解析

### 什么是 DNS

**DNS（Domain Name System，域名系统）** 是互联网的电话簿。人类记忆域名（如 `google.com`）比记忆 IP 地址（如 `142.250.80.46`）容易得多，DNS 的作用就是将域名翻译成对应的 IP 地址。

### DNS 解析过程

```
用户输入 api.example.com
    ↓
① 查本地 DNS 缓存（浏览器缓存 → 操作系统缓存）
    ↓ 缓存未命中
② 向本地 DNS 服务器查询（通常由 ISP 提供）
    ↓ 本地 DNS 服务器也没有
③ 本地 DNS 服务器向根域名服务器查询
    ↓ 根服务器返回 .com 顶级域服务器地址
④ 向 .com 顶级域服务器查询
    ↓ 返回 example.com 的权威 DNS 服务器地址
⑤ 向权威 DNS 服务器查询
    ↓ 返回 api.example.com 的 IP 地址
⑥ IP 地址层层回传，最终到达用户浏览器
```

### DNS 记录类型

| 记录类型 | 用途 | 示例 |
|----------|------|------|
| A | 域名 → IPv4 地址 | `example.com → 93.184.216.34` |
| AAAA | 域名 → IPv6 地址 | `example.com → 2606:2800:220:1:...` |
| CNAME | 域名 → 另一个域名 | `www.example.com → example.com` |
| MX | 邮件服务器地址 | `example.com → mail.example.com` |
| TXT | 文本记录（验证用） | SPF、DKIM 等邮件验证 |

### DNS 在部署中的角色

部署后端服务时，你需要：

1. **购买域名**：通过域名注册商（如阿里云万网、Cloudflare）
2. **配置 DNS 解析**：将域名指向你的服务器 IP
3. **设置 TTL**：控制 DNS 缓存时间，TTL 越短，DNS 变更生效越快

```python
# Python 中进行 DNS 查询
import socket

# 查询域名对应的 IP 地址
ip = socket.gethostbyname("www.baidu.com")
print(f"www.baidu.com 的 IP 地址: {ip}")

# 获取更详细的 DNS 信息
result = socket.getaddrinfo("www.baidu.com", 443)
for info in result[:3]:
    family, socktype, proto, canonname, sockaddr = info
    print(f"地址族: {family}, 地址: {sockaddr}")
```

## 环节 2：CDN 缓存

### 什么是 CDN

**CDN（Content Delivery Network，内容分发网络）** 是分布在全球各地的缓存服务器网络。它将你的网站内容缓存到离用户最近的节点，大幅减少访问延迟。

### CDN 工作原理

```
没有 CDN 的情况：
用户（北京） ──────────── 长距离 ────────────→ 服务器（美国）
延迟: ~200ms

有 CDN 的情况：
用户（北京） ── 短距离 ──→ CDN 节点（北京）→ 缓存命中，直接返回
延迟: ~10ms

                        如果缓存未命中：
CDN 节点（北京） ──────────→ 源服务器（美国）→ 获取内容并缓存
```

### CDN 适用场景

| 内容类型 | 是否适合 CDN | 原因 |
|----------|-------------|------|
| 图片、视频、CSS、JS | ✅ 非常适合 | 静态内容，不变化 |
| API 接口响应 | ❌ 通常不适合 | 动态内容，每次都不同 |
| 下载文件 | ✅ 适合 | 大文件，CDN 加速明显 |
| 实时数据 | ❌ 不适合 | 需要最新数据，不能缓存 |

### CDN 关键概念

- **源站（Origin）**：你的真实服务器
- **边缘节点（Edge）**：CDN 在全球各地的缓存服务器
- **缓存命中（Hit）**：CDN 节点有缓存，直接返回
- **缓存未命中（Miss）**：CDN 节点没有缓存，需要回源获取
- **回源（Origin Pull）**：CDN 节点向源站请求内容

## 环节 3-4：TCP 连接与 TLS 握手

### TCP 三次握手

在发送 HTTP 请求之前，客户端和服务器需要先建立 TCP 连接：

```
客户端                         服务器
  │                              │
  │──── SYN（我要连接）────────→ │  第 1 次握手
  │                              │
  │←─── SYN+ACK（好的，来吧）── │  第 2 次握手
  │                              │
  │──── ACK（确认，开始通信）──→ │  第 3 次握手
  │                              │
  │     连接建立，可以传输数据    │
```

### TLS 握手（HTTPS）

如果使用 HTTPS，还需要额外的 TLS 握手来协商加密参数：

```
客户端                              服务器
  │                                   │
  │─── ClientHello（支持的加密方式）→ │
  │                                   │
  │←── ServerHello（选定的加密方式）─ │
  │    + 服务器证书                    │
  │                                   │
  │─── 验证证书 + 密钥交换 ─────────→ │
  │                                   │
  │←── 确认 ─────────────────────────│
  │                                   │
  │      加密通道建立完成              │
```

这就是为什么 HTTPS 比 HTTP 慢的原因——多了 TLS 握手的开销。TLS 1.3 将握手从 2-RTT 优化到了 1-RTT。

## 环节 5-7：服务器处理请求

### Web 服务器（Nginx）

Nginx 作为反向代理，是请求到达后端应用前的第一道门：

```
外部请求
    ↓
Nginx 接收请求
    ↓
┌─────────────────────────────────┐
│  Nginx 的职责：                  │
│  1. SSL 终止（解密 HTTPS）      │
│  2. 静态文件直接返回             │
│  3. 动态请求转发给 Python 应用   │
│  4. 负载均衡（分发到多个后端）   │
│  5. 限流和访问控制              │
└─────────────────────────────────┘
    ↓
转发给 Gunicorn / uWSGI
```

### 应用服务器（Gunicorn + Python）

```python
# 一个简单的 Flask 应用
from flask import Flask, jsonify

app = Flask(__name__)

@app.route("/api/users")
def get_users():
    # ① 解析请求参数
    page = request.args.get("page", 1, type=int)

    # ② 查询数据库
    users = db.query("SELECT * FROM users LIMIT 10 OFFSET %s", (page - 1) * 10)

    # ③ 处理业务逻辑
    result = [{"id": u.id, "name": u.name} for u in users]

    # ④ 返回 JSON 响应
    return jsonify({"users": result, "page": page})
```

### 请求在 Python 应用中的处理流程

```
HTTP 请求到达
    ↓
① URL 路由匹配：找到对应的处理函数
    ↓
② 中间件处理：鉴权、日志、CORS 等
    ↓
③ 请求解析：提取参数、Body、Header
    ↓
④ 业务逻辑：核心处理（可能调用其他服务）
    ↓
⑤ 数据库操作：查询或修改数据
    ↓
⑥ 构造响应：序列化数据为 JSON
    ↓
⑦ 返回 HTTP 响应
```

## 环节 8：数据库查询

```
Python 应用
    ↓
① 获取数据库连接（从连接池）
    ↓
② 构造 SQL 查询
    ↓
③ 发送查询到数据库服务器
    ↓
④ 数据库执行查询（索引扫描、表连接等）
    ↓
⑤ 返回结果集
    ↓
⑥ Python 应用将结果转换为对象
    ↓
⑦ 归还数据库连接到连接池
```

## 环节 9：返回响应

响应沿着请求的反方向返回：

```
数据库 → Python 应用 → Gunicorn → Nginx → 网络传输 → CDN → 浏览器渲染
```

### HTTP 响应结构

```
HTTP/1.1 200 OK              ← 状态行
Content-Type: application/json  ← 响应头
Cache-Control: no-cache
Content-Length: 128
                                ← 空行
{"users": [...], "page": 1}    ← 响应体
```

## 性能指标

### 关键指标一览

| 指标 | 全称 | 含义 | 参考值 |
|------|------|------|--------|
| DNS | DNS Lookup Time | 域名解析耗时 | < 50ms |
| TCP | TCP Connection Time | 建立连接耗时 | < 100ms |
| TLS | TLS Handshake Time | 加密握手耗时 | < 100ms |
| TTFB | Time To First Byte | 从请求发出到收到第一个字节 | < 200ms |
| 总延迟 | Total Latency | 完整的请求-响应时间 | < 1s |

### TTFB 详解

**TTFB（Time To First Byte）** 是衡量后端性能最重要的指标：

```
|←────────────── TTFB ──────────────→|
| DNS | TCP | TLS | 请求发送 | 服务器处理 | 首字节返回 |
                                     ↑
                              这里体现后端性能
```

TTFB 高意味着：
- 服务器处理慢（业务逻辑复杂、数据库查询慢）
- 网络延迟高（服务器太远、没有 CDN）
- 服务器过载（CPU/内存不足）

### 延迟、带宽与吞吐

```
延迟（Latency）：数据从 A 到 B 需要的时间
  → 北京到纽约的延迟约 200ms
  → 同城延迟约 5ms

带宽（Bandwidth）：单位时间能传输的数据量
  → 100Mbps 带宽 = 每秒最多传输 12.5MB

吞吐量（Throughput）：实际传输的数据量
  → 受限于带宽和延迟的共同影响
```

### 测量请求耗时的 Python 代码

```python
import time
import urllib.request

url = "https://www.baidu.com"

# 测量总耗时
start = time.perf_counter()
response = urllib.request.urlopen(url)
data = response.read()
elapsed = time.perf_counter() - start

print(f"请求 URL: {url}")
print(f"状态码: {response.status}")
print(f"响应大小: {len(data)} bytes")
print(f"总耗时: {elapsed * 1000:.2f} ms")
```

```
    │ ⑤ 反向代理转发
    ▼

    ▼

    │
    │ ⑦ ORM 查询
    ▼

    │
    │ ⑧ 返回结果，原路回传
    ▼
用户浏览器渲染结果
```

## 常见坑

### 1. DNS 缓存导致的"部署后访问不到新服务器"

修改了 DNS 记录后，由于各级 DNS 缓存（TTL 未到期），用户可能仍然访问到旧服务器。

**解决方案**：部署前将 TTL 调低（如 60 秒），等待旧缓存过期后再切换。

### 2. CDN 缓存了不该缓存的内容

API 响应被 CDN 缓存，导致用户看到过期数据。

**解决方案**：为 API 路径设置 `Cache-Control: no-cache` 响应头。

### 3. HTTPS 证书过期

证书过期后浏览器会显示安全警告，用户无法访问。

**解决方案**：使用 Let's Encrypt 免费证书并配置自动续期。

### 4. 反向代理超时

Nginx 默认代理超时为 60 秒，长时间运行的请求会被中断。

**解决方案**：调整 `proxy_read_timeout` 参数，或改为异步任务。

## 小结

| 环节 | 作用 | 后端开发者可优化的点 |
|------|------|---------------------|
| DNS | 域名解析 | 选择可靠的 DNS 服务商 |
| CDN | 内容缓存加速 | 合理设置缓存策略 |
| TCP/TLS | 建立安全连接 | 启用 HTTP/2、TLS 1.3 |
| Nginx | 反向代理、负载均衡 | 优化配置、启用 gzip |
| Python 应用 | 业务逻辑处理 | 代码优化、异步处理 |
| 数据库 | 数据存储和查询 | 索引优化、查询优化、连接池 |
