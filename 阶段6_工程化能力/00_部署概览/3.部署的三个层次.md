# 3. 部署的三个层次

> 单机、容器、编排——根据项目规模选对部署方式，比堆工具更重要。

## 概念解释

部署并不是只有一种方式。根据项目的复杂度和规模，部署可以分为三个层次，每个层次对应不同的工具和方法。选择合适的部署方式，是后端开发者的重要决策。

```
复杂度递增 →

┌─────────────┐   ┌─────────────┐   ┌─────────────┐
│  静态部署    │   │  应用部署    │   │ 容器化部署   │
│  Static     │ → │  Application│ → │  Container   │
│             │   │             │   │             │
│ 纯前端文件  │   │ 带后端逻辑  │   │ 复杂应用    │
│ HTML/CSS/JS │   │ API/数据库  │   │ 微服务集群  │
└─────────────┘   └─────────────┘   └─────────────┘
```

## 为什么需要了解三个层次

| 问题 | 答案 |
|------|------|
| 个人博客该用什么部署？ | 静态部署，免费且简单 |
| 公司内部 API 该用什么？ | 应用部署，需要服务器和数据库 |
| 大型电商系统该用什么？ | 容器化部署，需要弹性和可扩展性 |

选择错误的部署方式会导致：过度复杂（杀鸡用牛刀）或能力不足（项目跑不起来）。

## 第一层：静态部署

### 什么是静态部署

静态部署用于托管**不需要服务器端处理**的文件：HTML、CSS、JavaScript、图片等。所有内容都是预先生成好的文件，服务器只需原样返回。

```
用户请求 index.html
    ↓
服务器找到文件 index.html
    ↓
直接返回文件内容（无需任何计算）
```

### 适用场景

| 场景 | 说明 |
|------|------|
| 个人博客 | Hugo、Hexo、Jekyll 生成的静态站点 |
| 项目文档 | 技术文档、API 文档 |
| 前端单页应用 | React/Vue 构建的 SPA |
| 作品集 | 个人作品集展示页 |
| 落地页 | 产品宣传页、活动页 |

### 典型工具对比

| 工具 | 费用 | 难度 | 特点 |
|------|------|------|------|
| GitHub Pages | 免费 | ⭐ | 与 GitHub 深度集成 |
| Vercel | 免费/付费 | ⭐ | 自动部署、全球 CDN |
| Netlify | 免费/付费 | ⭐ | 表单处理、Serverless 函数 |
| Cloudflare Pages | 免费 | ⭐ | 无限带宽、边缘计算 |
| 阿里云 OSS | 按量付费 | ⭐⭐ | 国内访问快 |

### 静态部署示例

```bash
# 使用 GitHub Pages 部署
# 1. 将代码推送到 GitHub 仓库
git push origin main

# 2. 在仓库 Settings → Pages 中选择分支
# 3. 自动生成 https://username.github.io/repo-name/

# 使用 Vercel 部署
# 1. 安装 Vercel CLI
npm install -g vercel

# 2. 在项目目录执行
vercel

# 3. 按提示操作，自动部署到 Vercel
```

### 静态部署的优缺点

```
✅ 优点：
├── 零服务器成本（大多数平台免费）
├── 零运维（平台处理一切）
├── 极快的加载速度（CDN + 静态文件）
├── 天然高并发（静态文件几乎不消耗 CPU）
└── 自动 HTTPS

❌ 缺点：
├── 无法运行后端逻辑
├── 无法直接操作数据库
├── 内容更新需要重新构建
└── 不适合需要用户认证的场景
```

## 第二层：应用部署

### 什么是应用部署

应用部署用于托管**需要服务器端处理**的项目：API 接口、数据库操作、用户认证等。这是后端开发者最常用的部署方式。

```
用户请求 /api/users
    ↓
服务器接收请求
    ↓
Python 应用执行业务逻辑
    ↓
查询数据库
    ↓
返回动态生成的响应
```

### 适用场景

| 场景 | 说明 |
|------|------|
| REST API | 为前端/移动端提供数据接口 |
| Web 应用 | Django/Flask/FastAPI 全栈应用 |
| 小程序后端 | 微信小程序、支付宝小程序后端 |
| 定时任务 | 需要定时执行的后台任务 |
| WebSocket 服务 | 实时通信、在线聊天 |

### 典型工具对比

| 工具 | 费用 | 难度 | 特点 |
|------|------|------|------|
| 阿里云 ECS | 月付 50+ | ⭐⭐⭐ | 完全控制、国内首选 |
| 腾讯云 CVM | 月付 50+ | ⭐⭐⭐ | 完全控制 |
| Railway | 免费/付费 | ⭐⭐ | 自动部署、简单易用 |
| Render | 免费/付费 | ⭐⭐ | 自动 HTTPS、数据库 |
| Heroku | 付费 | ⭐⭐ | 老牌 PaaS、生态丰富 |
| PythonAnywhere | 免费/付费 | ⭐ | 专为 Python 设计 |

### 应用部署典型架构

```
互联网
  │
  ▼
┌─────────────────────────────────┐
│  Nginx（反向代理 + 静态文件）   │
│  监听 80/443 端口               │
└─────────┬───────────────────────┘
          │
          ▼
┌─────────────────────────────────┐
│  Gunicorn（WSGI 应用服务器）    │
│  多个 worker 进程               │
└─────────┬───────────────────────┘
          │
          ▼
┌─────────────────────────────────┐
│  Flask / Django / FastAPI 应用  │
│  处理业务逻辑                   │
└─────────┬───────────────────────┘
          │
          ▼
┌─────────────────────────────────┐
│  MySQL / PostgreSQL 数据库      │
│  数据存储                       │
└─────────────────────────────────┘
```

### 在云服务器上部署的完整步骤

```bash
# 1. 购买云服务器，获取公网 IP
# 2. SSH 登录服务器
ssh root@47.96.xx.xx

# 3. 安装 Python 和依赖
apt update && apt install python3 python3-pip
pip3 install flask gunicorn

# 4. 上传代码（或通过 Git 拉取）
git clone https://github.com/user/project.git
cd project
pip3 install -r requirements.txt

# 5. 使用 Gunicorn 启动应用
gunicorn -w 4 -b 127.0.0.1:8000 app:app

# 6. 配置 Nginx 反向代理
# 7. 配置 systemd 保持服务运行
# 8. 配置防火墙、HTTPS
```

### 应用部署的优缺点

```
✅ 优点：
├── 能运行任意后端逻辑
├── 能连接数据库
├── 完全控制运行环境
├── 支持 WebSocket 等高级功能
└── 适合绝大多数后端项目

❌ 缺点：
├── 需要服务器费用
├── 需要运维知识（Linux、Nginx、防火墙）
├── 环境配置容易出问题（"我电脑上能跑啊"）
├── 扩容需要手动操作
└── 需要自行处理安全更新
```

## 第三层：容器化部署

### 什么是容器化部署

容器化部署将应用及其所有依赖打包成一个标准化的**容器镜像**，在任何支持 Docker 的机器上都能以完全相同的方式运行。

```
开发者电脑                    测试服务器                   生产服务器
┌──────────────┐          ┌──────────────┐          ┌──────────────┐
│ Docker 容器  │          │ Docker 容器  │          │ Docker 容器  │
│ ┌──────────┐ │          │ ┌──────────┐ │          │ ┌──────────┐ │
│ │ Python   │ │          │ │ Python   │ │          │ │ Python   │ │
│ │ 依赖库   │ │  ←─相同─→│ │ 依赖库   │ │  ←─相同─→│ │ 依赖库   │ │
│ │ 应用代码 │ │   镜像   │ │ 应用代码 │ │   镜像   │ │ 应用代码 │ │
│ │ 系统库   │ │          │ │ 系统库   │ │          │ │ 系统库   │ │
│ └──────────┘ │          │ └──────────┘ │          │ └──────────┘ │
└──────────────┘          └──────────────┘          └──────────────┘
```

### 适用场景

| 场景 | 说明 |
|------|------|
| 微服务架构 | 多个服务独立部署、独立扩展 |
| 团队协作 | 保证所有开发者环境一致 |
| 多环境部署 | 开发/测试/生产使用同一镜像 |
| 大规模应用 | 需要自动扩缩容、负载均衡 |
| 复杂依赖 | 应用需要特定的系统库或工具 |

### 典型工具对比

| 工具 | 费用 | 难度 | 特点 |
|------|------|------|------|
| Docker | 免费 | ⭐⭐⭐ | 容器化标准 |
| Docker Compose | 免费 | ⭐⭐⭐ | 多容器编排 |
| Kubernetes | 免费 | ⭐⭐⭐⭐⭐ | 容器编排王者 |
| 阿里云 ACK | 按量付费 | ⭐⭐⭐⭐ | 托管 K8s |
| AWS ECS/EKS | 按量付费 | ⭐⭐⭐⭐ | AWS 容器服务 |

### Dockerfile 示例

```dockerfile
# 基于官方 Python 镜像
FROM python:3.12-slim

# 设置工作目录
WORKDIR /app

# 复制依赖文件并安装（利用 Docker 缓存层）
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# 复制应用代码
COPY . .

# 暴露端口
EXPOSE 8000

# 启动命令
CMD ["gunicorn", "-w", "4", "-b", "0.0.0.0:8000", "app:app"]
```

### Kubernetes 部署示意

```yaml
# deployment.yaml - K8s 部署配置
apiVersion: apps/v1
kind: Deployment
metadata:
  name: my-api
spec:
  replicas: 3                    # 运行 3 个实例
  selector:
    matchLabels:
      app: my-api
  template:
    metadata:
      labels:
        app: my-api
    spec:
      containers:
      - name: my-api
        image: registry.cn-hangzhou.aliyuncs.com/myproject/my-api:v1.2
        ports:
        - containerPort: 8000
        resources:
          requests:
            memory: "256Mi"
            cpu: "250m"
          limits:
            memory: "512Mi"
            cpu: "500m"
```

### 容器化部署的优缺点

```
✅ 优点：
├── 环境一致性（告别"我电脑上能跑"）
├── 快速部署和回滚
├── 弹性伸缩（自动扩缩容）
├── 资源隔离（不同应用互不影响）
├── 便于微服务架构
└── CI/CD 友好

❌ 缺点：
├── 学习曲线陡峭（Docker + K8s 概念多）
├── 基础设施成本高
├── 对小项目来说过于复杂
├── 调试比直接部署更困难
└── 需要专门的运维知识
```

## 三个层次对比

| 维度 | 静态部署 | 应用部署 | 容器化部署 |
|------|----------|----------|-----------|
| 适用项目 | 前端/文档 | 后端 API/Web 应用 | 微服务/大型应用 |
| 服务器需求 | 无/极低 | 中等 | 较高 |
| 月成本 | 免费 | ¥50-500 | ¥200-2000+ |
| 运维难度 | ⭐ | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ |
| 部署速度 | 秒级 | 分钟级 | 分钟级 |
| 扩展性 | 天然可扩展 | 手动扩展 | 自动扩展 |
| 技术栈 | HTML/CSS/JS | Python + Linux | Docker + K8s |
| 数据库 | 不支持 | 支持 | 支持 |

## 个人开发者如何选择

### 决策流程图

```
你的项目需要后端逻辑吗？
    │
    ├── 否 → 静态部署（GitHub Pages / Vercel）
    │
    └── 是 → 项目规模大吗？需要微服务吗？
              │
              ├── 否 → 应用部署
              │        ├── 想省心？→ Railway / Render
              │        └── 想省钱？→ 阿里云 ECS / 腾讯云 CVM
              │
              └── 是 → 容器化部署
                       ├── 小团队？→ Docker Compose
                       └── 大团队？→ Kubernetes
```

### 推荐的成长路径

```
阶段 1：学会静态部署
  └── 用 GitHub Pages 部署个人博客

阶段 2：学会应用部署
  └── 在云服务器上部署 Flask API

阶段 3：学会容器化部署
  └── 用 Docker 打包和部署应用

阶段 4：学会编排和自动化
  └── 用 Docker Compose / K8s 管理多服务
```

## 常见坑

### 1. 一开始就上 Kubernetes

很多初学者看到 K8s 很火就直接上，结果花了大量时间在基础设施而非业务开发。

**建议**：除非你的项目确实需要，否则从简单的应用部署开始。

### 2. 忽视安全配置

应用部署时忘记配置防火墙、使用 root 用户运行应用、不启用 HTTPS。

**建议**：遵循最小权限原则，使用非 root 用户，始终启用 HTTPS。

### 3. 静态部署用了错误的工具

用云服务器托管纯静态网站，浪费金钱和精力。

**建议**：纯前端项目优先使用 Vercel / GitHub Pages。

### 4. 容器镜像过大

将不必要的文件打入镜像，导致镜像好几 GB，部署缓慢。

**建议**：使用 `.dockerignore` 文件，选择 slim 基础镜像，多阶段构建。

## 小结

| 层次 | 核心思想 | 典型工具 | 适合人群 |
|------|----------|----------|----------|
| 静态部署 | 只托管文件 | GitHub Pages, Vercel | 前端开发者、个人博客 |
| 应用部署 | 运行后端代码 | 云服务器 + Nginx | 后端开发者、中小项目 |
| 容器化部署 | 标准化打包 | Docker, K8s | DevOps、大型项目 |
