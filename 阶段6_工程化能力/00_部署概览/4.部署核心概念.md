# 4. 部署核心概念

> 环境、版本、回滚、健康检查——这些贯穿部署全流程的术语必须先打通。

## 概念解释

部署不只是"把代码放到服务器上"。在实际工程中，有一系列核心概念贯穿整个部署流程，理解这些概念是做好部署工作的基础。

本章涵盖以下核心概念：

```
部署核心概念
├── 环境隔离（Environment）
├── 构建（Build）
├── 回滚（Rollback）
├── 零停机部署（Zero-Downtime）
└── CI/CD（持续集成/持续部署）
```

## 一、环境隔离

### 什么是环境隔离

环境隔离是将软件运行在**多个独立的、用途明确的环境**中，每个环境有各自的配置、数据和依赖，互不干扰。

标准的三环境模型：

```
┌─────────────┐   ┌─────────────┐   ┌─────────────┐
│  开发环境    │   │  测试环境    │   │  生产环境    │
│ Development │──→│   Testing   │──→│ Production  │
│             │   │             │   │             │
│ 开发者本机   │   │ 独立服务器   │   │ 正式服务器   │
│ 假数据      │   │ 模拟数据    │   │ 真实数据    │
│ 可随意调试   │   │ 可调试      │   │ 严禁调试    │
└─────────────┘   └─────────────┘   └─────────────┘
```

### 为什么需要环境隔离

| 场景 | 没有环境隔离 | 有环境隔离 |
|------|-------------|-----------|
| 开发者测试新功能 | 直接修改生产数据库 💀 | 在开发环境自由测试 ✅ |
| 新版本上线前验证 | 无法验证，直接上线赌运气 | 在测试环境完整验证 ✅ |
| 配置不同 | 改配置要手动切换，容易出错 | 每个环境独立配置 ✅ |
| 多人协作 | 互相覆盖、冲突 | 各自独立，互不影响 ✅ |

### 环境变量：实现隔离的核心手段

环境变量是操作系统级别的键值对，不同环境设置不同的值，代码中读取即可。

```python
import os

# 通过环境变量区分不同配置
# 开发环境: DATABASE_URL=sqlite:///dev.db
# 测试环境: DATABASE_URL=mysql://test-server/mydb
# 生产环境: DATABASE_URL=mysql://prod-server/mydb

database_url = os.environ.get(
    "DATABASE_URL",
    "sqlite:///dev.db"  # 默认值用于开发环境
)

debug_mode = os.environ.get("DEBUG", "true").lower() == "true"

secret_key = os.environ.get("SECRET_KEY", "dev-secret-key")
# ⚠️ 生产环境必须设置强密钥，绝不能使用默认值！

print(f"数据库: {database_url}")
print(f"调试模式: {debug_mode}")
```

### 常见的环境变量

| 变量名 | 用途 | 开发环境示例 | 生产环境示例 |
|--------|------|-------------|-------------|
| `DATABASE_URL` | 数据库连接 | `sqlite:///dev.db` | `mysql://user:pass@rds.xxx` |
| `SECRET_KEY` | 加密密钥 | `dev-key` | 随机 64 位字符串 |
| `DEBUG` | 调试模式 | `true` | `false` |
| `LOG_LEVEL` | 日志级别 | `DEBUG` | `WARNING` |
| `REDIS_URL` | 缓存地址 | `redis://localhost` | `redis://redis-cluster` |
| `API_KEY` | 第三方服务密钥 | 测试 key | 正式 key |

### `.env` 文件管理

```bash
# .env.development（开发环境）
DATABASE_URL=sqlite:///dev.db
DEBUG=true
SECRET_KEY=dev-only-key
LOG_LEVEL=DEBUG

# .env.production（生产环境）
DATABASE_URL=mysql://user:strongpass@prod-db:3306/myapp
DEBUG=false
SECRET_KEY=a8f2k9d3m7n1p5...（随机生成）
LOG_LEVEL=WARNING
```

**⚠️ 重要规则**：`.env` 文件绝不能提交到 Git 仓库！

```bash
# .gitignore 中必须包含
.env
.env.local
.env.production
```

## 二、构建（Build）

### 什么是构建

构建是将**源代码**转换为**可运行产物**的过程。不同语言的构建方式不同：

| 语言 | 源代码 | 构建过程 | 可运行产物 |
|------|--------|----------|-----------|
| Python | `.py` 文件 | 安装依赖、打包 | wheel 包 / Docker 镜像 |
| Java | `.java` 文件 | 编译 | `.jar` 文件 |
| JavaScript | `.ts` / `.jsx` | 编译 + 打包 | `.js` + 静态文件 |
| Go | `.go` 文件 | 编译 | 二进制文件 |

### Python 项目的构建过程

Python 是解释型语言，不需要编译，但仍需要以下构建步骤：

```
源代码 + requirements.txt
    ↓
① 创建虚拟环境
    ↓
② 安装依赖（pip install -r requirements.txt）
    ↓
③ 收集静态文件（Django: collectstatic）
    ↓
④ 数据库迁移（migrate）
    ↓
⑤ 打包为可分发格式（wheel / Docker 镜像）
```

### Python 打包工具

```python
# setup.py - 传统的 Python 打包配置
from setuptools import setup, find_packages

setup(
    name="my-api",
    version="1.0.0",
    packages=find_packages(),
    install_requires=[
        "flask>=3.0",
        "gunicorn>=21.0",
        "sqlalchemy>=2.0",
    ],
    entry_points={
        "console_scripts": [
            "my-api=myapi.main:run",
        ],
    },
)
```

```toml
# pyproject.toml - 现代的 Python 打包配置（推荐）
[build-system]
requires = ["setuptools>=68.0", "wheel"]
build-backend = "setuptools.backends._legacy:_Backend"

[project]
name = "my-api"
version = "1.0.0"
dependencies = [
    "flask>=3.0",
    "gunicorn>=21.0",
    "sqlalchemy>=2.0",
]

[project.scripts]
my-api = "myapi.main:run"
```

### 构建命令示例

```bash
# 方式 1：传统方式
python setup.py sdist bdist_wheel

# 方式 2：现代方式（推荐）
pip install build
python -m build

# 方式 3：Docker 构建
docker build -t my-api:v1.0 .
```

## 三、回滚（Rollback）

### 什么是回滚

回滚是在新版本出现问题时，**快速恢复到之前稳定版本**的操作。这是部署安全网中最重要的一环。

```
时间线：
v1.0（稳定） → v1.1（稳定） → v1.2（出 bug！）→ 回滚到 v1.1 → v1.2.1（修复后重新部署）
                                     ↑
                              紧急回滚，恢复服务
```

### 为什么需要回滚机制

| 情况 | 没有回滚机制 | 有回滚机制 |
|------|-------------|-----------|
| 新版本有严重 bug | 手忙脚乱修 bug，用户持续受影响 | 一键回滚，秒级恢复 |
| 数据库迁移出错 | 数据可能损坏 | 回滚迁移 + 回滚代码 |
| 性能严重下降 | 排查问题耗时很长 | 先回滚恢复，再排查 |

### 回滚策略

#### 策略 1：Git 版本回滚

```bash
# 查看最近的部署历史
git log --oneline -5

# 回滚到上一个稳定版本
git revert HEAD
# 重新部署

# 或者回滚到特定版本
git revert abc1234..HEAD
```

#### 策略 2：保留多个版本

```
服务器上的部署目录：
/opt/myapp/
├── current → releases/20240115_v1.2/   （当前运行版本）
├── releases/
│   ├── 20240115_v1.2/   ← 当前版本（有问题）
│   ├── 20240110_v1.1/   ← 上一版本（稳定）
│   └── 20240105_v1.0/   ← 更早的版本
```

```bash
# 快速回滚：修改软链接指向上一版本
ln -sfn /opt/myapp/releases/20240110_v1.1 /opt/myapp/current
# 重启服务
systemctl restart myapp
```

#### 策略 3：数据库回滚

```python
# Alembic 数据库迁移回滚
# 回滚最近一次迁移
alembic downgrade -1

# 回滚到特定版本
alembic downgrade abc1234

# 查看迁移历史
alembic history
```

### 回滚的注意事项

1. **数据库兼容性**：新版本的数据库迁移可能不可逆，需要提前设计可逆迁移
2. **回滚时间窗口**：部署后保留旧版本至少 24 小时
3. **回滚演练**：定期练习回滚操作，确保真正需要时能顺利执行
4. **监控告警**：部署后密切监控错误率和响应时间，异常时自动触发回滚

## 四、零停机部署

### 什么是零停机部署

零停机部署（Zero-Downtime Deployment）是在部署新版本时，**用户完全感知不到服务中断**的部署方式。

```
传统部署：
旧版本停止 → 部署新版本 → 启动新版本
     ↑              ↑            ↑
   服务正常     服务中断 ❌    服务恢复

零停机部署：
旧版本运行中 ──────────────────────→ 旧版本停止
                新版本部署并启动 ↑
                流量切换 ────→ 新版本运行
                              ↑
                         用户无感知 ✅
```

### 蓝绿部署（Blue-Green Deployment）

同时运行两套完整的环境（蓝和绿），切换流量指向即可：

```
                    ┌─────────────────┐
                    │   负载均衡器     │
                    │  （流量切换）    │
                    └────┬────────┬───┘
                         │        │
              ┌──────────┘        └──────────┐
              ▼                              ▼
┌──────────────────────┐    ┌──────────────────────┐
│   🔵 蓝色环境        │    │   🟢 绿色环境        │
│   当前生产版本 v1.1   │    │   新版本 v1.2        │
│                      │    │                      │
│   ← 接收全部流量     │    │   部署完成，等待切换  │
└──────────────────────┘    └──────────────────────┘

切换步骤：
1. 在绿色环境部署新版本 v1.2
2. 在绿色环境运行测试，确认正常
3. 负载均衡器将流量从蓝色切换到绿色
4. 观察绿色环境运行状态
5. 如无问题，蓝色环境作为回滚备用
```

**优点**：切换速度快（秒级）、回滚简单（切回去即可）
**缺点**：需要两套环境，成本翻倍

### 滚动更新（Rolling Update）

逐步替换旧版本实例，始终保持部分实例在运行：

```
初始状态：4 个实例都是 v1.1
┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐
│ v1.1 │ │ v1.1 │ │ v1.1 │ │ v1.1 │
└──────┘ └──────┘ └──────┘ └──────┘

第 1 步：替换第 1 个实例
┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐
│ v1.2 │ │ v1.1 │ │ v1.1 │ │ v1.1 │  ← 3 个旧版本仍在服务
└──────┘ └──────┘ └──────┘ └──────┘

第 2 步：替换第 2 个实例
┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐
│ v1.2 │ │ v1.2 │ │ v1.1 │ │ v1.1 │  ← 2 个旧版本仍在服务
└──────┘ └──────┘ └──────┘ └──────┘

第 3 步：替换第 3 个实例
┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐
│ v1.2 │ │ v1.2 │ │ v1.2 │ │ v1.1 │
└──────┘ └──────┘ └──────┘ └──────┘

第 4 步：替换最后一个实例
┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐
│ v1.2 │ │ v1.2 │ │ v1.2 │ │ v1.2 │  ← 全部更新完成
└──────┘ └──────┘ └──────┘ └──────┘
```

**优点**：不需要额外环境，资源利用率高
**缺点**：更新过程中新旧版本共存，需要注意兼容性

### 对比

| 维度 | 蓝绿部署 | 滚动更新 |
|------|----------|----------|
| 资源需求 | 2 倍（两套环境） | 略多于正常 |
| 切换速度 | 秒级 | 分钟级 |
| 回滚速度 | 秒级（切回蓝色） | 需要再次滚动更新 |
| 新旧版本共存 | 不会 | 会（更新过程中） |
| 适用场景 | 重要服务、数据库变更 | 一般服务、无状态应用 |
| Kubernetes 支持 | 需要额外工具 | 原生支持 |

## 五、CI/CD

### 什么是 CI/CD

**CI/CD** 是持续集成（Continuous Integration）和持续部署（Continuous Deployment）的缩写，代表一套自动化代码交付流程。

```
开发者写代码
    ↓
① CI：持续集成
   ├── 代码推送到 Git 仓库
   ├── 自动运行测试
   ├── 自动检查代码质量
   └── 构建可运行产物
    ↓
② CD：持续部署
   ├── 自动部署到测试环境
   ├── 运行集成测试
   ├── 自动/手动部署到生产环境
   └── 监控部署结果
```

### CI/CD 流程图

```
Developer               CI Server              Production
    │                      │                       │
    │  git push            │                       │
    │─────────────────────→│                       │
    │                      │                       │
    │                      │  运行单元测试          │
    │                      │  运行代码检查          │
    │                      │  构建 Docker 镜像      │
    │                      │                       │
    │                      │  部署到测试环境        │
    │                      │──────────────────────→│ 测试环境
    │                      │                       │
    │                      │  运行集成测试          │
    │                      │                       │
    │                      │  部署到生产环境        │
    │                      │──────────────────────→│ 生产环境
    │                      │                       │
    │  部署完成通知        │                       │
    │←─────────────────────│                       │
```

### 常见的 CI/CD 工具

| 工具 | 类型 | 特点 |
|------|------|------|
| GitHub Actions | 云端 CI/CD | 与 GitHub 深度集成、免费额度充足 |
| GitLab CI | 云端/自建 | 功能全面、支持自建 Runner |
| Jenkins | 自建 | 老牌工具、插件生态丰富 |
| CircleCI | 云端 | 速度快、配置简洁 |
| Travis CI | 云端 | 开源项目免费 |

### GitHub Actions 示例

```yaml
# .github/workflows/deploy.yml
name: CI/CD Pipeline

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

jobs:
  # 持续集成
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: 设置 Python
        uses: actions/setup-python@v5
        with:
          python-version: "3.12"

      - name: 安装依赖
        run: pip install -r requirements.txt

      - name: 运行测试
        run: pytest tests/ -v

      - name: 代码检查
        run: ruff check .

  # 持续部署
  deploy:
    needs: test           # 测试通过后才部署
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'  # 只在 main 分支部署
    steps:
      - uses: actions/checkout@v4

      - name: 构建 Docker 镜像
        run: docker build -t my-api:${{ github.sha }} .

      - name: 推送到镜像仓库
        run: |
          docker tag my-api:${{ github.sha }} registry.example.com/my-api:${{ github.sha }}
          docker push registry.example.com/my-api:${{ github.sha }}

      - name: 部署到服务器
        run: |
          ssh user@server "docker pull registry.example.com/my-api:${{ github.sha }} && \
                           docker stop my-api || true && \
                           docker run -d --name my-api -p 8000:8000 \
                           registry.example.com/my-api:${{ github.sha }}"
```

### CI/CD 的核心原则

1. **单一代码源**：所有代码在 Git 仓库管理
2. **自动化测试**：每次提交自动运行测试
3. **快速反馈**：测试失败立即通知开发者
4. **构建一次，部署多次**：构建产物不变，只是在不同环境运行
5. **部署可重复**：同一个版本可以多次部署，结果一致
6. **部署可回滚**：任何版本都可以快速回滚

## 常见坑

### 1. 环境变量泄露

将 `.env` 文件提交到 Git，导致密钥泄露。

**解决方案**：`.env` 加入 `.gitignore`，使用密钥管理工具。

### 2. 回滚时忘记回滚数据库

代码回滚了，但数据库迁移没有回滚，导致代码和数据不匹配。

**解决方案**：设计可逆的数据库迁移，回滚时同步执行数据库回滚。

### 3. 新旧版本 API 不兼容

滚动更新过程中，新旧版本的 API 响应格式不同，前端报错。

**解决方案**：API 变更保持向后兼容，或使用蓝绿部署避免共存。

### 4. CI/CD 流程太慢

测试和构建耗时过长，开发者不愿意频繁提交。

**解决方案**：优化测试速度、使用缓存、并行执行任务。

### 5. 构建不可重复

同一份代码两次构建结果不同，导致"在我这能跑"的问题。

**解决方案**：锁定依赖版本（`requirements.txt` 或 `poetry.lock`），使用 Docker 固化环境。

## 概念之间的关系

```
开发者提交代码
    │
    ▼
CI/CD 流水线启动
    │
    ├── CI：运行测试 → 构建产物
    │                    │
    │                    ▼
    │              环境隔离：选择目标环境
    │                    │
    │                    ▼
    ├── CD：部署到测试环境 → 验证
    │                    │
    │                    ▼
    │              零停机部署：蓝绿/滚动
    │                    │
    │                    ▼
    │              部署到生产环境
    │                    │
    │                    ▼
    └── 出问题？→ 回滚到上一版本
```

## 小结

| 概念 | 核心思想 | 关键工具/方法 |
|------|----------|-------------|
| 环境隔离 | 不同环境互不干扰 | 环境变量、`.env`、虚拟环境 |
| 构建 | 源代码 → 可运行产物 | setuptools、Docker build |
| 回滚 | 快速恢复到稳定版本 | Git revert、版本目录、DB downgrade |
| 零停机部署 | 用户无感知地更新 | 蓝绿部署、滚动更新 |
| CI/CD | 自动化交付流程 | GitHub Actions、GitLab CI |
