核心概念

▼

部署的本质

部署（Deploy）是将代码从「仅本地可用」转变为「全球任何人都能访问」的过程。本质是让代码运行在满足 24h 在线、公网可达、固定地址、足够性能、有监控备份 五条件的机器上。

本地开发的局限性

你的电脑
┌─────────────────────────┐
│  python app.py          │
│  监听 127.0.0.1:8000    │
│  ✅ 你自己能访问         │
│  ❌ 同事无法访问         │
│  ❌ 用户无法访问         │
│  ❌ 关机就停止服务       │
└─────────────────────────┘

本地开发 vs 生产环境

维度本地开发生产环境
访问地址localhost / 127.0.0.1域名如 api.example.com
运行方式python app.py 直接运行Gunicorn/uWSGI 管理进程
数据库本地 SQLite云数据库 RDS，有备份
配置管理硬编码或 .env环境变量 / 配置中心
日志print() 输出终端结构化日志写入日志系统
错误处理显示详细堆栈返回通用错误，记详细日志
调试工具开启 debug 模式严禁开启 debug 模式
并发单进程多进程/多线程，高并发
安全无需考虑HTTPS、防火墙、鉴权

三种服务器类型

┌──────────────────────────────────────────────────────┐
│ 云虚拟机 (IaaS)              托管平台 (PaaS)           │
│ ┌──────────────────┐      ┌──────────────────┐      │
│ │ 你负责：          │      │ 你负责：          │      │
│ │ 应用/运行时/OS    │      │ 应用代码          │      │
│ └──────────────────┘      └──────────────────┘      │
│ 例：ECS、EC2                例：Vercel、Railway      │
├──────────────────────────────────────────────────────┤
│ 容器化 (Docker/K8s)                                   │
│ ┌──────────────────────────────────────────────┐    │
│ │ 容器 = 应用 + 运行时 + 系统依赖 → 一键运行    │    │
│ └──────────────────────────────────────────────┘    │
└──────────────────────────────────────────────────────┘
类型优点缺点代表
云虚拟机完全控制、性价比高需要运维知识阿里云 ECS
托管平台零运维、自动扩容灵活性受限Vercel
容器化环境一致、易迁移学习曲线陡Docker

第 1 步：获取网络信息

理解 127.0.0.1、局域网 IP 与公网 IP 的区别，是理解部署的基础。

import socket

def get_local_ip():
    s = socket.socket(socket.AF_INET, socket.SOCK_DGRAM)
    s.connect(("8.8.8.8", 80))
    ip = s.getsockname()[0]
    s.close(); return ip

# 127.0.0.1 → 仅本机
# 192.168.x.x → 同 Wi-Fi 可访问
# 公网 IP → 全球（需端口映射）

第 2 步：检查端口占用

for port in [80, 443, 3306, 8000]:
    s = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
    s.settimeout(1)
    r = s.connect_ex(("127.0.0.1", port))
    if r == 0: print(f"端口 {port}: ✅ 已占用")
    else:     print(f"端口 {port}: ⬜ 空闲")
    s.close()
地址类型示例谁能访问
回环地址127.0.0.1仅本机
局域网 IP192.168.1.5同一网络下的设备
公网 IP47.96.x.x全球（需端口映射）

第 3 步：启动 HTTP 服务器

from http.server import HTTPServer, BaseHTTPRequestHandler

class DemoHandler(BaseHTTPRequestHandler):
    def do_GET(self):
        if self.path == "/api/info":
            self.send_response(200)
            self.send_header("Content-Type", "application/json")
            self.end_headers()
            self.wfile.write(
                json.dumps({"status":"ok"}).encode("utf-8"))

server = HTTPServer(("127.0.0.1", 8000), DemoHandler)
print("http://127.0.0.1:8000")
server.serve_forever()  # ← 只监听本机，外网不可达

常见误区

▼
❌ 误区✅ 纠正
本地能运行 = 可以部署部署还需处理网络、安全、性能、监控等
部署就是买服务器传文件包含环境配置、进程管理、HTTPS、备份等全流程
不需要区分开发和生产环境配置和策略完全不同，混用会导致安全漏洞
python app.py 可用于生产必须用 Gunicorn/uWSGI 生产级 WSGI 服务器
⚠️ 安全警告：生产环境必须关闭 DEBUG 模式，否则错误堆栈会暴露源码路径和内部逻辑。

核心概念

▼

完整链路总览

用户输入 https://api.example.com/users
    ↓
① DNS 解析（域名 → IP 地址）
    ↓
② CDN 缓存检查
    ↓
③ TCP 三次握手
    ↓
④ TLS 加密握手（HTTPS）
    ↓
⑤ 发送 HTTP 请求到服务器
    ↓
⑥ Nginx 反向代理接收
    ↓
⑦ Python 应用处理业务逻辑
    ↓
⑧ 数据库查询（MySQL/PostgreSQL）
    ↓
⑨ 原路返回响应给用户

DNS 解析过程（环节 ①）

用户输入域名 → 查浏览器缓存 → 查操作系统缓存
    ↓ 缓存未命中
本地 DNS 服务器 → 根域名服务器 → .com 顶级域服务器
    → 权威 DNS 服务器 → 返回 IP → 层层回传
记录类型用途示例
A域名 → IPv4example.com → 93.184.216.34
CNAME域名 → 另一域名www → example.com
MX邮件服务器→ mail.example.com
TXT文本验证SPF、DKIM

TCP 三次握手 + TLS 握手

客户端                         服务器
  │──── SYN ────────────────→ │  第1次
  │←─── SYN+ACK ──────────── │  第2次
  │──── ACK ────────────────→ │  第3次 — 连接建立
  │─── ClientHello ─────────→ │
  │←── ServerHello + 证书 ── │  TLS 1.3: 1-RTT
  │─── 密钥交换 ────────────→ │
  │      加密通道完成          │

CDN 原理

无 CDN：北京 ──── 200ms ────→ 美国服务器
有 CDN：北京 ── 10ms ──→ 北京 CDN 节点（缓存命中）
            未命中 → CDN 回源 → 美国 → 缓存后返回

性能关键指标

指标含义参考值
DNS域名解析耗时< 50ms
TCP建立连接耗时< 100ms
TLS加密握手耗时< 100ms
TTFB收到首字节时间< 200ms
总延迟完整请求-响应< 1s
📖 TTFB 详解（点击展开）

TTFB（Time To First Byte）是从发出请求到收到第一个字节的总时间，是衡量后端性能的最重要指标。

|←──────── TTFB ────────→|
| DNS | TCP | TLS | 请求发送 | 服务器处理 | 首字节返回 |
                                    ↑ 后端性能体现

TTFB 高意味着：服务器处理慢、网络延迟高、或服务器过载。

import time, urllib.request

url = "https://httpbin.org/get"
start_dns = time.perf_counter()
ip = socket.gethostbyname("httpbin.org")
dns = (time.perf_counter()-start_dns)*1000

start = time.perf_counter()
r = urllib.request.urlopen(url, timeout=10)
data = r.read()
total = (time.perf_counter()-start)*1000

print(f"DNS: {dns:.2f}ms, 总耗时: {total:.2f}ms, 状态: {r.status}")

DNS 解析1-50ms首次慢，后续有缓存

常见误区

▼
❌ 误区✅ 纠正
改 DNS 立即生效有 TTL 缓存，部署前先调低 TTL 至 60s
所有资源都走 CDNAPI 动态响应应设置 Cache-Control: no-cache
HTTPS 只是多一层加密需 TLS 握手、证书验证、密钥协商，有额外开销
网站慢一定是后端问题慢可能发生在 DNS、CDN、网络、DB 任一环节
⚠️ 安全提醒：HTTPS 证书过期浏览器会阻止访问。用 Let's Encrypt 免费证书 + certbot 自动续期。

代码示例

▼

第 1 步：静态部署 — 生成 HTML 并托管

import tempfile, os
static_dir = tempfile.mkdtemp()
with open(os.path.join(static_dir,"index.html"),"w") as f:
    f.write("<h1>Hello Static!</h1>")
# 托管：python -m http.server 8000
# 生产：git push → Vercel / GitHub Pages

第 2 步：应用部署 — API 路由处理

class APIHandler(BaseHTTPRequestHandler):
    def do_GET(self):
        routes = {
            "/api/users": self._handle_users,
            "/api/status": self._handle_status,
        }
        h = routes.get(self.path)
        if h: h()
        else: self._send_json(404,{"error":"Not Found"})

    def _handle_users(self):
        self._send_json(200,{
            "data":[{"id":1,"name":"张三"}],
            "source":"模拟数据（实际查 MySQL）"})

第 3 步：容器化 — Dockerfile 模板

FROM python:3.12-slim                    # slim 版 ~120MB
WORKDIR /app
COPY requirements.txt .                # 先复制依赖→利用缓存
RUN pip install --no-cache-dir -r requirements.txt
COPY . .                                # 再复制代码
RUN useradd -m appuser && chown -R appuser:appuser /app
USER appuser                            # 非 root
EXPOSE 8000
CMD ["gunicorn","-w","4","-b","0.0.0.0:8000","app:app"]
部署方式典型启动命令
静态不需要
应用gunicorn -w 4 -b 0.0.0.0:8000 app:app
容器docker run -d -p 8000:8000 myapp:v1

Developer               CI Server              Production
  │ git push              │                       │
  │──────────────────────→│ 测试+检查+构建         │
  │                       │──────────────────────→│ 测试环境
  │                       │──────────────────────→│ 生产环境
  │←── 通知 ────────────│                       │
工具类型特点
GitHub Actions云端与 GitHub 深度集成，免费额度充足
GitLab CI云端/自建功能全面，支持自建 Runner
Jenkins自建老牌工具，插件丰富
📖 CI/CD 六条核心原则（点击展开）
  1. 单一代码源：所有代码在 Git 仓库管理
  2. 自动化测试：每次提交自动运行
  3. 快速反馈：失败立即通知
  4. 构建一次部署多次：产物不变只换环境
  5. 部署可重复：同版本多次部署结果一致
  6. 部署可回滚：任何版本可快速回滚

import os

第 3 步：GitHub Actions CI/CD

# .github/workflows/deploy.yml
name: CI/CD
on:
  push:
    branches: [main]

jobs:
  test:                           # CI 阶段
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with: {python-version: "3.12"}
      - run: pip install -r requirements.txt
      - run: pytest tests/ -v

  deploy:                        # CD 阶段
    needs: test                 # 测试通过才部署
    if: github.ref == 'refs/heads/main'
    steps:
      - run: docker build -t my-api:${{ github.sha }} .
      - run: |
          ssh user@server "docker pull ... && docker run ..."
CI/CD 原则说明
单一代码源所有代码在 Git 管理
自动化测试每次提交自动运行
快速反馈失败立即通知
构建一次部署多次同一产物不同环境运行
部署可回滚任何版本可快速恢复

常见误区

▼
❌ 误区✅ 纠正
将 .env 提交到 Git.env 含密钥，必须加入 .gitignore
代码回滚但忘记回滚 DB同步执行 alembic downgrade
滚动更新时新旧 API 不兼容API 变更向后兼容，或改用蓝绿部署
CI/CD 太慢就减少测试优化速度（缓存/并行），非减少覆盖率
构建两次结果不同锁定依赖版本，用 Docker 固化环境
⚠️ 安全警告：环境变量泄露是生产事故第一原因。生产密钥通过 Vault 或云厂商密钥管理注入，绝不写在配置文件里。
