第一节 · Git基础操作

Git 是后端协作的底座，先把 add/commit/push/pull/diff 这套日常动作打通。
🎯学习目标▼
  • 理解 Git 分布式版本控制系统的核心理念
  • 掌握工作区、暂存区、本地仓库、远程仓库四级模型
  • 熟练使用 init、add、commit、log、status、diff 命令

Git 是后端开发的必备工具。代码管理、团队协作、版本回溯、CI/CD 流水线都建立在 Git 之上。
📖核心概念▼

什么是 Git

Git 是 2005 年由 Linus Torvalds 创建的分布式版本控制系统（DVCS）。每个开发者本地都有一份完整的仓库副本，包括全部历史记录。这意味着：

  • 无网络也能查看历史、提交代码、创建分支
  • 不存在单点故障，任何副本都能恢复完整项目
  • 分支和合并操作极其轻量，几乎瞬间完成

四级工作模型

工作区 (Working Directory)
   │
   │  git add
   ▼
暂存区 (Staging Area / Index)
   │
   │  git commit
   ▼
本地仓库 (Local Repository)
   │
   │  git push
   ▼
远程仓库 (Remote Repository)

文件状态

状态含义
未跟踪 (Untracked)新创建但还没 git add 的文件
已修改 (Modified)已跟踪文件被修改，还没 add
已暂存 (Staged)已 add，等待下一次 commit
已提交 (Committed)已保存到本地仓库

HEAD 与分支

HEAD 是一个特殊指针，指向当前所在的 commit。通常它指向某个分支名（如 main），而分支名又指向最新的 commit：

HEAD → main → commit_c → commit_b → commit_a

.gitignore 核心规则

  • *.log — 忽略所有 .log 文件
  • build/ — 忽略 build 目录
  • !important.log — 不忽略 important.log（取反）
  • /TODO — 只忽略根目录的 TODO
重要：.gitignore 只能忽略未跟踪的文件。已 add 的文件需先 git rm --cached file 才能被忽略。

fetch vs pull

命令做了什么安全性
git fetch只下载远程新提交，不修改你的代码安全
git pullfetch + merge，直接修改你的代码可能产生冲突
💻代码示例▼

初始化与克隆

git init                                    # 在当前目录初始化仓库
git clone https://github.com/user/repo.git  # 克隆远程仓库
git clone https://github.com/user/repo.git my_folder  # 克隆到指定目录

首次配置（必须！）

git config --global user.name "你的名字"
git config --global user.email "your@email.com"

查看状态与历史

git status                    # 查看工作区和暂存区状态（最常用！）
git status -s                 # 简洁模式
git log                       # 查看提交历史
git log --oneline --graph     # 单行 + 图形化分支图
git log -3                    # 最近 3 次提交
git show <commit-hash>       # 查看某次提交的改动

暂存与提交

git add file.py               # 暂存单个文件
git add .                     # 暂存所有改动
git add src/                  # 暂存某个目录
git commit -m "feat: 添加用户注册"  # 提交
git commit -am "fix: 修复Bug"      # 跳过暂存区（已跟踪文件）
git commit --amend -m "新信息"     # 修改上一次的提交信息

查看差异

git diff                      # 工作区 vs 暂存区
git diff --staged             # 暂存区 vs 最新提交
git diff abc123 def456        # 两个 commit 之间的差异
git diff main feature/login   # 两个分支的差异

典型 .gitignore

# 忽略日志和缓存
*.log
__pycache__/
*.pyc

# 忽略敏感信息
.env

# 忽略 IDE 配置
.vscode/
.idea/

# 忽略 build 产物
build/
dist/

# 但不忽略 important.log
!important.log

远程仓库

git remote -v                              # 查看远程仓库
git remote add origin https://github.com/user/repo.git  # 关联远程
git push -u origin main                    # 首次推送（-u 建立追踪）
git push                                   # 之后直接推送
git pull                                   # 拉取并合并（= fetch + merge）
git fetch origin                           # 只拉取，不合并（更安全）

日常开发节奏

# 早上开始工作，先拉取最新代码
git pull

# 改完一个功能
git status
git diff

# 确认没问题，提交
git add .
git commit -m "feat: 添加用户登录接口"

# 推送到远程
git push
⚠️常见误区▼
❌ 首次提交时没配 user.name/email
✅ 用 git config --global 配置一次
❌ .gitignore 对已跟踪文件无效
✅ 先 git rm --cached file 再忽略
❌ git add . 误提交大文件或敏感文件
✅ 还没 push 就 reset，已 push 用 revert
❌ pull 时遇到冲突不知如何处理
✅ 先 stash 暂存 → pull → stash pop 再解冲突
❌ commit message 写错了直接 push
✅ amend 改信息，push 用 --force-with-lease
❌ 习惯性直接 git pull
✅ 不确定时先 git fetch 看看，再决定 merge
安全提示：git push --force 会覆盖远程历史，极端危险。推荐用 --force-with-lease，会先检查远程是否有新提交。

第二节 · 分支管理与 Git Flow

多人并行开发靠分支隔离，Git Flow 给出主流团队约定俗成的协作模板。
🎯学习目标▼
  • 理解分支的本质（指向 commit 的可移动指针）
  • 掌握 branch、checkout、merge 的基础操作
  • 理解 Git Flow 的 5 种分支角色与工作流
  • 掌握冲突的产生原因和解决步骤
  • 理解 rebase 与 merge 的差异及黄金法则
分支是协作的核心。多人并行开发、代码审查、版本发布——所有规范的工作流都建立在分支管理之上。
📖核心概念▼

什么是分支

分支就是一个指向某个 commit 的可移动指针。创建分支只是创建新指针，极其轻量（不像 SVN 需要复制目录）。

        commit_a ← commit_b ← commit_c  (main)
                            ↖
                              commit_d ← commit_e  (feature/login)

为什么需要分支

  • 隔离开发：新功能不影响主分支稳定性
  • 并行工作：多人可同时在不同分支上工作
  • 代码审查：通过 PR 在合并前审查代码
  • 实验性开发：失败的实验直接删除

fast-forward vs no-ff 合并

# fast-forward（默认）：直接把 main 指针移过去
main → A → B → C → D → E

# --no-ff：创建合并 commit，保留分支历史
main → A → B ──────── F
              ↗      ↗
      feature → C → D

Git Flow 五种分支

分支从哪创建合并到哪用途
main——生产代码，只接受 release/hotfix
develop——开发主分支，集成所有功能
feature/*developdevelop开发新功能
release/*developmain + develop准备发布，只修 bug
hotfix/*mainmain + develop紧急修复生产 Bug

冲突的产生

当两个分支修改了同一个文件的同一部分，Git 无法自动决定保留哪个版本，就产生冲突。冲突标记：

<<<<<<< HEAD          # 当前分支的内容
DB_HOST = 'production'
=======                  # 分隔线
DB_HOST = 'localhost'
>>>>>>> feature       # 要合并分支的内容

Merge vs Rebase

维度MergeRebase
结果创建合并 commit，保留完整历史把提交"重新接"到最新之后
优点历史真实、安全历史线性、干净
缺点历史可能杂乱改写历史，公共分支危险
适用合并到主分支个人分支同步主分支更新
黄金法则：绝对不要在公共分支（别人也在用的分支）上 rebase！

GitHub Flow（轻量替代）

只有两种分支：

  • main：始终保持可部署状态
  • feature/*：从 main 创建，PR 合并回 main

适合持续部署的 Web 应用。

💻代码示例▼

分支创建与切换

git branch                          # 查看本地分支
git branch -a                       # 查看所有分支（含远程）
git branch feature/login            # 创建新分支
git checkout feature/login          # 切换分支（旧语法）
git switch feature/login            # 切换分支（Git 2.23+）
git checkout -b feature/login       # 创建并切换（最常用）
git switch -c feature/login         # 同上（新语法）
git branch -d feature/login         # 删除已合并分支
git branch -D feature/login         # 强制删除未合并分支

合并分支

git checkout main
git merge feature/login             # 默认（可能 fast-forward）
git merge --no-ff feature/login     # 强制创建合并 commit
git merge --abort                   # 放弃合并（搞砸时救命）

Rebase

git checkout feature
git rebase main                     # 在自己的分支上 rebase
git push --force-with-lease         # 推送 rebase 后的分支

冲突解决

git mergetool                       # 打开图形化工具（VS Code 等）
# 手动解决步骤：
# 1. 打开冲突文件，删除冲突标记，保留正确内容
# 2. git add <file>               # 标记冲突已解决
# 3. git commit                     # 完成合并

git merge --abort                   # 放弃整个合并

Git Flow 完整工作流

# 1. 开始新功能
git checkout develop
git pull origin develop
git checkout -b feature/user-auth

# 2. 开发并提交
git add .
git commit -m "feat: 添加登录页面"
git commit -m "feat: 实现 JWT 验证"

# 3. 合并回 develop
git checkout develop
git merge --no-ff feature/user-auth
git branch -d feature/user-auth
git push origin develop

# 4. 准备发布 v1.2.0
git checkout -b release/1.2.0 develop
git commit -m "fix: 修复发布前发现的Bug"

# 5. 发布到 main
git checkout main
git merge --no-ff release/1.2.0
git tag -a v1.2.0 -m "Release v1.2.0"

# 6. 同步 release 改动到 develop
git checkout develop
git merge --no-ff release/1.2.0
git branch -d release/1.2.0

# 7. 紧急修复
git checkout -b hotfix/fix-crash main
git commit -m "fix: 修复崩溃"
git checkout main
git merge --no-ff hotfix/fix-crash
git tag -a v1.2.1 -m "Hotfix v1.2.1"
git checkout develop
git merge --no-ff hotfix/fix-crash
git branch -d hotfix/fix-crash

Tag 标签管理

git tag v1.0.0                                 # 轻量标签
git tag -a v1.0.0 -m "Release v1.0.0"          # 附注标签（推荐）
git tag -l                                     # 查看所有标签
git show v1.0.0                                # 查看标签信息

git push origin v1.0.0                         # 推送单个标签
git push origin --tags                         # 推送所有标签
git tag -d v1.0.0                              # 删除本地标签
git push origin --delete v1.0.0                # 删除远程标签

# 基于旧标签创建修复分支
git checkout -b hotfix/v1.0.1 v1.0.0

误操作恢复

git stash                       # 暂存当前改动
git checkout feature            # 切到正确分支
git stash pop                   # 恢复改动

git log main..feature/abandoned # 查看分支独有的提交
git reset --hard ORIG_HEAD      # 回到合并之前的状态

git push --force-with-lease     # 推送 rebase 后的分支（比 --force 安全）
⚠️常见误区▼
❌ 在错误的分支上开发了
✅ 用 stash 暂存 → 切到正确分支 → pop 恢复
❌ 删除分支提示 "not fully merged"
✅ 用 -D 强制删除（确认改动不需要）
❌ rebase 之后 push 不上去
✅ 用 --force-with-lease（比 --force 安全）
❌ 合并冲突解决错了且已 commit
✅ 还没 push 就 reset --hard ORIG_HEAD
❌ 分支命名混乱（中文、随机字符）
✅ feature/user-login、fix/login-crash 清晰规范
❌ 在公共分支上 rebase
✅ 永远只在自己的分支上 rebase 主分支
安全提示：git push --force 会覆盖远程历史，可能丢同事的提交。务必用 --force-with-lease，它会先检查远程是否有新提交。
