第一节

Docker 基础概念

Docker 用容器解决"在我电脑上能跑"，是现代部署绕不开的标准化封装。
  • 说清 Docker 解决的环境一致性问题
  • 区分镜像、容器、仓库、宿主机
  • 理解镜像分层和容器可写层
  • 比较 Docker 与虚拟机的差异
  • 理解 Client、Daemon、Registry 的工作流
与后端开发的关联：Docker 让 FastAPI、Redis、MySQL 等运行环境可以被团队共享、CI 复用、服务器复现。

Docker 的核心不是“装了一个系统”，而是把应用和依赖打包成标准化运行单元。镜像是只读模板，容器是镜像启动后的运行实例。

镜像到容器的关系：
┌─────────────────────────┐
│ Image: python-api:v1     │  只读模板
│  - Python runtime        │
│  - dependencies          │
│  - app code              │
└────────────┬────────────┘
             │ docker run
             ▼
┌─────────────────────────┐
│ Container: api-1         │  运行实例
│  + writable layer        │  临时可写层
└─────────────────────────┘
概念定位后端类比
Image只读模板，包含运行所需文件应用发布包
Container镜像的运行实例，有可写层正在运行的 API 服务
Dockerfile镜像构建说明书构建脚本
Registry镜像存储和分发服务镜像版代码仓库
展开：Docker 与虚拟机的核心区别

虚拟机会运行完整操作系统，所以隔离更强但启动慢、占用大。Docker 容器共享宿主机内核，本质上更接近隔离的进程，因此启动速度快、资源开销低，适合微服务、CI/CD 和开发环境统一。

基础脚本先检查 Docker Client 与 Daemon，再运行后续演示，避免环境不存在时直接报错。

def check_docker_installed() -> bool:
    result = run_cmd("docker --version", check=False)
    if result is None or result.returncode != 0:
        print("Docker 未安装或不在 PATH 中")
        return False

    result = run_cmd("docker info --format '{{.ServerVersion}}'", check=False)
    if result is None or result.returncode != 0:
        print("Docker Daemon 未运行")
        return False

    return True
输入`docker --version` 与 `docker info`
输出Docker 可用时继续；未安装或守护进程未运行时给出提示
错误理解纠正
Docker 就是轻量级虚拟机容器共享宿主机内核，虚拟机有独立内核，二者隔离机制不同。
一个容器里跑完整应用栈推荐一个容器一个主要进程，数据库、缓存、Web 服务分开编排。
容器里的数据天然安全容器删除会丢失可写层，数据库必须使用 Volume 或 Bind Mount。
安全提醒：不要把密钥、`.env`、证书文件打进镜像，容器镜像可能被推送到共享仓库。

1. 镜像和容器的关系最准确的是？

正确答案：B。镜像像类或设计图，容器像对象或运行实例。

2. 容器删除后仍需保留数据库数据，应该使用什么？

正确答案：C。可写层随容器删除而消失，持久化数据要挂载到卷。
第二节

Docker 常用命令

镜像、容器、数据卷、网络——日常 20 条命令就够覆盖 90% 的容器操作。
  • 掌握镜像拉取、构建、查看、删除
  • 熟悉容器运行、停止、日志、进入容器
  • 理解 `-d`、`-p`、`-v`、`-e` 参数
  • 知道数据卷和网络的基本命令
  • 识别高风险清理命令
与后端开发的关联：排查线上容器问题时，`ps`、`logs`、`exec`、`inspect` 是最常用的一组命令。

Docker 命令可以按对象记忆：镜像负责构建和分发，容器负责运行和调试，Volume 负责持久化，Network 负责容器通信。

常用命令地图：
Image
  ├─ pull      下载镜像
  ├─ images    查看本地镜像
  ├─ build     构建镜像
  └─ rmi       删除镜像

Container
  ├─ run       创建并运行
  ├─ ps        查看容器
  ├─ logs      查看日志
  ├─ exec      进入/执行命令
  └─ stop/rm   停止/删除

Storage & Network
  ├─ volume    数据持久化
  └─ network   服务通信
命令典型用途注意点
`docker run`从镜像创建并启动容器`run` 每次都会创建新容器
`docker exec`在运行中的容器内执行命令常用于进入 shell 调试
`docker logs -f`实时追踪容器日志适合看 API 启动报错
`docker volume`管理持久化数据数据库容器不能只靠可写层

`docker run` 的参数组合最值得熟练：后台运行、命名、端口映射、卷挂载和环境变量通常一起出现。

# 运行一个 FastAPI 容器
docker run -d \
  --name my-api \
  -p 8000:8000 \
  -v /host/project:/app \
  -e APP_ENV=development \
  --restart unless-stopped \
  python:3.12-slim \
  uvicorn app:app --host 0.0.0.0 --port 8000

# 调试和观察
docker ps
docker logs -f my-api
docker exec -it my-api /bin/sh
输入`docker run` 参数组合
输出宿主机访问 `localhost:8000`，容器内运行 FastAPI 服务
误区纠正
`docker stop` 和 `docker kill` 随便用`stop` 给应用优雅关闭时间，生产环境优先使用 `stop`。
`docker rm` 能删除运行中容器普通 `rm` 只能删除已停止容器，强制删除有数据风险。
端口映射失败就是镜像坏了先检查宿主机端口是否被占用。
清理命令可以随手执行`system prune -a --volumes` 可能删掉镜像和卷数据。
安全提醒：涉及 `prune`、`rm -f`、`down -v` 的命令都要先确认影响范围，尤其是数据卷。

1. 想实时查看容器日志，应该使用？

正确答案：A。`logs -f` 类似 `tail -f`，适合追踪服务输出。

2. 在运行中的容器内打开 shell，常用命令是？

正确答案：D。`exec` 是在已有容器中执行命令，`-it` 提供交互终端。
第三节

Dockerfile 编写

Dockerfile 是镜像的"配方文件"，写好它才能让构建可复现、镜像更精简。
  • 理解 Dockerfile 是镜像构建脚本
  • 掌握 FROM、WORKDIR、COPY、RUN、CMD
  • 会写 Python 后端生产级 Dockerfile
  • 理解多阶段构建和层缓存
  • 知道 `.dockerignore` 与非 root 用户的重要性
与后端开发的关联：一个好的 Dockerfile 会直接影响构建速度、镜像体积、容器安全和服务关闭行为。

Dockerfile 的每条关键指令都在定义镜像的某一层或运行时元数据。写 Dockerfile 的核心能力，是让依赖层稳定、源码层靠后、安全配置完整。

推荐构建顺序：
FROM python:3.12-slim
  │
  ├─ WORKDIR /app
  ├─ COPY requirements.txt .
  ├─ RUN pip install ...
  │        ↑ 依赖层：尽量稳定，便于缓存
  ├─ COPY . .
  │        ↑ 源码层：变化频繁，放在后面
  ├─ USER appuser
  ├─ EXPOSE 8000
  └─ CMD ["uvicorn", ...]
指令作用后端最佳实践
FROM指定基础镜像Python 项目优先 `python:3.12-slim`
RUN构建时执行命令合并命令并清理缓存
COPY复制构建上下文文件先复制依赖文件，再复制源码
CMD容器默认启动命令用 exec 格式，监听 `0.0.0.0`
USER切换运行用户生产环境避免 root 运行
展开：为什么多阶段构建能减小镜像

构建阶段可能需要编译器、构建工具和缓存文件，但运行阶段只需要应用与运行依赖。多阶段构建使用多个 `FROM`，最终镜像只保留运行阶段，再从构建阶段复制必要产物，所以镜像更小，也减少了攻击面。

demo 中的 Dockerfile 使用多阶段构建、层缓存、健康检查和非 root 用户，是 Python 后端容器化的核心模板。

# 阶段 1：依赖安装
FROM python:3.12-slim AS builder
WORKDIR /build
COPY requirements.txt .
RUN pip install --user --no-cache-dir -r requirements.txt

# 阶段 2：运行环境
FROM python:3.12-slim AS runtime
WORKDIR /app
COPY --from=builder /root/.local /root/.local
ENV PATH=/root/.local/bin:$PATH
COPY . .
RUN useradd -m -r appuser && chown -R appuser:appuser /app
USER appuser
EXPOSE 8000
HEALTHCHECK --interval=30s --timeout=10s --retries=3 \
    CMD curl -f http://localhost:8000/health || exit 1
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]
输入`docker build -t myapp:v1.0 .`
输出可用 `docker run -p 8000:8000 myapp:v1.0` 启动的 FastAPI 镜像
误区纠正
先 `COPY . .` 再装依赖源码一变就会重新安装依赖，构建慢。先复制 `requirements.txt`。
CMD 用 shell 格式shell 格式可能导致 SIGTERM 不传递，`docker stop` 无法优雅关闭。
监听 `127.0.0.1`容器外无法访问，应监听 `0.0.0.0`。
生产环境 root 运行创建非 root 用户，降低权限风险。
安全提醒：`.dockerignore` 应排除 `.env`、密钥、证书、虚拟环境和缓存目录。

1. 为什么要先 `COPY requirements.txt` 再 `COPY . .`？

正确答案：B。依赖文件稳定，放前面可最大化利用 Docker 层缓存。

2. Dockerfile 中推荐的 CMD 写法是？

正确答案：A。exec 格式有更好的信号处理，`0.0.0.0` 才能让容器外访问。
第四节

docker-compose 编排

一个 YAML 把 Web、DB、Redis 一起拉起来，本地开发与单机部署的最小编排工具。
  • 理解 compose 用一个 YAML 管理多容器
  • 掌握 services、volumes、networks
  • 会编排 FastAPI + MySQL + Redis
  • 理解服务名通信和健康检查依赖
  • 区分 `up`、`down`、`logs`、`exec`、`build`
与后端开发的关联：实际项目很少只有一个 Web 容器，compose 是本地开发和小型部署最直接的编排入口。

docker-compose 把多个 `docker run` 命令收敛到一个 `docker-compose.yml` 中。服务之间加入同一个网络后，可以直接用 service name 当主机名。

Compose 三服务关系：
┌───────────── app-network ─────────────┐
│                                       │
│  web                                  │
│   ├─ DATABASE_URL -> db:3306          │
│   └─ REDIS_URL    -> redis:6379       │
│                                       │
│  db      -> mysql-data volume         │
│  redis   -> redis-data volume         │
└───────────────────────────────────────┘
配置块作用记忆方式
services定义每个容器服务Web、DB、Redis 都是 service
volumes声明命名卷数据库数据落在卷里
networks声明容器网络服务名 DNS 解析靠网络
depends_on控制启动依赖配合 healthcheck 等待就绪
展开：depends_on 的真正含义

`depends_on` 默认只保证启动顺序，不保证数据库已经初始化完成。要等待服务就绪，需要被依赖服务配置 `healthcheck`，再在 `depends_on` 里使用 `condition: service_healthy`。

demo 的 compose 文件展示了 FastAPI、MySQL、Redis 三服务编排，重点看服务名、数据卷和健康检查依赖。

services:
  db:
    image: mysql:8.0
    volumes:
      - mysql-data:/var/lib/mysql
    healthcheck:
      test: ["CMD", "mysqladmin", "ping", "-h", "localhost"]

  redis:
    image: redis:7.2-alpine
    volumes:
      - redis-data:/data

  web:
    build:
      context: .
      dockerfile: Dockerfile
    ports:
      - "8000:8000"
    environment:
      - DATABASE_URL=mysql+aiomysql://appuser:apppass123@db:3306/demo_db
      - REDIS_URL=redis://redis:6379/0
    depends_on:
      db:
        condition: service_healthy
      redis:
        condition: service_healthy

volumes:
  mysql-data:
  redis-data:
输入`docker compose up -d --build`
输出启动 Web、MySQL、Redis，并通过服务名建立容器间通信
误区纠正
容器之间用 `localhost` 通信`localhost` 指容器自己，同一网络下应使用服务名，如 `db`、`redis`。
`depends_on` 等于服务可用默认只等容器启动，服务就绪要配 `healthcheck`。
`down -v` 是普通停止`-v` 会删除命名卷，数据库数据可能丢失。
改 Dockerfile 后直接 `up -d`需要 `up -d --build` 或先 `build`。
安全提醒：示例密码只适合学习环境，真实项目应通过环境变量或密钥管理注入，不提交 `.env`。

1. web 容器连接 MySQL 时，主机名应写成？

正确答案：C。同一 compose 网络下服务名会被解析为容器地址。

2. 修改 Dockerfile 后希望重新构建并启动，应使用？

正确答案：B。`--build` 会重新构建镜像再启动服务。
