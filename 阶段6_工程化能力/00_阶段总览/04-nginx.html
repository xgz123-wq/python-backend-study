第一节 · Nginx基础与反向代理

Nginx 是服务最前面的"门卫"，反向代理把外部请求安全地转给后端进程。
🎯学习目标▼
  • 理解 Nginx 的定位：高性能 Web 服务器 + 反向代理
  • 掌握正向代理与反向代理的核心区别
  • 熟悉 nginx.conf 的层级结构（main → events → http → server → location）
  • 掌握 location 5 种匹配规则及优先级
  • 能独立配置 FastAPI 反向代理 + 真实 IP 透传
Nginx 是后端部署的核心组件。FastAPI/Django 上线前几乎都需经 Nginx 转发，配置错误会导致 502、IP 丢失、HTTPS 失效等线上故障。
📖核心概念▼

Nginx 是什么

Nginx（engine-x）是 2004 年由 Igor Sysoev 开发的高性能 HTTP 服务器和反向代理服务器。在全球 Web 服务器市场占据约 36% 份额（2024），是部署 Python 后端应用最常用的网关组件。

核心能力

能力实际用途
Web 服务器直接提供 HTML/CSS/JS/图片
反向代理FastAPI/Django 部署
负载均衡高可用架构
HTTPS 终端SSL/TLS 加密解密
缓存减少后端重复计算
限流防止恶意请求

为什么需要 Nginx（不直接让 FastAPI 监听 80 端口）

方案 A（不推荐）：客户端 → FastAPI(uvicorn) 直接对外
方案 B（推荐）：  客户端 → Nginx → FastAPI(uvicorn)

Nginx 优势：C 编写事件驱动，处理静态比 Python 快 10-100 倍；隐藏后端技术栈；后端崩溃时返回友好错误页；统一处理 HTTPS、压缩、缓存。

正向代理 vs 反向代理

# 正向代理：代理代表「客户端」
客户端 → [正向代理] → 目标服务器
         ↑
    客户端知道目标服务器是谁

# 反向代理：代理代表「服务端」
客户端 → [反向代理] → 后端服务器
         ↑
    客户端不知道后端服务器是谁
维度正向代理反向代理
代理谁客户端服务端
客户端知道目标吗知道不知道
部署位置客户端一侧服务端一侧
典型工具VPN、SquidNginx、HAProxy
主要目的访问控制/翻墙负载均衡/安全/缓存

反向代理的三大作用

# 1. 安全隔离
互联网 → Nginx (80/443) → FastAPI (127.0.0.1:8000)

# 2. 负载均衡
Nginx ──┬→ FastAPI 实例 1 (8001)
        ├→ FastAPI 实例 2 (8002)
        └→ FastAPI 实例 3 (8003)

# 3. 统一入口
api.example.com/api/*    → FastAPI
api.example.com/static/* → Nginx 直接返回
api.example.com/admin/*  → Django

nginx.conf 层级结构

main (全局块)
├── events (事件块)
└── http (HTTP 块)
    ├── upstream (上游服务器组)
    └── server (虚拟主机)
        └── location (路径匹配规则)

location 匹配优先级（从高到低）

语法含义示例
= /path精确匹配= /health
^~ /path前缀匹配，不再检查正则^~ /static/
~ regex正则匹配（区分大小写）~ \.(jpg|png)$
~* regex正则匹配（不区分大小写）~* \.(gif|ico)$
/path普通前缀匹配/api/

关键 header 作用

Header作用
Host传递原始请求的域名
X-Real-IP客户端真实 IP
X-Forwarded-For完整代理链路 IP 列表
X-Forwarded-Proto原始请求协议（http/https）

nginx.conf 完整结构

# main 全局块
user nginx;
worker_processes auto;
error_log /var/log/nginx/error.log;
pid /run/nginx.pid;

# events 事件块
events {
    worker_connections 1024;
    use epoll;
}

# http 块
http {
    include       mime.types;
    default_type  application/octet-stream;
    sendfile on;
    keepalive_timeout 65;

    # upstream 上游服务器组
    upstream backend {
        server 127.0.0.1:8001;
        server 127.0.0.1:8002;
    }

    # server 虚拟主机
    server {
        listen 80;
        server_name example.com;

        location / {
            proxy_pass http://backend;
        }

        location /static/ {
            alias /var/www/static/;
        }
    }
}

反向代理 FastAPI 实战

server {
    listen 80;
    server_name api.example.com;

    # 反向代理到 FastAPI
    location / {
        proxy_pass http://127.0.0.1:8000;

        # 传递真实客户端信息
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # 超时设置
        proxy_connect_timeout 10s;
        proxy_read_timeout 30s;
        proxy_send_timeout 30s;
    }

    # 健康检查（直接返回，不经过后端）
    location = /nginx-health {
        return 200 "OK";
        add_header Content-Type text/plain;
    }
}

FastAPI 中获取真实 IP

from fastapi import FastAPI, Request

app = FastAPI()

@app.get("/whoami")
async def whoami(request: Request):
    # 如果 Nginx 设置了 X-Real-IP
    real_ip = request.headers.get("X-Real-IP", "unknown")
    # 如果没有 Nginx，拿到的是直连 IP
    direct_ip = request.client.host
    return {
        "real_ip": real_ip,
        "direct_ip": direct_ip,
    }
⚠️常见误区▼
❌ 修改配置后忘记 reload
✅ 先 nginx -t 测试，再 nginx -s reload
❌ proxy_pass 末尾斜杠写错导致路由错乱
✅ /api/ + /backend/ → /api/users 变 /users
❌ 502 Bad Gateway 直接放弃
✅ 检查后端是否运行、端口、防火墙
❌ 文件上传 413
✅ 调大 client_max_body_size
❌ server_name 写 IP 地址
✅ 用域名，IP 变化时只需改 DNS
安全提示：proxy_pass 末尾斜杠差异是新手最常踩的坑。有斜杠时 location 匹配部分会被替换，无斜杠则原样传递。

第二节 · 静态文件与HTTPS配置

让 Nginx 直接发静态资源、再用 Let's Encrypt 给站点挂上 HTTPS。
🎯学习目标▼
  • 理解 root 与 alias 的差异及正确使用
  • 掌握 gzip 压缩配置及适用场景
  • 理解 HTTP 缓存机制（Cache-Control、ETag）
  • 理解 HTTPS/TLS 握手原理
  • 能独立配置 Let's Encrypt 免费证书 + HTTP 自动跳转 HTTPS
生产环境 HTTPS 已是标配。Let's Encrypt 让零成本启用 HTTPS 成为可能，Certbot 可自动化整个流程。
📖核心概念▼

静态文件

不需要后端计算就能直接返回的文件：HTML/CSS/JS、图片、字体、媒体。Nginx 用 sendfile() 系统调用，数据直接从磁盘到网卡，不经过用户态内存。

root vs alias

指令路径计算示例
root /var/wwwroot + 完整 URI/static/a.css → /var/www/static/a.css
alias /opt/files/alias 替换匹配前缀/static/a.css → /opt/files/a.css
alias 结尾必须加 /，root 结尾不加 /。

gzip 压缩

级别压缩率CPU适用
1低极低高并发实时压缩
6中中推荐默认值
9高高低流量极致压缩

不要对图片/视频 gzip：JPEG/PNG/MP4 已是压缩格式，再 gzip 几乎无效且浪费 CPU。

缓存策略

文件类型推荐原因
JS/CSS（带 hash）max-age=31536000（1年）文件名含 hash，内容变化时文件名也变
图片/字体max-age=2592000（30天）变化不频繁
HTMLno-cache始终验证最新
API 响应no-store不缓存动态数据

ETag 工作原理

1. 首次请求：服务器返回文件 + ETag: "abc123"
2. 再次请求：浏览器发送 If-None-Match: "abc123"
3. 文件未变：服务器返回 304 Not Modified（不传文件体）
4. 文件已变：服务器返回 200 OK + 新文件 + 新 ETag

HTTPS = HTTP + TLS

HTTP 三大问题：窃听、篡改、冒充。HTTPS 全部解决。

TLS 握手过程（简化）

客户端                                    服务器
  │─── 1. ClientHello ──────────────────→  │  (加密算法列表、随机数)
  │←── 2. ServerHello ──────────────────── │  (选定算法、证书、随机数)
  │─── 3. 验证证书 + 加密的预主密钥 ─────→ │  (用公钥加密)
  │    5. 双方用三个随机数生成会话密钥        │
  │←── 6. 对称加密通信开始 ──────────────→  │  (后续用会话密钥加密)
概念说明
非对称加密公钥加密私钥解密，握手阶段传递密钥（慢）
对称加密同一密钥加密解密，传输数据（快）
证书CA 签发，证明公钥确实属于该域名
证书链根 CA → 中间 CA → 网站证书，逐级验证

Let's Encrypt vs 传统付费证书

对比传统付费Let's Encrypt
费用几百-几千元/年免费
验证邮件/人工审核域名验证（自动化）
有效期1-2 年90 天（自动续期）
适用企业级 EV 证书个人项目、中小网站

301 vs 302 重定向

  • 301 永久：浏览器会记住，下次直接访问 HTTPS
  • 302 临时：浏览器每次都先访问 HTTP 再跳转

生产环境应使用 301。

静态文件：root vs alias

server {
    listen 80;
    server_name example.com;

    # 方式 1：root（路径拼接）
    # /static/css/style.css → /var/www/html/static/css/style.css
    location /static/ {
        root /var/www/html;
    }

    # 方式 2：alias（路径替换）
    # /static/css/style.css → /opt/app/static/css/style.css
    location /static/ {
        alias /opt/app/static/;
    }
}

gzip 压缩配置

http {
    gzip on;
    gzip_min_length 1k;               # 小于 1KB 不压缩
    gzip_comp_level 6;                # 推荐 6 级
    gzip_types
        text/plain
        text/css
        text/javascript
        application/javascript
        application/json
        application/xml
        image/svg+xml;
    gzip_vary on;
    gzip_proxied any;
}

缓存配置

location /static/ {
    alias /var/www/static/;

    # 30 天过期（Nginx 自动设置 Cache-Control 和 Expires）
    expires 30d;

    # 更精细控制
    add_header Cache-Control "public, max-age=2592000";
}

location / {
    # HTML 不缓存，保证最新
    add_header Cache-Control "no-cache, no-store, must-revalidate";
}

HTTPS 完整配置

# HTTP → HTTPS 重定向
server {
    listen 80;
    server_name example.com www.example.com;
    return 301 https://$host$request_uri;
}

# HTTPS 主配置
server {
    listen 443 ssl;
    server_name example.com www.example.com;

    # 证书路径
    ssl_certificate     /etc/letsencrypt/live/example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/example.com/privkey.pem;

    # SSL 安全配置
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers HIGH:!aNULL:!MD5;
    ssl_prefer_server_ciphers on;

    # SSL 会话缓存
    ssl_session_cache shared:SSL:10m;
    ssl_session_timeout 10m;

    # HSTS
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;

    # 反向代理
    location / {
        proxy_pass http://127.0.0.1:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    # 静态文件
    location /static/ {
        alias /var/www/static/;
        expires 30d;
    }
}

SSL 安全加固

# 禁止 SSLv3、TLSv1.0、TLSv1.1
ssl_protocols TLSv1.2 TLSv1.3;

# 强加密算法
ssl_ciphers 'ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256';

# OCSP Stapling
ssl_stapling on;
ssl_stapling_verify on;
resolver 8.8.8.8 8.8.4.4 valid=300s;

# DH 参数
ssl_dhparam /etc/nginx/dhparam.pem;
⚠️常见误区▼
❌ HTTPS 页面引用 HTTP 资源（混合内容）
✅ 用 HTTPS 或协议相对路径 //
❌ 证书域名不匹配（签给 example.com 但访问 www.example.com）
✅ 申请时同时包含两个域名
❌ Certbot 续期失败不去排查
✅ 用 certbot renew --dry-run 模拟看报错
❌ 静态文件 403 以为是路径错
✅ 检查目录权限和 Nginx 运行用户
❌ gzip 对所有文件都开
✅ 只对文本类（HTML/CSS/JS/JSON）开启
安全提示：测试 SSL 配置用 [SSL Labs](https://www.ssllabs.com/ssltest/)，目标是 A 或 A+ 评级。

第三节 · 负载均衡与高级配置

单实例扛不住时，用 Nginx 把流量分发到多台后端，并加上限流与缓存。
🎯学习目标▼
  • 理解负载均衡的工作层次（DNS/四层/七层）
  • 掌握 Nginx 四种负载均衡算法及适用场景
  • 理解 upstream 高级参数（weight、max_fails、backup）
  • 掌握限流（limit_req）与限并发（limit_conn）配置
  • 能配置 WebSocket 代理、长连接超时、跨域等高级功能
负载均衡是高可用的基石。生产环境单实例宕机不能影响服务，限流是防止恶意流量压垮后端的关键手段。
📖核心概念▼

负载均衡的层次

层次说明工具
DNS域名解析到多个 IPRoute53、CloudFlare
四层（TCP/UDP）按 IP+端口转发LVS、HAProxy (TCP mode)
七层（HTTP）解析 HTTP 头/路径/CookieNginx、HAProxy (HTTP mode)

Nginx 工作在七层，可按 URL、Header、Cookie 做精细路由。

负载均衡算法

算法配置关键字Session 保持适用
轮询（默认）无❌后端配置相同
加权轮询weight=N❌后端性能不同
IP Haship_hash✅ 按 IP需要 Session 粘性
最少连接least_conn❌请求处理时间差异大

upstream 服务器状态参数

参数说明
weight=N权重，默认为 1
max_fails=Nfail_timeout 内最大失败次数
fail_timeout=Ns失败统计窗口 & 标记不可用后等待时间
backup备用服务器，其他都不可用才接收请求
down标记为永久离线

限流

参数说明
$binary_remote_addr按客户端 IP 限流
zone=name:10m共享内存区域（10MB ≈ 16万 IP）
rate=10r/s每秒 10 个（也可用 100r/m 每分钟）
burst=20允许突发排队 20 个
nodelay突发请求立即处理，不延迟

常用日志变量

变量说明示例
$remote_addr客户端 IP203.0.113.1
$request完整请求行GET /api/users HTTP/1.1
$status响应状态码200
$request_time请求处理总时间（秒）0.032
$upstream_response_time后端响应时间0.028
$body_bytes_sent发送字节数1234

WebSocket 特殊性

WebSocket 是长连接协议，基于 HTTP Upgrade 机制建立。Nginx 默认不支持，需显式配置 Upgrade 头。WebSocket 长连接必须增大 proxy_read_timeout，默认 60s 会被断开。

💻代码示例▼

四种负载均衡算法

# 1. 轮询（默认）
upstream backend {
    server 127.0.0.1:8001;
    server 127.0.0.1:8002;
    server 127.0.0.1:8003;
}

# 2. 加权轮询
upstream backend {
    server 127.0.0.1:8001 weight=5;
    server 127.0.0.1:8002 weight=3;
    server 127.0.0.1:8003 weight=2;
}

# 3. IP Hash
upstream backend {
    ip_hash;
    server 127.0.0.1:8001;
    server 127.0.0.1:8002;
}

# 4. 最少连接
upstream backend {
    least_conn;
    server 127.0.0.1:8001;
    server 127.0.0.1:8002;
}

upstream 高级配置

upstream backend {
    server 127.0.0.1:8001 weight=5 max_fails=3 fail_timeout=30s;
    server 127.0.0.1:8002 weight=3 max_fails=3 fail_timeout=30s;
    server 127.0.0.1:8003 backup;              # 备用服务器
    server 127.0.0.1:8004 down;                # 永久离线
}

server {
    location / {
        proxy_pass http://backend;
        # 502/503/超时/出错时自动切换到下一个后端
        proxy_next_upstream error timeout http_502 http_503;
    }
}

限流配置

http {
    # 定义限流区域：每个客户端 IP 每秒 10 个请求
    limit_req_zone $binary_remote_addr zone=api_limit:10m rate=10r/s;

    # 定义限并发区域
    limit_conn_zone $binary_remote_addr zone=conn_limit:10m;

    server {
        location /api/ {
            # 限流：突发 20 个排队 + nodelay 立即处理
            limit_req zone=api_limit burst=20 nodelay;
            # 限并发：每 IP 最多 10 个连接
            limit_conn conn_limit 10;

            proxy_pass http://backend;
        }
    }

    # 限流返回 429 而非默认 503
    limit_req_status 429;
    limit_conn_status 429;
}

JSON 日志格式

http {
    log_format json_log escape=json '{'
        '"time": "$time_iso8601",'
        '"remote_addr": "$remote_addr",'
        '"request": "$request",'
        '"status": $status,'
        '"body_bytes_sent": $body_bytes_sent,'
        '"request_time": $request_time,'
        '"upstream_response_time": "$upstream_response_time"'
    '}';

    access_log /var/log/nginx/access.log json_log;
}

# 按站点配置 + 静态文件不记录日志
server {
    server_name api.example.com;
    access_log /var/log/nginx/api_access.log json_log;
    error_log  /var/log/nginx/api_error.log warn;

    location /static/ {
        access_log off;
        alias /var/www/static/;
    }
}

WebSocket 代理配置

# map 必须在 http 块中
map $http_upgrade $connection_upgrade {
    default upgrade;
    ''      close;
}

upstream websocket_backend {
    server 127.0.0.1:8001;
    server 127.0.0.1:8002;
    ip_hash;  # WebSocket 长连接建议 ip_hash 保持粘性
}

server {
    listen 80;
    server_name ws.example.com;

    location /ws/ {
        proxy_pass http://websocket_backend;
        proxy_http_version 1.1;

        # 关键：传递 Upgrade 和 Connection 头
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection $connection_upgrade;

        # WebSocket 长连接超时（默认 60s 太短）
        proxy_read_timeout 3600s;
        proxy_send_timeout 3600s;

        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}

按路径分流到不同后端

upstream api_backend   { server 127.0.0.1:8001; }
upstream admin_backend { server 127.0.0.1:9001; }
upstream static_backend{ server 127.0.0.1:7001; }

server {
    listen 80;
    server_name example.com;

    location /api/    { proxy_pass http://api_backend; }
    location /admin/  { proxy_pass http://admin_backend; }
    location /static/ { proxy_pass http://static_backend; }
}

CORS 跨域配置

location /api/ {
    add_header Access-Control-Allow-Origin    "https://frontend.example.com" always;
    add_header Access-Control-Allow-Methods   "GET, POST, PUT, DELETE, OPTIONS" always;
    add_header Access-Control-Allow-Headers   "Authorization, Content-Type" always;
    add_header Access-Control-Allow-Credentials "true" always;

    # OPTIONS 预检请求直接返回 204
    if ($request_method = OPTIONS) {
        return 204;
    }

    proxy_pass http://backend;
}

FastAPI WebSocket 示例

from fastapi import FastAPI, WebSocket

app = FastAPI()

@app.websocket("/ws/chat")
async def chat(websocket: WebSocket):
    await websocket.accept()
    try:
        while True:
            data = await websocket.receive_text()
            await websocket.send_text(f"Echo: {data}")
    except Exception:
        pass  # 客户端断开连接
⚠️常见误区▼
❌ upstream 写 localhost
✅ 用 127.0.0.1 明确 IPv4
❌ WebSocket 60 秒后断开
✅ 增大 proxy_read_timeout 到 3600s
❌ limit_req_zone 放在 server/location 中
✅ 必须在 http 块顶层
❌ 负载均衡后 Session 丢失（用户被踢下线）
✅ IP Hash / Redis 共享 / JWT
❌ 上传文件 413
✅ 调大 client_max_body_size
❌ CORS 在 Nginx 和应用层都配置
✅ 只在一处配置，避免重复头
安全提示：被 CDN/反向代理覆盖了真实 IP 时，所有请求看起来来自同一 IP，限流可能误伤。需配置 real_ip_header 和 set_real_ip_from 还原真实客户端。
