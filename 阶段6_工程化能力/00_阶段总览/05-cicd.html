第一节 · CI/CD 概念与流程

🎯 学习目标 ▶
  • 区分 CI 与 CD（持续交付 vs 持续部署）
  • 掌握 Pipeline / Job / Step / Trigger / Runner 概念
  • 理解「快速失败」原则的工程价值
  • 能对比 GitHub Actions / GitLab CI / Jenkins 选型
  • 能识别 CI/CD 上线后 7 类常见坑
与后端开发的关联：后端服务的发布频率最高（每日多次），CI/CD 是把 Git commit 变成线上可访问接口的「最后一公里」自动化通道。
📖 核心概念 ▶

1. CI（持续集成）

开发人员频繁（通常每天多次）将代码合并到主干，每次合并自动触发：

  • 代码拉取：从 VCS 获取最新代码
  • 依赖安装：自动安装第三方库
  • 静态检查：lint / 类型检查
  • 自动测试：单元测试 + 集成测试

核心目标：尽早发现问题，修复成本最低。

2. CD（持续交付 / 持续部署）

层次英文关键差异
持续交付Continuous Delivery代码始终「可发布」，但需要人工点击发布
持续部署Continuous Deployment测试通过后全自动部署到生产

两者差在「最后一步是否需要人工」。持续部署 = 持续交付 + 自动发布。

3. CI/CD 全景

代码提交 → 自动测试 → 构建打包 → 预发布验证 → 生产部署
  push     pytest     docker       staging       prod

4. Pipeline / Job / Step / Trigger / Runner / Artifact

组件角色典型配置
Pipeline顶层容器，一个完整流程.github/workflows/ci.yml
Trigger启动 Pipeline 的事件push / pull_request / schedule / workflow_dispatch
Job独立执行单元默认并行，needs 串行
StepJob 内最小动作uses（Action）或 run（shell）
Runner执行 Job 的机器GitHub-hosted（ubuntu-latest）或自托管
ArtifactJob 产物（报告、镜像）upload-artifact / download-artifact
📐 为什么默认并行？
GitHub Actions 设计为「Job 独立运行在不同 Runner」，每个 Job 启动新虚拟机、互相隔离。并行能节省总耗时，needs 才是显式声明依赖。

1) Python 后端 API 完整 Pipeline（概念示意）

下面是从 push 到生产部署的 4 个 Job 串行执行：

# .github/workflows/backend.yml —— 概念示意
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.11'
      - run: pip install -r requirements.txt
      - run: pytest --cov=app

  build:
    needs: test          # 串行：等 test 通过
    steps:
      - run: docker build -t app:${{ github.sha }} .

  deploy-staging:
    needs: build
    steps:
      - run: ssh staging "docker compose up -d"
输入命令预期输出（成功场景）
git push origin maintest ✓ → build ✓ → deploy-staging ✓
PR 修改 src/api/users.py 后 push仅触发 test Job，PR 显示 ✅ 状态
只改 README.md 后 push若配置了 paths 过滤，则不触发

2) 并行 Job + needs 依赖

jobs:
  lint:
    runs-on: ubuntu-latest
    steps: [- run: ruff check .]

  test:
    runs-on: ubuntu-latest
    steps: [- run: pytest]

  security-scan:
    runs-on: ubuntu-latest
    steps: [- run: bandit -r src/]

  build:
    needs: [lint, test, security-scan]   # 等三者都过
    steps: [- run: docker build .]

lint / test / security-scan 三个 Job 同时启动，build 在它们都完成后才执行 → 总耗时 = max(三个) + build。

3) demo/1_cicd_concept.py 核心结构

class StageStatus(Enum):
    PENDING  = "pending"
    RUNNING  = "running"
    PASSED   = "passed"
    FAILED   = "failed"
    SKIPPED  = "skipped"   # 前序失败时本阶段的状态

class CICDPipeline:
    def run(self, force_fail_stage=None) -> bool:
        # 关键逻辑：前序失败 → 后续全部 SKIPPED
        for i, stage in enumerate(self.stages):
            if not all_passed:
                stage.status = StageStatus.SKIPPED
                continue
            if not self._simulate_stage(stage):
                all_passed = False      # 标记失败，下一轮全部 SKIPPED

这个 30 行核心逻辑就是真实 CI/CD 系统的简化模型：任意阶段失败 → 立即终止 → 后续阶段被跳过。

⚠️ 常见误区 ▶
❌ 误区 ✅ 纠正
「上了 CI/CD 就不用管代码质量」 CI/CD 只是工具。测试覆盖率 10% 时绿色勾毫无意义
Pipeline 跑 30 分钟，团队开始绕过测试 用 cache 缓存依赖、拆分并行 Job、区分冒烟 vs 回归
把数据库密码写在 YAML 里 用 Settings → Secrets 注入，${{ secrets.DB_PASS }}
「在我机器上能跑」 用 Docker 固化构建环境；锁死 Python 与依赖版本
部署没有回滚机制 每次构建打 git SHA 标签，保留最近 N 个镜像
所有 Job 串行执行 无依赖的 Job 默认并行，needs 声明串行
Pipeline 悄悄失败没人发现 配置 Slack/邮件/钉钉告警
🔐 安全提醒：Secrets 绝不能 commit 到代码仓库。即便私有仓库，泄露后影响范围极广。GitHub 提供了三级作用域（Organization / Repository / Environment），建议敏感凭证只用最低作用域。

3. 完整 Workflow YAML 骨架

name:        CI/CD Pipeline          # 显示在 GitHub Actions 页面
on:          push / PR / cron / 手动  # 触发条件
env:         PYTHON_VERSION=3.11      # 全局环境变量
concurrency: group: ...               # 同分支只保留最新一次运行
jobs:
  job-id:
    runs-on: ubuntu-latest
    timeout-minutes: 10
    needs: [other-job]                # 依赖关系（可选）
    if: <条件表达式>                   # 条件执行（可选）
    env: { JOB_VAR: x }               # Job 级别环境变量
    steps:
      - name: 
        uses: actions/checkout@v4     # 引用预定义 Action
      - name: 
        run: echo hello               # 执行 shell
        env: { STEP_VAR: y }          # Step 级别环境变量

5. 环境变量三级作用域

env:        GLOBAL_VAR = "hello"     ← 所有 Job 共享

  job1:
    env:    JOB_VAR = "world"        ← 当前 Job 共享

      - env: STEP_VAR = "foo"        ← 仅当前 Step 可见

6. Secrets 三级作用域

作用域作用范围
Organization Secrets组织内所有仓库
Repository Secrets当前仓库所有 Workflow
Environment Secrets只限特定环境（如 prod），支持审批

日志中 Secrets 自动遮蔽为 ***。

7. 矩阵构建（Matrix）

同一份代码在多个 Python 版本 / 操作系统上并行测试：

strategy:
  matrix:
    python-version: ['3.10', '3.11', '3.12']
    os: [ubuntu-latest, windows-latest]
  exclude:                            # 排除
    - python-version: '3.10'
      os: windows-latest
→ 生成 3×2-1 = 5 个并行 Job
   py3.10/ubuntu, py3.11/ubuntu+windows, py3.12/ubuntu+windows

8. Job 间依赖与数据传递

jobs:
  test:
    outputs:                          # ① 当前 Job 定义输出
      result: ${{ steps.t.outputs.result }}
    steps:
      - id: t
        run: echo "result=passed" >> $GITHUB_OUTPUT

  build:
    needs: test                       # ② 显式声明依赖
    steps:
      - run: echo "${{ needs.test.outputs.result }}"   # ③ 读取上游输出

3) demo/2_github_actions.py —— 动态生成 Workflow

class WorkflowGenerator:
    def __init__(self, name: str):
        self.workflow = {
            "name": name,
            "on": {},
            "env": {},
            "jobs": {},
        }

    def add_trigger_push(self, branches: list | None = None):
        self.workflow["on"]["push"] = (
            {"branches": branches} if branches else None
        )

    def to_yaml(self) -> str:
        return yaml.dump(self.workflow, sort_keys=False, allow_unicode=True)

# 调用
gen = WorkflowGenerator("Python Backend CI/CD")
gen.add_trigger_push(branches=["main"])
print(gen.to_yaml())   # 替代手写 YAML，避免缩进错误

用 Python 字典构建 Workflow 比手写 YAML 更安全：不会出现缩进 / 拼写错误，也方便按环境动态生成。

⚠️ 常见误区 ▶
❌ 误区 / 坑✅ 正确做法
YAML 混用 Tab 和空格缩进 统一 2 空格缩进，禁用 Tab
忘记 actions/checkout@v4 每个 Job 几乎都需要，第一步先 checkout
把密钥写在 YAML 明文里 用 ${{ secrets.NAME }} 引用，存入 Settings → Secrets
缓存 key 写死（如 pip-cache） 用 hashFiles('requirements.txt') 让依赖变更新缓存
缺少 .dockerignore，构建上下文过大 排除 .git、__pycache__、venv/ 等
不设 timeout-minutes 显式设置合理超时，避免挂起耗尽额度
Fork 仓库的 PR 访问不到 Secrets 用 mock 数据 / 跳过依赖 secrets 的步骤
多人同时 push 触发多个 Workflow 部署 用 concurrency + cancel-in-progress: true
🔐 安全提醒：来自 fork 仓库的 PR 无法访问仓库 Secrets，这是 GitHub 的安全设计。如果你的测试需要数据库密码，要么用 SQLite 内存库，要么在 PR 上跳过敏感步骤。

第三节 · 端到端流水线与部署脚本

🎯 学习目标 ▶
  • 能串联 GitHub Actions + 部署脚本实现端到端发布
  • 掌握健康检查 + 自动回滚的部署策略
  • 理解并发控制、镜像标签、SSH 部署等关键点
与后端开发的关联：真正「上线」不只是 docker compose up -d，还要回答「上线失败怎么办？」「怎么回滚到上一个稳定版本？」，这一节把生产级部署的完整套路串起来。
📖 核心概念 ▶

1. 端到端发布链路

┌──────────┐  push   ┌──────────────────┐  ssh   ┌──────────────────┐
│  GitHub  │ ──────▶ │  GitHub Actions  │ ──────▶ │  生产服务器      │
│  仓库    │         │  (CI/CD)         │        │  deploy.sh       │
└──────────┘         └──────────────────┘        └──────────────────┘
                              │                          │
                              ▼                          ▼
                       ┌──────────────┐           ┌──────────────┐
                       │ 构建 Docker  │           │ 健康检查失败 │
                       │ 镜像 + 推仓库 │           │ → 自动回滚   │
                       └──────────────┘           └──────────────┘

2. 部署脚本六阶段

阶段动作失败处理
1. 拉取代码git fetch && git reset --hard origin/main直接退出
2. 安装依赖pip install -r requirements.txt直接退出
3. 跑测试pytest -x -q终止部署
4. 构建镜像docker build -t app:${SHA}-${ts} .直接退出
5. 启动服务docker compose up -d进入健康检查
6. 健康检查curl /health 重试 N 次自动 rollback()

3. 自动回滚原理

  • 每次部署成功后，把 image_tag + commit_sha 写入 .deploy_info
  • 健康检查失败时，读取 .deploy_info 的上一版本
  • docker compose down && IMAGE_TAG=旧版 docker compose up -d
  • 回滚后再次健康检查，仍失败则人工介入

4. 关键工程策略

策略作用
镜像标签 = ${commit_sha}-${ts}每次构建唯一可追溯
保留最近 N 个镜像回滚有目标
健康检查重试 N 次容忍服务冷启动
set -euo pipefail任一命令失败立即终止
concurrency: cancel-in-progress同分支只保留最新一次运行
💻 代码示例 ▶

1) deploy.sh 主流程（节选）

#!/usr/bin/env bash
set -euo pipefail   # 任一命令失败立即退出

APP_NAME="my-python-backend"
APP_DIR="/opt/app"
HEALTH_URL="http://localhost:8000/health"
HEALTH_RETRIES=10

log_info() { echo -e "\033[0;34m[INFO]\033[0m $(date '+%F %T') $*"; }

# 1. 拉取代码
cd ${APP_DIR}
git fetch origin main
git reset --hard origin/main
local commit_sha=$(git rev-parse --short HEAD)

# 2. 装依赖
python3 -m venv venv 2>/dev/null || true
source venv/bin/activate
pip install -q -r requirements.txt

# 3. 跑测试（-x 遇到失败即停）
pytest tests/ -x -q || { log_error "测试失败"; exit 1; }

# 4. 构建镜像（带 commit SHA 标签）
local tag="${commit_sha}-$(date +%Y%m%d%H%M%S)"
docker build -t ${APP_NAME}:${tag} \
             --build-arg COMMIT_SHA=${commit_sha} .

# 5. 启服务 + 6. 健康检查
docker compose -f docker-compose.yml up -d
if health_check; then
    save_deploy_info ${tag} ${commit_sha}    # 记录当前版本
else
    rollback                              # 自动回滚到上一版
fi

2) 健康检查函数

health_check() {
    local attempt=1
    while [[ ${attempt} -le ${HEALTH_RETRIES} ]]; do
        local code
        code=$(curl -s -o /dev/null -w "%{http_code}" ${HEALTH_URL} 2>/dev/null || echo 000)
        if [[ ${code} == "200" ]]; then
            return 0   # 通过
        fi
        sleep ${HEALTH_INTERVAL}
        attempt=$((attempt + 1))
    done
    return 1   # 失败
}

3) 自动回滚函数

rollback() {
    local prev_tag
    prev_tag=$(read_prev_deploy_info)   # 从 .deploy_info 读上一版
    [[ -z ${prev_tag} ]] && { log_error "无上一版"; exit 1; }

    log_warn "回滚到 ${prev_tag}"
    docker compose down --timeout 30
    IMAGE_TAG=${prev_tag} docker compose up -d

    # 回滚后再次健康检查
    if health_check; then
        log_success "回滚成功"
    else
        log_error "回滚后仍失败，请人工介入"
        exit 1
    fi
}

4) 端到端 Workflow（部署 Job 片段）

deploy:
  needs: [lint, test, build]
  if: github.ref == 'refs/heads/main'
  environment:
    name: production
    url: https://api.example.com
  steps:
    - name: Deploy
      uses: appleboy/ssh-action@v1
      with:
        host: ${{ secrets.SERVER_HOST }}
        username: ${{ secrets.SERVER_USER }}
        key: ${{ secrets.SSH_PRIVATE_KEY }}
        script: |
          cd /opt/app
          ./deploy.sh
          # 内部已含健康检查 + 自动回滚
场景deploy.sh 行为
测试失败第 3 阶段退出，不构建、不部署
健康检查通过保存 .deploy_info，清理旧镜像
健康检查失败自动回滚到 .deploy_info 中的上一版
回滚后仍失败退出码 1，GitHub Actions 显示红色 ✗
手动 --rollback跳过所有阶段，直接回滚
⚠️ 常见误区 ▶
❌ 误区 / 坑✅ 正确做法
部署成功就完事，没有回滚脚本 每次保存 .deploy_info，失败时自动 rollback
镜像全部打 latest 标签 用 ${commit_sha}-${ts} 唯一标识
部署后立刻判定成功 等容器冷启动，curl /health 重试 N 次
多人同时 push 导致部署冲突 concurrency.cancel-in-progress: true
部署脚本不写 set -e 开 set -euo pipefail，任一失败即终止
所有镜像都堆积在服务器上 保留最近 N 个（如 3 个），定期清理
