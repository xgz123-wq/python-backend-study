第一节 · 微服务架构概念

把单体拆成多个独立服务，换来横向扩展，但也带来分布式复杂度。
🎯学习目标▼
  • 理解单体应用与微服务架构的本质差异
  • 掌握微服务通信方式（HTTP REST、gRPC、消息队列）
  • 理解服务发现、配置中心、链路追踪等治理机制
  • 理解 API 网关在微服务架构中的核心作用
  • 明确「何时该拆微服务」的判断标准
微服务是后端架构演进的重要方向。理解它的核心理念、优缺点和适用场景，能让你在系统设计时做出更合理的判断。
📖核心概念▼

1.1 单体应用 vs 微服务

单体（Monolith）是把所有功能模块打包在一个代码库和进程中运行的架构；微服务（Microservices）则是将应用拆分为多个独立部署的小型服务，每个服务负责一个业务领域，通过轻量级协议通信。

┌──────────────────────────────┐
│         单体应用              │
├──────────────────────────────┤
│ 用户 │ 订单 │ 支付 │ 库存    │
│ 模块 │ 模块 │ 模块 │ 模块    │
├──────────────────────────────┤
│      共享数据库               │
└──────────────────────────────┘
┌────────┐  ┌────────┐  ┌────────┐  ┌────────┐
│用户服务│  │订单服务│  │支付服务│  │库存服务│
│ :8001  │  │ :8002  │  │ :8003  │  │ :8004  │
└───┬────┘  └───┬────┘  └───┬────┘  └───┬────┘
    │          │          │          │
    └──────────┴──────────┴──────────┘
                    │
            ┌───────▼────────┐
            │   API 网关     │
            │     :8080      │
            └────────────────┘
维度单体应用微服务
复杂度低高（分布式复杂性）
部署简单复杂（需要编排工具）
扩展性差（整体扩容）好（独立扩容）
技术栈统一可异构
团队规模小团队大团队
适用阶段初期验证业务成熟后

1.2 什么时候该拆微服务？

核心原则：不要过早拆分！ 大多数项目初期用单体就够了。Martin Fowler 提出「Monolith First」——先用单体快速验证业务，等到痛点明显时再逐步拆分。

适合使用单体的阶段（点击展开）
  • 初创团队，产品还在验证阶段（PMF 之前）
  • 团队人数 < 10 人，沟通成本低
  • 业务量小，单体足以支撑
  • 团队缺乏运维能力（微服务需要成熟的 DevOps 支撑）
考虑拆分微服务的信号（点击展开）
  • 团队规模增长：超过 2 个披萨团队（约 10-15 人），协作开始混乱
  • 部署频率高：每天需要部署多次，但每次都要走完整流程
  • 扩展瓶颈：只有少数模块需要扩容，但必须整体扩容
  • 业务边界清晰：可以明确划分出独立的业务领域（DDD 限界上下文）
  • 技术栈需要异构：某个模块需要用 Go 写高性能服务，其他用 Python

1.4 微服务治理三大机制

① 服务发现（Service Discovery）：服务实例地址是动态的（容器重启后 IP 可能变化），需要一个注册中心来管理。

用户服务启动 ──▶ 注册到 Consul/Eureka
                   │
                   ▼
注册中心记录：user-service: [ip1, ip2, ip3]
                   ▲
订单服务 ──查询──┘
                   │
                   ▼
        负载均衡调用某个 ip

常见工具：Consul、Eureka、Nacos、etcd

② 配置中心（Configuration Center）：微服务数量多，配置文件散落在各个服务中，难以统一管理。

user-service:
  database_url: "postgresql://..."
  cache_ttl: 3600
order-service:
  database_url: "postgresql://..."
  max_retries: 3

常见工具：Apollo、Nacos、Consul KV、Spring Cloud Config

③ 链路追踪（Distributed Tracing）：一个请求可能经过 5-10 个服务，需要给每个请求分配一个 Trace ID 跟踪完整调用链。

请求 A (trace_id: abc123)
  ├── 订单服务 (span_id: 001, 20ms)
  │   ├── 用户服务 (span_id: 002, 15ms)
  │   └── 库存服务 (span_id: 003, 30ms)  ◀── 瓶颈！
  └── 总耗时: 65ms

常见工具：Jaeger、Zipkin、SkyWalking、OpenTelemetry

1.5 API 网关

API 网关是微服务架构的统一入口，处理请求路由、认证、限流等横切关注点。

客户端 ──▶ API 网关 ──┬──▶ 用户服务
                     ├──▶ 订单服务
                     ├──▶ 支付服务
                     └──▶ 库存服务

网关职责：
├── 路由：/users/* ──▶ user-service
├── 认证：验证 JWT Token
├── 限流：每分钟最多 1000 请求
├── 日志：统一记录访问日志
└── 聚合：将多个服务的数据组合后返回

常见工具：Kong、Nginx、Traefik、Spring Cloud Gateway

为什么需要 API 网关？客户端不需要知道内部服务的地址；统一的认证和授权入口；避免每个服务都重复实现限流、日志等功能；可以对外隐藏内部微服务的细节。

① 模拟服务注册中心

class ServiceRegistry:
    """模拟服务注册中心（类似 Consul、Eureka、Nacos）"""

    def __init__(self):
        # 存储格式：{服务名: [{地址、端口、状态、注册时间}, ...]}
        self._services = {}
        self._lock = threading.Lock()

    def register(self, service_name, host, port):
        """服务注册：服务启动时调用"""
        with self._lock:
            instance = {"host": host, "port": port,
                        "status": "UP", "registered_at": time.time()}
            self._services.setdefault(service_name, []).append(instance)

    def discover(self, service_name):
        """服务发现：获取某个服务的所有可用实例地址"""
        with self._lock:
            instances = self._services.get(service_name, [])
            available = [i for i in instances if i["status"] == "UP"]
            return available[0] if available else None

    def deregister(self, service_name, host, port):
        """服务注销：服务关闭时调用"""
        with self._lock:
            self._services[service_name] = [
                i for i in self._services.get(service_name, [])
                if not (i["host"] == host and i["port"] == port)
            ]

def start_service(name, handler_class, port):
    """在后台线程启动一个 HTTP 服务"""
    server = HTTPServer(("127.0.0.1", port), handler_class)
    thread = threading.Thread(target=server.serve_forever, daemon=True)
    thread.start()
    # 向注册中心注册
    registry.register(name, "127.0.0.1", port)
    return server

class OrderServiceHandler(BaseHTTPRequestHandler):

            with urlopen(user_url, timeout=3) as resp:
                user_data = json.loads(resp.read().decode("utf-8"))

            # 组合订单 + 用户信息返回
            self._send_json(200, {**order, "user": user_data})

④ API 网关的认证 + 路由

class APIGatewayHandler(BaseHTTPRequestHandler):
    # 路由规则：URL 前缀 → 服务名
    ROUTES = {
        "/api/users":  "user-service",
        "/api/orders": "order-service",
    }

    def do_GET(self):
        # ★ 认证检查
        api_key = self.headers.get("X-API-Key", "")
        if self.path != "/api/health" and api_key != "demo-key-123":
            self._send_json(401, {"error": "认证失败"})
            return

        # ★ 路由匹配 + 转发
        for prefix, service_name in self.ROUTES.items():
            if self.path.startswith(prefix):
                service = registry.discover(service_name)
                forward_url = f"http://{service['host']}:{service['port']}{self.path[4:]}"
                with urlopen(forward_url, timeout=3) as resp:
                    return self._send_json(200, json.loads(resp.read()))

⑤ 网关的数据聚合

def _handle_dashboard(self):
    """聚合接口：从多个服务获取数据并组合"""
    result = {}
    for svc_name, path in [("user-service", "/users"),
                            ("order-service", "/orders")]:
        svc = registry.discover(svc_name)
        if svc:
            url = f"http://{svc['host']}:{svc['port']}{path}"
            with urlopen(url, timeout=3) as resp:
                result[svc_name.replace("-service", "s")] = json.loads(resp.read())
    result["total_users"]  = len(result.get("users", []))
    result["total_orders"] = len(result.get("orders", []))
    self._send_json(200, result)

⚠️常见误区▼
❌ 项目初期就拆 20 个微服务
✅ 单体优先，痛点驱动拆分（Monolith First）
❌ 拆微服务后用本地事务保证一致性
✅ 用 Saga 模式或最终一致性（消息队列）
❌ 同步调用链 A→B→C→D→E（10 层）
✅ 控制调用深度，长链改异步消息
❌ 每个服务都有自己的配置文件，改密码改 N 处
✅ 用配置中心（Apollo/Nacos）统一管理 + 热更新
❌ 拆了 50 个服务但没容器编排和 CI/CD
✅ 配套 Docker + K8s + CI/CD + 监控告警
❌ 服务挂了调用方长时间等待
✅ 引入熔断器（Circuit Breaker）快速失败
运维成本警告：微服务带来的分布式复杂性（网络延迟、分布式事务、数据一致性、调试困难）远超其带来的好处，除非业务和团队规模真的需要，否则不要轻易拆分。

📖核心概念▼

2.1 gRPC 是什么？

gRPC 是 Google 开源的高性能 RPC（远程过程调用） 框架。让你像调用本地函数一样调用远程服务器上的函数，而且速度很快。

客户端                                  服务端
┌──────────┐                            ┌──────────┐
│ client.  │ ─── gRPC 调用 ──────────▶ │ GetUser()│
│ GetUser  │   Protobuf 序列化         │          │
│ (id=1)   │ ◀──────────── 响应 ─────── │ return   │
│          │   Protobuf 反序列化        │ User{...}│
└──────────┘                            └──────────┘

gRPC 的核心特性：

  • 使用 Protocol Buffers 定义接口和序列化数据
  • 基于 HTTP/2 协议（支持多路复用、头部压缩）
  • 支持四种通信模式：一元调用、服务端流、客户端流、双向流
  • 自动生成多语言的客户端和服务端代码

2.2 Protobuf vs JSON

特性JSONProtobuf
格式文本（人可读）二进制（不可读）
体积较大小（约 1/3 ~ 1/2）
速度较慢快 20~100 倍
类型安全无（弱类型）强类型（.proto 严格定义）
接口定义OpenAPI/Swagger.proto 文件
版本兼容需手动处理原生支持（按字段编号）

2.3 .proto 文件核心语法

语法含义
message定义数据结构（类似 class）
service定义 RPC 服务及其方法
rpc Method(Req) returns (Resp)一元 RPC
rpc Method(Req) returns (stream Resp)服务端流
rpc Method(stream Req) returns (stream Resp)双向流
enum枚举（proto3 要求第一个值必须是 0）
= 1, = 2字段编号，唯一标识，不能修改

维度REST (JSON)gRPC (Protobuf)

2.5 Kubernetes（K8s）核心思想

Kubernetes（k 和 s 之间有 8 个字母，简称 K8s）是 Google 开源的容器编排平台。当你有 20 个微服务 × 3 个副本 = 60 个容器时，K8s 帮你解决：

  • 如何部署这 60 个容器？
  • 某个容器挂了怎么办？
  • 流量高峰时如何自动扩容？
  • 如何做滚动更新？
  • 容器之间如何通信？

K8s 就是解决这些问题的「容器管家」。

2.6 K8s 架构总览

┌─────────────────────────────────────────────────────┐
│              Master Node（控制平面）                  │
├─────────────────────────────────────────────────────┤
│  API Server  │  Scheduler  │ Controller Manager    │
│  (集群入口)  │  (调度器)    │  (状态控制器)         │
├─────────────────────────────────────────────────────┤
│              etcd（分布式键值存储）                    │
└─────────────────────────────────────────────────────┘
                          │
                          ▼
┌─────────────────────────────────────────────────────┐
│              Worker Nodes（工作节点）                 │
├─────────────────────────────────────────────────────┤
│  ┌────────┐  ┌────────┐  ┌────────┐                │
│  │ Pod 1  │  │ Pod 2  │  │ Pod 3  │  ◀── 运行容器  │
│  └────────┘  └────────┘  └────────┘                │
│  kubelet  │  kube-proxy  ◀── 节点代理               │
└─────────────────────────────────────────────────────┘

控制平面核心组件：

  • API Server：所有操作的入口（接收 kubectl 命令）
  • Scheduler：决定 Pod 应该运行在哪个节点
  • Controller Manager：确保实际状态 = 期望状态（如副本数不足自动补）
  • etcd：分布式键值存储，保存集群所有配置和状态

2.7 K8s 四大核心对象

① Pod（最小部署单元）

一个 Pod 通常运行 1 个主容器
+ 可选 sidecar（如日志收集）
+ 有独立 IP，但 IP 是临时的
+ 不要直接创建 Pod，用 Deployment 管

② Deployment（部署控制器）——管理 Pod 副本数 + 滚动更新 + 回滚

spec:
  replicas: 3               ◀── 始终保持 3 个副本
  strategy:
    type: RollingUpdate     ◀── 滚动更新（零停机）
    rollingUpdate:
      maxSurge: 1           ◀── 更新时最多多 1 个
      maxUnavailable: 0     ◀── 更新时不允许有不可用

③ Service（服务发现 + 负载均衡）——为动态的 Pod 提供稳定访问入口

类型说明使用场景
ClusterIP仅集群内部访问微服务间内部通信
NodePort通过节点 IP + 端口访问开发测试、对外暴露
LoadBalancer云厂商负载均衡器生产环境对外暴露
订单服务可通过 DNS 名称 http://user-service:80 访问用户服务，不需要知道 Pod IP——Service 帮你屏蔽了底层的动态变化。

④ ConfigMap（配置管理）——将配置与镜像解耦

ConfigMap                  Deployment
┌──────────────────┐       ┌──────────────────┐
│ DATABASE_URL=... │──env──▶│ envFrom:         │
│ CACHE_TTL=3600   │       │   configMapRef:  │
│ LOG_LEVEL=info   │       │     name: app-cfg│
└──────────────────┘       └──────────────────┘
普通配置用 ConfigMap，敏感信息（密码/API Key）用 Secret

① 典型 .proto 文件

// user.proto — 用户服务接口定义
syntax = "proto3";
package user;

// ---------- 数据结构 ----------
message GetUserRequest {
    int64 user_id = 1;    // 字段编号 1，类型 int64
}

message UserResponse {
    int64 id = 1;
    string name = 2;
    string email = 3;
    int32 age = 4;
    UserStatus status = 5;

    enum UserStatus {
        UNKNOWN  = 0;     // proto3 要求第一个枚举值为 0
        ACTIVE   = 1;
        INACTIVE = 2;
    }
}

message CreateUserRequest {
    string name  = 1;
    string email = 2;
    int32  age   = 3;
}

// ---------- 服务定义 ----------
service UserService {
    rpc GetUser    (GetUserRequest)    returns (UserResponse);
    rpc CreateUser (CreateUserRequest) returns (UserResponse);
    rpc ListUsers  (GetUserRequest)    returns (stream UserResponse);
}

⚠️常见误区▼
❌ Protobuf 字段编号随意修改
✅ 一旦使用永不修改，废弃字段用 reserved 保留编号
❌ 期望 gRPC 二进制能像 JSON 一样可读
✅ 用 grpcurl 或 Postman 的 gRPC 支持调试
❌ 浏览器直接调用 gRPC
✅ 走 gRPC-Web 代理（Envoy）或 API 网关转换
❌ Pod Pending 不排查直接等
✅ kubectl describe pod 看 Events，定位资源/调度问题
❌ ConfigMap 改了以为 Pod 会自动更新
✅ 手动 kubectl rollout restart 触发 Pod 重建
❌ 2-3 人 CRUD 项目也上 K8s
✅ 简单应用用 Docker Compose，中等规模用 PaaS
HTTP/2 负载均衡陷阱：gRPC 基于 HTTP/2 单连接多路复用，L4 负载均衡器（如 kube-proxy iptables）只在连接建立时分配一次，会导致所有请求都打到同一个 Pod——需要用客户端负载均衡或支持 gRPC 的 L7 负载均衡器（Envoy）。
