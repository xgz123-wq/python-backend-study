# 权限与用户管理

> 多用户系统里，rwx 与用户组决定谁能动什么，是部署安全的第一道闸。

## 一、为什么需要权限管理？

Linux 是一个**多用户多任务**系统。权限机制解决的问题：

1. **隔离**：用户 A 不能偷看用户 B 的文件
2. **保护**：普通用户不能修改系统配置文件
3. **安全**：防止恶意程序获取系统控制权

> 后端场景：你的 Web 应用用 `www-data` 用户运行，它不应该能读取 `/etc/shadow`（密码文件）或修改其他用户的代码。权限配置错误是最常见的安全漏洞之一。

## 二、文件权限详解

### 2.1 查看权限

```bash
ls -l file.txt
# 输出示例：
# -rw-r--r-- 1 alice developers 1024 May 30 10:00 file.txt
#  │││││││││
#  │├┤├┤├┤
#  │ │ │ └── 其他用户权限（other）: r-- (只读)
#  │ │ └──── 所属用户组权限（group）: r-- (只读)
#  │ └────── 文件所有者权限（owner）: rw- (读写)
#  └──────── 文件类型：- 普通文件，d 目录，l 软链接
```

### 2.2 权限字符含义

| 字符 | 含义 | 对文件的作用 | 对目录的作用 |
|------|------|-------------|-------------|
| `r` | 读（read） | 查看文件内容 | 列出目录内容（`ls`） |
| `w` | 写（write） | 修改文件内容 | 在目录中创建/删除文件 |
| `x` | 执行（execute） | 运行文件（脚本/程序） | 进入目录（`cd`） |
| `-` | 无该权限 | — | — |

### 2.3 权限的数字表示法

每个权限位对应一个数字：`r=4, w=2, x=1, -=0`，三组权限各求和得到一个三位数。

```
rw-r--r--  →  owner: r+w = 4+2 = 6
                group: r   = 4   = 4
                other: r   = 4   = 4
              → 644

rwxr-xr-x  →  owner: r+w+x = 4+2+1 = 7
                group: r+x   = 4+1   = 5
                other: r+x   = 4+1   = 5
              → 755
```

**常见权限速查表**：

| 数字 | 字符 | 含义 | 典型用途 |
|------|------|------|---------|
| `755` | `rwxr-xr-x` | 所有者全部权限，其他人只读+执行 | 目录、可执行脚本 |
| `644` | `rw-r--r--` | 所有者读写，其他人只读 | 普通文件 |
| `700` | `rwx------` | 仅所有者有全部权限 | 私密目录（如 `.ssh/`） |
| `600` | `rw-------` | 仅所有者可读写 | 私密文件（如 SSH 私钥） |
| `777` | `rwxrwxrwx` | 所有人全部权限 | ⚠️ 极其危险，绝不在生产环境使用 |

### 2.4 chmod - 修改权限

```bash
# 数字模式（推荐，简洁明了）
chmod 755 script.sh          # 设置 rwxr-xr-x
chmod 644 config.txt         # 设置 rw-r--r--
chmod 600 ~/.ssh/id_rsa      # SSH 私钥必须是 600
chmod +x deploy.sh           # 添加执行权限（符号模式）

# 符号模式（适合微调）
chmod u+x script.sh          # 给 owner 添加 x 权限
chmod g-w file.txt           # 从 group 移除 w 权限
chmod o=r file.txt           # 设置 other 为只读
chmod a+r file.txt           # 给所有人添加 r 权限（a = all）

# 递归修改
chmod -R 755 project/        # 递归修改目录及其下所有文件
```

### 2.5 chown - 修改所有者

```bash
# 修改文件所有者
sudo chown alice file.txt              # 改所有者为 alice
sudo chown alice:developers file.txt   # 改所有者和所属组
sudo chown -R www-data:www-data /var/www/  # 递归修改 Web 目录

# chgrp - 只修改所属组
sudo chgrp developers file.txt
```

## 三、用户和用户组

### 3.1 用户概念

```bash
# 查看当前用户
whoami                         # 输出: alice
id                             # 输出: uid=1000(alice) gid=1000(alice) groups=1000(alice),27(sudo)

# 用户信息存储在 /etc/passwd
cat /etc/passwd
# 格式: 用户名:密码占位:UID:GID:描述:家目录:登录Shell
# alice:x:1000:1000:Alice Wang:/home/alice:/bin/bash

# 特殊用户
# root (UID=0): 超级管理员，拥有所有权限
# www-data: Web 服务器运行用户
# nobody: 最小权限用户
```

### 3.2 用户管理命令

```bash
# 创建用户
sudo useradd -m -s /bin/bash alice     # -m 创建家目录，-s 指定 Shell
sudo passwd alice                       # 设置密码

# 更友好的创建方式（Ubuntu 推荐）
sudo adduser alice                      # 交互式，自动创建家目录等

# 修改用户
sudo usermod -aG docker alice          # 把 alice 加入 docker 组（-aG = append Group）
sudo usermod -s /bin/zsh alice         # 修改默认 Shell

# 删除用户
sudo userdel alice                     # 删除用户但保留家目录
sudo userdel -r alice                  # 删除用户并删除家目录

# 切换用户
su - alice                             # 切换到 alice（- 加载环境变量）
sudo -u alice whoami                   # 以 alice 身份执行命令
exit                                   # 退出切换
```

### 3.3 用户组管理

```bash
# 查看组信息
groups                        # 当前用户所属的所有组
groups alice                  # 查看 alice 所属的组
cat /etc/group                # 所有组的信息

# 创建/删除组
sudo groupadd developers
sudo groupdel developers

# 把用户加入组
sudo usermod -aG developers alice

# ⚠️ 加入组后需要重新登录才能生效，或执行：
newgrp developers
```

## 四、sudo 权限管理

### 4.1 什么是 sudo

`sudo`（Super User Do）让普通用户可以**临时以其他用户（通常是 root）的身份执行命令**。

```bash
# 以 root 身份执行命令
sudo apt update                # 以 root 执行 apt update
sudo systemctl restart nginx   # 以 root 重启 Nginx

# 查看当前用户的 sudo 权限
sudo -l

# 以指定用户身份执行
sudo -u www-data touch /var/www/test.txt
```

### 4.2 配置 sudoers

```bash
# 编辑 sudo 配置（必须用 visudo，不能直接编辑文件！）
sudo visudo

# 常见配置格式：
# 用户名 主机=(可切换的用户) 可执行的命令

# 允许 alice 执行所有命令（需要密码）
alice ALL=(ALL:ALL) ALL

# 允许 alice 执行所有命令（不需要密码）
alice ALL=(ALL:ALL) NOPASSWD: ALL

# 允许 deploy 用户只重启 nginx（最小权限原则）
deploy ALL=(ALL) NOPASSWD: /usr/bin/systemctl restart nginx

# 把用户加入 sudo 组（Ubuntu 默认 sudo 组有全部权限）
sudo usermod -aG sudo alice
```

> ⚠️ **安全原则**：给用户的 sudo 权限应该尽可能小。生产环境中不要给 `NOPASSWD: ALL`。

## 五、SSH 密钥登录

### 5.1 为什么用 SSH 密钥

| 方式 | 安全性 | 便捷性 |
|------|--------|--------|
| 密码登录 | 容易被暴力破解 | 每次都要输密码 |
| 密钥登录 | 几乎不可能破解 | 配置好后免密登录 |

### 5.2 生成密钥对

```bash
# 生成 RSA 密钥对（推荐 4096 位）
ssh-keygen -t rsa -b 4096 -C "alice@example.com"

# 或使用 Ed25519（更现代、更安全）
ssh-keygen -t ed25519 -C "alice@example.com"

# 生成两个文件：
# ~/.ssh/id_rsa       ← 私钥（绝不能泄露！权限 600）
# ~/.ssh/id_rsa.pub   ← 公钥（可以公开）
```

### 5.3 配置免密登录

```bash
# 方法 1：ssh-copy-id（最简单）
ssh-copy-id alice@server.example.com
# 自动把公钥追加到远程服务器的 ~/.ssh/authorized_keys

# 方法 2：手动复制
cat ~/.ssh/id_rsa.pub | ssh alice@server.example.com "mkdir -p ~/.ssh && cat >> ~/.ssh/authorized_keys"

# 设置正确权限（非常重要！）
chmod 700 ~/.ssh
chmod 600 ~/.ssh/id_rsa          # 私钥必须 600
chmod 644 ~/.ssh/id_rsa.pub      # 公钥 644
chmod 600 ~/.ssh/authorized_keys # authorized_keys 必须 600
```

### 5.4 SSH 配置文件

```bash
# ~/.ssh/config 示例
Host myserver
    HostName 192.168.1.100
    User alice
    Port 22
    IdentityFile ~/.ssh/id_rsa

# 之后可以直接用别名连接
ssh myserver                   # 等价于 ssh alice@192.168.1.100 -p 22 -i ~/.ssh/id_rsa
```

### 5.5 禁用密码登录（生产环境推荐）

```bash
# 编辑 /etc/ssh/sshd_config
sudo nano /etc/ssh/sshd_config

# 修改以下配置：
# PasswordAuthentication no          # 禁用密码登录
# PubkeyAuthentication yes           # 启用密钥登录
# PermitRootLogin no                 # 禁止 root 直接登录

# 重启 SSH 服务
sudo systemctl restart sshd
```

## 六、常见坑

### 坑 1：SSH 私钥权限太宽松

```bash
# 报错信息
# @@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@
# @         WARNING: UNPROTECTED PRIVATE KEY FILE!          @
# @@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@@

# 原因：私钥权限不是 600
# 解决：
chmod 600 ~/.ssh/id_rsa
```

### 坑 2：sudo 配置错误导致无法使用 sudo

```bash
# 如果 visudo 编辑出错，可能导致所有用户都无法 sudo
# 解决：重启进入 recovery mode，以 root 修复 /etc/sudoers

# 预防措施：
# 1. 永远用 visudo 编辑（它有语法检查）
# 2. 保持一个 root 终端不关闭，万一出错可以修复
# 3. 编辑前先备份：sudo cp /etc/sudoers /etc/sudoers.bak
```

### 坑 3：`chmod 777` 的诱惑

```bash
# 错误做法：权限问题就 chmod 777
chmod 777 /var/www/          # ← 所有人都可以修改你的网站文件！

# 正确做法：找到权限不足的原因
# 1. 确认进程运行在哪个用户下：ps aux | grep nginx
# 2. 给该用户正确的权限：sudo chown -R www-data:www-data /var/www/
# 3. 设置合理权限：chmod 755 /var/www/
```

### 坑 4：忘记 `sudo` 后文件变成 root 所有

```bash
# 问题场景
sudo vim /home/alice/config.txt     # 用 sudo 编辑，文件变成 root 所有
vim /home/alice/config.txt           # 之后普通用户无法保存

# 修复
sudo chown alice:alice /home/alice/config.txt
```

### 坑 5：加入用户组后不生效

```bash
# 问题：usermod -aG 之后权限没有变化
sudo usermod -aG docker alice
docker ps    # 仍然报 permission denied

# 原因：组变更需要重新登录才生效
# 解决（二选一）：
# 1. 退出终端重新登录
# 2. 当前会话临时生效：
newgrp docker
```

### 坑 6：`-aG` 中漏掉 `-a`

```bash
# 错误：-G 会覆盖用户已有的其他组！
sudo usermod -G docker alice        # alice 只属于 docker 组了，其他组被移除

# 正确：-aG 是 append + group，追加到已有组列表
sudo usermod -aG docker alice       # alice 在原有组基础上增加 docker 组
```
