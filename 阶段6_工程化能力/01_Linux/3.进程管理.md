# 进程管理

> 你的服务在服务器上就是一个进程，能看、能起、能停、能查端口才算掌控。

## 一、什么是进程？

**进程**（Process）是正在运行的程序的实例。当你执行 `python app.py` 时，操作系统就创建了一个进程来运行这个程序。

### 核心概念

| 术语 | 含义 | 查看方式 |
|------|------|---------|
| **PID** | 进程 ID（Process ID），系统唯一标识 | `echo $$`（当前 Shell 的 PID） |
| **PPID** | 父进程 ID（Parent PID），创建该进程的进程 | `ps -o pid,ppid,cmd` |
| **前台进程** | 占据终端、接收键盘输入的进程 | 普通命令默认在前台运行 |
| **后台进程** | 不占据终端、在后台运行的进程 | 命令末尾加 `&` |

### 进程的生命周期

```
创建（fork）→ 就绪（等待 CPU）→ 运行（执行中）→ 阻塞（等待 I/O）→ 终止（退出）
                                                         ↑
                                              常见场景：读文件、等网络、等用户输入
```

## 二、查看进程

### 2.1 ps - 快照式查看

```bash
# 最常用的组合
ps aux
# a = 所有用户, u = 显示用户, x = 包含无终端的进程
# 输出列含义：
# USER    PID  %CPU  %MEM    VSZ    RSS  TTY   STAT  START   TIME  COMMAND
# alice  1234  2.5   1.2   123456  7890  ?     Ss   10:00   0:05  python app.py

# STAT 列的含义（进程状态）
# R - 运行中（Running）
# S - 休眠中（Sleeping），等待事件
# D - 不可中断的休眠（通常是等 I/O，kill 不掉）
# Z - 僵尸进程（Zombie）
# T - 已停止（Stopped）

# 常用过滤方式
ps aux | grep python                 # 查找 Python 进程
ps aux | grep -v grep                # 排除 grep 自身
ps aux --sort=-%cpu | head -10       # CPU 使用最多的 10 个进程
ps aux --sort=-%mem | head -10       # 内存使用最多的 10 个进程

# 查看指定进程的详细信息
ps -p 1234 -o pid,ppid,user,%cpu,%mem,cmd
```

### 2.2 top / htop - 实时监控

```bash
# top - 系统自带，实时刷新
top
# 快捷键（在 top 界面中）：
# P - 按 CPU 排序
# M - 按内存排序
# k - 输入 PID 终止进程
# q - 退出

# htop - 更好用的替代品（需安装）
sudo apt install htop -y
htop
# 优势：彩色显示、支持鼠标、可以树形展示进程关系（F5）
```

### 2.3 其他实用命令

```bash
# pgrep - 按名称查找进程 PID
pgrep python                     # 所有 Python 进程的 PID
pgrep -f "app.py"                # 按完整命令行搜索（-f）
pgrep -u alice                   # alice 用户的所有进程

# lsof - 列出打开的文件（Linux 中一切皆文件）
lsof -i :8080                    # 查看谁占用了 8080 端口
lsof -p 1234                     # 查看 PID 1234 打开了哪些文件
lsof -c python                   # 查看所有 Python 进程打开的文件
```

## 三、前台与后台

### 3.1 基本操作

```bash
# & - 把命令放到后台运行
python long_task.py &            # 后台运行，终端可以继续使用
# 输出: [1] 12345  （[作业号] PID）

# Ctrl + Z - 暂停当前前台进程
# 暂停后进程状态变为 T（Stopped）

# jobs - 查看当前终端的后台作业
jobs
# [1]-  Running    python task1.py &
# [2]+  Stopped    python task2.py

# fg - 把后台作业放回前台
fg %1                            # 把作业 1 放回前台

# bg - 让暂停的作业在后台继续运行
bg %2                            # 让作业 2 在后台继续
```

### 3.2 nohup - 断开终端也不停止

```bash
# 问题：直接 & 后台运行的进程，关闭终端后会被终止
python app.py &                  # 关闭终端后进程也会死

# 解决：nohup（No Hang Up）
nohup python app.py &            # 即使终端关闭，进程也不会停止
# 输出重定向到 nohup.out（默认）

nohup python app.py > app.log 2>&1 &    # 自定义日志文件
# > app.log     stdout 重定向到 app.log
# 2>&1          stderr 也重定向到 app.log
# &             放到后台

# 查看 nohup 启动的进程
ps aux | grep app.py
```

## 四、终止进程

### 4.1 kill 命令

```bash
# kill 发送信号给进程
kill 1234                        # 发送 SIGTERM（15），请求进程优雅退出
kill -9 1234                     # 发送 SIGKILL（9），强制杀死进程
kill -15 1234                    # 等价于 kill 1234（默认就是 15）

# 重要信号
# SIGTERM (15) - 请求终止，进程可以捕获并做清理工作（推荐先用这个）
# SIGKILL (9)  - 强制终止，进程无法捕获、无法忽略（最后手段）
# SIGHUP (1)   - 挂断信号，常用于让进程重新加载配置
# SIGINT (2)   - 中断信号（等价于 Ctrl+C）

# 正确终止进程的顺序
# 1. 先尝试 SIGTERM
kill 1234

# 2. 等几秒，如果还没退出，再用 SIGKILL
sleep 3 && ps -p 1234 && kill -9 1234
```

### 4.2 killall 和 pkill

```bash
# killall - 按名称终止所有匹配的进程
killall python                   # 终止所有 Python 进程（慎用！）
killall -9 node                  # 强制终止所有 Node.js 进程

# pkill - 更灵活的按条件终止
pkill -f "app.py"                # 终止命令行包含 "app.py" 的进程
pkill -u alice                   # 终止 alice 的所有进程
pkill -HUP nginx                 # 让 Nginx 重新加载配置
```

## 五、systemd 服务管理

### 5.1 什么是 systemd

`systemd` 是现代 Linux 的**初始化系统和服务管理器**（PID 1），负责：
- 系统启动时自动启动服务
- 管理后台服务（启动/停止/重启）
- 服务崩溃后自动重启
- 管理日志（journalctl）

### 5.2 systemctl 常用命令

```bash
# 服务管理
sudo systemctl start nginx          # 启动服务
sudo systemctl stop nginx           # 停止服务
sudo systemctl restart nginx        # 重启服务
sudo systemctl reload nginx         # 重新加载配置（不中断服务）
sudo systemctl status nginx         # 查看服务状态

# 开机自启
sudo systemctl enable nginx         # 设置开机自启
sudo systemctl disable nginx        # 取消开机自启
sudo systemctl is-enabled nginx     # 查看是否已设置自启

# 查看失败的服务
systemctl --failed

# 重新加载 systemd 配置（修改了 .service 文件后必须执行）
sudo systemctl daemon-reload
```

### 5.3 创建自定义服务

```bash
# 创建服务文件
sudo nano /etc/systemd/system/myapp.service
```

```ini
[Unit]
Description=My Python Web App
After=network.target

[Service]
Type=simple
User=www-data
Group=www-data
WorkingDirectory=/home/alice/myapp
# 启动命令（必须使用绝对路径）
ExecStart=/home/alice/myapp/venv/bin/python /home/alice/myapp/app.py
# 崩溃后自动重启
Restart=always
RestartSec=5
# 环境变量
Environment=FLASK_ENV=production
Environment=DATABASE_URL=postgresql://localhost/mydb

[Install]
WantedBy=multi-user.target
```

```bash
# 启用并启动服务
sudo systemctl daemon-reload
sudo systemctl enable myapp
sudo systemctl start myapp
sudo systemctl status myapp
```

## 六、常见坑

### 坑 1：僵尸进程（Zombie Process）

**僵尸进程**：子进程已经终止，但父进程没有回收它的退出状态。进程表中仍保留着一条记录（状态为 Z）。

```bash
# 查看僵尸进程
ps aux | grep 'Z'
# 或
ps -eo pid,ppid,stat,cmd | grep Z

# 僵尸进程本身不占资源（不占 CPU、不占内存），但占 PID 槽位
# 大量僵尸进程说明父进程有 bug

# 解决方法：
# 1. 找到父进程并修复（让它正确 wait 子进程）
# 2. 杀死父进程，僵尸进程会被 init (PID 1) 收养并清理
kill <parent_pid>
```

### 坑 2：孤儿进程（Orphan Process）

**孤儿进程**：父进程先终止，子进程还在运行。孤儿进程会被 init（PID 1）收养，通常无害。

```bash
# 查看孤儿进程（PPID 为 1）
ps -eo pid,ppid,cmd | awk '$2 == 1'
```

### 坑 3：kill -9 的副作用

```bash
# kill -9 是"核弹"，进程无法做任何清理工作
# 可能导致：
# - 数据库写入中断，数据损坏
# - 临时文件未删除
# - Socket 连接未正确关闭

# 正确做法：先 TERM，再 KILL
kill <pid>              # 先发送 SIGTERM
sleep 5                 # 等待 5 秒
kill -9 <pid>           # 如果还没退出，强制杀死
```

### 坑 4：nohup 的日志文件越来越大

```bash
# 问题：nohup.out 无限增长，吃满磁盘
# 解决：
# 1. 使用 logrotate 轮转日志
# 2. 或直接丢弃不需要的输出
nohup python app.py > /dev/null 2>&1 &

# 3. 生产环境不用 nohup，用 systemd 管理
# systemd 自带日志管理（journalctl），不会无限增长
```

### 坑 5：D 状态进程杀不掉

```bash
# 问题：进程状态为 D（不可中断的休眠），kill -9 也杀不掉
ps aux | grep 'D'
# 原因：进程正在等待 I/O（通常是 NFS 网络文件系统、磁盘故障）
# 解决：
# 1. 解决底层 I/O 问题（修复 NFS、检查磁盘）
# 2. 如果是 NFS 挂死，只能重启
```

### 坑 6：端口被占用但找不到进程

```bash
# 查看谁占用了端口
sudo lsof -i :8080
# 或
sudo ss -tlnp | grep 8080
# 或
sudo netstat -tlnp | grep 8080

# 找到 PID 后终止
sudo kill <pid>
```
