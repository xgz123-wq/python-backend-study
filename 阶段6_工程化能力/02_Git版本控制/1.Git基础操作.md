# Git 基础操作

> Git 是后端协作的底座，先把 add/commit/push/pull/diff 这套日常动作打通。

## 1. 概念解释

### 什么是 Git？

Git 是一个**分布式版本控制系统**（Distributed Version Control System, DVCS），由 Linus Torvalds 于 2005 年创建，最初是为了管理 Linux 内核源代码而设计。

与集中式版本控制系统（如 SVN）不同，Git 每个开发者本地都有一份**完整的仓库副本**，包括全部历史记录。这意味着：

- 即使没有网络，也能查看历史、提交代码、创建分支
- 不存在单点故障，任何一个副本都可以恢复完整项目
- 分支和合并操作极其轻量，几乎瞬间完成

### 为什么需要版本控制？

想象以下场景：

1. **代码被误删**：你删了一段代码，三天后发现还需要它——没有版本控制，只能靠记忆重写
2. **多人协作冲突**：两个人同时修改同一个文件，怎么合并？
3. **线上出了 Bug**：想快速回退到上一个稳定版本，怎么办？
4. **想尝试新功能**：又怕改坏了现有代码，需要一个安全的"试验场"

版本控制系统就是为了解决这些问题而诞生的。Git 是目前最流行的 VCS，几乎所有开源项目和企业开发都在使用。

## 2. 核心概念

理解 Git 必须掌握以下四个核心概念，它们构成了 Git 的工作模型：

```
工作区 (Working Directory)
   │
   │  git add
   ▼
暂存区 (Staging Area / Index)
   │
   │  git commit
   ▼
本地仓库 (Local Repository)
   │
   │  git push
   ▼
远程仓库 (Remote Repository)
```

### 工作区（Working Directory）

你电脑上能看到的项目目录，你在这里编辑文件。工作区里的文件有两种状态：

- **已跟踪（Tracked）**：已被 Git 管理的文件（已提交过）
- **未跟踪（Untracked）**：新创建但还没被 `git add` 的文件

### 暂存区（Staging Area / Index）

暂存区是一个"准备区"，存放你**下一次要提交的内容**。

为什么需要暂存区？因为你可能同时改了 5 个文件，但只想提交其中 3 个。暂存区让你**精确控制每次提交包含哪些改动**。

```bash
# 只暂存指定文件
git add file1.py file2.py
git commit -m "只提交这两个文件的改动"
```

### 本地仓库（Local Repository）

执行 `git commit` 后，暂存区的内容会被永久保存到本地仓库，形成一个**提交（commit）**。每个 commit 都有：

- 一个唯一的 SHA-1 哈希值（如 `a1b2c3d4...`）
- 作者信息、时间戳
- 提交信息（commit message）
- 指向父 commit 的指针（形成链表/树状历史）

### HEAD

`HEAD` 是一个特殊指针，指向**当前所在的 commit**。通常它指向某个分支名（如 `main`），而分支名又指向最新的 commit：

```
HEAD → main → commit_c → commit_b → commit_a
```

当你切换分支时，HEAD 就指向新的分支；当你做 `git reset` 时，HEAD 会移动。

## 3. 实际场景

### 场景一：个人项目从零开始

```bash
# 1. 创建项目目录并初始化 Git
mkdir my_project && cd my_project
git init

# 2. 写代码、暂存、提交
echo "# My Project" > README.md
git add README.md
git commit -m "init: 添加 README"

# 3. 关联远程仓库并推送
git remote add origin https://github.com/user/my_project.git
git push -u origin main
```

### 场景二：参与开源项目

```bash
# 1. 克隆别人的仓库
git clone https://github.com/someone/cool_project.git
cd cool_project

# 2. 查看项目历史
git log --oneline -20

# 3. 拉取最新代码（别人可能已经更新了）
git pull origin main
```

### 场景三：日常开发节奏

```bash
# 早上开始工作，先拉取最新代码
git pull

# 改完一个功能，查看改了什么
git status
git diff

# 确认没问题，提交
git add .
git commit -m "feat: 添加用户登录接口"

# 推送到远程
git push
```

## 4. 核心用法

### 4.1 初始化与克隆

```bash
# 在现有目录初始化 Git 仓库
git init

# 克隆远程仓库到本地
git clone https://github.com/user/repo.git

# 克隆到指定目录
git clone https://github.com/user/repo.git my_folder
```

### 4.2 查看状态与历史

```bash
# 查看工作区和暂存区状态（最常用的命令！）
git status

# 简洁模式
git status -s

# 查看提交历史
git log

# 单行显示，更直观
git log --oneline --graph

# 查看最近 3 次提交
git log -3

# 查看某次提交的具体改动
git show <commit-hash>
```

### 4.3 暂存与提交

```bash
# 暂存单个文件
git add file.py

# 暂存所有改动
git add .

# 暂存某个目录
git add src/

# 提交（附帶说明）
git commit -m "feat: 添加用户注册功能"

# 跳过暂存区，直接提交所有已跟踪文件的修改
git commit -am "fix: 修复登录 Bug"

# 修改上一次的提交信息
git commit --amend -m "feat: 完善用户注册功能"
```

### 4.4 查看差异

```bash
# 工作区 vs 暂存区（还没 add 的改动）
git diff

# 暂存区 vs 最新提交（下次 commit 会包含什么）
git diff --staged
# 或
git diff --cached

# 两个 commit 之间的差异
git diff abc123 def456

# 两个分支的差异
git diff main feature/login
```

### 4.5 .gitignore 配置

`.gitignore` 文件告诉 Git 哪些文件**不要跟踪**。常见的忽略规则：

```gitignore
# 忽略所有 .log 文件
*.log

# 忽略 build/ 目录
build/

# 忽略 .env 文件（包含敏感信息）
.env

# 忽略 __pycache__ 目录
__pycache__/
*.pyc

# 忽略 IDE 配置
.vscode/
.idea/

# 但不忽略 important.log（! 表示取反）
!important.log

# 忽略根目录下的 TODO，不忽略子目录下的
/TODO
```

**重要规则**：
- `.gitignore` 只能忽略**未跟踪**的文件
- 如果文件已经被 `git add` 过，需要先 `git rm --cached file` 才能被忽略
- `.gitignore` 本身应该被提交到仓库，让团队共享忽略规则

### 4.6 远程仓库操作

```bash
# 查看已配置的远程仓库
git remote -v

# 添加远程仓库
git remote add origin https://github.com/user/repo.git

# 推送到远程（首次推送需要 -u 建立追踪关系）
git push -u origin main
# 之后可以直接
git push

# 从远程拉取并合并
git pull
# 等价于
git fetch + git merge

# 只拉取，不合并（更安全，可以先看看别人改了什么）
git fetch origin
git log origin/main  # 查看远程的提交
git merge origin/main  # 确认后再合并
```

## 5. 常见坑

### 坑 1：忘记配置用户信息

```bash
# 提交时 Git 报错：Please tell me who you are
# 解决：配置全局用户信息（只需一次）
git config --global user.name "你的名字"
git config --global user.email "your@email.com"
```

### 坑 2：.gitignore 对已跟踪文件无效

```bash
# 错误做法：把已经被 git add 的文件写进 .gitignore，发现没用
# 正确做法：先从 Git 缓存中移除，再忽略
git rm --cached unwanted_file.log
# 然后把 unwanted_file.log 加入 .gitignore
```

### 坑 3：`git add .` 误提交了大文件或敏感文件

```bash
# 如果还没 push，可以撤销提交
git reset --soft HEAD~1   # 撤销 commit，保留改动在暂存区
git reset HEAD~1          # 撤销 commit，改动回到工作区

# 如果已经 push 了，需要用 git revert 创建反向提交
git revert <commit-hash>
```

### 坑 4：pull 时遇到冲突不知道怎么处理

```bash
# 如果本地有未提交的改动，先 stash 起来
git stash
git pull
git stash pop   # 恢复暂存的改动，可能需要解冲突
```

### 坑 5：commit message 写错了

```bash
# 修改最近一次提交的信息
git commit --amend -m "正确的提交信息"

# 如果已经 push，需要强制推送（谨慎！）
git push --force-with-lease  # 比 --force 更安全
```

### 坑 6：分不清 fetch 和 pull

| 命令 | 做了什么 | 安全性 |
|------|---------|--------|
| `git fetch` | 只下载远程的新提交到本地，**不修改你的代码** | 安全 |
| `git pull` | fetch + merge，**直接修改你的代码** | 可能产生冲突 |

**建议**：在不确定远程有什么改动时，先 `fetch` 看看，再决定是否 `merge`。
