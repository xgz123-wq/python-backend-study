# 分支管理与 Git Flow

> 多人并行开发靠分支隔离，Git Flow 给出主流团队约定俗成的协作模板。

## 1. 概念解释

### 什么是分支（Branch）？

分支是 Git 最强大的功能之一。本质上，分支就是一个**指向某个 commit 的可移动指针**。

```
        commit_a ← commit_b ← commit_c  (main)
                            ↖
                              commit_d ← commit_e  (feature/login)
```

上图中 `main` 和 `feature/login` 各是一个指针，分别指向不同的 commit。创建分支只是创建了一个新指针，**极其轻量**（不像 SVN 需要复制整个目录）。

### 为什么需要分支？

1. **隔离开发**：新功能开发不影响主分支的稳定性
2. **并行工作**：多人可以同时在不同分支上工作
3. **代码审查**：通过 Pull Request 在合并前审查代码
4. **快速切换**：随时切换到另一个分支处理紧急 Bug
5. **实验性开发**：创建实验分支，失败了直接删除，不影响主代码

## 2. 核心分支操作

### 2.1 创建与切换分支

```bash
# 查看所有本地分支（当前分支前有 * 号）
git branch

# 查看包括远程分支在内的所有分支
git branch -a

# 创建新分支（但不切换过去）
git branch feature/login

# 切换到该分支
git checkout feature/login
# 或者用新语法（Git 2.23+）
git switch feature/login

# 创建并切换（最常用）
git checkout -b feature/login
# 或
git switch -c feature/login

# 删除本地分支（已合并的）
git branch -d feature/login

# 强制删除（未合并的）
git branch -D feature/login
```

### 2.2 合并分支

```bash
# 先切回目标分支
git checkout main

# 合并 feature 分支
git merge feature/login

# 如果两个分支没有分叉，Git 会做 fast-forward 合并
# 如果想保留合并记录（即使可以 fast-forward）
git merge --no-ff feature/login
```

**fast-forward vs no-ff 合并：**

```
# fast-forward（默认，如果可能）：直接把 main 指针移过去
main → A → B → C → D → E  (feature 的内容直接接上)

# --no-ff：创建一个合并 commit，保留分支历史
main → A → B ──────── F  (F 是合并 commit)
              ↗      ↗
      feature → C → D
```

## 3. 分支策略：Git Flow

### 什么是 Git Flow？

Git Flow 是由 Vincent Driessen 提出的一套**分支管理规范**，适合有明确版本发布节奏的项目。它定义了 5 种分支：

```
main ────────────────●────────●──── (稳定版本，只接受合并)
                    ↗          ↗
release ──────●───●────●───●─── (准备发布的版本，只做 bug 修复)
             ↗                  ↗
develop ──●───●───●───●───●─── (开发主分支，集成所有功能)
         ↗       ↗           ↗
feature ──●───●─── (功能分支，完成后合并到 develop)
```

### 五种分支角色

| 分支类型 | 命名规范 | 从哪里创建 | 合并到哪里 | 用途 |
|---------|---------|-----------|-----------|------|
| `main` | 固定 | — | — | 生产环境代码，只接受 release 和 hotfix 合并 |
| `develop` | 固定 | — | — | 开发主分支，集成所有完成的功能 |
| `feature/*` | feature/功能名 | develop | develop | 开发新功能 |
| `release/*` | release/版本号 | develop | main + develop | 准备发布，只做 bug 修复和文档更新 |
| `hotfix/*` | hotfix/问题描述 | main | main + develop | 紧急修复生产环境 Bug |

### Git Flow 工作流示例

```bash
# 1. 开始一个新功能
git checkout develop
git pull origin develop
git checkout -b feature/user-auth

# 2. 在功能分支上开发（多次提交）
git add .
git commit -m "feat: 添加登录页面"
git commit -m "feat: 实现 JWT 验证"

# 3. 功能完成，合并回 develop
git checkout develop
git merge --no-ff feature/user-auth
git branch -d feature/user-auth  # 删除功能分支
git push origin develop

# 4. 准备发布 v1.2.0
git checkout -b release/1.2.0 develop
# 在 release 分支上修复发现的问题
git commit -m "fix: 修复发布前发现的 Bug"

# 5. 发布到 main
git checkout main
git merge --no-ff release/1.2.0
git tag -a v1.2.0 -m "Release v1.2.0"

# 6. 把 release 上的修复也合并回 develop
git checkout develop
git merge --no-ff release/1.2.0
git branch -d release/1.2.0

# 7. 生产环境发现紧急 Bug
git checkout -b hotfix/fix-crash main
git commit -m "fix: 修复崩溃问题"
git checkout main
git merge --no-ff hotfix/fix-crash
git tag -a v1.2.1 -m "Hotfix v1.2.1"
git checkout develop
git merge --no-ff hotfix/fix-crash
git branch -d hotfix/fix-crash
```

### 简化版：GitHub Flow

如果你觉得 Git Flow 太重，可以用 GitHub Flow——只有两种分支：

1. `main`：始终保持可部署状态
2. `feature/*`：从 main 创建，完成后通过 Pull Request 合并回 main

适合持续部署的 Web 应用，没有版本号的约束。

## 4. 冲突解决

### 冲突是怎么产生的

当**两个分支修改了同一个文件的同一部分**时，Git 无法自动决定保留哪个版本，就会产生冲突。

```bash
# 模拟冲突场景
# 在 main 分支修改 config.py 第 5 行
echo "DB_HOST = 'production'" >> config.py
git add config.py && git commit -m "update db host for prod"

# 在 feature 分支也修改 config.py 第 5 行
git checkout feature
echo "DB_HOST = 'localhost'" >> config.py
git add config.py && git commit -m "update db host for dev"

# 合并时产生冲突
git checkout main
git merge feature
# CONFLICT (content): Merge conflict in config.py
```

### 冲突标记

冲突文件里会出现这样的标记：

```python
<<<<<<< HEAD
DB_HOST = 'production'
=======
DB_HOST = 'localhost'
>>>>>>> feature
```

- `<<<<<<< HEAD` 到 `=======`：当前分支（main）的内容
- `=======` 到 `>>>>>>> feature`：要合并进来的分支的内容

### 手动解决冲突

1. 打开冲突文件，删除冲突标记，保留正确的内容
2. 保存文件
3. `git add <file>` 标记冲突已解决
4. `git commit` 完成合并

```python
# 解决后的 config.py（根据环境选择保留哪个）
DB_HOST = os.getenv('DB_HOST', 'localhost')
```

### 工具辅助解决冲突

```bash
# 使用图形化工具解决冲突
git mergetool

# 常用工具：VS Code、Beyond Compare、meld
# VS Code 打开冲突文件时会自动高亮冲突区域，提供 Accept 按钮
```

### 放弃合并

```bash
# 如果合并搞砸了，可以放弃
git merge --abort
```

## 5. Rebase vs Merge

两种整合分支改动的方式，各有优缺点：

### Merge（合并）

```bash
git checkout main
git merge feature
```

**结果**：创建一个新的"合并 commit"，保留完整的分支历史。

```
main:    A → B ──────── M  (M 是合并 commit)
              ↗        ↗
feature:    C → D → E
```

**优点**：历史真实反映开发过程，安全（不改写已有提交）
**缺点**：历史可能很杂乱，合并 commit 很多

### Rebase（变基）

```bash
git checkout feature
git rebase main
```

**结果**：把 feature 分支的提交"摘下来"，重新接到 main 的最新提交之后。

```
# rebase 前
main:    A → B → C
feature: A → B → D → E

# rebase 后（D 和 E 被重新应用到 C 之后）
main:    A → B → C
feature: A → B → C → D' → E'  (D' E' 是新的 commit，内容同 D E)
```

**优点**：历史线性、干净
**缺点**：改写了 commit 历史，如果在公共分支上 rebase 会导致团队其他人混乱

### 黄金法则

> **绝对不要在公共分支（别人也在用的分支）上 rebase！**

```bash
# ✅ 正确：在自己的功能分支上 rebase 主分支的更新
git checkout feature
git rebase main

# ❌ 错误：在 main 上 rebase（会改写公共历史）
git checkout main
git rebase feature
```

### 团队建议

- **个人分支**：用 rebase 保持历史整洁
- **合并到主分支时**：用 merge（`--no-ff`）保留合并记录
- **不确定时**：用 merge，更安全

## 6. Tag 标签管理

Tag 是给某个 commit 打上**不可变的标记**，通常用于标记版本号。

```bash
# 创建轻量标签（只是指向 commit 的指针）
git tag v1.0.0

# 创建附注标签（推荐，包含作者、时间、说明）
git tag -a v1.0.0 -m "Release v1.0.0: 首个正式版本"

# 查看所有标签
git tag

# 查看某个标签的信息
git show v1.0.0

# 给过去的 commit 打标签
git tag -a v0.9.0 abc1234 -m "Beta version"

# 推送标签到远程（默认 push 不会推送标签）
git push origin v1.0.0

# 推送所有标签
git push origin --tags

# 删除本地标签
git tag -d v1.0.0

# 删除远程标签
git push origin --delete v1.0.0

# 基于某个标签创建分支（用于修复旧版本）
git checkout -b hotfix/v1.0.1 v1.0.0
```

## 7. 常见坑

### 坑 1：在错误的分支上开发了

```bash
# 场景：以为在 feature 分支，结果在 main 上写了一堆代码
# 解决：用 stash 暂存改动，切到正确分支再恢复
git stash
git checkout feature
git stash pop
git add . && git commit -m "feat: ..."
```

### 坑 2：删除分支时提示"not fully merged"

```bash
# 说明该分支有改动还没合并
# 如果确定不需要那些改动了：
git branch -D feature/abandoned  # 大写 D 强制删除

# 如果不确定，先看看有什么改动
git log main..feature/abandoned
```

### 坑 3：rebase 之后 push 不上去

```bash
# 因为 rebase 改写了历史，普通 push 会被拒绝
# 用 --force-with-lease（比 --force 安全，会检查远程是否有别人的新提交）
git push --force-with-lease
```

### 坑 4：合并冲突解决错了

```bash
# 如果 commit 还没做完（还在冲突状态）
git merge --abort  # 放弃整个合并

# 如果已经 commit 了，但还没 push
git reset --hard ORIG_HEAD  # 回到合并之前的状态

# 如果已经 push 了，只能再做一个修正 commit
```

### 坑 5：分支命名混乱

建议统一命名规范：

```
feature/用户登录       →  ✗ 避免中文
feature/user-login     →  ✓ 清晰明了
feature/PROJ-123-login →  ✓ 带任务编号
fix/login-crash        →  ✓ 修复类
hotfix/payment-bug     →  ✓ 紧急修复
```

### 坑 6：merge 和 rebase 混用导致历史混乱

**团队必须统一选择一种策略**：

- 策略 A（推荐新手）：所有分支都用 merge
- 策略 B（进阶）：个人分支用 rebase，合入主分支用 merge --no-ff
