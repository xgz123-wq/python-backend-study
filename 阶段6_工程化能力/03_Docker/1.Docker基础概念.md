# Docker 基础概念

> Docker 用容器解决"在我电脑上能跑"，是现代部署绕不开的标准化封装。

## 一、什么是 Docker

### 1.1 一句话定义

Docker 是一种**容器化技术**，它能把你的应用和所有依赖打包成一个标准化的"集装箱"，在任何机器上都能一模一样地运行。

### 1.2 为什么叫"集装箱"

在集装箱发明之前，货物运输是一件极其麻烦的事——散货装卸效率低、容易损坏、到了不同码头还要适配不同的搬运方式。集装箱的出现彻底改变了航运业：**不管里面装什么，外面的规格统一，到哪儿都能卸**。

Docker 对软件做了同样的事：

| 现实世界         | Docker 世界               |
|----------------|---------------------------|
| 集装箱设计图       | **镜像（Image）**          |
| 实际运输的集装箱    | **容器（Container）**       |
| 码头/港口         | **Docker Host（宿主机）**   |
| 货物清单          | **Dockerfile**            |
| 全球港口网络       | **Docker Registry（仓库）** |

### 1.3 解决的核心问题

> "在我电脑上明明能跑啊！"

这是每个开发者都经历过的痛苦。代码在开发机上运行正常，到了测试服务器就报错，上线后又是另一番景象。原因无非：

- 操作系统版本不同
- Python 版本不同
- 依赖库版本冲突
- 环境变量配置不一致
- 系统库缺失

Docker 的回答是：**把应用连同它运行所需的一切，打包在一起。不依赖宿主机的任何环境**。

## 二、镜像（Image）vs 容器（Container）

### 2.1 镜像（Image）

镜像是一个**只读模板**，包含了运行应用所需的一切：

- 操作系统基础层（如 Ubuntu、Alpine）
- 运行时环境（如 Python 3.12）
- 应用代码
- 依赖库
- 配置文件
- 环境变量

镜像是**分层（Layer）存储**的，每一层代表 Dockerfile 中的一条指令。层与层之间是叠加关系，上层可以覆盖下层的文件。

```
镜像层次结构示例：
┌─────────────────────────┐
│ Layer 5: CMD 启动命令     │  ← 最顶层
├─────────────────────────┤
│ Layer 4: COPY 应用代码    │
├─────────────────────────┤
│ Layer 3: RUN pip install │
├─────────────────────────┤
│ Layer 2: RUN apt-get     │
├─────────────────────────┤
│ Layer 1: FROM ubuntu     │  ← 基础层
└─────────────────────────┘
```

### 2.2 容器（Container）

容器是镜像的**运行实例**。你可以从一个镜像启动多个容器，每个容器之间互相隔离。

容器在镜像之上添加了一个**可写层**，容器运行时的所有修改都发生在这一层。容器删除后，可写层也随之消失。

```python
# 类比理解
class Image:
    """镜像 = 类（模板）"""
    pass

class Container(Image):
    """容器 = 实例（对象）"""
    def __init__(self):
        self.writable_layer = {}  # 可写层
        self.is_running = True
```

### 2.3 对比总结

| 维度     | 镜像（Image）      | 容器（Container）         |
|--------|-------------------|--------------------------|
| 本质     | 只读模板           | 运行中的实例               |
| 类比     | 类 / 设计图         | 对象 / 实际建筑            |
| 是否可写  | 不可写             | 有可写层                  |
| 生命周期  | 持久存储           | 临时运行，删除即可销毁        |
| 数量关系  | 1 个镜像          | 可启动 N 个容器            |

## 三、Docker vs 虚拟机

### 3.1 架构对比

**虚拟机（VM）架构：**
```
┌──────┐ ┌──────┐ ┌──────┐
│App A │ │App B │ │App C │
├──────┤ ├──────┤ ├──────┤
│Libs  │ │Libs  │ │Libs  │
├──────┤ ├──────┤ ├──────┤
│OS A  │ │OS B  │ │OS C  │  ← 每个 VM 都有完整 OS
├──────┴─┴──────┴─┴──────┤
│      Hypervisor         │
├─────────────────────────┤
│      Host OS / 硬件      │
└─────────────────────────┘
```

**Docker 容器架构：**
```
┌──────┐ ┌──────┐ ┌──────┐
│App A │ │App B │ │App C │
├──────┤ ├──────┤ ├──────┤
│Libs  │ │Libs  │ │Libs  │  ← 共享内核，无需完整 OS
├──────┴─┴──────┴─┴──────┤
│     Docker Engine        │
├─────────────────────────┤
│      Host OS 内核        │
├─────────────────────────┤
│         硬件             │
└─────────────────────────┘
```

### 3.2 性能对比

| 维度       | 虚拟机（VM）         | Docker 容器          |
|----------|-------------------|---------------------|
| 启动速度   | 分钟级（需启动完整 OS）| 秒级（直接启动进程）    |
| 占用空间   | GB 级              | MB 级               |
| 运行开销   | 高（独立 OS 开销）   | 极低（共享内核）       |
| 隔离程度   | 强（硬件级隔离）     | 较弱（进程级隔离）     |
| 密度      | 一台机器跑十几个 VM  | 一台机器跑几百个容器    |
| 跨平台    | 可以运行不同 OS      | 只能运行相同内核的 OS  |

### 3.3 什么时候用哪个

- **用 Docker**：微服务部署、CI/CD 流水线、开发环境统一、快速扩缩容
- **用虚拟机**：需要运行不同操作系统、强安全隔离需求、运行遗留的老系统

## 四、Docker 架构

Docker 采用**客户端-服务端（C/S）架构**，核心组件如下：

### 4.1 Docker Client（客户端）

你直接交互的部分。当你在终端输入 `docker run`、`docker build` 等命令时，就是在操作 Client。

Client 通过 REST API 与 Daemon 通信，可以向本地或远程的 Daemon 发送指令。

### 4.2 Docker Daemon（守护进程）

运行在宿主机上的后台服务（`dockerd`），负责：

- 构建、运行、停止容器
- 管理镜像、网络、数据卷
- 监听 Client 的请求并执行

### 4.3 Docker Registry（镜像仓库）

存储和分发镜像的服务：

- **Docker Hub**：官方公共仓库，最大的镜像库
- **私有 Registry**：企业内部使用，如 Harbor
- **GitHub Container Registry**：GitHub 提供的容器镜像托管

```bash
# 从 Registry 拉取镜像
docker pull python:3.12-slim

# 推送镜像到 Registry
docker push myrepo/myapp:latest
```

### 4.4 完整工作流程

```
用户在 Client 输入：docker run python:3.12 python -c "print('hello')"
                    │
                    ▼
            Docker Daemon 收到请求
                    │
                    ▼
        检查本地是否有 python:3.12 镜像
           │                │
          有                没有
           │                │
           ▼                ▼
      直接创建容器    从 Registry 拉取镜像
           │                │
           ◄────────────────┘
                    │
                    ▼
            启动容器并执行命令
                    │
                    ▼
             输出结果 "hello"
```

## 五、为什么后端开发必须会 Docker

### 5.1 开发环境一致性

团队里每个人的电脑配置不同，用 Docker 后，`docker-compose up` 一行命令就能启动整套开发环境，所有人运行的环境完全一致。

### 5.2 CI/CD 流水线的基础

现代 CI/CD 工具（GitHub Actions、GitLab CI、Jenkins）几乎都基于 Docker。测试在容器里运行，构建在容器里完成，部署的也是容器镜像。

### 5.3 微服务架构的标配

当你的系统从单体变成十几个微服务时，不用 Docker 管理简直噩梦。每个服务一个容器，独立部署、独立扩缩、独立升级。

### 5.4 云原生时代的基本功

Kubernetes（K8s）是容器编排的事实标准，而 Docker 容器是 K8s 的基本调度单元。不会 Docker，就不可能学 K8s。

### 5.5 快速体验新技术

想试试 Redis、MongoDB、PostgreSQL？不用安装，一行命令就能跑起来：

```bash
docker run -d -p 6379:6379 redis
docker run -d -p 27017:27017 mongo
docker run -d -p 5432:5432 postgres
```

用完 `docker rm` 删掉，不留垃圾。

## 六、安装 Docker Desktop

### 6.1 系统要求

| 操作系统          | 要求                                 |
|----------------|-------------------------------------|
| Windows        | Windows 10/11（64 位），需启用 WSL 2    |
| macOS          | macOS 12 及以上                       |
| Linux          | 推荐用 Docker Engine 而非 Desktop      |

### 6.2 Windows 安装步骤

1. **启用 WSL 2**（Windows Subsystem for Linux）：
   ```powershell
   # 以管理员身份运行 PowerShell
   wsl --install
   ```

2. **下载 Docker Desktop**：
   访问 https://www.docker.com/products/docker-desktop/ 下载安装包

3. **安装并启动**：
   运行下载的安装程序，安装完成后重启电脑，启动 Docker Desktop

4. **验证安装**：
   ```bash
   docker --version
   docker run hello-world
   ```

### 6.3 macOS 安装步骤

1. 下载对应芯片（Apple Silicon / Intel）的 Docker Desktop
2. 将 Docker 拖入 Applications 文件夹
3. 启动 Docker Desktop
4. 终端验证：`docker --version`

### 6.4 Linux 安装（推荐用 Docker Engine）

```bash
# Ubuntu / Debian
sudo apt-get update
sudo apt-get install docker.io docker-compose-v2

# 启动服务
sudo systemctl start docker
sudo systemctl enable docker

# 将当前用户加入 docker 组（避免每次 sudo）
sudo usermod -aG docker $USER

# 重新登录后验证
docker --version
docker run hello-world
```

### 6.5 配置国内镜像加速

国内拉取 Docker Hub 镜像可能很慢，需要配置镜像加速器：

```json
// Docker Desktop → Settings → Docker Engine，添加：
{
    "registry-mirrors": [
        "https://mirror.ccs.tencentyun.com",
        "https://registry.docker-cn.com"
    ]
}
```

## 七、常见误区

### 误区 1：Docker 就是轻量级虚拟机

**错**。Docker 容器和虚拟机是两种完全不同的隔离方式。容器共享宿主机内核，虚拟机有独立的内核。容器的隔离性弱于虚拟机，但性能远优于虚拟机。

### 误区 2：一个容器里应该跑整个应用栈

**错**。Docker 的最佳实践是**一个容器只运行一个进程**。Web 服务、数据库、缓存应该分别用不同的容器，通过 docker-compose 编排。

### 误区 3：容器里的数据是安全的

**错**。容器删除后，可写层的数据也会丢失。需要持久化的数据必须挂载**数据卷（Volume）**或**绑定挂载（Bind Mount）**。

### 误区 4：Docker 只适合 Linux

**错**。虽然 Docker 的原生运行环境是 Linux，但 Docker Desktop 在 Windows 和 macOS 上通过 WSL 2 / HyperKit 提供了完整的 Linux 虚拟化层，体验一致。

## 八、本章小结

| 概念            | 要点                                         |
|---------------|---------------------------------------------|
| Docker 是什么   | 容器化技术，让应用在任何机器上一致运行             |
