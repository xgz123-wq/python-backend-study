# Dockerfile 编写

> Dockerfile 是镜像的"配方文件"，写好它才能让构建可复现、镜像更精简。

## 一、什么是 Dockerfile

Dockerfile 是一个文本文件，包含了构建 Docker 镜像所需的全部指令。Docker 读取 Dockerfile 中的指令，自动构建出镜像。

```
Dockerfile（配方） → docker build（烹饪） → Image（成品菜）
```

一个最简单的 Python 应用 Dockerfile：

```dockerfile
FROM python:3.12-slim
WORKDIR /app
COPY requirements.txt .
RUN pip install -r requirements.txt
COPY . .
EXPOSE 8000
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]
```

## 二、核心指令详解

### 2.1 FROM — 基础镜像

每条 Dockerfile 必须以 `FROM` 开头，指定基础镜像。

```dockerfile
# 完整 Python 镜像（约 900MB）
FROM python:3.12

# 精简版（约 125MB）— 推荐
FROM python:3.12-slim

# 极简版（约 50MB）— 需要编译的依赖可能装不上
FROM python:3.12-alpine
```

选择原则：
- **slim** 版本是最佳平衡点：体积小，包含常用工具
- **alpine** 体积最小，但使用 musl libc，某些 Python C 扩展编译会失败
- 避免使用完整版，体积太大

### 2.2 WORKDIR — 工作目录

设置后续指令的工作目录。如果目录不存在，Docker 会自动创建。

```dockerfile
WORKDIR /app
# 后续所有命令都在 /app 下执行
COPY . .
RUN pip install -r requirements.txt
```

> **最佳实践**：始终使用绝对路径。

### 2.3 COPY — 复制文件

从宿主机（构建上下文）复制文件到镜像中。

```dockerfile
# 复制单个文件
COPY requirements.txt /app/

# 复制整个目录
COPY . /app/

# 复制多个文件
COPY src/ config/ /app/

# 使用 --chown 设置文件所有者
COPY --chown=appuser:appuser . /app/
```

### 2.4 ADD — 增强版复制

`ADD` 与 `COPY` 类似，但有两个额外功能：

- 自动解压本地 `.tar` 文件
- 支持从 URL 下载文件（但不推荐，用 `RUN curl` 更清晰）

```dockerfile
# ADD 会自动解压
ADD app.tar.gz /app/

# 不推荐用 ADD 下载远程文件
# ADD https://example.com/file.tar.gz /tmp/
```

**最佳实践**：除非需要自动解压，否则一律使用 `COPY`。

### 2.5 RUN — 执行命令

在构建阶段执行 shell 命令，结果会保存在新的镜像层中。

```dockerfile
# 安装系统依赖
RUN apt-get update && apt-get install -y \
    gcc \
    libpq-dev \
    && rm -rf /var/lib/apt/lists/*

# 安装 Python 依赖
RUN pip install --no-cache-dir -r requirements.txt

# 多行命令用 && 连接（一个 RUN 完成多步，减少层数）
RUN apt-get update && \
    apt-get install -y curl && \
    apt-get clean && \
    rm -rf /var/lib/apt/lists/*
```

> **关键原则**：每个 `RUN` 指令都会创建一个新的镜像层。合并相关命令到一个 `RUN` 中可以减小镜像体积。

### 2.6 EXPOSE — 声明端口

声明容器运行时监听的端口。这只是一个**文档性质的声明**，不会真正映射端口。真正的端口映射在 `docker run -p` 中完成。

```dockerfile
EXPOSE 8000
EXPOSE 8000/tcp
EXPOSE 8000/udp
```

### 2.7 CMD — 默认启动命令

容器启动时执行的默认命令。**Dockerfile 中只有最后一条 CMD 生效**。

```dockerfile
# exec 格式（推荐，不会启动 shell 进程）
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]

# shell 格式（会启动 /bin/sh -c，信号处理可能有问题）
CMD uvicorn main:app --host 0.0.0.0 --port 8000

# 参数格式（配合 ENTRYPOINT 使用）
CMD ["--host", "0.0.0.0"]
```

### 2.8 ENTRYPOINT — 入口点

与 CMD 类似，但 **ENTRYPOINT 不会被 `docker run` 的参数覆盖**，而是将参数追加到 ENTRYPOINT 之后。

```dockerfile
# 场景：固定执行 python，CMD 提供默认参数
ENTRYPOINT ["python"]
CMD ["app.py"]

# docker run myimage             → python app.py
# docker run myimage script.py   → python script.py（覆盖 CMD）
# docker run myimage -c "pass"   → python -c "pass"
```

### 2.9 ENV — 环境变量

```dockerfile
# 单个变量
ENV APP_ENV=production

# 多个变量
ENV APP_ENV=production \
    APP_DEBUG=false \
    APP_PORT=8000

# 在后续指令中引用
RUN echo "Running in $APP_ENV mode"
```

### 2.10 ARG — 构建参数

与 ENV 类似，但 **ARG 只在构建阶段有效**，运行时不可用。

```dockerfile
ARG PYTHON_VERSION=3.12
FROM python:${PYTHON_VERSION}-slim

ARG APP_VERSION=1.0.0
RUN echo "Building version ${APP_VERSION}"
```

构建时传参：
```bash
docker build --build-arg PYTHON_VERSION=3.11 --build-arg APP_VERSION=2.0.0 -t myapp .
```

### 2.11 USER — 切换用户

```dockerfile
# 创建非 root 用户
RUN useradd -m appuser

# 切换到该用户
USER appuser

# 后续所有指令都以 appuser 身份运行
```

### 2.12 HEALTHCHECK — 健康检查

```dockerfile
HEALTHCHECK --interval=30s --timeout=10s --retries=3 \
    CMD curl -f http://localhost:8000/health || exit 1
```

参数说明：
- `--interval`：检查间隔（默认 30s）
- `--timeout`：单次检查超时（默认 30s）
- `--retries`：连续失败多少次后标记为 unhealthy（默认 3）
- `--start-period`：容器启动后的宽限期（默认 0s）

### 2.13 VOLUME — 声明匿名卷

```dockerfile
VOLUME /data
```

声明 `/data` 目录应该使用数据卷存储。运行时 Docker 会自动创建匿名卷挂载到该路径。

## 三、多阶段构建

### 3.1 问题

构建阶段可能需要大量工具（gcc 编译器、node 构建工具等），但运行阶段并不需要这些工具。如果全塞在一个镜像里，体积会很大。

### 3.2 解决方案

多阶段构建允许在一个 Dockerfile 中使用多个 `FROM` 指令，每个阶段可以使用不同的基础镜像。最终只保留最后一个阶段（或指定的阶段）的产物。

```dockerfile
# ===== 阶段 1：构建 =====
FROM python:3.12-slim AS builder

WORKDIR /build

# 安装构建依赖
RUN pip install --user --no-cache-dir \
    cython \
    numpy \
    pandas

COPY requirements.txt .
RUN pip install --user --no-cache-dir -r requirements.txt

COPY . .

# ===== 阶段 2：运行 =====
FROM python:3.12-slim

WORKDIR /app

# 从 builder 阶段复制已安装的依赖
COPY --from=builder /root/.local /root/.local
COPY --from=builder /build /app

ENV PATH=/root/.local/bin:$PATH

# 创建非 root 用户
RUN useradd -m appuser && chown -R appuser:appuser /app
USER appuser

EXPOSE 8000
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]
```

优势：最终镜像不包含构建工具和中间文件，体积大幅减小。

## 四、.dockerignore

`.dockerignore` 告诉 Docker 在构建时忽略哪些文件，避免把不必要的文件复制到镜像中。

```dockerignore
# 版本控制
.git
.gitignore

# Python 缓存
__pycache__
*.pyc
*.pyo
.pytest_cache

# 虚拟环境
venv/
.venv/
env/

# IDE 配置
.vscode/
.idea/

# Docker 自身文件
Dockerfile
docker-compose*.yml
.dockerignore

# 文档和测试
*.md
tests/
docs/

# 系统文件
.DS_Store
Thumbs.db

# 敏感文件
.env
.env.*
*.pem
*.key
```

为什么要用 `.dockerignore`：
- 减小构建上下文大小，加快构建速度
- 避免敏感文件（密钥、配置文件）被意外打入镜像
- 防止缓存文件干扰构建结果

## 五、Python 后端 Dockerfile 最佳实践

### 5.1 完整示例

```dockerfile
# ========================================
# Python FastAPI 应用 Dockerfile（生产级）
# ========================================

# --- 阶段 1：依赖安装 ---
FROM python:3.12-slim AS deps

WORKDIR /deps

# 先复制依赖文件，利用层缓存
COPY requirements.txt .
RUN pip install --user --no-cache-dir -r requirements.txt

# --- 阶段 2：运行环境 ---
FROM python:3.12-slim AS runtime

WORKDIR /app

# 安装必要的系统工具（如 curl 用于健康检查）
RUN apt-get update && \
    apt-get install -y --no-install-recommends curl && \
    apt-get clean && \
    rm -rf /var/lib/apt/lists/*

# 从 deps 阶段复制 Python 依赖
COPY --from=deps /root/.local /root/.local
ENV PATH=/root/.local/bin:$PATH

# 复制应用代码
COPY . .

# 创建非 root 用户并设置权限
RUN useradd -m -r appuser && \
    chown -R appuser:appuser /app
USER appuser

# 声明端口
EXPOSE 8000

# 健康检查
HEALTHCHECK --interval=30s --timeout=10s --start-period=5s --retries=3 \
    CMD curl -f http://localhost:8000/health || exit 1

# 启动命令
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000", "--workers", "4"]
```

### 5.2 最佳实践清单

| 实践                        | 原因                                        |
|---------------------------|-------------------------------------------|
| 使用 slim 基础镜像           | 体积小（~125MB），够用                         |
| 先 COPY requirements.txt  | 利用层缓存，依赖不变时不用重新安装                   |
| `pip install --no-cache-dir` | 不缓存 pip 下载文件，减小层体积                 |
| 合并 RUN 指令               | 减少层数                                     |
| 使用多阶段构建                | 最终镜像不包含构建工具                           |
| 创建非 root 用户             | 安全最佳实践                                   |
| 添加 HEALTHCHECK            | 容器编排工具可以检测服务健康状态                    |
| 使用 .dockerignore          | 避免无关文件进入镜像                            |
| 不硬编码配置                  | 通过环境变量注入，同一镜像适配不同环境               |

## 六、镜像优化

### 6.1 减小镜像体积

```dockerfile
# 不好的写法 — 每层都保留了 apt 缓存
RUN apt-get update
RUN apt-get install -y gcc
RUN apt-get install -y libpq-dev

# 好的写法 — 合并 + 清理缓存
RUN apt-get update && \
    apt-get install -y --no-install-recommends gcc libpq-dev && \
    apt-get clean && \
    rm -rf /var/lib/apt/lists/*
```

体积对比：

| 优化措施                   | 节省空间        |
|--------------------------|----------------|
| python → python:slim     | ~750MB         |
| `--no-cache-dir` (pip)   | ~50MB          |
| 清理 apt 缓存              | ~20-100MB      |
| 多阶段构建                  | 取决于构建工具大小  |
| `.dockerignore`           | 视项目而定       |

### 6.2 利用层缓存

Docker 构建时，每一层会检查是否有变化。如果某一层的输入没有变，就使用缓存（秒级完成），否则该层及之后的所有层都要重新构建。

```dockerfile
# 差的写法 — 代码一变，依赖也要重新安装
COPY . .                              # ← 代码经常变
RUN pip install -r requirements.txt   # ← 每次都重新执行

# 好的写法 — 先复制依赖文件，利用缓存
COPY requirements.txt .               # ← 依赖文件很少变
RUN pip install -r requirements.txt   # ← 使用缓存，秒级完成
COPY . .                              # ← 只有代码层重新构建
```

## 七、常见坑

### 坑 1：CMD 和 ENTRYPOINT 的 shell 格式 vs exec 格式

```dockerfile
# shell 格式 — 会启动 /bin/sh -c，PID 1 是 sh 而不是你的应用
CMD uvicorn main:app --host 0.0.0.0 --port 8000

# exec 格式 — 应用直接作为 PID 1 运行，能正确接收信号
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]
```

shell 格式的问题是：`docker stop` 发送 SIGTERM 给 PID 1（sh），但 sh 不会转发信号给子进程，导致应用无法正常关闭，只能等 10 秒后被 SIGKILL 强杀。

### 坑 2：监听 127.0.0.1 导致外部无法访问

```dockerfile
# 错误 — 只监听容器内部的 localhost
CMD ["uvicorn", "main:app", "--host", "127.0.0.1", "--port", "8000"]

# 正确 — 监听所有网络接口
CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000"]
```

### 坑 3：用 root 用户运行应用

虽然容器已经有隔离，但以 root 运行仍然存在容器逃逸风险。生产环境应该创建非 root 用户。

### 坑 4：在 Dockerfile 中使用 `pip install -r` 不加 `--no-cache-dir`

pip 默认会缓存下载的包文件。在 Docker 中这些缓存没有意义（不会复用），白白增加镜像体积。

### 坑 5：忘记清理 apt 缓存

`apt-get install` 会在 `/var/lib/apt/lists/` 中留下包索引文件。不清理的话，这些文件会永久存在于镜像层中。

### 坑 6：构建上下文过大

没有 `.dockerignore` 文件时，`docker build` 会把当前目录所有文件（包括 `node_modules/`、`.git/`、大文件等）发送给 Docker daemon，导致构建非常慢。

## 八、本章小结

| 指令           | 作用                    | 注意事项                  |
|--------------|------------------------|--------------------------|
| FROM         | 指定基础镜像              | 用 slim 版本              |
| WORKDIR      | 设置工作目录              | 用绝对路径                 |
| COPY         | 复制文件到镜像            | 先复制依赖文件利用缓存        |
| RUN          | 构建时执行命令            | 合并命令、清理缓存           |
| CMD          | 容器启动命令              | 用 exec 格式              |
| ENTRYPOINT   | 固定入口点               | 配合 CMD 使用              |
| EXPOSE       | 声明端口（文档性）         | 真正映射靠 `docker run -p` |
| ENV / ARG    | 环境变量 / 构建参数        | ARG 只在构建时有效          |
| USER         | 切换用户                 | 生产环境用非 root           |
| HEALTHCHECK  | 健康检查                 | 配合编排工具使用             |
| 多阶段构建     | 减小镜像体积              | 构建阶段和运行阶段分离        |
