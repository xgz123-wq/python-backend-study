# docker-compose 编排

> 一个 YAML 把 Web、DB、Redis 一起拉起来，本地开发与单机部署的最小编排工具。

## 一、为什么需要 docker-compose

### 1.1 问题场景

一个典型的后端应用至少包含：

- Web 服务（FastAPI / Django / Flask）
- 数据库（MySQL / PostgreSQL）
- 缓存（Redis）
- 可能还有消息队列（RabbitMQ / Kafka）

如果只用 `docker run`，你需要：

```bash
# 1. 创建网络
docker network create my-app-net

# 2. 启动数据库
docker run -d --name mysql \
  --network my-app-net \
  -e MYSQL_ROOT_PASSWORD=root123 \
  -e MYSQL_DATABASE=myapp \
  -v mysql-data:/var/lib/mysql \
  -p 3306:3306 \
  mysql:8.0

# 3. 启动 Redis
docker run -d --name redis \
  --network my-app-net \
  -v redis-data:/data \
  -p 6379:6379 \
  redis:7.2

# 4. 启动 Web 应用
docker run -d --name web \
  --network my-app-net \
  -e DATABASE_URL=mysql://root:root123@mysql:3306/myapp \
  -e REDIS_URL=redis://redis:6379/0 \
  -p 8000:8000 \
  myapp:latest
```

每次启动都要敲一遍，还要保证顺序和参数正确。如果团队成员要搭建开发环境，还得写一份详细的文档一步步教。

### 1.2 docker-compose 的解决方案

**一条命令，启动所有服务：**

```bash
docker compose up -d
```

所有服务的配置都写在一个 `docker-compose.yml` 文件中，版本化管理，团队共享。

## 二、docker-compose.yml 语法

### 2.1 基本结构

```yaml
# docker-compose.yml 基本结构
version: "3.9"       # Compose 文件版本

services:            # 服务定义（核心部分）
  web:
    ...
  db:
    ...
  redis:
    ...

volumes:             # 命名卷声明
  mysql-data:
  redis-data:

networks:            # 网络声明
  app-network:
```

### 2.2 服务（Service）配置项

每个服务就是一个容器的完整配置：

```yaml
services:
  web:
    # --- 镜像来源 ---
    build:                      # 从 Dockerfile 构建
      context: .                # 构建上下文目录
      dockerfile: Dockerfile    # Dockerfile 路径
    # 或
    image: myapp:latest         # 直接使用镜像

    # --- 容器配置 ---
    container_name: my-web      # 容器名称
    restart: unless-stopped     # 重启策略

    # --- 网络 ---
    ports:
      - "8000:8000"             # 端口映射
    expose:
      - "8000"                  # 仅内部暴露（不映射到宿主机）
    networks:
      - app-network

    # --- 存储 ---
    volumes:
      - ./src:/app              # 绑定挂载（开发用）
      - static-data:/app/static # 命名卷

    # --- 环境变量 ---
    environment:
      - APP_ENV=development
      - DATABASE_URL=mysql://root:root123@db:3306/myapp
    # 或从文件加载
    env_file:
      - .env

    # --- 依赖 ---
    depends_on:
      db:
        condition: service_healthy    # 等 db 健康后再启动 web
      redis:
        condition: service_started    # 等 redis 启动后再启动 web

    # --- 健康检查 ---
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8000/health"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 10s

    # --- 资源限制 ---
    deploy:
      resources:
        limits:
          cpus: "1.0"
          memory: 512M
        reservations:
          cpus: "0.5"
          memory: 256M
```

### 2.3 depends_on 的正确使用

`depends_on` 只控制**启动顺序**，不等待服务就绪。

```yaml
# 基础版：只等容器启动，不等服务就绪
depends_on:
  - db
  - redis

# 高级版：等待服务健康检查通过（推荐）
depends_on:
  db:
    condition: service_healthy    # 等 db 的 healthcheck 通过
  redis:
    condition: service_started
```

要让 `service_healthy` 生效，被依赖的服务必须配置 `healthcheck`。

## 三、实战：一键启动 FastAPI + MySQL + Redis

### 3.1 完整 docker-compose.yml

```yaml
# docker-compose.yml
# FastAPI + MySQL + Redis 全栈编排

version: "3.9"

services:
  # ============================================
  # MySQL 数据库
  # ============================================
  db:
    image: mysql:8.0
    container_name: myapp-db
    restart: unless-stopped
    environment:
      MYSQL_ROOT_PASSWORD: root123
      MYSQL_DATABASE: myapp
      MYSQL_USER: appuser
      MYSQL_PASSWORD: apppass123
    ports:
      - "3306:3306"
    volumes:
      - mysql-data:/var/lib/mysql
    networks:
      - app-network
    healthcheck:
      test: ["CMD", "mysqladmin", "ping", "-h", "localhost"]
      interval: 10s
      timeout: 5s
      retries: 5
      start_period: 30s

  # ============================================
  # Redis 缓存
  # ============================================
  redis:
    image: redis:7.2-alpine
    container_name: myapp-redis
    restart: unless-stopped
    ports:
      - "6379:6379"
    volumes:
      - redis-data:/data
    networks:
      - app-network
    healthcheck:
      test: ["CMD", "redis-cli", "ping"]
      interval: 10s
      timeout: 5s
      retries: 5

  # ============================================
  # FastAPI Web 应用
  # ============================================
  web:
    build:
      context: .
      dockerfile: Dockerfile
    container_name: myapp-web
    restart: unless-stopped
    ports:
      - "8000:8000"
    volumes:
      - ./src:/app        # 开发模式：代码热更新
    environment:
      - APP_ENV=development
      - DATABASE_URL=mysql+aiomysql://appuser:apppass123@db:3306/myapp
      - REDIS_URL=redis://redis:6379/0
    env_file:
      - .env
    depends_on:
      db:
        condition: service_healthy
      redis:
        condition: service_healthy
    networks:
      - app-network
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:8000/health"]
      interval: 30s
      timeout: 10s
      retries: 3
      start_period: 15s

# ============================================
# 命名卷
# ============================================
volumes:
  mysql-data:
    driver: local
  redis-data:
    driver: local

# ============================================
# 自定义网络
# ============================================
networks:
  app-network:
    driver: bridge
```

### 3.2 对应的 Dockerfile

```dockerfile
FROM python:3.12-slim

WORKDIR /app

# 安装 curl（用于健康检查）
RUN apt-get update && \
    apt-get install -y --no-install-recommends curl && \
    apt-get clean && \
    rm -rf /var/lib/apt/lists/*

COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

COPY . .

RUN useradd -m appuser && chown -R appuser:appuser /app
USER appuser

EXPOSE 8000

HEALTHCHECK --interval=30s --timeout=10s --start-period=5s --retries=3 \
    CMD curl -f http://localhost:8000/health || exit 1

CMD ["uvicorn", "main:app", "--host", "0.0.0.0", "--port", "8000", "--reload"]
```

## 四、环境变量管理

### 4.1 三种方式

```yaml
services:
  web:
    # 方式 1：直接写在 compose 文件中
    environment:
      - APP_ENV=development
      - DEBUG=true

    # 方式 2：从 .env 文件加载
    env_file:
      - .env

    # 方式 3：从 .env 文件加载，同时使用变量替换
    environment:
      - DATABASE_URL=${DATABASE_URL}
      - SECRET_KEY=${SECRET_KEY}
```

### 4.2 .env 文件

```bash
# .env（不要提交到 Git！）
DATABASE_URL=mysql+aiomysql://appuser:apppass123@db:3306/myapp
REDIS_URL=redis://redis:6379/0
SECRET_KEY=your-super-secret-key-here
APP_ENV=development
DEBUG=true
```

### 4.3 变量替换

docker-compose.yml 中可以直接引用环境变量：

```yaml
services:
  web:
    image: myapp:${APP_VERSION:-latest}    # 默认 latest
    ports:
      - "${WEB_PORT:-8000}:8000"           # 默认 8000
```

## 五、网络和数据卷配置

### 5.1 网络详解

```yaml
networks:
  # 桥接网络（默认）
  app-network:
    driver: bridge

  # 使用已有的外部网络
  external-network:
    external: true
    name: my-existing-network
```

**同一 network 下的容器可以用容器名（service name）互相访问**：
- `web` 容器访问 `db` 容器：`mysql://db:3306/myapp`
- `web` 容器访问 `redis` 容器：`redis://redis:6379/0`

### 5.2 数据卷详解

```yaml
volumes:
  # Docker 管理的命名卷
  mysql-data:
    driver: local

  # 绑定挂载（直接映射宿主机目录）
  # 不需要在顶层声明，直接在 service 中用即可

  # 带驱动选项的卷
  shared-data:
    driver: local
    driver_opts:
      type: nfs
      o: addr=10.0.0.1,rw
      device: ":/shared"
```

## 六、docker compose 常用命令

```bash
# ===== 启动与停止 =====

# 构建并启动所有服务（后台运行）
docker compose up -d

# 构建并启动（前台运行，看日志）
docker compose up

# 只启动指定服务
docker compose up -d web redis

# 停止所有服务
docker compose down

# 停止并删除数据卷（慎用！会丢数据）
docker compose down -v

# 重新构建镜像
docker compose up -d --build

# ===== 查看状态 =====

# 查看服务状态
docker compose ps

# 查看日志（所有服务）
docker compose logs

# 实时查看某个服务的日志
docker compose logs -f web

# ===== 操作单个服务 =====

# 在某个服务中执行命令
docker compose exec web python manage.py shell

# 重启某个服务
docker compose restart web

# 停止某个服务
docker compose stop web

# ===== 构建 =====

# 只构建，不启动
docker compose build

# 构建指定服务
docker compose build web

# ===== 清理 =====

# 删除停止的容器
docker compose rm

# 停止并删除容器、网络（保留卷和数据）
docker compose down

# 停止并删除一切（包括卷，慎用）
docker compose down -v --rmi all
```

## 七、多环境配置

### 7.1 使用多个 compose 文件

```
docker-compose.yml          # 基础配置
docker-compose.override.yml # 开发环境覆盖（自动加载）
docker-compose.prod.yml     # 生产环境覆盖
```

```yaml
# docker-compose.override.yml（开发环境）
services:
  web:
    build: .                          # 开发时从本地构建
    volumes:
      - ./src:/app                    # 挂载源码，支持热更新
    environment:
      - APP_ENV=development
      - DEBUG=true
    command: uvicorn main:app --host 0.0.0.0 --port 8000 --reload

# docker-compose.prod.yml（生产环境）
services:
  web:
    image: registry.example.com/myapp:latest  # 使用预构建镜像
    environment:
      - APP_ENV=production
      - DEBUG=false
    command: uvicorn main:app --host 0.0.0.0 --port 8000 --workers 4
    deploy:
      replicas: 3
```

```bash
# 开发环境（自动加载 override）
docker compose up -d

# 生产环境
docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d
```

## 八、常见坑

### 坑 1：depends_on 不等于"等服务就绪"

`depends_on` 默认只保证容器启动，不保证服务可用。MySQL 启动后还需要几十秒初始化。解决方案是使用 `condition: service_healthy` 配合 `healthcheck`。

### 坑 2：容器间用 localhost 通信

容器之间不能用 `localhost` 或 `127.0.0.1` 通信。必须使用**服务名**（如 `db`、`redis`）作为主机名。`localhost` 指的是容器自己。

### 坑 3：`docker compose down -v` 会删除数据卷

`-v` 参数会删除所有命名卷，数据库数据全丢。开发时想保留数据就不要加 `-v`。

### 坑 4：修改了 Dockerfile 但没重新构建

`docker compose up -d` 不会自动重新构建镜像。修改 Dockerfile 后需要：

```bash
docker compose up -d --build
# 或
docker compose build && docker compose up -d
```

### 坑 5：端口冲突

宿主机端口被占用时，`docker compose up` 会报错。检查端口占用，或修改 compose 文件中的端口映射。

### 坑 6：环境变量优先级

优先级从高到低：
1. `docker compose run -e KEY=VAL` 命令行参数
2. `environment:` 在 compose 文件中直接定义
3. `env_file:` 引用的文件
4. 宿主机环境变量
5. `.env` 文件（compose 自动读取）

## 九、docker-compose v1 vs v2

| 维度       | v1（docker-compose）      | v2（docker compose）           |
|----------|--------------------------|-------------------------------|
| 命令格式   | `docker-compose up`      | `docker compose up`           |
| 实现      | Python 编写的独立工具       | Go 编写，作为 Docker CLI 插件   |
| 性能      | 较慢                      | 更快                          |
| 维护状态   | 已废弃（2023 年停止维护）    | 当前推荐使用                     |

现在应该统一使用 `docker compose`（v2，无连字符）。
