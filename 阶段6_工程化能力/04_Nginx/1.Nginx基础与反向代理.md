# Nginx 基础与反向代理

> Nginx 是服务最前面的"门卫"，反向代理把外部请求安全地转给后端进程。

## 一、Nginx 是什么？

Nginx（读作 "engine-x"）是一个**高性能的 HTTP 服务器和反向代理服务器**，由 Igor Sysoev 于 2004 年开发。

在全球 Web 服务器市场中，Nginx 占据约 **36%** 的份额（2024 年数据），是部署 Python 后端应用（FastAPI / Django / Flask）时最常用的网关组件。

### Nginx 的核心能力

| 能力 | 说明 | 实际用途 |
|------|------|----------|
| Web 服务器 | 直接提供静态文件（HTML/CSS/JS/图片） | 前端资源托管 |
| 反向代理 | 接收客户端请求，转发给后端应用服务器 | FastAPI/Django 部署 |
| 负载均衡 | 将请求分发到多个后端实例 | 高可用架构 |
| HTTPS 终端 | 处理 SSL/TLS 加密解密 | 安全通信 |
| 缓存 | 缓存后端响应，减少重复计算 | 性能优化 |
| 限流 | 限制客户端请求频率 | 防止恶意请求 |

### 为什么不用 Python 直接监听 80/443 端口？

```
方案 A（不推荐）：客户端 → FastAPI(uvicorn) 直接对外
方案 B（推荐）：  客户端 → Nginx → FastAPI(uvicorn)
```

Nginx 的优势：

1. **性能**：C 语言编写，事件驱动架构，处理静态文件比 Python 快 10-100 倍
2. **安全**：隐藏后端技术栈，防止直接暴露 Python 进程
3. **稳定性**：后端崩溃时 Nginx 可返回友好错误页面
4. **功能丰富**：HTTPS、压缩、缓存等由 Nginx 统一处理，应用代码更简洁

## 二、正向代理 vs 反向代理

这是面试高频考点，必须搞清楚。

### 正向代理（Forward Proxy）

```
客户端 → [正向代理] → 目标服务器
         ↑
    客户端知道目标服务器是谁
    代理代表「客户端」
```

**典型场景**：
- VPN / 科学上网工具
- 公司内网代理上网
- Python 中的 `requests` 的 `proxies` 参数

```python
import requests

# 正向代理示例：客户端通过代理访问外部网站
proxies = {
    "http": "http://proxy.company.com:8080",
    "https": "http://proxy.company.com:8080",
}
response = requests.get("https://api.github.com", proxies=proxies)
```

### 反向代理（Reverse Proxy）

```
客户端 → [反向代理] → 后端服务器
         ↑
    客户端不知道后端服务器是谁
    代理代表「服务端」
```

**典型场景**：
- Nginx 作为 FastAPI 的前置网关
- CDN 节点
- API Gateway（Kong / Traefik）

### 对比总结

| 维度 | 正向代理 | 反向代理 |
|------|----------|----------|
| 代理谁 | 客户端 | 服务端 |
| 客户端知道目标吗 | 知道 | 不知道 |
| 部署位置 | 客户端一侧 | 服务端一侧 |
| 典型工具 | VPN、Squid | Nginx、HAProxy |
| 主要目的 | 访问控制/翻墙 | 负载均衡/安全/缓存 |

## 三、为什么需要反向代理？

### 1. 安全隔离

```
没有反向代理：
  互联网 → FastAPI (IP:端口 直接暴露)

有反向代理：
  互联网 → Nginx (只暴露 80/443) → FastAPI (仅监听 127.0.0.1:8000)
```

- 后端服务器不直接暴露 IP 和端口
- 可以限制只允许特定路径通过
- 可以屏蔽恶意请求（限流、黑名单）

### 2. 负载均衡

```
                    ┌→ FastAPI 实例 1 (127.0.0.1:8001)
客户端 → Nginx ─────┼→ FastAPI 实例 2 (127.0.0.1:8002)
                    └→ FastAPI 实例 3 (127.0.0.1:8003)
```

- 多个后端实例分担请求压力
- 某个实例宕机时自动切换到其他实例

### 3. 统一入口

```
api.example.com/api/*      → FastAPI (后端 API)
api.example.com/static/*   → Nginx 直接返回静态文件
api.example.com/admin/*    → Django (管理后台)
```

- 多个服务共享同一个域名和端口
- HTTPS 证书只需在 Nginx 层配置一次
- 前端和后端可以分别部署，但用户看到同一个域名

## 四、Nginx 安装和基本配置

### 安装

```bash
# Ubuntu / Debian
sudo apt update
sudo apt install nginx

# CentOS / RHEL
sudo yum install epel-release
sudo yum install nginx

# macOS（Homebrew）
brew install nginx

# Docker
docker run -d -p 80:80 --name my-nginx nginx:alpine

# 验证安装
nginx -v          # 查看版本
sudo nginx -t     # 测试配置文件语法
sudo systemctl start nginx    # 启动
sudo systemctl enable nginx   # 开机自启
```

### 目录结构

```
/etc/nginx/
├── nginx.conf              # 主配置文件
├── conf.d/                 # 额外的配置文件（推荐在这里写站点配置）
│   └── mysite.conf
├── sites-available/        # 可用站点配置（Debian 系列）
├── sites-enabled/          # 已启用站点（symlink 到 sites-available）
├── mime.types              # MIME 类型映射
└── snippets/               # 可复用的配置片段
```

### 常用命令

```bash
sudo nginx -t              # 测试配置语法（修改后必做）
sudo nginx -s reload       # 重新加载配置（不中断服务）
sudo nginx -s stop         # 立即停止
sudo nginx -s quit         # 优雅停止（处理完当前请求）
sudo tail -f /var/log/nginx/access.log   # 实时查看访问日志
sudo tail -f /var/log/nginx/error.log    # 实时查看错误日志
```

## 五、nginx.conf 结构详解

Nginx 配置文件采用**层级块结构**：

```
main (全局块)
├── events (事件块)
└── http (HTTP 块)
    ├── upstream (上游服务器组)
    └── server (虚拟主机)
        └── location (路径匹配规则)
```

### 完整结构示例

```nginx
# ===== main 全局块 =====
user nginx;                          # 运行 Nginx 的系统用户
worker_processes auto;               # 工作进程数（auto = CPU 核心数）
error_log /var/log/nginx/error.log;  # 全局错误日志
pid /run/nginx.pid;                  # PID 文件

# ===== events 事件块 =====
events {
    worker_connections 1024;         # 每个 worker 最大并发连接数
    use epoll;                       # Linux 推荐的事件模型
}

# ===== http 块 =====
http {
    include       mime.types;        # MIME 类型映射
    default_type  application/octet-stream;

    sendfile on;                     # 高效传输静态文件
    keepalive_timeout 65;            # 长连接超时时间

    # ----- upstream 上游服务器组 -----
    upstream backend {
        server 127.0.0.1:8001;
        server 127.0.0.1:8002;
    }

    # ----- server 虚拟主机 -----
    server {
        listen 80;                   # 监听端口
        server_name example.com;     # 域名

        # ----- location 路径匹配 -----
        location / {
            proxy_pass http://backend;   # 反向代理到 upstream
        }

        location /static/ {
            alias /var/www/static/;      # 静态文件目录
        }
    }
}
```

### location 匹配规则（优先级从高到低）

| 语法 | 含义 | 示例 |
|------|------|------|
| `= /path` | 精确匹配 | `= /health` |
| `^~ /path` | 前缀匹配，不再检查正则 | `^~ /static/` |
| `~ regex` | 正则匹配（区分大小写） | `~ \.(jpg\|png)$` |
| `~* regex` | 正则匹配（不区分大小写） | `~* \.(gif\|ico)$` |
| `/path` | 普通前缀匹配 | `/api/` |

```nginx
# 匹配优先级示例
location = /api/health { ... }      # 精确匹配 /api/health（最高优先）
location ^~ /api/ { ... }           # 前缀匹配 /api/* 开头
location ~ \.php$ { ... }           # 正则匹配 .php 结尾
location / { ... }                  # 兜底匹配（最低优先）
```

## 六、反向代理配置实战

### 场景：FastAPI 应用部署

```nginx
server {
    listen 80;
    server_name api.example.com;

    # 反向代理到 FastAPI（uvicorn 运行在 127.0.0.1:8000）
    location / {
        proxy_pass http://127.0.0.1:8000;

        # 传递真实客户端信息给后端
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;

        # 超时设置
        proxy_connect_timeout 10s;
        proxy_read_timeout 30s;
        proxy_send_timeout 30s;
    }

    # 健康检查接口（直接返回，不经过后端）
    location = /nginx-health {
        return 200 "OK";
        add_header Content-Type text/plain;
    }
}
```

### 关键 header 解释

| Header | 作用 |
|--------|------|
| `Host` | 传递原始请求的域名，后端据此判断是哪个站点 |
| `X-Real-IP` | 客户端真实 IP（不经过代理时 `$remote_addr` 就是客户端 IP） |
| `X-Forwarded-For` | 完整的代理链路 IP 列表 |
| `X-Forwarded-Proto` | 原始请求协议（http 或 https），后端据此构造正确的重定向 URL |

### FastAPI 中获取真实 IP

```python
from fastapi import FastAPI, Request

app = FastAPI()

@app.get("/whoami")
async def whoami(request: Request):
    # 如果 Nginx 设置了 X-Real-IP
    real_ip = request.headers.get("X-Real-IP", "unknown")
    # 如果没有 Nginx，拿到的是直连 IP
    direct_ip = request.client.host
    return {
        "real_ip": real_ip,
        "direct_ip": direct_ip,
    }
```

## 七、常见坑

### 1. 修改配置后忘记 reload

```bash
# 修改完 nginx.conf 后一定要：
sudo nginx -t           # 先测试语法
sudo nginx -s reload    # 再重新加载
```

不 reload 的话，Nginx 还在用旧配置运行。

### 2. proxy_pass 末尾斜杠的影响

```nginx
# 有斜杠：location 匹配的部分会被替换掉
location /api/ {
    proxy_pass http://backend/;    # /api/users → /users
}

# 无斜杠：原始路径原样传递
location /api/ {
    proxy_pass http://backend;     # /api/users → /api/users
}
```

这一个斜杠的差异会导致路由完全错乱，是新手最常踩的坑。

### 3. 502 Bad Gateway

原因：Nginx 无法连接到后端服务。

排查步骤：
1. 后端服务是否在运行？`curl http://127.0.0.1:8000`
2. 端口是否写对？检查 `proxy_pass` 的端口
3. 防火墙是否阻止？`sudo ufw status`

### 4. 文件上传 413 Request Entity Too Large

```nginx
# 默认只允许 1MB 上传，需要调大
client_max_body_size 50m;    # 允许 50MB
```

### 5. server_name 写了 IP 地址

```nginx
# 不推荐
server_name 192.168.1.100;

# 推荐：用域名，IP 变化时只需改 DNS
server_name api.example.com;
```

## 八、总结

| 知识点 | 核心要点 |
|--------|----------|
| Nginx 定位 | 高性能 Web 服务器 + 反向代理 |
| 正向代理 | 代表客户端，隐藏客户端身份 |
| 反向代理 | 代表服务端，隐藏后端服务器 |
| 安装命令 | `apt install nginx` / `brew install nginx` |
| 配置结构 | main → events → http → server → location |
| location 优先级 | `=` > `^~` > `~` > 普通前缀 |
| 反向代理核心 | `proxy_pass` + 传递真实 IP 的 header |
| 改完配置必做 | `nginx -t` 测试 + `nginx -s reload` |
