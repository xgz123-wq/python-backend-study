# 静态文件与 HTTPS 配置

> 让 Nginx 直接发静态资源、再用 Let's Encrypt 给站点挂上 HTTPS。

## 一、静态文件托管

### 什么是静态文件？

静态文件是指**不需要后端计算就能直接返回**的文件：

- HTML / CSS / JavaScript
- 图片（PNG / JPG / SVG / WebP）
- 字体文件（WOFF2 / TTF）
- 用户上传的媒体文件

Nginx 处理静态文件的性能远高于 Python 应用，因为 Nginx 使用 `sendfile()` 系统调用，数据直接从磁盘到网卡，**不经过用户态内存**。

### 基本配置

```nginx
server {
    listen 80;
    server_name example.com;

    # 方式 1：root（路径拼接）
    # 请求 /static/css/style.css → 查找 /var/www/html/static/css/style.css
    location /static/ {
        root /var/www/html;    # root + 完整 URI = 文件路径
    }

    # 方式 2：alias（路径替换）
    # 请求 /static/css/style.css → 查找 /opt/app/static/css/style.css
    location /static/ {
        alias /opt/app/static/;  # alias 替换 location 匹配的部分
    }
}
```

### root vs alias 的区别

| 指令 | 文件路径计算方式 | 示例 |
|------|-----------------|------|
| `root /var/www` | root + 完整 URI | `/static/a.css` → `/var/www/static/a.css` |
| `alias /opt/files/` | alias 替换匹配前缀 | `/static/a.css` → `/opt/files/a.css` |

```nginx
# root 示例：location 路径是文件路径的一部分
location /static/ {
    root /var/www;
    # /static/logo.png → /var/www/static/logo.png
}

# alias 示例：location 路径和文件路径可以完全不同
location /images/ {
    alias /data/media/photos/;
    # /images/cat.jpg → /data/media/photos/cat.jpg
}
```

**常见坑**：`alias` 结尾必须加 `/`，`root` 结尾不加 `/`。

## 二、gzip 压缩

### 为什么需要 gzip？

文本类文件（HTML / CSS / JS / JSON）压缩率通常在 **60%-90%**，可以显著减少传输时间和带宽消耗。

```
未压缩：style.css = 200KB → 传输 200KB
压缩后：style.css = 200KB → gzip → 40KB → 传输 40KB（节省 80%）
```

### 配置

```nginx
http {
    gzip on;                          # 开启 gzip
    gzip_min_length 1k;               # 小于 1KB 不压缩（压缩收益小）
    gzip_comp_level 6;                # 压缩级别 1-9（6 是性能和压缩率的平衡点）
    gzip_types                        # 需要压缩的 MIME 类型
        text/plain
        text/css
        text/javascript
        application/javascript
        application/json
        application/xml
        image/svg+xml;
    gzip_vary on;                     # 添加 Vary: Accept-Encoding 头
    gzip_proxied any;                 # 对所有代理请求也启用 gzip
}
```

### 压缩级别选择

| 级别 | 压缩率 | CPU 消耗 | 适用场景 |
|------|--------|----------|----------|
| 1 | 低 | 极低 | 高并发实时压缩 |
| 6 | 中 | 中 | **推荐默认值** |
| 9 | 高 | 高 | 低流量、追求极致压缩 |

**不要对图片/视频开启 gzip**：JPEG / PNG / MP4 已经是压缩格式，再 gzip 几乎无效且浪费 CPU。

## 三、缓存策略

### HTTP 缓存机制

浏览器缓存可以大幅减少重复请求，提升页面加载速度。

```nginx
location /static/ {
    alias /var/www/static/;

    # Cache-Control：控制缓存行为
    expires 30d;                      # 30 天过期（Nginx 自动设置 Cache-Control 和 Expires）

    # 更精细的控制
    add_header Cache-Control "public, max-age=2592000";  # 30天 = 2592000秒

    # 对 HTML 不缓存（保证用户始终看到最新版本）
    # 通常在另一个 location 中配置
}

location / {
    add_header Cache-Control "no-cache, no-store, must-revalidate";
}
```

### 常见缓存策略

| 文件类型 | 推荐策略 | 原因 |
|----------|----------|------|
| JS/CSS（带 hash） | `max-age=31536000`（1年） | 文件名含 hash，内容变化时文件名也变 |
| 图片/字体 | `max-age=2592000`（30天） | 变化不频繁 |
| HTML | `no-cache` | 始终验证，保证最新 |
| API 响应 | `no-store` | 不缓存动态数据 |

### ETag 机制

```nginx
# Nginx 默认开启 ETag
etag on;    # 默认就是 on，通常不需要显式配置
```

ETag 工作原理：
1. 首次请求：服务器返回文件 + `ETag: "abc123"`
2. 再次请求：浏览器发送 `If-None-Match: "abc123"`
3. 文件未变：服务器返回 `304 Not Modified`（不传文件体，极快）
4. 文件已变：服务器返回 `200 OK` + 新文件 + 新 ETag

## 四、HTTPS 原理

### 为什么需要 HTTPS？

HTTP 是明文传输，存在三大问题：
1. **窃听**：中间人可以读取传输内容（密码、Cookie）
2. **篡改**：中间人可以修改传输内容（注入广告、恶意脚本）
3. **冒充**：无法确认对方身份（DNS 劫持、钓鱼网站）

HTTPS = HTTP + TLS（Transport Layer Security），解决以上所有问题。

### TLS 握手过程（简化版）

```
客户端                                    服务器
  │                                        │
  │─── 1. ClientHello ──────────────────→  │
  │    (支持的加密算法列表、随机数)            │
  │                                        │
  │←── 2. ServerHello ──────────────────── │
  │    (选定加密算法、证书、随机数)            │
  │                                        │
  │─── 3. 验证证书 ─────────────────────→   │
  │    (检查证书链、域名、有效期)             │
  │    生成「预主密钥」用服务器公钥加密         │
  │                                        │
  │─── 4. 加密的预主密钥 ────────────────→  │
  │                                        │
  │    5. 双方各自用三个随机数生成会话密钥      │
  │                                        │
  │←── 6. 对称加密通信开始 ──────────────→  │
  │    (后续通信用会话密钥加密，速度快)         │
```

### 关键概念

| 概念 | 说明 |
|------|------|
| 非对称加密 | 公钥加密、私钥解密，用于握手阶段传递密钥（慢） |
| 对称加密 | 同一个密钥加密和解密，用于实际数据传输（快） |
| 证书 | 由 CA（证书颁发机构）签发，证明"这个公钥确实属于这个域名" |
| 证书链 | 根 CA → 中间 CA → 网站证书，逐级验证 |

## 五、Let's Encrypt 免费证书

### Let's Encrypt 简介

Let's Encrypt 是一个**免费、自动化、开放**的 CA 机构，由 Mozilla、EFF 等组织支持。

| 对比 | 传统付费证书 | Let's Encrypt |
|------|-------------|---------------|
| 费用 | 几百-几千元/年 | 免费 |
| 验证方式 | 邮件 / 人工审核 | 域名验证（自动化） |
| 有效期 | 1-2 年 | 90 天（自动续期） |
| 信任度 | 所有浏览器 | 所有现代浏览器 |
| 适用场景 | 企业级、EV 证书 | 个人项目、中小网站 |

### Certbot 安装和使用

Certbot 是 Let's Encrypt 官方的自动化工具。

```bash
# 安装 Certbot + Nginx 插件（Ubuntu）
sudo apt install certbot python3-certbot-nginx

# 一键申请证书并自动配置 Nginx
sudo certbot --nginx -d example.com -d www.example.com

# 手动续期测试
sudo certbot renew --dry-run

# Certbot 会自动添加 cron 任务，每天检查是否需要续期
# 证书到期前 30 天内自动续期
```

### Certbot 做了什么？

```bash
# certbot --nginx 执行的操作：
1. 在 Nginx 配置中添加临时 location 用于验证域名所有权
2. Let's Encrypt 服务器访问该 location 确认域名归属
3. 签发证书并保存到 /etc/letsencrypt/live/example.com/
4. 修改 Nginx 配置，添加 SSL 相关指令
5. reload Nginx
```

## 六、HTTPS Nginx 配置

### 完整配置示例

```nginx
# HTTP → HTTPS 重定向
server {
    listen 80;
    server_name example.com www.example.com;

    # 所有 HTTP 请求 301 永久重定向到 HTTPS
    return 301 https://$host$request_uri;
}

# HTTPS 主配置
server {
    listen 443 ssl;
    server_name example.com www.example.com;

    # SSL 证书路径（Certbot 自动生成的路径）
    ssl_certificate     /etc/letsencrypt/live/example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/example.com/privkey.pem;

    # SSL 安全配置
    ssl_protocols TLSv1.2 TLSv1.3;          # 只允许 TLS 1.2 和 1.3
    ssl_ciphers HIGH:!aNULL:!MD5;           # 高强度加密算法
    ssl_prefer_server_ciphers on;           # 优先使用服务器的加密算法

    # SSL 会话缓存（减少重复握手）
    ssl_session_cache shared:SSL:10m;
    ssl_session_timeout 10m;

    # HSTS：告诉浏览器"以后永远用 HTTPS 访问"
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;

    # 反向代理到 FastAPI
    location / {
        proxy_pass http://127.0.0.1:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    # 静态文件
    location /static/ {
        alias /var/www/static/;
        expires 30d;
    }
}
```

### HTTP 重定向到 HTTPS 的几种方式

```nginx
# 方式 1：return（推荐，最简单高效）
return 301 https://$host$request_uri;

# 方式 2：rewrite（灵活但稍慢）
rewrite ^(.*)$ https://$host$1 permanent;

# 方式 3：if 判断（不推荐，但某些场景需要）
if ($scheme = http) {
    return 301 https://$host$request_uri;
}
```

**301 vs 302**：
- 301（永久重定向）：浏览器会记住，下次直接访问 HTTPS
- 302（临时重定向）：浏览器每次都先访问 HTTP 再跳转

生产环境应使用 301。

## 七、SSL 安全加固

### 推荐的 SSL 配置

```nginx
# 禁止 SSLv3、TLSv1.0、TLSv1.1（都有已知漏洞）
ssl_protocols TLSv1.2 TLSv1.3;

# 使用强加密算法
ssl_ciphers 'ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256';

# OCSP Stapling：服务器代替客户端向 CA 验证证书吊销状态
ssl_stapling on;
ssl_stapling_verify on;
resolver 8.8.8.8 8.8.4.4 valid=300s;

# DH 参数（增强密钥交换安全性）
# 生成命令：openssl dhparam -out /etc/nginx/dhparam.pem 2048
ssl_dhparam /etc/nginx/dhparam.pem;
```

### SSL Labs 评分测试

访问 [SSL Labs](https://www.ssllabs.com/ssltest/) 输入域名，可以检测 SSL 配置安全性并给出 A+ 到 F 的评分。

目标：达到 **A** 或 **A+** 评级。

## 八、常见坑

### 1. 混合内容（Mixed Content）

HTTPS 页面中引用了 HTTP 资源，浏览器会阻止或警告。

```html
<!-- 错误：HTTPS 页面中引用 HTTP 资源 -->
<script src="http://cdn.example.com/jquery.js"></script>

<!-- 正确：使用 HTTPS 或协议相对路径 -->
<script src="https://cdn.example.com/jquery.js"></script>
<script src="//cdn.example.com/jquery.js"></script>
```

### 2. 证书域名不匹配

```
证书签发给 example.com，但用户访问 www.example.com
→ 浏览器报证书错误
```

解决：申请证书时同时包含两个域名：
```bash
sudo certbot --nginx -d example.com -d www.example.com
```

### 3. Certbot 续期失败

常见原因：
- 80 端口被占用（Certbot 需要 80 端口做验证）
- DNS 未正确指向服务器
- Nginx 配置语法错误导致 reload 失败

排查：
```bash
sudo certbot renew --dry-run    # 模拟续期，查看报错
sudo certbot certificates       # 查看已安装的证书
```

### 4. 静态文件 403 Forbidden

```bash
# 原因：Nginx 用户（nginx/www-data）没有文件读取权限
# 检查权限
ls -la /var/www/static/

# 修复
sudo chown -R www-data:www-data /var/www/static/
sudo chmod -R 755 /var/www/static/
```

### 5. gzip 不生效

排查步骤：
```bash
# 检查请求头是否包含 Accept-Encoding
curl -H "Accept-Encoding: gzip" -I http://example.com/style.css

# 检查响应头是否包含 Content-Encoding: gzip
# 如果没有，检查 gzip_types 是否包含该 MIME 类型
# 注意：文件太小（< gzip_min_length）也不会压缩
```

## 九、总结

| 知识点 | 核心要点 |
|--------|----------|
| 静态文件 | 用 `alias` 或 `root`，优先 Nginx 直接返回 |
| root vs alias | root 拼接完整路径，alias 替换匹配前缀 |
| gzip | 压缩文本文件，级别 6，不对图片压缩 |
| 缓存策略 | 带 hash 的资源长期缓存，HTML 不缓存 |
| ETag | 文件未变返回 304，节省带宽 |
| HTTPS | HTTP + TLS，解决窃听、篡改、冒充问题 |
| TLS 握手 | 非对称加密传递密钥，对称加密传输数据 |
| Let's Encrypt | 免费证书，90 天有效期，Certbot 自动续期 |
| HTTP→HTTPS | 使用 `return 301` 永久重定向 |
| SSL 加固 | TLSv1.2+，强加密算法，HSTS |
