# 负载均衡与高级配置

> 单实例扛不住时，用 Nginx 把流量分发到多台后端，并加上限流与缓存。

## 一、什么是负载均衡？

当单个后端服务器无法承受所有请求时，需要**多个服务器实例**共同分担流量。负载均衡器（Load Balancer）负责决定每个请求发给哪个后端实例。

```
                        ┌→ 后端 A (8001) ─── 处理请求
客户端 → Nginx (LB) ────┼→ 后端 B (8002) ─── 处理请求
                        └→ 后端 C (8003) ─── 处理请求
```

### 负载均衡的层次

| 层次 | 说明 | 典型工具 |
|------|------|----------|
| DNS 负载均衡 | 域名解析到多个 IP | AWS Route53、CloudFlare |
| 四层（TCP/UDP） | 按 IP+端口转发，不解析 HTTP | LVS、HAProxy (TCP mode) |
| 七层（HTTP） | 解析 HTTP 头/路径/Cookie 转发 | **Nginx**、HAProxy (HTTP mode) |

Nginx 的负载均衡工作在**七层（HTTP 层）**，可以根据 URL、Header、Cookie 等做精细化的路由决策。

## 二、负载均衡算法

Nginx 开源版支持四种负载均衡算法：

### 1. 轮询（Round Robin）—— 默认

```nginx
upstream backend {
    server 127.0.0.1:8001;
    server 127.0.0.1:8002;
    server 127.0.0.1:8003;
}
```

请求按顺序依次分发：A → B → C → A → B → C → ...

**优点**：简单公平
**适用**：所有后端实例配置相同、处理能力一致

### 2. 加权轮询（Weighted Round Robin）

```nginx
upstream backend {
    server 127.0.0.1:8001 weight=5;    # 权重 5，处理 50% 请求
    server 127.0.0.1:8002 weight=3;    # 权重 3，处理 30% 请求
    server 127.0.0.1:8003 weight=2;    # 权重 2，处理 20% 请求
}
```

分发比例约为 5:3:2。

**适用**：后端服务器配置不同（如 8001 是 8 核，8002 和 8003 是 4 核）

### 3. IP Hash

```nginx
upstream backend {
    ip_hash;                           # 根据客户端 IP 做哈希
    server 127.0.0.1:8001;
    server 127.0.0.1:8002;
    server 127.0.0.1:8003;
}
```

同一个客户端 IP 始终被分配到同一个后端实例。

**优点**：天然支持 Session 保持（无需 Session 共享）
**缺点**：新增/删除后端时，大量客户端会被重新分配（可用 `consistent hash` 缓解）

**适用**：使用内存 Session 且无法共享的应用

### 4. 最少连接（Least Connections）

```nginx
upstream backend {
    least_conn;                        # 分配给当前连接数最少的服务器
    server 127.0.0.1:8001;
    server 127.0.0.1:8002;
    server 127.0.0.1:8003;
}
```

每次把请求发给当前活跃连接数最少的后端。

**适用**：后端请求处理时间差异大（如有的请求处理 10ms，有的处理 5s）

### 算法对比

| 算法 | 配置关键字 | Session 保持 | 适用场景 |
|------|-----------|-------------|----------|
| 轮询 | （默认） | ❌ | 后端配置相同 |
| 加权 | `weight=N` | ❌ | 后端性能不同 |
| IP Hash | `ip_hash` | ✅ 按 IP | 需要 Session 粘性 |
| 最少连接 | `least_conn` | ❌ | 请求处理时间差异大 |

## 三、upstream 高级配置

### 服务器状态参数

```nginx
upstream backend {
    server 127.0.0.1:8001 weight=5 max_fails=3 fail_timeout=30s;
    server 127.0.0.1:8002 weight=3 max_fails=3 fail_timeout=30s;
    server 127.0.0.1:8003 backup;              # 备用服务器
    server 127.0.0.1:8004 down;                # 暂时下线
}
```

| 参数 | 说明 |
|------|------|
| `weight=N` | 权重，默认为 1 |
| `max_fails=N` | 在 `fail_timeout` 时间内允许的最大失败次数 |
| `fail_timeout=Ns` | 失败统计窗口 & 被标记为不可用后的等待时间 |
| `backup` | 备用服务器，只在其他服务器都不可用时才接收请求 |
| `down` | 标记为永久离线，不参与负载均衡 |

### 被动健康检查（开源版）

Nginx 开源版只支持**被动健康检查**：通过实际请求来发现后端故障。

```nginx
upstream backend {
    server 127.0.0.1:8001 max_fails=3 fail_timeout=30s;
    server 127.0.0.1:8002 max_fails=3 fail_timeout=30s;
}

server {
    location / {
        proxy_pass http://backend;
        proxy_next_upstream error timeout http_502 http_503;
        # 当后端返回 502/503 或超时/出错时，自动切换到下一个后端
    }
}
```

工作流程：
1. 请求发到 8001，返回 502
2. Nginx 自动将请求重试到 8002
3. 如果 8001 在 30 秒内失败 3 次，标记为不可用
4. 30 秒后重新尝试向 8001 发送请求

### 主动健康检查（Nginx Plus / 第三方模块）

```nginx
# Nginx Plus 专属功能（付费版）
upstream backend {
    server 127.0.0.1:8001;
    server 127.0.0.1:8002;
    health_check interval=5s fails=3 passes=2 uri=/health;
    # 每 5 秒主动请求 /health 接口检查后端状态
}
```

开源替代方案：使用 `nginx_upstream_check_module` 第三方模块。

## 四、限流和限并发

### 为什么需要限流？

- 防止恶意爬虫大量请求消耗资源
- 防止 DDoS 攻击
- 防止接口被滥用（如短信验证码接口）
- 保护后端服务不被突发流量压垮

### 限制请求速率（limit_req）

```nginx
http {
    # 定义限流区域：每个客户端 IP 每秒允许 10 个请求
    limit_req_zone $binary_remote_addr zone=api_limit:10m rate=10r/s;

    server {
        location /api/ {
            # burst=20：允许突发 20 个请求排队
            # nodelay：排队请求立即处理（不加则按速率限制匀速处理）
            limit_req zone=api_limit burst=20 nodelay;

            proxy_pass http://backend;
        }
    }
}
```

**参数解释**：

| 参数 | 说明 |
|------|------|
| `$binary_remote_addr` | 按客户端 IP 限流 |
| `zone=api_limit:10m` | 共享内存区域名称和大小（10MB ≈ 16 万个 IP） |
| `rate=10r/s` | 每秒 10 个请求（也可以用 `100r/m` 表示每分钟 100 个） |
| `burst=20` | 允许突发排队 20 个请求 |
| `nodelay` | 突发请求立即处理，不延迟 |

### 限制并发连接数（limit_conn）

```nginx
http {
    # 定义限并发区域：按客户端 IP 统计
    limit_conn_zone $binary_remote_addr zone=conn_limit:10m;

    server {
        location / {
            limit_conn conn_limit 10;    # 每个 IP 最多 10 个并发连接
            proxy_pass http://backend;
        }
    }
}
```

### 限流返回自定义错误页

```nginx
# 默认限流返回 503，可以自定义
limit_req_status 429;          # 改为 429 Too Many Requests
limit_conn_status 429;

# 自定义错误页
error_page 429 /429.html;
location = /429.html {
    root /var/www/error_pages;
    internal;                   # 只允许内部跳转
}
```

## 五、日志配置

### 默认日志格式

```nginx
# Nginx 内置的 combined 格式
log_format combined '$remote_addr - $remote_user [$time_local] '
                    '"$request" $status $body_bytes_sent '
                    '"$http_referer" "$http_user_agent"';
```

### 自定义 JSON 日志格式

```nginx
http {
    # 定义 JSON 格式日志（方便 ELK/Loki 等日志系统解析）
    log_format json_log escape=json '{'
        '"time": "$time_iso8601",'
        '"remote_addr": "$remote_addr",'
        '"request": "$request",'
        '"status": $status,'
        '"body_bytes_sent": $body_bytes_sent,'
        '"request_time": $request_time,'
        '"upstream_response_time": "$upstream_response_time",'
        '"http_user_agent": "$http_user_agent",'
        '"http_x_forwarded_for": "$http_x_forwarded_for"'
    '}';

    # 全局访问日志
    access_log /var/log/nginx/access.log json_log;
}
```

### 按站点配置日志

```nginx
server {
    server_name api.example.com;

    access_log /var/log/nginx/api_access.log json_log;
    error_log  /var/log/nginx/api_error.log warn;

    # 静态文件请求不记录日志（减少磁盘 IO）
    location /static/ {
        access_log off;
        alias /var/www/static/;
    }

    # 健康检查接口不记录日志
    location = /health {
        access_log off;
        return 200 "OK";
    }
}
```

### 常用日志变量

| 变量 | 说明 | 示例 |
|------|------|------|
| `$remote_addr` | 客户端 IP | `203.0.113.1` |
| `$request` | 完整请求行 | `GET /api/users HTTP/1.1` |
| `$status` | 响应状态码 | `200` |
| `$request_time` | 请求处理总时间（秒） | `0.032` |
| `$upstream_response_time` | 后端响应时间 | `0.028` |
| `$body_bytes_sent` | 发送给客户端的字节数 | `1234` |
| `$http_user_agent` | 客户端 User-Agent | `Mozilla/5.0 ...` |

## 六、WebSocket 代理

### WebSocket 的特殊性

WebSocket 是长连接协议，基于 HTTP Upgrade 机制建立：

```
客户端 → 服务器：GET /ws HTTP/1.1 + Upgrade: websocket
服务器 → 客户端：101 Switching Protocols
（之后双方直接通过 TCP 双向通信）
```

Nginx 默认不支持 WebSocket，需要显式配置 Upgrade 头。

### 配置

```nginx
# 定义 WebSocket 后端的连接升级映射
map $http_upgrade $connection_upgrade {
    default upgrade;
    ''      close;
}

upstream websocket_backend {
    server 127.0.0.1:8001;
    server 127.0.0.1:8002;
    # WebSocket 是长连接，建议使用 ip_hash 保持粘性
    ip_hash;
}

server {
    listen 80;
    server_name ws.example.com;

    location /ws/ {
        proxy_pass http://websocket_backend;
        proxy_http_version 1.1;             # WebSocket 需要 HTTP 1.1

        # 关键：传递 Upgrade 和 Connection 头
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection $connection_upgrade;

        # WebSocket 长连接超时设置
        proxy_read_timeout 3600s;           # 默认 60s 太短，长连接会被断开
        proxy_send_timeout 3600s;

        # 传递真实客户端信息
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}
```

### FastAPI WebSocket 示例

```python
from fastapi import FastAPI, WebSocket

app = FastAPI()

@app.websocket("/ws/chat")
async def chat(websocket: WebSocket):
    await websocket.accept()
    try:
        while True:
            data = await websocket.receive_text()
            await websocket.send_text(f"Echo: {data}")
    except Exception:
        pass  # 客户端断开连接
```

## 七、其他高级配置

### 按路径分流到不同后端

```nginx
upstream api_backend {
    server 127.0.0.1:8001;
}

upstream admin_backend {
    server 127.0.0.1:9001;
}

upstream static_backend {
    server 127.0.0.1:7001;
}

server {
    listen 80;
    server_name example.com;

    location /api/ {
        proxy_pass http://api_backend;
    }

    location /admin/ {
        proxy_pass http://admin_backend;
    }

    location /static/ {
        proxy_pass http://static_backend;
    }
}
```

### 跨域配置（CORS）

```nginx
location /api/ {
    # 添加 CORS 响应头
    add_header Access-Control-Allow-Origin "https://frontend.example.com" always;
    add_header Access-Control-Allow-Methods "GET, POST, PUT, DELETE, OPTIONS" always;
    add_header Access-Control-Allow-Headers "Authorization, Content-Type" always;
    add_header Access-Control-Allow-Credentials "true" always;

    # OPTIONS 预检请求直接返回 204
    if ($request_method = OPTIONS) {
        return 204;
    }

    proxy_pass http://backend;
}
```

**注意**：CORS 最好在 FastAPI 层用 `CORSMiddleware` 处理，Nginx 层做 CORS 容易和应用的 CORS 配置冲突，导致重复的 `Access-Control-Allow-Origin` 头。

### 请求体大小限制

```nginx
# 默认 1MB，上传文件时经常不够
client_max_body_size 50m;    # 允许 50MB 上传

# 如果要允许任意大小（不推荐）
client_max_body_size 0;
```

## 八、常见坑

### 1. upstream 中 server 写了 localhost

```nginx
# 可能有问题：localhost 可能解析为 IPv6 ::1
upstream backend {
    server localhost:8000;       # ❌ 可能解析为 [::1]:8000
}

# 推荐：明确使用 127.0.0.1
upstream backend {
    server 127.0.0.1:8000;       # ✅ 明确 IPv4
}
```

### 2. WebSocket 60 秒后断开

```nginx
# 默认 proxy_read_timeout 是 60s
# WebSocket 长连接必须加大这个值
proxy_read_timeout 3600s;
```

### 3. 限流配置不生效

常见原因：
- `limit_req_zone` 必须放在 `http` 块中，不能放在 `server` 或 `location` 中
- 没加 `burst` 参数，导致正常突发请求也被拒绝
- 被 CDN / 反向代理覆盖了真实 IP，所有请求看起来来自同一个 IP

### 4. 负载均衡后 Session 丢失

```
用户请求 1 → 后端 A（创建 Session）
用户请求 2 → 后端 B（没有这个 Session → 要求重新登录）
```

解决方案：
1. **IP Hash**：同一 IP 始终到同一后端
2. **Session 共享**：用 Redis 存储 Session（推荐）
3. **JWT Token**：无状态认证，不依赖 Session

### 5. 配置 reload 后旧 worker 进程不退出

```bash
# 查看 Nginx 进程
ps aux | grep nginx

# 如果有大量旧 worker 进程，可能是因为旧请求还在处理
# 设置 worker_shutdown_timeout 强制退出时间
worker_shutdown_timeout 10s;
```

### 6. 健康检查误判

```nginx
# fail_timeout 太短会导致偶发超时就把后端标记为不可用
server 127.0.0.1:8001 max_fails=3 fail_timeout=30s;
# 推荐：至少 3 次失败 + 30 秒窗口
```

## 九、总结

| 知识点 | 核心要点 |
|--------|----------|
| 负载均衡算法 | 轮询、加权、IP Hash、最少连接 |
| upstream 参数 | weight / max_fails / fail_timeout / backup / down |
| 被动健康检查 | 通过 `proxy_next_upstream` + `max_fails` 实现 |
| 限流 | `limit_req_zone` + `limit_req`（按 IP、按速率） |
| 限并发 | `limit_conn_zone` + `limit_conn` |
| 日志 | 自定义 JSON 格式便于日志系统分析 |
| WebSocket | 需要 `proxy_http_version 1.1` + Upgrade 头 |
| Session 问题 | 用 Redis 共享 Session 或 JWT 无状态认证 |
