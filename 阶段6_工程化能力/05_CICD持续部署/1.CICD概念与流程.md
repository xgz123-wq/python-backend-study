# CI/CD 概念与流程

> CI 自动测、CD 自动发——把"提交代码到上线"这条链路从手工变全自动。

## 1. 概念解释

### CI（Continuous Integration，持续集成）

持续集成是指**开发人员频繁地将代码合并到主干分支**（通常每天多次），每次合并都会自动触发一系列操作：

- **代码拉取**：从版本控制系统获取最新代码
- **依赖安装**：自动安装项目所需的第三方库
- **静态检查**：对代码风格、类型注解等进行 lint 检查
- **自动测试**：运行单元测试、集成测试，确保新代码没有破坏已有功能

**核心目标**：尽早发现问题。当问题刚引入时就捕获它，修复成本最低。

### CD（Continuous Delivery / Continuous Deployment）

CD 实际上包含两个层次：

| 层次 | 英文 | 含义 |
|------|------|------|
| 持续交付 | Continuous Delivery | 代码始终处于"可发布"状态，但**发布到生产环境需要人工审批** |
| 持续部署 | Continuous Deployment | 代码通过所有测试后**自动部署到生产环境**，全程无人工干预 |

两者的区别在于最后一步是否需要人工点击"发布"按钮。持续部署是持续交付的进一步自动化。

### CI/CD 合在一起

CI/CD 就是将从代码提交到生产部署的整个流程**自动化**：

```
代码提交 → 自动测试 → 构建打包 → 预发布验证 → 生产部署
```

## 2. 为什么需要 CI/CD

### 没有 CI/CD 的痛苦

想象一个没有 CI/CD 的团队：

1. 开发人员 A 写完代码，手动跑了一下测试，"好像没问题"
2. 通过 U 盘或手动 scp 把代码传到服务器
3. 手动执行 `pip install -r requirements.txt`，发现某个依赖版本冲突
4. 手动启动服务，发现端口被占用
5. 上线后发现 bug，需要回滚，但忘了上一个稳定版本是哪个 commit
6. 整个发布过程耗时 2 小时，团队提心吊胆

### CI/CD 解决的问题

| 问题 | CI/CD 如何解决 |
|------|---------------|
| 手动操作容易出错 | 全流程自动化，消除人为失误 |
| 发布周期长 | 自动化流水线，分钟级完成 |
| bug 发现晚 | 每次提交都跑测试，即时发现 |
| 回滚困难 | 每次构建都有记录，一键回滚 |
| 环境不一致 | 统一的构建环境，保证一致性 |
| 团队协作冲突 | 强制代码审查 + 自动化验证 |
| 部署知识集中在个别人 | 流程文档化在 Pipeline 配置中 |

### 量化收益

根据 DORA（DevOps Research and Assessment）报告：

- 使用 CI/CD 的团队，**部署频率高 208 倍**
- **变更前置时间**（从代码提交到生产部署）快 106 倍
- **变更失败率**低 7 倍
- **故障恢复时间**快 2604 倍

## 3. 实际场景

### 场景 1：Python 后端 API 项目

```
开发人员 git push
    ↓
GitHub Actions 自动触发
    ↓
┌─────────────────────────────────────────┐
│  Job 1: 测试                            │
│  ├── checkout 代码                      │
│  ├── 安装 Python 3.11                   │
│  ├── pip install -r requirements.txt    │
│  ├── flake8 代码风格检查                 │
│  ├── pytest 运行单元测试                 │
│  └── pytest --cov 生成覆盖率报告         │
└─────────────────────────────────────────┘
    ↓ 测试通过
┌─────────────────────────────────────────┐
│  Job 2: 构建                            │
│  ├── Docker build 构建镜像               │
│  ├── 打标签: app:v1.2.3-commit_sha      │
│  └── 推送到容器镜像仓库                   │
└─────────────────────────────────────────┘
    ↓ 构建成功
┌─────────────────────────────────────────┐
│  Job 3: 部署到预发布环境                  │
│  ├── SSH 连接预发布服务器                 │
│  ├── docker pull 最新镜像                │
│  ├── docker-compose up -d               │
│  └── 运行冒烟测试                        │
└─────────────────────────────────────────┘
    ↓ 预发布验证通过
┌─────────────────────────────────────────┐
│  Job 4: 部署到生产环境                    │
│  ├── SSH 连接生产服务器                   │
│  ├── docker pull 镜像                    │
│  ├── docker-compose up -d               │
│  ├── 健康检查: curl /health              │
│  └── 失败则自动回滚到上一版本             │
└─────────────────────────────────────────┘
```

### 场景 2：多人协作的开源项目

- 有人提交 Pull Request → 自动运行测试 → 维护者看到绿色勾才合并
- 合并到 main 分支 → 自动构建并发布到 PyPI
- 定时任务（cron）→ 每天凌晨跑一次全量测试

### 场景 3：微服务架构

- 每个服务有独立的 CI/CD Pipeline
- 服务 A 部署完成后，自动触发依赖它的服务 B 的集成测试
- 通过 Git tag 触发特定服务的发布

## 4. 核心概念详解

### Pipeline（流水线）

Pipeline 是整个 CI/CD 流程的**顶层容器**，定义在配置文件（如 `.github/workflows/ci.yml`）中。一个 Pipeline 代表一个完整的自动化流程。

### Trigger（触发器）

定义什么事件会启动 Pipeline：

| 触发器 | 说明 | 典型用法 |
|--------|------|---------|
| `push` | 代码推送时触发 | 提交代码后自动测试 |
| `pull_request` | PR 创建/更新时触发 | PR 合并前自动验证 |
| `schedule` | 定时触发（cron 表达式） | 每日凌晨跑回归测试 |
| `workflow_dispatch` | 手动触发 | 紧急部署或调试 |
| `release` | 发布版本时触发 | 发布到 PyPI |

### Job（作业）

Job 是 Pipeline 中的一个**独立执行单元**。一个 Pipeline 可以包含多个 Job：

- 多个 Job **默认并行执行**
- 可以通过 `needs` 关键字设置**依赖关系**，实现串行执行
- 每个 Job 运行在独立的 Runner（执行环境）上

```yaml
jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - run: pytest

  deploy:
    needs: test          # deploy 必须等 test 完成后才执行
    runs-on: ubuntu-latest
    steps:
      - run: deploy.sh
```

### Step（步骤）

Step 是 Job 中的**最小执行单元**，每个 Step 要么是一个 shell 命令（`run`），要么是一个预定义的 Action（`uses`）：

```yaml
steps:
  - name: 检出代码        # Step 1: 使用预定义 Action
    uses: actions/checkout@v4

  - name: 运行测试        # Step 2: 执行 shell 命令
    run: pytest tests/
```

### Runner（执行器）

Runner 是运行 Job 的虚拟机或容器：

- **GitHub-hosted runner**：GitHub 提供的云端虚拟机（ubuntu-latest, windows-latest 等）
- **Self-hosted runner**：你自己搭建的服务器，适合需要特定环境的场景

### Artifact（产物）

Job 执行后生成的文件（测试报告、构建产物等），可以在不同 Job 之间传递：

```yaml
- uses: actions/upload-artifact@v4
  with:
    name: test-results
    path: reports/

# 在另一个 Job 中下载
- uses: actions/download-artifact@v4
  with:
    name: test-results
```

## 5. CI/CD 工具对比

| 特性 | GitHub Actions | GitLab CI | Jenkins |
|------|---------------|-----------|---------|
| 托管方式 | GitHub 云端 | GitLab 云端/自托管 | 完全自托管 |
| 配置格式 | YAML | YAML | Groovy / Jenkinsfile |
| 学习曲线 | 低 | 中 | 高 |
| 生态市场 | Actions Marketplace 丰富 | 内置模板丰富 | 插件生态庞大但老旧 |
| 与代码仓库集成 | 原生 GitHub 集成 | 原生 GitLab 集成 | 需要额外配置 |
| 免费额度 | 公开仓库免费，私有仓库 2000 分钟/月 | 免费 400 分钟/月 | 完全免费（自付费服务器） |
| 适合场景 | 中小型项目、开源项目 | 全规模项目 | 大型企业、复杂流水线 |
| Runner 管理 | 云端免运维 | 支持自托管 | 完全自主管理 |

### 为什么推荐 GitHub Actions

对于 Python 后端学习者：

1. **零基础设施**：不需要搭建服务器，GitHub 账号即可使用
2. **YAML 配置简单**：比起 Jenkins 的 Groovy 脚本，YAML 更易读
3. **Actions 市场**：大量社区贡献的现成 Action，拿来就用
4. **与 GitHub 深度集成**：PR 状态检查、Issue 自动化、Package Registry
5. **免费额度充足**：个人学习完全够用

## 6. 常见坑

### 坑 1：把 CI/CD 当成"万能药"

**误区**："上了 CI/CD 就不用管代码质量了。"

**现实**：CI/CD 只是工具。如果你的测试覆盖率只有 10%，CI 跑出来的绿色勾毫无意义。CI/CD 的效果取决于你写了多少有意义的测试。

### 坑 2：Pipeline 运行时间过长

**现象**：一次 Pipeline 跑 30 分钟，开发人员开始绕过测试直接部署。

**解决**：
- 使用缓存（cache）避免重复安装依赖
- 将大型测试套件拆分为并行 Job
- 区分快速冒烟测试和完整回归测试

### 坑 3：Secrets 硬编码

**现象**：把数据库密码、API Key 写在代码或配置文件里。

**正确做法**：使用平台提供的 Secrets 管理功能（GitHub → Settings → Secrets），通过环境变量注入：

```yaml
env:
  DATABASE_URL: ${{ secrets.DATABASE_URL }}
```

### 坑 4：忽略构建产物的一致性

**现象**："在我机器上能跑" → CI 环境和本地环境不一致。

**解决**：使用 Docker 固化构建环境，或者在 CI 中严格锁定 Python 版本和依赖版本。

### 坑 5：部署没有回滚机制

**现象**：新版本上线出问题，团队手忙脚乱找上一个稳定版本。

**解决**：
- 每次构建打上唯一的 tag（如 git commit SHA）
- 保留最近 N 个版本的镜像
- 部署脚本中包含自动回滚逻辑

### 坑 6：所有 Job 串行执行

**现象**：测试、lint、构建、安全扫描全部串行，Pipeline 非常慢。

**解决**：无依赖关系的 Job 并行执行：

```yaml
jobs:
  lint:
    runs-on: ubuntu-latest
    steps: [...]

  test:
    runs-on: ubuntu-latest
    steps: [...]

  build:
    needs: [lint, test]    # 等 lint 和 test 都完成
    runs-on: ubuntu-latest
    steps: [...]
```

### 坑 7：不监控 Pipeline 本身

**现象**：Pipeline 悄悄失败了，没人发现。

**解决**：配置通知（Slack、邮件、钉钉），Pipeline 失败时自动告警。

## 7. 总结

CI/CD 的本质是**用自动化取代人工操作，用代码定义部署流程**：

- **CI** 保证每次代码变更都经过验证
- **CD** 保证验证通过的代码能快速、安全地到达生产环境
- **Pipeline** 是整个流程的"代码化"表达，版本化、可审计、可复用
