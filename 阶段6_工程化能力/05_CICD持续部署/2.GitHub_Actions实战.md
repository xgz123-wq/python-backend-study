# GitHub Actions 实战

> 一份 YAML 工作流跑通"push 触发 → 测试 → 构建镜像 → 部署"全流程。

## 1. 概念解释

GitHub Actions 是 GitHub 内置的 CI/CD 平台，通过 YAML 配置文件定义自动化工作流，直接托管在代码仓库中。它让你无需额外搭建 Jenkins 等工具，即可实现从代码提交到生产部署的全流程自动化。

### 核心架构

```
┌─────────────────────────────────────────────────┐
│                  Workflow                         │
│  （定义在 .github/workflows/*.yml 中）            │
│                                                   │
│  ┌─────────── Trigger（触发器）                   │
│  │   push / pull_request / schedule / manual     │
│  │                                               │
│  │  ┌─────────── Job 1 ──────────────────────┐   │
│  │  │  runs-on: ubuntu-latest                │   │
│  │  │  ┌─────────── Step 1 ──────────────┐   │   │
│  │  │  │  uses: actions/checkout@v4      │   │   │
│  │  │  └─────────────────────────────────┘   │   │
│  │  │  ┌─────────── Step 2 ──────────────┐   │   │
│  │  │  │  run: pytest                    │   │   │
│  │  │  └─────────────────────────────────┘   │   │
│  │  └────────────────────────────────────────┘   │
│  │                                               │
│  │  ┌─────────── Job 2 (needs: Job1) ────────┐   │
│  │  │  runs-on: ubuntu-latest                │   │
│  │  │  ┌─────────── Step 1 ──────────────┐   │   │
│  │  │  │  run: docker build ...          │   │   │
│  │  │  └─────────────────────────────────┘   │   │
│  │  └────────────────────────────────────────┘   │
│  └───────────────────────────────────────────────┘
│                                                   │
│  ┌─────────── Runner（执行环境）                   │
│  │  GitHub 托管的虚拟机 / 自托管服务器              │
│  └───────────────────────────────────────────────┘
└─────────────────────────────────────────────────┘
```

### 五大核心组件

| 组件 | 说明 | 类比 |
|------|------|------|
| **Workflow** | 定义在 `.github/workflows/` 下的 YAML 文件 | 菜谱 |
| **Event** | 触发 Workflow 的事件（push, PR 等） | 点菜 |
| **Job** | Workflow 中的一个独立任务 | 一道菜的制作过程 |
| **Step** | Job 中的单个操作步骤 | 切菜、炒菜、装盘 |
| **Runner** | 执行 Job 的虚拟机环境 | 厨房 |

## 2. 为什么需要 GitHub Actions

### 相比其他方案的优势

- **零配置启动**：不需要搭建服务器，在 GitHub 仓库里直接写 YAML 就能用
- **与 GitHub 深度集成**：PR 状态检查、Issue 自动化、GitHub Packages 原生支持
- **Actions 市场**：数万个社区贡献的 Action，覆盖常见操作
- **按需计费**：公开仓库完全免费，私有仓库有 2000 分钟/月免费额度
- **矩阵构建**：轻松测试多版本 Python、多操作系统的兼容性

## 4. 核心用法

### 4.1 Workflow YAML 语法详解

一个完整的 Workflow 文件结构：

```yaml
# 文件位置：.github/workflows/ci.yml
name: CI/CD Pipeline              # Workflow 名称，显示在 GitHub Actions 页面

on:                               # 触发条件
  push:                           # 代码推送时触发
    branches: [main, develop]     # 只监听指定分支
    paths:                        # 只监听指定路径的文件变更
      - 'src/**'
      - 'tests/**'
  pull_request:                   # PR 创建或更新时触发
    branches: [main]
  schedule:                       # 定时触发（UTC 时间）
    - cron: '0 2 * * *'           # 每天凌晨 2 点
  workflow_dispatch:              # 允许手动触发
    inputs:
      environment:                # 手动触发时可选择的参数
        description: '部署环境'
        required: true
        default: 'staging'
        type: choice
        options: [staging, production]

env:                              # 全局环境变量（所有 Job 共享）
  PYTHON_VERSION: '3.11'
  REGISTRY: ghcr.io

jobs:                             # 定义所有 Job
  test:                           # Job ID（自定义）
    name: Run Tests               # Job 显示名称
    runs-on: ubuntu-latest        # Runner 环境
    timeout-minutes: 10           # 超时时间

    steps:                        # Job 包含的步骤
      - name: Checkout code       # Step 1: 检出代码
        uses: actions/checkout@v4 # 使用预定义 Action

      - name: Set up Python       # Step 2: 设置 Python 环境
        uses: actions/setup-python@v5
        with:
          python-version: ${{ env.PYTHON_VERSION }}
          cache: 'pip'            # 自动缓存 pip 依赖

      - name: Install dependencies  # Step 3: 安装依赖
        run: |
          python -m pip install --upgrade pip
          pip install -r requirements.txt
          pip install pytest pytest-cov flake8

      - name: Lint with flake8    # Step 4: 代码检查
        run: flake8 src/ --count --max-line-length=120

      - name: Run tests           # Step 5: 运行测试
        run: pytest tests/ -v --cov=src --cov-report=xml

      - name: Upload coverage     # Step 6: 上传覆盖率报告
        uses: actions/upload-artifact@v4
        with:
          name: coverage-report
          path: coverage.xml
```

### 4.2 常用触发条件详解

```yaml
on:
  # 1. push 触发 — 最常用
  push:
    branches:                     # 分支过滤
      - main                      # 精确匹配
      - 'release/**'              # 通配符匹配
    tags:
      - 'v*'                      # tag 推送时触发（如 v1.0.0）
    paths:
      - 'src/**'                  # 路径过滤
      - '!docs/**'                # 排除 docs 目录（感叹号取反）

  # 2. pull_request 触发
  pull_request:
    types: [opened, synchronize]  # 只在 PR 创建和更新时触发
                                  # 默认: opened, synchronize, reopened

  # 3. schedule 触发（cron 表达式）
  schedule:
    - cron: '30 5,17 * * *'       # 每天 5:30 和 17:30 UTC
    # 格式: 分 时 日 月 周几
    # * = 每个值
    # , = 列举多个值
    # - = 范围

  # 4. workflow_dispatch 手动触发
  workflow_dispatch:
    inputs:
      log_level:
        description: '日志级别'
        required: false
        default: 'warning'
        type: choice
        options: [debug, info, warning, error]
```

### 4.3 常用预定义 Actions

```yaml
# 检出代码（几乎所有 Workflow 的第一步）
uses: actions/checkout@v4

# 设置 Python 环境
uses: actions/setup-python@v5
with:
  python-version: '3.11'

# 缓存依赖（加速构建）
uses: actions/cache@v4
with:
  path: ~/.cache/pip
  key: ${{ runner.os }}-pip-${{ hashFiles('**/requirements.txt') }}
  restore-keys: |
    ${{ runner.os }}-pip-

# 上传/下载构建产物
uses: actions/upload-artifact@v4
uses: actions/download-artifact@v4

# Docker 登录和构建
uses: docker/login-action@v3
uses: docker/build-push-action@v5

# SSH 远程执行命令
uses: appleboy/ssh-action@v1
with:
  host: ${{ secrets.SERVER_HOST }}
  username: ${{ secrets.SERVER_USER }}
  key: ${{ secrets.SSH_PRIVATE_KEY }}
  script: |
    cd /app && docker-compose pull && docker-compose up -d
```

### 4.4 环境变量和 Secrets 管理

```yaml
# 三种环境变量作用域
env:                              # 全局（所有 Job 共享）
  GLOBAL_VAR: "hello"

jobs:
  job1:
    env:                          # Job 级别（当前 Job 内所有 Step 共享）
      JOB_VAR: "world"
    steps:
      - name: 使用环境变量
        env:                      # Step 级别（仅当前 Step）
          STEP_VAR: "foo"
        run: |
          echo $GLOBAL_VAR
          echo $JOB_VAR
          echo $STEP_VAR

      - name: 动态设置变量（跨 Step 传递）
        run: |
          echo "BUILD_VERSION=1.0.$(date +%s)" >> $GITHUB_ENV
      - name: 使用上一步设置的变量
        run: echo "Version is $BUILD_VERSION"
```

**Secrets 管理最佳实践**：

1. 在 GitHub 仓库 → Settings → Secrets and variables → Actions 中添加
2. 支持三级作用域：Organization → Repository → Environment
3. 使用 Environment secrets 区分 staging 和 production 的凭证
4. 日志中自动遮蔽 secret 值（显示为 `***`）

```yaml
# 使用 Environment 保护生产部署
jobs:
  deploy-prod:
    environment:
      name: production
      url: https://myapp.com
    steps:
      - run: deploy.sh
        env:
          # 只有审批通过后才能访问 production 环境的 secrets
          DB_PASSWORD: ${{ secrets.PROD_DB_PASSWORD }}
          API_KEY: ${{ secrets.PROD_API_KEY }}
```

### 4.5 矩阵构建（多版本测试）

矩阵构建让你在一组不同的配置上运行同一个 Job：

```yaml
jobs:
  test:
    strategy:
      fail-fast: false            # 一个失败不取消其他
      matrix:
        python-version: ['3.10', '3.11', '3.12']
        os: [ubuntu-latest, windows-latest]
        exclude:                  # 排除某些组合
          - python-version: '3.10'
            os: windows-latest
        include:                  # 添加额外组合
          - python-version: '3.13'
            os: ubuntu-latest
            experimental: true

    runs-on: ${{ matrix.os }}
    continue-on-error: ${{ matrix.experimental || false }}

    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: ${{ matrix.python-version }}
      - run: pytest tests/ -v
```

上述配置会生成以下组合：
- Python 3.10 on Ubuntu
- Python 3.11 on Ubuntu
- Python 3.11 on Windows
- Python 3.12 on Ubuntu
- Python 3.12 on Windows
- Python 3.13 on Ubuntu（实验性，失败不阻塞）

### 4.6 Job 间依赖与数据传递

```yaml
jobs:
  test:
    runs-on: ubuntu-latest
    outputs:                      # 定义 Job 输出
      test-result: ${{ steps.test-step.outputs.result }}
    steps:
      - id: test-step
        run: echo "result=passed" >> $GITHUB_OUTPUT

  build:
    needs: test                   # 依赖 test Job
    runs-on: ubuntu-latest
    steps:
      - run: echo "Test result was ${{ needs.test.outputs.test-result }}"

  deploy:
    needs: [test, build]          # 依赖多个 Job
    if: github.ref == 'refs/heads/main'  # 条件执行
    runs-on: ubuntu-latest
    steps:
      - run: echo "Deploying..."
```

### 4.7 实战：Python 后端完整 CI/CD 配置

```yaml
name: Python Backend CI/CD

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

env:
  REGISTRY: ghcr.io
  IMAGE_NAME: ${{ github.repository }}

jobs:
  # ========== 测试阶段 ==========
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-python@v5
        with:
          python-version: '3.11'
          cache: 'pip'

      - name: Install dependencies
        run: |
          pip install -r requirements.txt
          pip install pytest pytest-cov httpx

      - name: Run tests
        run: pytest tests/ -v --cov=app --cov-report=term-missing
        env:
          DATABASE_URL: sqlite:///./test.db
          SECRET_KEY: test-secret-key

  # ========== 构建阶段 ==========
  build:
    needs: test
    if: github.event_name == 'push' && github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write

    steps:
      - uses: actions/checkout@v4

      - name: Log in to Container Registry
        uses: docker/login-action@v3
        with:
          registry: ${{ env.REGISTRY }}
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Build and push Docker image
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: |
            ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:latest
            ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}:${{ github.sha }}

  # ========== 部署阶段 ==========
  deploy:
    needs: build
    runs-on: ubuntu-latest
    environment:
      name: production
      url: https://api.myapp.com

    steps:
      - name: Deploy via SSH
        uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.SERVER_HOST }}
          username: ${{ secrets.SERVER_USER }}
          key: ${{ secrets.SSH_PRIVATE_KEY }}
          script: |
            cd /opt/app
            docker compose pull
            docker compose up -d
            sleep 5
            curl -f http://localhost:8000/health || exit 1
```

## 5. 常见坑

### 坑 1：YAML 缩进错误

YAML 对缩进非常敏感，**必须使用空格，不能用 Tab**。常见错误：

```yaml
# 错误！混用 Tab 和空格
steps:
    - name: Test
	  run: pytest        # Tab 缩进会报错

# 正确：统一 2 空格缩进
steps:
  - name: Test
    run: pytest
```

### 坑 2：忘记 checkout 代码

每个 Job 运行在独立的 Runner 上，**不会自动包含你的代码**。第一步几乎总是：

```yaml
- uses: actions/checkout@v4
```

忘记这一步会导致 `pytest: command not found` 或 `No such file or directory` 等莫名其妙的错误。

### 坑 3：Secrets 在 PR 中不可用

出于安全考虑，来自 fork 仓库的 Pull Request **无法访问仓库 Secrets**。如果你的测试依赖数据库密码等敏感信息，需要：

- 使用测试专用的 mock 数据
- 用 `if: github.event_name != 'pull_request'` 跳过依赖 secrets 的步骤
- 使用 Environment protection rules 控制访问

### 坑 4：缓存 key 不正确

缓存的 key 需要包含依赖文件的 hash，否则依赖更新后仍然使用旧缓存：

```yaml
# 错误：固定 key，依赖更新后不会刷新缓存
key: pip-cache

# 正确：依赖文件变更时自动刷新
key: ${{ runner.os }}-pip-${{ hashFiles('requirements.txt') }}
restore-keys: |
  ${{ runner.os }}-pip-
```

### 坑 5：Docker 构建上下文过大

没有 `.dockerignore` 文件会导致整个项目目录（包括 `.git`、`__pycache__`、`node_modules`）被发送到 Docker daemon，构建极慢。

### 坑 6：不设置超时时间

一个挂起的 Job 可能白白消耗几十分钟的 Runner 时间（和额度）：

```yaml
jobs:
  test:
    timeout-minutes: 15    # 设置合理超时
```

### 坑 7：条件表达式语法

`if` 条件中引用 context 变量时不需要 `${{ }}`，但也可以写：

```yaml
# 两种写法都正确
if: github.ref == 'refs/heads/main'
if: ${{ github.ref == 'refs/heads/main' }}

# 但字符串比较要注意类型
if: github.event.inputs.deploy == 'true'   # 字符串比较
if: github.event.inputs.deploy == true      # 布尔比较（可能不等价）
```

### 坑 8：并发冲突

多人同时 push 可能导致多个 Workflow 同时运行，争抢资源或部署冲突：

```yaml
# 同一分支只保留最新的一个 Workflow 运行
concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true
```

## 6. 调试技巧

### 启用调试日志

在 Workflow 的 Secrets 中添加 `ACTIONS_RUNNER_DEBUG: true` 和 `ACTIONS_STEP_DEBUG: true`，可以看到详细的执行日志。

### 使用 act 本地测试

[act](https://github.com/nektos/act) 工具可以在本地用 Docker 运行 GitHub Actions Workflow：

```bash
# 安装 act
brew install act    # macOS
# 或从 GitHub releases 下载

# 本地运行 Workflow
act push
act -j test         # 只运行指定 Job
```

### 手动触发调试

```yaml
on:
  workflow_dispatch:    # 添加手动触发
```

在 GitHub 仓库 → Actions → 选择 Workflow → Run workflow，可以手动传入参数并观察执行过程。

---

## 7. 总结

GitHub Actions 的核心优势是**简单、集成度高、生态丰富**：

- **YAML 配置**：声明式定义，易读易维护
- **Events 触发**：灵活响应代码变更、定时任务、手动操作
- **Jobs 编排**：并行 / 串行灵活组合
- **Actions 市场**：开箱即用的组件
- **Secrets 管理**：安全的敏感信息处理

掌握 GitHub Actions 后，你将能够用几十行 YAML 配置，搭建一个专业级的 CI/CD 流水线。
