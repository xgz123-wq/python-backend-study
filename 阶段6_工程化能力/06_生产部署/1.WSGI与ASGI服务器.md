# WSGI 与 ASGI 服务器

> Gunicorn 跑同步框架、Uvicorn 跑异步框架，这是 Python 上生产的标准前置。

## 1. 概念解释

### WSGI（Web Server Gateway Interface）

WSGI 是 Python 定义的 Web 服务器与 Web 应用之间的标准接口（PEP 3333）。它规定了：

- **服务器**如何调用**应用**（传入 `environ` 字典和 `start_response` 回调）
- **应用**如何返回**响应**（返回可迭代的字节串）

WSGI 是**同步**的：每个请求占用一个线程/进程，请求处理完毕后才释放。

```python
# WSGI 应用的标准签名
def application(environ, start_response):
    start_response('200 OK', [('Content-Type', 'text/plain')])
    return [b'Hello World']
```

### ASGI（Asynchronous Server Gateway Interface）

ASGI 是 WSGI 的异步继任者，由 Django Channels 团队提出。它支持：

- **异步请求处理**（`async def`）
- **WebSocket**、**长连接**、**后台任务**
- **双向通信**（服务器可以主动推送给客户端）

```python
# ASGI 应用的标准签名
async def application(scope, receive, send):
    await send({
        'type': 'http.response.start',
        'status': 200,
        'headers': [[b'content-type', b'text/plain']],
    })
    await send({
        'type': 'http.response.body',
        'body': b'Hello World',
    })
```

### 对比

| 特性 | WSGI | ASGI |
|------|------|------|
| 同步/异步 | 同步 | 异步 |
| WebSocket | 不支持 | 支持 |
| 适用框架 | Flask, Django（同步） | FastAPI, Django Channels |
| 典型服务器 | Gunicorn | Uvicorn |
| 并发模型 | 多进程/多线程 | 事件循环 + 协程 |

## 2. 为什么需要专用服务器

Python 内置的 `uvicorn`、`flask run` 等开发服务器：

- **不安全**：未经安全加固，容易被攻击
- **性能差**：单进程，无法利用多核 CPU
- **不稳定**：崩溃后无法自动重启
- **无监控**：缺少日志、指标、健康检查

生产环境需要专用的 **WSGI/ASGI 服务器**来处理：

- 多进程管理（充分利用 CPU）
- 进程监控与自动重启
- 优雅重启（零停机部署）
- 访问日志与错误日志
- 超时控制（防止请求卡死）

---

## 3. 实际场景

| 场景 | 推荐方案 |
|------|----------|
| FastAPI 项目（异步） | Gunicorn + Uvicorn Worker |
| Flask/Django（同步） | Gunicorn（sync worker） |
| 纯异步高性能场景 | 单独使用 Uvicorn（配合进程管理器） |
| 容器化部署 | 单 worker Uvicorn（由 K8s/Docker 管理进程） |

## 4. 核心用法

### 4.1 Uvicorn：ASGI 服务器

Uvicorn 是基于 `uvloop` 和 `httptools` 的高性能 ASGI 服务器。

```bash
# 安装
pip install uvicorn

# 开发环境启动（带热重载）
uvicorn app.main:app --reload --host 0.0.0.0 --port 8000

# 生产环境启动（不带 --reload）
uvicorn app.main:app --host 0.0.0.0 --port 8000 --workers 4
```

**关键参数**：
- `--host`：监听地址，`0.0.0.0` 表示接受所有来源
- `--port`：监听端口
- `--workers`：工作进程数（生产环境使用）
- `--reload`：**仅用于开发**，文件变更自动重启

### 4.2 Gunicorn：WSGI 服务器 + 进程管理器

Gunicorn（Green Unicorn）是 Python 最流行的 WSGI HTTP 服务器，核心功能是**进程管理**。

```bash
# 安装
pip install gunicorn

# 启动 Flask 应用（4 个 worker）
gunicorn -w 4 -b 0.0.0.0:8000 app:app

# 启动 Django 应用
gunicorn -w 4 -b 0.0.0.0:8000 myproject.wsgi:application
```

**Gunicorn 的架构**：

```
                    ┌──────────────────────┐
                    │     Master Process   │
                    │   (进程管理器)         │
                    └──────────┬───────────┘
                               │
              ┌────────────────┼────────────────┐
              │                │                │
        ┌─────▼──────┐  ┌─────▼──────┐  ┌─────▼──────┐
        │  Worker 1  │  │  Worker 2  │  │  Worker 3  │
        │ (处理请求)  │  │ (处理请求)  │  │ (处理请求)  │
        └────────────┘  └────────────┘  └────────────┘
```

Master 进程负责：
- 启动/停止 worker
- 监控 worker 健康状态
- worker 崩溃时自动重启
- 优雅重启（接收 HUP 信号）

### 4.3 Gunicorn + Uvicorn Worker（最佳组合）

对于 FastAPI 等 ASGI 框架，推荐使用 Gunicorn 作为进程管理器，Uvicorn 作为 worker：

```bash
# 安装
pip install gunicorn uvicorn

# 启动（Gunicorn 管进程，Uvicorn 处理请求）
gunicorn app.main:app \
    --workers 4 \
    --worker-class uvicorn.workers.UvicornWorker \
    --bind 0.0.0.0:8000
```

**为什么这样组合**：
- Gunicorn 负责**进程管理**（成熟稳定）
- Uvicorn 负责**异步请求处理**（高性能）
- 两者分工明确，各取所长

### 4.4 配置文件方式（推荐）

生产环境推荐使用配置文件，而不是命令行参数：

```python
# gunicorn_config.py

# 监听地址
bind = "0.0.0.0:8000"

# Worker 数量：CPU 核数 * 2 + 1
import multiprocessing
workers = multiprocessing.cpu_count() * 2 + 1

# Worker 类型
worker_class = "uvicorn.workers.UvicornWorker"

# 超时（秒）
timeout = 120

# 日志
accesslog = "/var/log/app/access.log"
errorlog = "/var/log/app/error.log"
loglevel = "info"
```

启动时指定配置文件：

```bash
gunicorn -c gunicorn_config.py app.main:app
```

### 4.5 Worker 数量选择

经典公式：**workers = 2 * CPU 核数 + 1**

| CPU 核数 | 推荐 worker 数 | 适用场景 |
|----------|---------------|----------|
| 1 核 | 3 | 低流量开发服务器 |
| 2 核 | 5 | 小型生产环境 |
| 4 核 | 9 | 中型生产环境 |
| 8 核 | 17 | 大型生产环境 |

**注意**：
- 如果应用是 **CPU 密集型**，worker 数 = CPU 核数
- 如果应用是 **I/O 密集型**（大多数 Web 应用），worker 数 = 2 * CPU + 1
- 如果使用 **异步 worker**（UvicornWorker），worker 数可以适当减少

## 5. 生产环境为什么不能直接用 `uvicorn --reload`

```bash
# ❌ 错误：生产环境使用 --reload
uvicorn app.main:app --reload --host 0.0.0.0 --port 8000
```

**问题**：

1. **性能损耗**：`--reload` 会持续监控文件变化，消耗 CPU 和 I/O
2. **安全风险**：任何人上传 `.py` 文件即可触发代码执行
3. **稳定性差**：文件变更导致频繁重启，服务不稳定
4. **单进程**：`--reload` 模式只启动单进程，无法利用多核

```bash
# ✅ 正确：生产环境去掉 --reload，指定 workers
uvicorn app.main:app --host 0.0.0.0 --port 8000 --workers 4
```

## 6. 常见坑

### 6.1 忘记安装 worker 依赖

```bash
# 错误：ModuleNotFoundError: No module named 'uvicorn.workers'
gunicorn -c gunicorn_config.py app.main:app

# 解决：安装 uvicorn[standard]
pip install "uvicorn[standard]"
```

### 6.2 Worker 数设置过多

```bash
# 错误：1 核 CPU 设置 20 个 worker
workers = 20

# 后果：进程切换开销大，内存占用高，性能反而下降
# 解决：遵循 2 * CPU + 1 公式
```

### 6.3 超时设置过短

```python
# 错误：timeout = 10（10 秒）
# 如果接口需要调用外部 API（可能耗时 30 秒），worker 会被杀掉

# 解决：根据实际业务设置
timeout = 120  # 120 秒
```

### 6.4 日志路径权限不足

```bash
# 错误：Permission denied: '/var/log/app/access.log'

# 解决：创建目录并设置权限
sudo mkdir -p /var/log/app
sudo chown www-data:www-data /var/log/app
```

### 6.5 忘记绑定 0.0.0.0

```bash
# 错误：只绑定 localhost，外部无法访问
bind = "127.0.0.1:8000"

# 解决：绑定 0.0.0.0（由 Nginx 反向代理保护）
bind = "0.0.0.0:8000"
```

## 7. 总结

| 要点 | 说明 |
|------|------|
| WSGI | 同步接口，适合 Flask/Django |
| ASGI | 异步接口，适合 FastAPI |
| Gunicorn | 进程管理器，负责多进程和稳定性 |
| Uvicorn | ASGI 服务器，负责异步请求处理 |
| 最佳组合 | Gunicorn + UvicornWorker |
| Worker 数 | 2 * CPU 核数 + 1 |
| 生产禁忌 | 禁止使用 `--reload` |
