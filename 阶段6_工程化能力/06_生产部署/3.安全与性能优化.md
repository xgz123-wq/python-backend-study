# 安全与性能优化

> 防火墙、密钥、连接池、缓存——线上服务比能跑更要紧的是"稳"与"快"。

## 1. 概念解释

### 安全

将应用部署到生产环境后，它暴露在公网上，面临各种攻击：

- **端口扫描**：攻击者扫描开放端口，寻找可利用的服务
- **暴力破解**：尝试猜测 SSH 密码、数据库密码
- **SQL 注入**：通过恶意输入操纵数据库
- **DDoS**：大量请求打垮服务
- **中间人攻击**：窃听未加密的 HTTP 流量

### 性能优化

性能优化的目标是：

- **降低响应时间**：用户请求尽快得到响应
- **提高吞吐量**：单位时间内处理更多请求
- **减少资源消耗**：用更少的 CPU/内存完成同样的工作
- **稳定性**：高负载下服务不崩溃

---

## 2. 为什么需要

| 风险 | 后果 |
|------|------|
| 数据库端口暴露 | 被暴力破解，数据泄露 |
| SSH 使用密码登录 | 被暴力破解，服务器被控制 |
| HTTP 明文传输 | 用户数据被窃听 |
| 敏感信息硬编码 | 代码泄露 = 密钥泄露 |
| Worker 数量不当 | 性能差或资源浪费 |
| 无监控告警 | 故障发生很久后才发现 |

---

## 3. 实际场景

```
攻击者扫描你的服务器
    │
    ├── 22 端口（SSH）
    │   ├── 密码登录 → 暴力破解 → 控制服务器
    │   └── 密钥登录 → 攻击失败
    │
    ├── 80/443 端口（HTTP/HTTPS）
    │   ├── HTTP → 中间人攻击 → 窃取用户数据
    │   └── HTTPS → 加密传输 → 攻击失败
    │
    ├── 5432 端口（PostgreSQL）
    │   ├── 暴露 → 暴力破解 → 数据库泄露
    │   └── 只监听 localhost → 攻击失败
    │
    └── 8000 端口（应用）
        ├── 直接暴露 → 绕过 Nginx 防护
        └── 只监听 localhost → 必须通过 Nginx → 安全
```

## 4. 核心用法

### 4.1 安全清单

#### 端口管理

```bash
# 查看开放端口
sudo netstat -tulpn
sudo ss -tulpn

# 使用 UFW 防火墙（Ubuntu）
sudo ufw default deny incoming
sudo ufw default allow outgoing
sudo ufw allow ssh        # 22
sudo ufw allow http       # 80
sudo ufw allow https      # 443
sudo ufw enable
sudo ufw status

# 云服务器安全组（阿里云/腾讯云）
# 只开放 80、443 端口
# 22 端口限制 IP 来源（仅允许你的 IP）
```

#### SSH 加固

```bash
# 编辑 /etc/ssh/sshd_config

# 禁用密码登录（使用密钥）
PasswordAuthentication no

# 禁用 root 登录
PermitRootLogin no

# 修改默认端口（减少扫描）
Port 2222

# 限制登录用户
AllowUsers deploy

# 重启 SSH 服务
sudo systemctl restart sshd
```

**生成 SSH 密钥**：

```bash
# 本地机器生成密钥
ssh-keygen -t ed25519 -C "your_email@example.com"

# 复制公钥到服务器
ssh-copy-id -p 22 deploy@server-ip

# 之后可以用密钥登录
ssh -p 22 deploy@server-ip
```

#### HTTPS 配置

```bash
# 安装 Certbot（Let's Encrypt 免费证书）
sudo apt install certbot python3-certbot-nginx

# 自动配置 HTTPS
sudo certbot --nginx -d yourdomain.com -d www.yourdomain.com

# 自动续期（Let's Encrypt 证书 90 天过期）
sudo certbot renew --dry-run
```

**密码强度**：

```python
import secrets
import string

def generate_password(length=32):
    """生成强密码"""
    chars = string.ascii_letters + string.digits + "!@#$%^&*"
    return ''.join(secrets.choice(chars) for _ in range(length))

# 生成数据库密码
db_password = generate_password()
print(f"数据库密码: {db_password}")
```

### 4.4 性能优化：Gunicorn Worker 调优

```python
# gunicorn_config.py

import multiprocessing

# CPU 密集型应用：worker 数 = CPU 核数
# workers = multiprocessing.cpu_count()

# I/O 密集型应用（大多数 Web 应用）：worker 数 = 2 * CPU + 1
workers = multiprocessing.cpu_count() * 2 + 1

# 使用异步 worker（FastAPI 项目）
worker_class = "uvicorn.workers.UvicornWorker"

# 超时设置
timeout = 120          # 请求超时（秒）
graceful_timeout = 30  # 优雅停止超时
keepalive = 5          # Keep-Alive 超时

# 预加载应用（减少内存，但热更新需要重启所有 worker）
preload_app = True

# 最大请求数（防止内存泄漏，worker 处理 N 个请求后重启）
max_requests = 1000
max_requests_jitter = 50  # 随机偏移，避免所有 worker 同时重启
```

**Worker 调优策略**：

```bash
# 1. 监控当前性能
ab -n 1000 -c 10 http://localhost:8000/api/health

# 2. 调整 worker 数量
workers = 4   # 测试 4 个 worker
workers = 8   # 测试 8 个 worker
workers = 16  # 测试 16 个 worker

# 3. 对比结果，选择最优值
```

### 4.5 静态文件由 Nginx 托管

```nginx
# /etc/nginx/sites-available/myapp

server {
    listen 80;
    server_name yourdomain.com;

    # 静态文件直接由 Nginx 返回（不经过 Python）
    location /static/ {
        alias /opt/myapp/static/;
        expires 30d;           # 缓存 30 天
        add_header Cache-Control "public, immutable";
    }

    # 上传文件
    location /media/ {
        alias /opt/myapp/media/;
        expires 7d;
    }

    # API 请求转发给 Gunicorn
    location /api/ {
        proxy_pass http://127.0.0.1:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    # 前端 SPA（Vue/React）
    location / {
        root /opt/myapp/frontend/dist;
        try_files $uri $uri/ /index.html;
    }
}
```

**为什么静态文件由 Nginx 托管**：

| 方式 | 性能 | 原因 |
|------|------|------|
| Python 返回静态文件 | 慢 | 需要经过 Python 进程 |
| Nginx 直接返回 | 快 | Nginx 专为静态文件优化，零拷贝 |

### 4.6 监控与告警基础

#### 应用健康检查

```python
# FastAPI 健康检查端点
from fastapi import APIRouter
from sqlalchemy import text
from app.database import engine

router = APIRouter()

@router.get("/health")
async def health_check():
    """健康检查（供负载均衡器/监控系统调用）"""
    try:
        # 检查数据库连接
        with engine.connect() as conn:
            conn.execute(text("SELECT 1"))

        return {
            "status": "healthy",
            "database": "connected",
        }
    except Exception as e:
        return {
            "status": "unhealthy",
            "database": "disconnected",
            "error": str(e),
        }, 503
```

#### 基础监控指标

```python
# 使用 Prometheus 指标
from prometheus_client import Counter, Histogram, generate_latest
from fastapi import Response
import time

# 定义指标
REQUEST_COUNT = Counter(
    "http_requests_total",
    "Total HTTP requests",
    ["method", "endpoint", "status"]
)

REQUEST_DURATION = Histogram(
    "http_request_duration_seconds",
    "HTTP request duration",
    ["method", "endpoint"]
)

@app.middleware("http")
async def metrics_middleware(request, call_next):
    start = time.time()
    response = await call_next(request)
    duration = time.time() - start

    REQUEST_COUNT.labels(
        method=request.method,
        endpoint=request.url.path,
        status=response.status_code
    ).inc()

    REQUEST_DURATION.labels(
        method=request.method,
        endpoint=request.url.path
    ).observe(duration)

    return response

@app.get("/metrics")
async def metrics():
    return Response(content=generate_latest(), media_type="text/plain")
```

#### 告警配置

```bash
# 简单的告警脚本（配合 cron 定时执行）
# /opt/myapp/scripts/alert.sh

#!/bin/bash

# 检查服务是否存活
if ! systemctl is-active --quiet myapp; then
    echo "❌ 服务 myapp 已停止！" | mail -s "服务告警" admin@example.com
    sudo systemctl restart myapp
fi

# 检查磁盘空间
USAGE=$(df / | grep / | awk '{print $5}' | sed 's/%//')
if [ "$USAGE" -gt 90 ]; then
    echo "⚠️ 磁盘使用率超过 90%：${USAGE}%" | mail -s "磁盘告警" admin@example.com
fi

# 检查内存使用
MEM_USAGE=$(free | grep Mem | awk '{print $3/$2 * 100.0}')
if (( $(echo "$MEM_USAGE > 90" | bc -l) )); then
    echo "⚠️ 内存使用率超过 90%" | mail -s "内存告警" admin@example.com
fi
```

## 5. 常见坑

### 5.1 数据库端口暴露到公网

```bash
# ❌ 错误：PostgreSQL 监听所有地址
listen_addresses = '*'

# 后果：攻击者可以直接连接数据库，尝试暴力破解

# ✅ 正确：只监听本地
listen_addresses = 'localhost'
```

### 5.2 DEBUG 模式未关闭

```python
# ❌ 错误：生产环境 DEBUG = True
DEBUG = True

# 后果：
# - 异常页面暴露源代码、数据库结构
# - 性能下降（每次请求重新加载配置）

# ✅ 正确：生产环境 DEBUG = False
DEBUG = False
```

### 5.3 敏感信息泄露到日志

```python
# ❌ 错误：日志中包含密码
logger.info(f"用户登录: username={username}, password={password}")

# ✅ 正确：脱敏处理
logger.info(f"用户登录: username={username}")
```

### 5.4 HTTPS 配置不当

```nginx
# ❌ 错误：HTTP 和 HTTPS 都可以访问，没有强制跳转

# ✅ 正确：强制 HTTPS
server {
    listen 80;
    server_name yourdomain.com;
    return 301 https://$server_name$request_uri;
}

server {
    listen 443 ssl;
    server_name yourdomain.com;
    # ... SSL 配置
}
```

### 5.5 Worker 数量设置不当

```python
# ❌ 错误：Worker 数过多
workers = 100  # 1 核 CPU

# 后果：
# - 进程切换开销大
# - 内存占用高（每个 worker 占用 50-100 MB）
# - 性能反而下降

# ✅ 正确：遵循公式
workers = 2 * cpu_count() + 1
```

### 5.6 忽略日志轮转

```bash
# ❌ 错误：日志文件无限增长
ls -lh /var/log/myapp/app.log
-rw-r--r-- 1 root root 50G Jan 1 12:00 app.log

# 后果：磁盘空间耗尽，服务崩溃

# ✅ 正确：配置日志轮转（systemd journal 自动处理）
# 或者使用 RotatingFileHandler
```

### 5.7 没有健康检查

```bash
# 问题：服务已经崩溃，但负载均衡器还在转发流量
# 用户看到 502 Bad Gateway

# 解决：配置健康检查
# Nginx:
upstream backend {
    server 127.0.0.1:8000;
    # 健康检查（商业版 Nginx Plus）
    # health_check uri=/health;
}

# 或者使用 systemd 的 WatchdogSec
[Service]
WatchdogSec=30  # 30 秒内必须报告健康状态
```

## 7. 总结

| 要点 | 说明 |
|------|------|
| 防火墙 | 只开放 80、443，22 限制 IP |
| SSH | 密钥登录，禁用 root |
| HTTPS | Let's Encrypt 免费证书 |
| 数据库 | 只监听 localhost，强密码 |
| 敏感信息 | 环境变量，不硬编码 |
| Worker 调优 | 2 * CPU + 1，异步 worker |
| 静态文件 | Nginx 直接返回 |
| 监控 | 健康检查 + 告警 |
| 日志 | 结构化 + 轮转 |
