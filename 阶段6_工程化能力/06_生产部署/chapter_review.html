第一节 · WSGI 与 ASGI 服务器

  • 理解 WSGI 同步接口与 ASGI 异步接口的差异
  • 区分 Uvicorn / Gunicorn 各自定位与组合用法
  • 掌握 workers = 2 × CPU + 1 公式
  • 解释生产环境为什么禁用 --reload
与后端开发的关联：Flask/Django 同步项目配 Gunicorn（WSGI），FastAPI/Channels 异步项目配 Gunicorn + UvicornWorker（ASGI）；错配会让 WebSocket 不可用或性能浪费。

WSGI（Web Server Gateway Interface，PEP 3333）是 Python 的同步 Web 接口：每个请求占一个线程/进程，不支持 WebSocket，代表是 Flask / Django（同步模式）。

ASGI（Asynchronous Server Gateway Interface）是 WSGI 的异步继任者，由 Django Channels 团队提出，支持 async def、WebSocket、长连接、双向通信，代表是 FastAPI / Starlette。

④ Gunicorn Master-Worker 架构

                    ┌──────────────────────┐
                    │     Master Process   │
                    │   （进程管理器）        │
                    └──────────┬───────────┘
                               │
              ┌────────────────┼────────────────┐
              │                │                │
        ┌─────▼──────┐  ┌─────▼──────┐  ┌─────▼──────┐
        │  Worker 1  │  │  Worker 2  │  │  Worker 3  │
        │ (处理请求)  │  │ (处理请求)  │  │ (处理请求)  │
        └────────────┘  └────────────┘  └────────────┘

Master 负责：启停 worker、监控健康、崩溃自动重启、HUP 信号优雅重启（完成当前请求后重启，零停机）。

⑤ Worker 数量公式

经典公式：workers = 2 × CPU 核数 + 1。逻辑是 I/O 等待时 CPU 可切到其他 worker。

应用类型推荐 worker 数
CPU 密集型（图像处理、计算）≈ CPU 核数
I/O 密集型（绝大多数 Web）2 × CPU + 1
异步 worker（UvicornWorker）可适当减少
生产环境禁用 --reload：1) 持续监控文件消耗 CPU/I/O；2) 单进程浪费多核；3) 文件变动即重启服务不稳定；4) 任意能写 .py 文件的位置都成为 RCE 入口。

① Uvicorn 直接启动（ASGI）

# 开发环境（带热重载）
uvicorn app.main:app --reload --host 0.0.0.0 --port 8000

# 生产环境（关闭 reload，开启多 worker）
uvicorn app.main:app --host 0.0.0.0 --port 8000 --workers 4

② Gunicorn 启动（WSGI）

# Flask
gunicorn -w 4 -b 0.0.0.0:8000 app:app

# Django
gunicorn -w 4 -b 0.0.0.0:8000 myproject.wsgi:application

③ Gunicorn + Uvicorn Worker（推荐组合）

# Gunicorn 管进程（崩溃重启、优雅升级），Uvicorn 处理异步请求
gunicorn app.main:app \
    --workers 4 \
    --worker-class uvicorn.workers.UvicornWorker \
    --bind 0.0.0.0:8000
Gunicorn 负责成熟的进程管理，Uvicorn 负责高性能异步 IO，分工明确、各取所长。

生产环境使用 --reload关掉 --reload，改用 --workers N 配合 Gunicorn 进程管理

⑥ 结构化 JSON 日志

import logging, json
from datetime import datetime

class JSONFormatter(logging.Formatter):
    def format(self, record):
        data = {
            "timestamp": datetime.utcnow().isoformat(),
            "level": record.levelname,
            "message": record.getMessage(),
            "module": record.module,
        }
        if hasattr(record, "request_id"):
            data["request_id"] = record.request_id
        return json.dumps(data, ensure_ascii=False)

logger = logging.getLogger("myapp")

⑧ Prometheus 指标

from prometheus_client import Counter, Histogram, generate_latest
import time

REQUEST_COUNT = Counter("http_requests_total", "Total HTTP requests",
                        ["method", "endpoint", "status"])
REQUEST_DURATION = Histogram("http_request_duration_seconds", "HTTP latency",
                             ["method", "endpoint"])

@app.middleware("http")
async def metrics(request, call_next):
    start = time.time()
    response = await call_next(request)
    REQUEST_COUNT.labels(request.method, request.url.path, response.status_code).inc()
    REQUEST_DURATION.labels(request.method, request.url.path).observe(time.time() - start)
    return response

@app.get("/metrics")
async def metrics():
    return Response(content=generate_latest(), media_type="text/plain")
