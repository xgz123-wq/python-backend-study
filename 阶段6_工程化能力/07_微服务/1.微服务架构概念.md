# 微服务架构概念

> 把单体拆成多个独立服务，换来横向扩展，但也带来分布式复杂度。

## 一、什么是微服务？

### 1.1 单体应用（Monolith）

单体应用是将所有功能模块（用户、订单、支付、库存等）打包在一个代码库和进程中运行的架构。

```
┌─────────────────────────────────────────┐
│              单体应用                     │
├─────────────────────────────────────────┤
│  用户模块  │  订单模块  │  支付模块        │
│  库存模块  │  通知模块  │  报表模块        │
├─────────────────────────────────────────┤
│              共享数据库                   │
└─────────────────────────────────────────┘
```

**特点：**
- 一个代码库，一个部署包
- 所有模块共享同一个数据库
- 任何修改都需要重新部署整个应用
- 初期开发速度快，后期维护成本高

### 1.2 微服务架构（Microservices）

微服务是将应用拆分为多个**独立部署的小型服务**，每个服务负责一个业务领域，服务之间通过轻量级协议通信。

```
┌──────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐
│ 用户服务  │  │ 订单服务  │  │ 支付服务  │  │ 库存服务  │
│   :8001  │  │   :8002  │  │   :8003  │  │   :8004  │
└────┬─────┘  └────┬─────┘  └────┬─────┘  └────┬─────┘
     │              │              │              │
     └──────────────┴──────────────┴──────────────┘
                            │
                    ┌───────▼────────┐
                    │   API 网关      │
                    │    :8080       │
                    └────────────────┘
```

**特点：**
- 每个服务有独立的代码库和部署流程
- 每个服务有自己的数据库（Database per Service）
- 服务可以独立扩缩容
- 技术栈可以异构（Python + Java + Go 混合）

## 二、为什么需要微服务？

### 2.1 单体应用的痛点

| 痛点 | 说明 |
|------|------|
| 部署风险高 | 改一行代码也要重新部署整个应用 |
| 扩展困难 | 只能整体扩容，不能只扩展瓶颈服务 |
| 技术栈锁定 | 所有模块必须使用相同语言和框架 |
| 团队协作冲突 | 多个团队修改同一个代码库，合并冲突频繁 |
| 启动缓慢 | 代码量增大后，本地启动可能需要数分钟 |

### 2.2 微服务带来的好处

- **独立部署**：每个服务可以单独发布，不影响其他服务
- **弹性伸缩**：可以只扩容负载高的服务（如订单服务），而不必扩容整个系统
- **技术多样性**：不同服务可以选择最合适的技术栈
- **团队自治**：每个团队负责一个或多个服务，独立迭代
- **故障隔离**：一个服务崩溃不会拖垮整个系统（如果设计了熔断机制）

## 三、什么时候该拆微服务？

> ⚠️ **核心原则：不要过早拆分！** 大多数项目初期用单体就够了。

### 3.1 适合使用单体的阶段

- 初创团队，产品还在验证阶段（PMF 之前）
- 团队人数 < 10 人，沟通成本低
- 业务量小，单体足以支撑
- 团队缺乏运维能力（微服务需要成熟的 DevOps 支撑）

### 3.2 考虑拆分微服务的信号

- **团队规模增长**：超过 2 个披萨团队（约 10-15 人），协作开始混乱
- **部署频率高**：每天需要部署多次，但每次都要走完整流程
- **扩展瓶颈**：只有少数模块需要扩容，但必须整体扩容
- **业务边界清晰**：可以明确划分出独立的业务领域（DDD 限界上下文）
- **技术栈需要异构**：某个模块需要用 Go 写高性能服务，其他用 Python

### 3.3 Martin Fowler 的「单体优先」原则

> "Monolith First" —— 先用单体快速验证业务，等到痛点明显时再逐步拆分。

## 四、微服务通信方式

微服务之间需要通信，常见的方式有三种：

### 4.1 HTTP REST（同步）

最常用的方式，基于 HTTP 协议，使用 JSON 格式传输数据。

```python
# 订单服务调用用户服务
import requests

response = requests.get("http://user-service:8001/api/users/123")
user_data = response.json()
```

**优点：** 简单、通用、易于调试
**缺点：** 性能相对较低（JSON 序列化开销大）、同步阻塞

### 4.2 gRPC（同步，高性能）

基于 HTTP/2 协议，使用 Protocol Buffers 序列化数据，性能远高于 REST。

```
# .proto 文件定义接口
service UserService {
    rpc GetUser (UserRequest) returns (UserResponse);
}
```

**优点：** 高性能、强类型、支持双向流
**缺点：** 学习曲线陡、调试不如 REST 直观

### 4.3 消息队列（异步）

使用 RabbitMQ、Kafka 等中间件实现服务间的异步通信。

```
订单服务  ──[订单创建事件]──▶  消息队列  ──▶  库存服务（扣减库存）
                                      ──▶  通知服务（发送通知）
```

**优点：** 解耦、削峰填谷、支持异步
**缺点：** 增加系统复杂度、消息丢失和重复消费问题

### 4.4 对比总结

| 通信方式 | 协议 | 性能 | 耦合度 | 适用场景 |
|---------|------|------|--------|---------|
| HTTP REST | HTTP/1.1 | 中 | 中 | 简单的服务间调用 |
| gRPC | HTTP/2 | 高 | 低 | 高性能、内部服务通信 |
| 消息队列 | AMQP等 | 中 | 低 | 异步、解耦、事件驱动 |

## 五、微服务治理

微服务架构引入了分布式系统的复杂性，需要一系列治理机制：

### 5.1 服务发现（Service Discovery）

服务实例的地址是动态的（容器重启后 IP 可能变化），需要一个注册中心来管理。

```
用户服务启动 ──▶ 注册到 Consul/Eureka ──▶ 注册中心记录：user-service: [ip1, ip2, ip3]
订单服务需要调用 ──▶ 查询注册中心 ──▶ 获取用户服务的地址列表 ──▶ 负载均衡调用
```

**常见工具：** Consul、Eureka、Nacos、etcd

### 5.2 配置中心（Configuration Center）

微服务数量多，配置文件散落在各个服务中，难以统一管理。

```yaml
# 配置中心统一管理
user-service:
  database_url: "postgresql://..."
  cache_ttl: 3600
order-service:
  database_url: "postgresql://..."
  max_retries: 3
```

**常见工具：** Apollo、Nacos、Consul KV、Spring Cloud Config

### 5.3 链路追踪（Distributed Tracing）

一个请求可能经过 5-10 个服务，出问题时很难定位。链路追踪给每个请求分配一个 Trace ID，跟踪完整调用链。

```
请求 A (trace_id: abc123)
  ├── 订单服务 (span_id: 001, 耗时 20ms)
  │   ├── 用户服务 (span_id: 002, 耗时 15ms)
  │   └── 库存服务 (span_id: 003, 耗时 30ms)  ◀── 瓶颈在这里！
  └── 总耗时: 65ms
```

**常见工具：** Jaeger、Zipkin、SkyWalking、OpenTelemetry

### 5.4 熔断与限流

- **熔断（Circuit Breaker）**：当某个下游服务故障时，快速失败而不是长时间等待，防止级联故障
- **限流（Rate Limiting）**：控制每个服务的请求速率，防止过载

```
正常状态 ──[失败率超阈值]──▶ 熔断开启（快速失败）──[超时后]──▶ 半开（尝试恢复）
```

## 六、API 网关

API 网关是微服务架构的统一入口，处理请求路由、认证、限流等横切关注点。

```
客户端 ──▶ API 网关 ──┬──▶ 用户服务
                      ├──▶ 订单服务
                      ├──▶ 支付服务
                      └──▶ 库存服务

API 网关职责：
├── 路由：/users/* ──▶ user-service
├── 认证：验证 JWT Token
├── 限流：每分钟最多 1000 请求
├── 日志：统一记录访问日志
└── 聚合：将多个服务的数据组合后返回
```

**常见工具：** Kong、Nginx、Traefik、Spring Cloud Gateway

**为什么需要 API 网关？**
- 客户端不需要知道内部服务的地址
- 统一的认证和授权入口
- 避免每个服务都重复实现限流、日志等功能
- 可以对外隐藏内部微服务的细节

## 七、常见坑

### 坑 1：过早拆分微服务

**现象：** 项目刚开始就设计成 20 个微服务，结果团队只有 3 个人。

**后果：** 80% 的时间花在基础设施和运维上，业务功能进展缓慢。

**建议：** 单体优先，等到痛点明确时再拆分。

### 坑 2：分布式事务

**现象：** 订单服务创建订单成功，但库存服务扣减库存失败，数据不一致。

**原因：** 微服务各自有数据库，不能像单体那样使用本地事务。

**解决方案：**
- Saga 模式：将分布式事务拆分为一系列本地事务，每个事务有对应的补偿操作
- 最终一致性：接受短时间内的数据不一致，通过消息队列保证最终一致

### 坑 3：服务间调用链路太长

**现象：** A 调用 B，B 调用 C，C 调用 D…… 一个请求经过了 10 个服务。

**后果：** 延迟叠加，任何一个环节超时都会导致整个请求失败。

**建议：**
- 控制调用深度，避免过长的同步调用链
- 使用异步消息代替部分同步调用
- 引入链路追踪工具快速定位瓶颈

### 坑 4：日志和调试困难

**现象：** 请求出错了，但不知道在哪个服务出的问题。

**解决方案：**
- 统一的日志收集（ELK Stack：Elasticsearch + Logstash + Kibana）
- 链路追踪（Jaeger、Zipkin）
- 给每个请求传递 Trace ID，贯穿所有服务

### 坑 5：配置管理混乱

**现象：** 每个服务都有自己的配置文件，修改数据库密码需要改 N 个文件。

**解决方案：** 使用配置中心（Apollo、Nacos）统一管理，支持热更新。

### 坑 6：忽视运维成本

**现象：** 拆分了 50 个微服务，但没有容器编排、没有 CI/CD、没有监控。

**后果：** 运维团队疲于奔命，每次发布都是「人肉部署」。

**建议：** 微服务需要成熟的 DevOps 能力支撑，包括：
- 容器化（Docker）
- 容器编排（Kubernetes）
- CI/CD 流水线
- 监控告警（Prometheus + Grafana）
- 日志聚合

## 八、总结

| 维度 | 单体应用 | 微服务 |
|------|---------|--------|
| 复杂度 | 低 | 高（分布式复杂性） |
| 部署 | 简单 | 复杂（需要编排工具） |
| 扩展性 | 差（整体扩容） | 好（独立扩容） |
| 技术栈 | 统一 | 可异构 |
| 团队规模 | 小团队 | 大团队 |
| 适用阶段 | 初期验证 | 业务成熟后 |

**核心建议：**
1. 不要为了微服务而微服务
2. 单体优先，痛点驱动拆分
3. 微服务需要成熟的 DevOps 和运维能力
4. 了解概念即可，实际工作中根据团队和业务规模决策
