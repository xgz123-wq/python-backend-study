# gRPC 与 Kubernetes 基础

> gRPC 让服务间通信比 HTTP+JSON 更快，K8s 则把容器编排做成行业标准。

## 一、gRPC 概念

### 1.1 什么是 gRPC？

gRPC 是 Google 开源的高性能 **RPC（Remote Procedure Call，远程过程调用）** 框架。

简单理解：让你像调用本地函数一样调用远程服务器上的函数，而且速度很快。

```
客户端                                    服务端
┌──────────┐                              ┌──────────┐
│          │  gRPC 调用                    │          │
│  client. │ ─────────────────────────────▶│ GetUser()│
│  GetUser │  Protocol Buffers 序列化      │          │
│  (id=1)  │◀─────────────────────────────│ return   │
│          │  Protocol Buffers 反序列化    │ User{...}│
└──────────┘                              └──────────┘
```

**gRPC 的核心特性：**
- 使用 **Protocol Buffers** 定义接口和序列化数据
- 基于 **HTTP/2** 协议（支持多路复用、头部压缩）
- 支持四种通信模式：一元调用、服务端流、客户端流、双向流
- 自动生成多语言的客户端和服务端代码

### 1.2 Protocol Buffers（Protobuf）

Protobuf 是 Google 开发的数据序列化格式，类似于 JSON 但更紧凑、更快。

**Protobuf vs JSON 对比：**

```
# JSON 格式（文本，可读）
{
    "id": 123,
    "name": "张三",
    "email": "zhangsan@example.com",
    "age": 28
}
# 大约 70+ 字节

# Protobuf 格式（二进制，不可读，但更小）
# 序列化后大约 20-30 字节（约为 JSON 的 1/3 到 1/2）
```

**Protobuf 的优势：**
- **体积小**：二进制编码，比 JSON 小 3-10 倍
- **速度快**：序列化/反序列化速度比 JSON 快 20-100 倍
- **强类型**：通过 .proto 文件严格定义字段类型
- **向后兼容**：可以添加新字段而不破坏旧代码

### 1.3 .proto 文件示例

`.proto` 文件是 gRPC 的接口定义文件，同时定义了数据结构和 RPC 服务。

```protobuf
// user.proto — 用户服务接口定义
syntax = "proto3";  // 使用 proto3 语法

package user;  // 包名，避免命名冲突

// 请求消息：获取用户
message GetUserRequest {
    int64 user_id = 1;  // 字段编号，不是默认值
}

// 响应消息：用户信息
message UserResponse {
    int64 id = 1;
    string name = 2;
    string email = 3;
    int32 age = 4;
    UserStatus status = 5;

    enum UserStatus {
        UNKNOWN = 0;
        ACTIVE = 1;
        INACTIVE = 2;
    }
}

// 请求消息：创建用户
message CreateUserRequest {
    string name = 1;
    string email = 2;
    int32 age = 3;
}

// 用户服务定义
service UserService {
    // 一元 RPC：获取用户
    rpc GetUser (GetUserRequest) returns (UserResponse);

    // 一元 RPC：创建用户
    rpc CreateUser (CreateUserRequest) returns (UserResponse);

    // 服务端流式 RPC：列出所有用户（服务端持续推送）
    rpc ListUsers (GetUserRequest) returns (stream UserResponse);

    // 双向流式 RPC：实时聊天（双方持续通信）
    rpc Chat (stream ChatMessage) returns (stream ChatMessage);
}

message ChatMessage {
    string sender = 1;
    string content = 2;
    int64 timestamp = 3;
}
```

**关键字段说明：**

| 语法 | 含义 |
|------|------|
| `message` | 定义数据结构（类似 class） |
| `service` | 定义 RPC 服务及其方法 |
| `rpc` | 定义一个远程调用方法 |
| `stream` | 表示流式传输（持续发送数据） |
| `= 1, = 2` | 字段编号，唯一标识，不能修改 |

## 二、gRPC vs REST 对比

| 维度 | REST (JSON) | gRPC (Protobuf) |
|------|-------------|-----------------|
| 协议 | HTTP/1.1 | HTTP/2 |
| 数据格式 | JSON（文本） | Protobuf（二进制） |
| 可读性 | 高（人可读） | 低（需要工具解码） |
| 性能 | 中 | 高（体积小、速度快） |
| 接口定义 | OpenAPI/Swagger | .proto 文件 |
| 代码生成 | 需要额外工具 | 自动生成 |
| 浏览器支持 | 原生支持 | 需要 gRPC-Web 代理 |
| 流式传输 | 有限（SSE/WebSocket） | 原生支持 |
| 适用场景 | 对外 API | 内部微服务通信 |

**什么时候用 gRPC？**
- 微服务之间的内部通信（追求高性能）
- 需要双向流式传输的场景（如实时推送）
- 多语言团队（.proto 文件是通用接口定义）

**什么时候用 REST？**
- 对外暴露的 API（浏览器、移动端调用）
- 需要高可读性、易调试的场景
- 团队对 gRPC 不熟悉时

**gRPC 工作流程：**

```
1. 编写 .proto 文件定义接口
2. 使用 protoc 编译器生成 Python 代码
   $ python -m grpc_tools.protoc -I. --python_out=. --grpc_python_out=. user.proto
3. 服务端实现生成的 Servicer 类
4. 客户端使用生成的 Stub 类调用
5. 启动服务，测试调用
```

## 四、Kubernetes（K8s）基本概念

### 4.1 什么是 Kubernetes？

Kubernetes（简称 K8s，因为 k 和 s 之间有 8 个字母）是 Google 开源的**容器编排平台**，用于自动化部署、扩展和管理容器化应用。

**为什么需要 K8s？**

假设你有 20 个微服务，每个服务有 3 个副本，一共 60 个容器。你面临的问题：
- 如何部署这 60 个容器？
- 某个容器挂了怎么办？
- 流量高峰时如何自动扩容？
- 如何做滚动更新？
- 容器之间如何通信？

**K8s 就是解决这些问题的「容器管家」。**

### 4.2 K8s 架构图示

```
┌─────────────────────────────────────────────────────────────────┐
│                      Master Node（控制平面）                      │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐            │
│  │  API Server  │  │  Scheduler  │  │ Controller  │            │
│  │  (集群入口)  │  │ (调度器)     │  │  Manager    │            │
│  └──────┬──────┘  └─────────────┘  └─────────────┘            │
│         │                                                       │
│  ┌──────▼──────┐                                               │
│  │    etcd      │  ◀── 分布式数据库，存储集群所有状态            │
│  └─────────────┘                                               │
└─────────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────────────┐
│                     Worker Nodes（工作节点）                       │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐                     │
│  │  Pod 1   │  │  Pod 2   │  │  Pod 3   │  ◀── 运行容器        │
│  │ (用户服务) │  │ (订单服务) │  │ (支付服务) │                     │
│  └──────────┘  └──────────┘  └──────────┘                     │
│                                                                 │
│  ┌─────────────┐  ┌─────────────┐                              │
│  │   kubelet   │  │  kube-proxy │  ◀── 节点代理和网络代理        │
│  └─────────────┘  └─────────────┘                              │
└─────────────────────────────────────────────────────────────────┘
```

**控制平面组件说明：**
- **API Server**：所有操作的入口，接收 kubectl 命令
- **Scheduler**：决定 Pod 应该运行在哪个节点上
- **Controller Manager**：确保实际状态与期望状态一致（如：期望 3 个副本，挂了 1 个就自动创建）
- **etcd**：分布式键值存储，保存集群的所有配置和状态

## 五、K8s 核心组件

### 5.1 Pod（最小部署单元）

Pod 是 K8s 中最小的可部署单元，包含一个或多个紧密关联的容器。

```yaml
# pod.yaml
apiVersion: v1
kind: Pod
metadata:
  name: user-service-pod
  labels:
    app: user-service
spec:
  containers:
    - name: user-service
      image: myregistry/user-service:v1.0.0
      ports:
        - containerPort: 8001
      resources:
        requests:
          memory: "256Mi"   # 最少需要 256MB 内存
          cpu: "250m"       # 最少需要 0.25 个 CPU 核心
        limits:
          memory: "512Mi"   # 最多使用 512MB 内存
          cpu: "500m"       # 最多使用 0.5 个 CPU 核心
```

**Pod 的特点：**
- 一个 Pod 通常运行一个主容器（有时加 sidecar 容器，如日志收集）
- Pod 有临时性，可能被随时销毁和重建
- 每个 Pod 有独立的 IP 地址
- 不要直接创建 Pod，而是通过 Deployment 管理

### 5.2 Deployment（部署控制器）

Deployment 管理 Pod 的副本数、更新策略和回滚。

```yaml
# deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: user-service-deployment
spec:
  replicas: 3  # 始终保持 3 个 Pod 副本
  selector:
    matchLabels:
      app: user-service
  template:
    metadata:
      labels:
        app: user-service
    spec:
      containers:
        - name: user-service
          image: myregistry/user-service:v1.0.0
          ports:
            - containerPort: 8001
  strategy:
    type: RollingUpdate  # 滚动更新策略
    rollingUpdate:
      maxSurge: 1         # 更新时最多多出 1 个 Pod
      maxUnavailable: 0   # 更新时不允许有 Pod 不可用
```

**Deployment 的核心能力：**
- **副本管理**：始终保持指定数量的 Pod 运行
- **滚动更新**：逐步替换旧版本 Pod，实现零停机更新
- **回滚**：新版本有问题可以一键回滚到上一个版本
- **自愈**：Pod 挂了自动重建

### 5.3 Service（服务发现与负载均衡）

Service 为 Pod 提供稳定的访问入口（IP 和 DNS 名称），并在多个 Pod 之间负载均衡。

```yaml
# service.yaml
apiVersion: v1
kind: Service
metadata:
  name: user-service
spec:
  selector:
    app: user-service  # 选择标签为 app=user-service 的 Pod
  ports:
    - protocol: TCP
      port: 80          # Service 暴露的端口
      targetPort: 8001  # Pod 容器的端口
  type: ClusterIP  # 集群内部访问（默认类型）
```

**Service 的类型：**

| 类型 | 说明 | 使用场景 |
|------|------|---------|
| ClusterIP | 仅集群内部可访问 | 微服务间内部通信 |
| NodePort | 通过节点 IP + 端口访问 | 开发测试、对外暴露 |
| LoadBalancer | 云厂商提供的负载均衡器 | 生产环境对外暴露 |

**为什么需要 Service？**

Pod 的 IP 是动态的（重启后会变），而 Service 提供稳定的 DNS 名称：
```
# 订单服务可以通过 DNS 名称访问用户服务，不需要知道具体 IP
http://user-service.default.svc.cluster.local:80/api/users/123
# 简写为：http://user-service:80
```

### 5.4 ConfigMap（配置管理）

ConfigMap 用于将配置信息与容器镜像解耦，方便统一管理。

```yaml
# configmap.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: user-service-config
data:
  DATABASE_URL: "postgresql://db:5432/users"
  CACHE_TTL: "3600"
  LOG_LEVEL: "info"
  APP_ENV: "production"
```

在 Deployment 中引用 ConfigMap：
```yaml
containers:
  - name: user-service
    image: myregistry/user-service:v1.0.0
    envFrom:
      - configMapRef:
          name: user-service-config  # 注入所有配置项为环境变量
```

**ConfigMap vs Secret：**
- ConfigMap：存储普通配置（数据库 URL、日志级别）
- Secret：存储敏感信息（密码、API Key），数据会 Base64 编码

## 六、K8s 常用命令速查

```bash
# 查看集群信息
kubectl cluster-info

# 查看所有 Pod
kubectl get pods

# 查看 Pod 详细信息
kubectl describe pod user-service-pod

# 查看 Pod 日志
kubectl logs user-service-pod

# 进入 Pod 容器内部
kubectl exec -it user-service-pod -- /bin/sh

# 创建资源
kubectl apply -f deployment.yaml

# 删除资源
kubectl delete -f deployment.yaml

# 扩缩容
kubectl scale deployment user-service-deployment --replicas=5

# 滚动更新（修改镜像版本）
kubectl set image deployment/user-service-deployment \
    user-service=myregistry/user-service:v2.0.0

# 回滚到上一个版本
kubectl rollout undo deployment/user-service-deployment

# 查看所有 Service
kubectl get services

# 查看所有 Deployment
kubectl get deployments
```

## 七、K8s 与微服务的关系

```
┌─────────────────────────────────────────────────────────────┐
│                    Kubernetes 集群                            │
│                                                             │
│  ┌─────────────┐  Deployment + Service                      │
│  │ 用户服务     │  3 个 Pod 副本                              │
│  │ :8001       │  ClusterIP Service 负载均衡                 │
│  └──────┬──────┘                                            │
│         │                                                   │
│  ┌──────▼──────┐  Deployment + Service                      │
│  │ 订单服务     │  5 个 Pod 副本（负载更高，所以副本更多）      │
│  │ :8002       │                                            │
│  └──────┬──────┘                                            │
│         │                                                   │
│  ┌──────▼──────┐  Deployment + Service                      │
│  │ 支付服务     │  3 个 Pod 副本                              │
│  │ :8003       │                                            │
│  └─────────────┘                                            │
│                                                             │
│  ┌─────────────┐                                            │
│  │ Ingress     │  ◀── 对外暴露 HTTP 路由（类似 API 网关）     │
│  │ (Nginx)     │                                            │
│  └─────────────┘                                            │
└─────────────────────────────────────────────────────────────┘
```

**K8s 为微服务提供的基础能力：**
- 服务发现：通过 Service 的 DNS 名称
- 负载均衡：Service 自动在 Pod 之间分配流量
- 自愈：Pod 挂了自动重建
- 弹性伸缩：HPA（Horizontal Pod Autoscaler）根据 CPU/内存自动扩缩容
- 滚动更新和回滚：Deployment 内置支持
- 配置管理：ConfigMap 和 Secret

## 八、常见坑

### 坑 1：Pod 一直处于 Pending 状态

**现象：** `kubectl get pods` 显示 Pod 状态为 `Pending`。

**原因：**
- 节点资源不足（CPU/内存不够）
- 没有节点满足调度条件（nodeSelector、affinity）
- PVC（持久卷声明）未绑定

**排查：**
```bash
kubectl describe pod <pod-name>  # 查看 Events 部分的调度信息
```

### 坑 2：Service 无法访问 Pod

**现象：** 通过 Service 的 DNS 名称访问不到后端 Pod。

**原因：**
- Service 的 `selector` 标签与 Pod 的标签不匹配
- Pod 没有处于 `Running` 状态
- 端口配置错误（`port` vs `targetPort`）

**排查：**
```bash
kubectl get endpoints <service-name>  # 查看 Service 关联的 Pod IP
```

### 坑 3：ConfigMap 修改后 Pod 没有更新

**现象：** 修改了 ConfigMap，但 Pod 内的环境变量没有变化。

**原因：** ConfigMap 的更新不会自动触发 Pod 重启（除非使用特殊的 Operator）。

**解决：** 手动重启 Deployment：
```bash
kubectl rollout restart deployment/user-service-deployment
```

### 坑 4：忽略资源限制

**现象：** 某个 Pod 内存泄漏，吃光节点资源，导致同节点其他 Pod 也被驱逐。

**建议：** 始终为每个容器设置 `resources.requests` 和 `resources.limits`。

### 坑 5：不了解 gRPC 与 HTTP/1.1 负载均衡的差异

**现象：** gRPC 客户端始终连接到同一个 Pod，负载不均衡。

**原因：** gRPC 基于 HTTP/2，一个连接上复用多个请求。L4 负载均衡器（如 kube-proxy 的 iptables）只在连接建立时分配一次。

**解决：**
- 使用客户端负载均衡（gRPC 内置）
- 使用支持 gRPC 的 L7 负载均衡器（如 Envoy）

### 坑 6：过度使用 Kubernetes

**现象：** 团队只有 2-3 个人，项目是简单的 CRUD 应用，却部署了 K8s。

**后果：** 80% 的时间花在维护 K8s 集群上，而不是开发业务功能。

**建议：**
- 简单应用使用 Docker Compose 就够了
- 中等规模可以考虑 PaaS 平台（如 Heroku、Railway）
- 确实需要 K8s 时，优先使用托管服务（如 GKE、EKS、ACK）

## 九、总结

| 技术 | 核心作用 | 学习重点 |
|------|---------|---------|
| gRPC | 高性能 RPC 通信 | 理解 Protobuf、会写 .proto 文件 |
| Kubernetes | 容器编排 | 理解 Pod/Service/Deployment/ConfigMap |
| Protocol Buffers | 数据序列化 | 比 JSON 更小更快、强类型 |
| .proto 文件 | 接口定义 | 定义 message 和 service |

**了解级别的建议：**
1. 知道 gRPC 是什么、为什么比 REST 快，会看 .proto 文件
2. 知道 K8s 的核心概念，能看懂 YAML 配置
3. 不需要从零搭建 K8s 集群，实际工作中用托管服务
4. 重点理解「为什么需要这些技术」，而不是「如何从零搭建」
