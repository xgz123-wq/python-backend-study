## 排障决策树：用户说"网站打不开"时怎么办

> 部署上线只是开始，运维才是真正的长期战。
> 遇到故障时，沿着请求链路**从外到内**逐层排查：

```
用户反馈："网站打不开"
│
├─ Step 1: DNS 能解析吗？
│   命令: dig resume-ai.example.com
│   ├─ 不通 → DNS 配置错误，检查域名解析记录
│   └─ 通 ↓
│
├─ Step 2: 服务器能 ping 通吗？
│   命令: ping 47.xxx.xxx.xxx
│   ├─ 不通 → 服务器宕机 / 安全组未放行 / 防火墙拦截
│   └─ 通 ↓
│
├─ Step 3: Nginx 返回什么状态码？
│   命令: curl -I https://resume-ai.example.com
│   ├─ 502/504 → 后端 FastAPI 挂了（转 Step 4）
│   ├─ 403 → Nginx 权限配置问题
│   ├─ 连接拒绝 → Nginx 没启动
│   └─ 200 但页面空白 → 前端问题或 API 异常
│
├─ Step 4: FastAPI 进程在运行吗？
│   命令: docker compose ps
│   ├─ 容器不在 → docker compose logs 查看崩溃原因
│   ├─ 容器在但 unhealthy → 应用内部错误（转 Step 5）
│   └─ 容器正常 → 可能是 Nginx 配置指向了错误端口
│
├─ Step 5: 应用日志说了什么？
│   命令: docker compose logs --tail=50 web
│   ├─ ImportError → 依赖缺失，检查 Dockerfile
│   ├─ ConnectionError → MySQL/Redis 连接失败（转 Step 6）
│   ├─ Timeout → AI API 调用超时，检查网络/API Key
│   └─ 无异常 → 检查请求参数和业务逻辑
│
└─ Step 6: MySQL/Redis 健康吗？
    命令: docker compose exec mysql mysqladmin ping
    命令: docker compose exec redis redis-cli ping
    ├─ 不通 → 容器挂了或配置错误
    └─ 通 → 检查连接池/慢查询/内存溢出
```

### 常见故障速查表

| 现象 | 最可能原因 | 一条排查命令 | 复习章节 |
|------|-----------|-------------|---------|
| 域名解析不到 IP | DNS 配置错误 | `dig your-domain.com` | [ch00](00_部署概览/2.用户请求的完整链路.md) |
| SSH 连不上服务器 | 安全组未放行 22 端口 | `telnet IP 22` | [ch01](01_Linux/2.权限与用户管理.md) |
| 502 Bad Gateway | FastAPI 进程崩溃 | `docker compose ps` | [ch04](04_Nginx/1.Nginx基础与反向代理.md) + [ch06](06_生产部署/1.WSGI与ASGI服务器.md) |
| "在我电脑能跑" | 环境不一致 | `docker compose build --no-cache` | [ch03](03_Docker/3.Dockerfile编写.md) |
| HTTPS 证书过期 | Let's Encrypt 未续期 | `certbot certificates` | [ch04](04_Nginx/2.静态文件与HTTPS配置.md) |
| Celery 任务不执行 | Worker 挂了或 Redis 连接断 | `docker compose logs celery` | [ch03](03_Docker/4.docker-compose编排.md) |
| 数据库连接超时 | 连接池耗尽或 MySQL 宕机 | `docker compose exec mysql mysqladmin ping` | [ch06](06_生产部署/2.进程管理与环境配置.md) |

### 题目 1：解释"部署"

> **场景**：你的大学室友（非技术背景）问你："你最近在学部署？部署是干嘛的？"

**用你自己的话解释**（试着写下来）：

<details>
<summary>✅ 点击展开评价标准</summary>
**合格线**：

- 能说清楚"让只有你能用的东西变成所有人都能用"
- 能提到"服务器"的概念（24小时开机的电脑）
- 能举一个生活中的类比（开餐厅、出版书、开网店）

**不合格的表现**：
- ❌ 只说"把代码放到服务器上"但解释不了服务器是什么
- ❌ 用了一堆术语（DNS、CDN、反向代理）但室友更困惑了
- ❌ 说不清本地开发和部署后的区别

</details>

- [ ] `docker compose build` 构建成功
- [ ] `docker compose up` 所有容器 healthy
- [ ] 镜像使用非 root 用户运行
- [ ] `.dockerignore` 排除了 `.env`、`.git`、`__pycache__`

### Nginx
- [ ] 反向代理配置正确（proxy_pass 指向正确端口）
- [ ] HTTPS 证书已安装且未过期
- [ ] HTTP 自动重定向到 HTTPS

- [ ] Gunicorn Worker 数量合理（2 × CPU + 1）
- [ ] systemd / Docker restart policy 配置自动重启
- [ ] 日志有轮转策略（不会撑爆磁盘）

### 安全
- [ ] 服务器仅开放 22/80/443 端口
- [ ] SSH 禁用密码登录，仅密钥认证
- [ ] 敏感信息通过环境变量注入，不在代码中
- [ ] 数据库端口（3306/6379）不对外暴露

### 运维
- [ ] 数据库有自动备份策略
- [ ] 有回滚方案（上一个可用镜像/tag）
- [ ] 配置了基本的监控告警

## 学习建议

### 推荐学习顺序

```
ch00 部署概览（1-2 天）  ← 先建立全局画面
  ↓
ch01 Linux（3-4 天）     ← 服务器基本功
  ↓
ch02 Git（2-3 天）       ← 代码管理
  ↓
ch03 Docker（4-5 天）    ← 核心中的核心，多花时间
  ↓
ch04 Nginx（3-4 天）     ← 开门迎客
  ↓
ch05 CI/CD（2-3 天）     ← 自动化
  ↓
ch06 生产部署（3-4 天）   ← 最后一公里
  ↓
ch07 微服务（1-2 天）    ← 了解即可，不急
```

### 三条原则

1. **动手 > 只读**：每个 Demo 必须亲自跑一遍，不要只看代码
2. **贯穿项目实操**：用 AI 简历分析系统作为主线，每学完一个 Stage 就实际部署一次
3. **费曼输出**：每学完一个 Stage，试着写一篇部署笔记——假装写给完全不懂部署的朋友看。写不出来 = 没学透，回去重看
